# 🧠 Rolling Updates — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Zero-downtime update mechanism** for Deployments — replaces old Pods with new Pods incrementally
- Old Pods are terminated **gradually** while new Pods are created **gradually**
- At no point during the update is the application fully down (when configured correctly)
- Controlled by two parameters: `maxSurge` (extra Pods allowed) and `maxUnavailable` (Pods that can be down)
- Each rolling update creates a **new ReplicaSet** — old RS is kept for rollback

## How It Works — Step by Step
```
BEFORE UPDATE: Deployment with replicas=3, image=nginx:1.25
  ReplicaSet-v1: [Pod-A (1.25)] [Pod-B (1.25)] [Pod-C (1.25)]

TRIGGER: kubectl set image deployment/myapp web=nginx:1.26

STEP 1: New RS created, starts scaling up
  ReplicaSet-v1: [Pod-A (1.25)] [Pod-B (1.25)] [Pod-C (1.25)]
  ReplicaSet-v2: [Pod-D (1.26)] ← new Pod creating

STEP 2: New Pod Ready → old Pod removed
  ReplicaSet-v1: [Pod-A (1.25)] [Pod-B (1.25)] ← one removed
  ReplicaSet-v2: [Pod-D (1.26)] [Pod-E (1.26)] ← scaling up

STEP 3: Continue until all replaced
  ReplicaSet-v1: ← scaled to 0 (kept for rollback)
  ReplicaSet-v2: [Pod-D (1.26)] [Pod-E (1.26)] [Pod-F (1.26)]

AFTER UPDATE: All Pods running new version
  Old RS preserved (replicas=0) → rollback target
```

## Real-World Use Cases
- Deploy new application version without downtime
- Update container image tag (`v1.0` → `v1.1`)
- Change environment variables or config references
- Update resource requests/limits
- Any change to `spec.template` triggers a rolling update

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────┐
│  Rolling Update is a DEPLOYMENT STRATEGY, not a K8s object   │
│                                                               │
│  Deployment                                                   │
│    spec.strategy:                                            │
│      type: RollingUpdate     ← THIS is the rolling update   │
│      rollingUpdate:                                          │
│        maxSurge: 25%         ← Controls speed & resources   │
│        maxUnavailable: 25%   ← Controls availability         │
│                                                               │
│  Deployment Controller (kube-controller-manager):            │
│    Watches Deployment → detects template change              │
│    Creates new RS → scales new up, old down                  │
│    Respects maxSurge / maxUnavailable at every step          │
└──────────────────────────────────────────────────────────────┘
```

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Deployment Controller** — Orchestrates the rolling update by managing ReplicaSets
- **ReplicaSets** — Old RS scales down, new RS scales up (both exist during update)
- **Scheduler** — Must place new Pods on suitable nodes during the update
- **Service** — Continues routing traffic; mix of old and new Pods during transition
- **Readiness Probes** — Gate for when new Pods are considered "ready" to receive traffic
- **HPA** — May conflict with or pause during rolling updates

## What Resources It Interacts With
- **Deployment** — Rolling update IS a Deployment strategy
- **ReplicaSets** — Two RS active during update (old and new)
- **Pods** — Old Pods terminated, new Pods created
- **Service endpoints** — Updated as Pods become Ready
- **PDB** (PodDisruptionBudget) — Can slow or block rolling updates
- **ResourceQuota** — May block new Pod creation if quota exhausted

## Upstream & Downstream Dependencies
```
TRIGGER:
  Any change to Deployment spec.template:
    → image change
    → env/envFrom change
    → volumeMount change
    → resource request/limit change
    → command/args change
    → label change on template

NOT A TRIGGER (no rolling update):
    → spec.replicas change (just scales current RS)
    → spec.strategy change (applies to next rollout)
    → metadata on Deployment itself
    → spec.revisionHistoryLimit

DOWNSTREAM DURING ROLLOUT:
  Deployment Controller
    → Creates new ReplicaSet
      → New RS creates new Pods
        → Scheduler places new Pods
          → Readiness probe passes → Pod added to Service
            → Old Pod removed from Service → terminates
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Rolling Update Configuration in Deployment
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: rolling-demo
spec:
  replicas: 4
  selector:
    matchLabels:
      app: rolling-demo
  strategy:
    type: RollingUpdate              # ← This enables rolling updates
    rollingUpdate:
      maxSurge: 1                    # ← Max extra Pods during update
      maxUnavailable: 1              # ← Max Pods down during update
  template:
    metadata:
      labels:
        app: rolling-demo
    spec:
      containers:
      - name: web
        image: nginx:1.25            # ← Changing this triggers rolling update
        ports:
        - containerPort: 80
        readinessProbe:               # ← CRITICAL for safe rolling updates
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 5
```

## maxSurge & maxUnavailable Explained
```
Given: replicas=4

┌─────────────────────────────────────────────────────────────────┐
│  maxSurge=1, maxUnavailable=1                                    │
│                                                                  │
│  Total Pods allowed: replicas + maxSurge = 4 + 1 = 5           │
│  Min available:      replicas - maxUnavailable = 4 - 1 = 3     │
│                                                                  │
│  At any moment: 3 to 5 Pods exist, at least 3 serve traffic    │
│  Speed: Moderate (replaces 1 at a time)                         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  maxSurge=2, maxUnavailable=0                                    │
│                                                                  │
│  Total Pods allowed: 4 + 2 = 6                                  │
│  Min available: 4 - 0 = 4 (ALL must be available always!)       │
│                                                                  │
│  Old Pods ONLY removed after new Pod is Ready                   │
│  Zero capacity loss, but uses more resources temporarily        │
│  Speed: Moderate (needs spare node capacity)                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  maxSurge=50%, maxUnavailable=50%                                │
│                                                                  │
│  Total Pods: 4 + 2 = 6 (50% of 4 = 2)                          │
│  Min available: 4 - 2 = 2 (50% capacity during update)         │
│                                                                  │
│  Fastest rollout, but significant capacity reduction             │
│  Speed: Fast (half the Pods replaced at once)                    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  ❌ maxSurge=0, maxUnavailable=0                                 │
│                                                                  │
│  IMPOSSIBLE — can't add OR remove Pods → DEADLOCK               │
│  Kubernetes rejects this configuration at validation            │
└─────────────────────────────────────────────────────────────────┘
```

## ⚠️ Common Traps

### Trap 1: Confusing integer vs percentage
```yaml
# Integer: exact Pod count
maxSurge: 2           # Up to 2 extra Pods (regardless of replicas count)
maxUnavailable: 1     # At most 1 Pod can be down

# Percentage: relative to replicas (rounded UP for surge, DOWN for unavailable)
maxSurge: 25%         # If replicas=4: ceil(4 * 0.25) = 1 extra Pod
maxUnavailable: 25%   # If replicas=4: floor(4 * 0.25) = 1 Pod can be down

# ⚠️ Percentage with small replica count can round to 0:
# replicas=1, maxSurge=25% → ceil(0.25) = 1 (OK)
# replicas=1, maxUnavailable=25% → floor(0.25) = 0 (means NO Pods can go down!)
```

### Trap 2: No readiness probe → unsafe rolling update
```yaml
# ❌ DANGEROUS — no readiness probe
spec:
  containers:
  - name: web
    image: nginx:1.25
    # No readinessProbe → Pod considered Ready as soon as container starts
    # New Pod may not actually be serving yet → old Pod killed too early → downtime!

# ✅ SAFE — readiness probe gates traffic
  containers:
  - name: web
    image: nginx:1.25
    readinessProbe:
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 5
```

### Trap 3: Using Recreate instead of RollingUpdate
```yaml
# ❌ This causes DOWNTIME
strategy:
  type: Recreate          # All old Pods killed FIRST → then new Pods created
  # rollingUpdate field is INVALID with Recreate type

# ✅ This avoids downtime
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0     # Zero capacity loss
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `strategy.type` | `RollingUpdate` or `Recreate` | Recreate = downtime; wrong value = validation error | 🔴 |
| `strategy.rollingUpdate.maxSurge` | Max extra Pods allowed during update | Too high = resource spike; 0 = must have maxUnavailable | 🔴 |
| `strategy.rollingUpdate.maxUnavailable` | Max Pods down during update | Too high = capacity loss; 0 = slow but safe | 🔴 |
| `spec.template.spec.containers[].image` | Changing this triggers rolling update | Wrong image → stuck rollout | 🔴 |
| Readiness probe on containers | Gates when new Pod receives traffic | No probe → premature traffic routing → downtime | 🔴 |
| `spec.replicas` | How many total Pods | Low replicas + high maxUnavailable = no capacity during update | 🟡 |
| `spec.revisionHistoryLimit` | How many old RS kept | 0 = can't rollback | 🟡 |
| `spec.progressDeadlineSeconds` | How long before rollout is marked failed | Default 600s; rollout marked `ProgressDeadlineExceeded` | 🟡 |
| `spec.minReadySeconds` | How long new Pod must be Ready before old terminated | 0 = immediate; higher = slower but safer | 🟡 |

## Key Relationships
```
maxSurge + maxUnavailable → control rollout behavior:
  ┌───────────────────┬──────────────────┬───────────────────────────┐
  │ maxSurge          │ maxUnavailable   │ Behavior                  │
  ├───────────────────┼──────────────────┼───────────────────────────┤
  │ 1                 │ 0                │ Safe: 1 extra, 0 down     │
  │ 0                 │ 1                │ No extra, 1 can go down   │
  │ 25%               │ 25%             │ Balanced (K8s default)     │
  │ 100%              │ 0               │ Blue-green style           │
  │ 0                 │ 100%            │ Recreate-like (all at once)│
  │ 0                 │ 0               │ ❌ DEADLOCK (rejected)     │
  └───────────────────┴──────────────────┴───────────────────────────┘

readinessProbe → new Pod Ready → added to Service → old Pod safe to remove
  Without probe: K8s thinks Pod is Ready immediately → premature switchover
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `strategy.type` = `RollingUpdate`? (Not `Recreate`?)
- `maxSurge` and `maxUnavailable` — are they both non-zero?
- Is there a `readinessProbe` on the container? (Safety gate)
- What `image` is being used?
- What `replicas` count?

## 🟢 Usually Irrelevant for CKA
- `progressDeadlineSeconds` specific value
- `minReadySeconds` specific value
- `revisionHistoryLimit` specific value (default 10 is fine)
- Internal RS naming convention
- Pod disruption budget interactions

## 🔴 Red Flags in 10 Seconds

| Red Flag | Problem |
|----------|---------|
| `strategy.type: Recreate` | Downtime during update |
| `maxSurge: 0` and `maxUnavailable: 0` | Deadlock — rollout impossible |
| No `readinessProbe` | New Pods get traffic before app is ready → errors |
| `maxUnavailable: 100%` with few replicas | ALL Pods can be down simultaneously |
| `revisionHistoryLimit: 0` | Can't rollback to any previous version |
| `rollingUpdate` block under `Recreate` strategy | Invalid — `rollingUpdate` only with `RollingUpdate` type |
| Very wrong image tag | Rollout will be stuck — new Pods never become Ready |

## 📋 `kubectl describe deployment <name>` During Rollout

### What to Look For
```
Replicas:     4 desired | 5 total | 3 available | 2 unavailable
                          ↑ maxSurge allowing extra
                                     ↑ some old still serving
                                                    ↑ new Pods not Ready yet

OldReplicaSets:  rolling-demo-v1 (2/2 replicas created)    ← old still running
NewReplicaSet:   rolling-demo-v2 (2/4 replicas created)    ← new scaling up

Conditions:
  Type           Status  Reason
  Available      True    MinimumReplicasAvailable        ← ✅ enough Pods available
  Progressing    True    ReplicaSetUpdated                ← rollout in progress

Events:
  Normal  ScalingReplicaSet  Scaled up replica set rolling-demo-v2 to 1
  Normal  ScalingReplicaSet  Scaled down replica set rolling-demo-v1 to 3
  Normal  ScalingReplicaSet  Scaled up replica set rolling-demo-v2 to 2
```

### Failure Signals
| Signal | Meaning |
|--------|---------|
| `Progressing: False, Reason: ProgressDeadlineExceeded` | Rollout stuck — timed out |
| `NewReplicaSet: rolling-demo-v2 (1/4)` stuck for minutes | New Pods not becoming Ready |
| `OldReplicaSets` still has replicas AND `NewReplicaSet` has 0 Ready | Update hasn't progressed |
| `unavailable: 4` | ALL Pods down (maxUnavailable too high or crash) |
| Events show rapid scale up/down repeatedly | Thrashing — possibly HPA conflict |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Stuck Rollout — Wrong Image
```
SYMPTOM:        kubectl rollout status → hangs
                kubectl get deploy → READY x/y (not all ready)
CAUSE:          New image doesn't exist → ImagePullBackOff on new Pods
RS STATUS:      Old RS still has some Pods; new RS has 0 Ready
BEHAVIOR:       Rollout frozen. maxUnavailable Pods may have been terminated.
                Service still routes to remaining old Pods.
FIX:            kubectl rollout undo deployment/<name>
                OR: kubectl set image deployment/<name> web=<correct-image>
EXAM SCENARIO:  "A deployment update is stuck. Fix it."
```

## Pattern 2: Stuck Rollout — New App Crashes
```
SYMPTOM:        Rollout hangs; new Pods in CrashLoopBackOff
CAUSE:          New image has bad entry point, missing config, dependency issue
RS STATUS:      New RS has Pods but READY=0
BEHAVIOR:       Rollout progresses partially then stalls
FIX:            kubectl rollout undo deployment/<name>
EXAM TIP:       Always undo FIRST to restore service → then debug
```

## Pattern 3: Stuck Rollout — New Pods Pending
```
SYMPTOM:        Rollout hangs; new Pods in Pending state
CAUSE:          Resource requests too high, no nodes available, PVC unbound
RS STATUS:      New RS has Pods but none scheduled
BEHAVIOR:       Old Pods keep serving (up to maxUnavailable may be gone)
FIX:            Reduce resource requests OR add nodes → OR rollback
```

## Pattern 4: Rollout Causes Downtime (Recreate Strategy)
```
SYMPTOM:        Brief or extended 5xx errors during deployment
CAUSE:          strategy.type: Recreate → all old Pods killed before new start
BEHAVIOR:       Zero Pods serving during startup gap
FIX:            Switch to strategy.type: RollingUpdate
```

## Pattern 5: Rollout Causes Errors (No Readiness Probe)
```
SYMPTOM:        Intermittent 502/503 errors during rolling update
CAUSE:          No readiness probe → Pod marked Ready before app fully starts
                Traffic reaches new Pod before it can serve → errors
BEHAVIOR:       Old Pod removed as soon as new Pod starts (not when Ready)
FIX:            Add readiness probe to container spec
EXAM TIP:       This is a subtler failure — Pod shows Running but app not ready
```

## Pattern 6: ProgressDeadlineExceeded
```
SYMPTOM:        kubectl describe deploy → Progressing: False
                Reason: ProgressDeadlineExceeded
CAUSE:          Rollout hasn't progressed within progressDeadlineSeconds (default: 600s)
BEHAVIOR:       Deployment marked as failed, but existing Pods keep running
                K8s does NOT auto-rollback!
FIX:            Diagnose why new Pods aren't Ready → fix → or rollback manually
```

## Pattern 7: PDB Blocking Rollout
```
SYMPTOM:        Rollout very slow; old Pods not being removed
CAUSE:          PodDisruptionBudget prevents eviction of old Pods
EVENTS:         "Cannot evict pod as it would violate the pod's disruption budget"
BEHAVIOR:       Rollout crawls or stalls — maxUnavailable effectively reduced by PDB
FIX:            Adjust PDB minAvailable, or wait for PDB to allow disruption
```

## Pattern 8: Rollback Impossible (Revision History Cleared)
```
SYMPTOM:        kubectl rollout undo → no previous revision
CAUSE:          revisionHistoryLimit: 0 (or too low) → old RS deleted
BEHAVIOR:       Can't rollback; must fix forward
FIX:            Manually set correct image: kubectl set image deployment/...
                PREVENTION: Keep revisionHistoryLimit >= 5
```

# 8️⃣ Misconfiguration Behavior

## What Happens If maxSurge=0 AND maxUnavailable=0?
- **Behavior:** Kubernetes rejects this at validation
- **Error:** `Invalid value: intstr.IntOrString{Type:0, IntVal:0, StrVal:""}: may not be 0 when maxSurge is 0`
- Rolling update **impossible** — can't add or remove any Pod
- **Fix:** Set at least one to ≥ 1

## What Happens If maxUnavailable Is Too High?
- **Behavior:** Too many old Pods killed at once → significant capacity loss during update
- Example: `replicas=4, maxUnavailable=3` → could drop to 1 running Pod during update
- If new Pods fail → only 1 old Pod serves ALL traffic → overloaded
- **Fix:** Lower `maxUnavailable` (use 0 or 1 for production safety)

## What Happens If Image Is Wrong?
- **Behavior:** New RS creates Pods → Pods can't pull image → `ImagePullBackOff`
- Old Pods stay (up to maxUnavailable limit)
- Rollout frozen — doesn't progress or complete
- **ProgressDeadlineExceeded** after timeout
- **Fix:** `kubectl rollout undo` → restores old RS

## What Happens If New App Crashes?
- **Behavior:** New Pods created → CrashLoopBackOff → never Ready
- Old Pods keep serving but some may have been removed (maxUnavailable)
- Service capacity reduced but not zero (unless maxUnavailable too high)
- **Fix:** `kubectl rollout undo` → old RS scales back up

## What Happens Without Readiness Probe?
- **Behavior:** New Pod starts → immediately marked Ready → added to Service
- Old Pod removed → but new Pod's app isn't actually ready → request errors
- Brief failures for each Pod swap during update
- Users experience intermittent errors
- **Fix:** Add `readinessProbe` to container spec

## What Happens If Resources Are Too High for New Pods?
- **Behavior:** New Pods created but Pending (unschedulable)
- Old Pods may keep serving (depends on maxUnavailable)
- Rollout stuck — new RS can't get Pods Ready
- **Fix:** Reduce resource requests or add node capacity

## What Happens If Labels Change in Template?
- **Behavior:** Template label change triggers new RS → BUT selector must still match
- If new template labels still match selector → rolling update proceeds
- If new template labels DON'T match selector → validation error, no update

# 9️⃣ Fast Detection Strategy

## 60-Second Rolling Update Debug Flow

```bash
# ⏱️ 0-10 sec: Is there an active rollout?
kubectl rollout status deployment/<name>
# "successfully rolled out" → ✅ done
# Hanging → rollout stuck

# ⏱️ 10-20 sec: What does the Deployment show?
kubectl get deploy <name>
# READY: 2/4 → only 2 Pods healthy
# UP-TO-DATE: 1 → only 1 Pod has new template
# AVAILABLE: 2 → 2 Pods receiving traffic

# ⏱️ 20-30 sec: Which RS is the problem?
kubectl get rs -l app=<label>
# Old RS: DESIRED=2, READY=2 (still serving)
# New RS: DESIRED=2, READY=0 (failing!)

# ⏱️ 30-40 sec: What's wrong with new Pods?
kubectl get pods -l app=<label>
# Look for: ImagePullBackOff, CrashLoopBackOff, Pending

# ⏱️ 40-55 sec: Drill into failing new Pod
kubectl describe pod <new-failing-pod> | tail -15
kubectl logs <new-failing-pod> --previous

# ⏱️ 55-60 sec: Decide action
# Quick fix available? → Fix forward
# Not sure? → kubectl rollout undo deployment/<name>
```

## Quick Diagnosis During Rollout
```
SCENARIO                                → ACTION
──────────────────────────────────────────────────────────
New Pods: ImagePullBackOff              → Fix image OR rollback
New Pods: CrashLoopBackOff             → Rollback first, debug later
New Pods: Pending                       → Check resources/nodes OR rollback
New Pods: Running but not Ready         → Check readiness probe
Old Pods still serving                  → Service OK, just rollout stuck
No Pods serving at all                  → Critical: maxUnavailable too high
  (READY 0/N, AVAILABLE 0)             → Rollback immediately
Rollout OK but intermittent errors      → Missing readiness probe
```

# 🔟 Fast Fix Strategy

## The CKA Rolling Update Toolkit

```bash
# === ROLLOUT MANAGEMENT ===

# Trigger rolling update
kubectl set image deployment/<name> <container>=<new-image>

# Check rollout progress
kubectl rollout status deployment/<name>

# View rollout history
kubectl rollout history deployment/<name>

# View specific revision
kubectl rollout history deployment/<name> --revision=<N>

# UNDO (rollback to previous) — THE MOST IMPORTANT COMMAND
kubectl rollout undo deployment/<name>

# Undo to specific revision
kubectl rollout undo deployment/<name> --to-revision=<N>

# Pause rollout (canary-style)
kubectl rollout pause deployment/<name>

# Resume rollout
kubectl rollout resume deployment/<name>

# Restart all Pods (same spec, fresh Pods)
kubectl rollout restart deployment/<name>

# === STRATEGY CHANGES ===

# Switch to Recreate (causes downtime)
kubectl patch deployment <name> \
  -p '{"spec":{"strategy":{"type":"Recreate","rollingUpdate":null}}}'

# Set specific maxSurge / maxUnavailable
kubectl patch deployment <name> \
  -p '{"spec":{"strategy":{"type":"RollingUpdate","rollingUpdate":{"maxSurge":1,"maxUnavailable":0}}}}'

# === RECORD CHANGE CAUSE ===
kubectl annotate deployment/<name> kubernetes.io/change-cause="Updated to v2.0"
```

## Common Fix Scenarios

### Fix: Stuck rollout (bad image)
```bash
# Step 1: Rollback instantly
kubectl rollout undo deployment/<name>

# Step 2: Verify service restored
kubectl rollout status deployment/<name>
kubectl get deploy <name>
# READY=4/4 ✅

# Step 3: Fix image and re-deploy
kubectl set image deployment/<name> web=nginx:1.26-correct
kubectl rollout status deployment/<name>
```

### Fix: Stuck rollout (debug without rollback)
```bash
# Check what's wrong with new Pods
kubectl get pods -l app=<label> | grep -v Running
kubectl describe pod <failing-pod> | tail -20
kubectl logs <failing-pod> --previous

# Fix forward
kubectl set image deployment/<name> web=<correct-image>
```

### Fix: Missing readiness probe (intermittent errors)
```bash
kubectl edit deployment <name>
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

```bash
# Create Deployment
kubectl create deployment roll-demo --image=nginx:1.25 --replicas=4

# Add readiness probe (important for safe rollouts)
kubectl patch deployment roll-demo -p '{
  "spec": {
    "strategy": {
      "type": "RollingUpdate",
      "rollingUpdate": {
        "maxSurge": 1,
        "maxUnavailable": 0
      }
    },
    "template": {
      "spec": {
        "containers": [{
          "name": "nginx",
          "readinessProbe": {
            "httpGet": {"path": "/", "port": 80},
            "initialDelaySeconds": 2,
            "periodSeconds": 3
          }
        }]
      }
    }
  }
}'
```

### Validation Steps
```bash
# All Pods running
kubectl get deploy roll-demo
# READY 4/4

# Trigger rolling update
kubectl set image deployment/roll-demo nginx=nginx:1.26

# Watch the rollout live
kubectl rollout status deployment/roll-demo
# "successfully rolled out" ✅

# Verify new image
kubectl get pods -l app=roll-demo -o jsonpath='{.items[0].spec.containers[0].image}'
# nginx:1.26

# Check revision history
kubectl rollout history deployment/roll-demo
# REVISION 1 (1.25), REVISION 2 (1.26)

kubectl rollout undo deployment/roll-demo
kubectl rollout status deployment/roll-demo
# Back to 1.25

# Watch RS behavior
kubectl get rs -l app=roll-demo
# Two RS: one at replicas=4 (current), one at replicas=0 (previous)
```

## Step 2 — Break It Intentionally

### Break 1: Update to Non-Existent Image
```bash
kubectl set image deployment/roll-demo nginx=nginx:FAKEVERSION
kubectl rollout status deployment/roll-demo
# Hangs... ❌
kubectl get pods -l app=roll-demo
# New Pod: ImagePullBackOff; old Pods still running (maxUnavailable=0)
# RECOVERY:
kubectl rollout undo deployment/roll-demo
```

### Break 2: Update to Crashing Image
```bash
kubectl set image deployment/roll-demo nginx=busybox:1.36
# busybox has no HTTP server → readiness probe fails → or container exits
kubectl rollout status deployment/roll-demo
# Hangs... ❌
kubectl get pods -l app=roll-demo
# New Pods: CrashLoopBackOff or not Ready
# RECOVERY:
kubectl rollout undo deployment/roll-demo
```

### Break 3: Set maxSurge=0,maxUnavailable=0 (Deadlock)
```bash
kubectl patch deployment roll-demo -p \
  '{"spec":{"strategy":{"rollingUpdate":{"maxSurge":0,"maxUnavailable":0}}}}'
# Error: ... may not be 0 when maxSurge is 0
# K8s catches this ✅
```

### Break 4: Use Recreate Strategy (Observe Downtime)
```bash
kubectl patch deployment roll-demo \
  -p '{"spec":{"strategy":{"type":"Recreate","rollingUpdate":null}}}'
kubectl set image deployment/roll-demo nginx=nginx:1.26
# Observe: ALL old Pods killed → gap → new Pods start
# Brief period with 0 available Pods
kubectl get pods -l app=roll-demo -w
# You'll see all Terminating, then all ContainerCreating
```

### Break 5: Remove Readiness Probe (Unsafe Rollout)
```bash
kubectl edit deployment roll-demo
# Remove the readinessProbe section
kubectl set image deployment/roll-demo nginx=nginx:1.25
# Rollout "succeeds" but new Pods serve traffic before app is ready
# Could cause brief errors if app has slow startup
```

### Break 6: Very High maxUnavailable
```bash
kubectl patch deployment roll-demo -p \
  '{"spec":{"strategy":{"type":"RollingUpdate","rollingUpdate":{"maxSurge":0,"maxUnavailable":4}}}}'
kubectl set image deployment/roll-demo nginx=nginx:1.26
# ALL old Pods removed at once (Recreate behavior)
# Service capacity drops to 0 briefly
kubectl get pods -l app=roll-demo -w
# All old Terminating simultaneously
```

### Break 7: Trigger Rollout Then Trigger Another (Overlapping)
```bash
kubectl set image deployment/roll-demo nginx=nginx:1.25
# While first rollout is in progress:
kubectl set image deployment/roll-demo nginx=nginx:1.27
# K8s cancels old rollout → starts new one to 1.27
# Old intermediate RS cleaned up
kubectl get rs -l app=roll-demo
# May show 3 RS: original, intermediate (0 replicas), latest
```

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] Rolling update is the DEFAULT strategy for Deployments
- [ ] `maxSurge` = max extra Pods during update (integer or %)
- [ ] `maxUnavailable` = max Pods that can be down (integer or %)
- [ ] Both cannot be 0 simultaneously — deadlock
- [ ] Changing `spec.template` triggers rollout; changing `spec.replicas` does NOT
- [ ] `kubectl rollout` commands: status, history, undo, restart, pause, resume
- [ ] `kubectl set image` to trigger update
- [ ] Readiness probe is critical for safe rolling updates
- [ ] `RollingUpdate` vs `Recreate` strategy types
- [ ] `revisionHistoryLimit` controls how many old RS are kept
- [ ] Kubernetes does NOT auto-rollback on failure (must use `rollout undo`)

### What to Focus On
- **Triggering and monitoring** rolling updates (`set image` → `rollout status`)
- **Rolling back** stuck or failed updates (`rollout undo`)
- **Debugging** stuck rollouts (new Pods failing → why)
- **Understanding** maxSurge / maxUnavailable impact on availability
- **Readiness probes** as the safety gate for traffic switching

### What to Practice Repeatedly
- Deploy → update image → watch rollout → rollback → all under 2 minutes
- Break a rollout (bad image) → detect stuck → undo → verify restored
- Set different maxSurge/maxUnavailable values and observe rollout behavior
- Use `kubectl rollout history --revision=N` to inspect old versions
- Compare RollingUpdate vs Recreate behavior in real time

### What to Ignore (Not CKA Relevant)
- How the Deployment controller calculates exact Pod creation/deletion order
- proportionalScaling algorithm internals
- Deployment status condition transition logic
- How maxSurge percentage is rounded internally (just know it exists)
- Advanced canary patterns (Argo Rollouts, Flagger)

### 5 Quiz Questions

**Q1:** A Deployment has `replicas=4, maxSurge=1, maxUnavailable=0`. During a rolling update, what's the maximum and minimum number of Pods running at any time?
> **A:** Maximum = 5 (4 + 1 surge). Minimum = 4 (0 unavailable). At least 4 Pods serve traffic at all times. Safest configuration — zero capacity loss.

**Q2:** A rolling update is stuck because the new image doesn't exist. Old Pods are still serving. How do you fix it?
> **A:** `kubectl rollout undo deployment/<name>`. This scales old RS back up and scales new RS to 0. Service is fully restored to previous version.

**Q3:** After `kubectl rollout undo`, what happens to the ReplicaSets?
> **A:** The old RS (previous version) scales back up to desired replicas. The new RS (failed version) scales to 0 but is kept for history. The undo becomes a new revision in rollout history.

**Q4:** Why should you always have a readiness probe when doing rolling updates?
> **A:** Without a readiness probe, Kubernetes considers a Pod Ready as soon as the container starts. Old Pods get removed before the new Pod's app is actually ready to serve. This causes brief errors for each Pod swap. Readiness probe ensures traffic only goes to Pods that are actually ready.

**Q5:** You change `spec.replicas` from 3 to 5 on a Deployment. Does this trigger a rolling update?
> **A:** No. Only changes to `spec.template` trigger a rolling update (new RS). Replica changes just scale the current RS. No new revision is created.

# 1️⃣3️⃣ Component Interaction Map

```
┌──────────────────────────────────────────────────────────────────┐
│              ROLLING UPDATE IN ACTION                              │
│                                                                   │
│  kubectl set image                                                │
│       │                                                           │
│       ▼                                                           │
│  Deployment (spec.template changed)                              │
│       │                                                           │
│       ▼ Deployment Controller                                    │
│  ┌─────────────┐          ┌─────────────┐                        │
│  │ OLD RS      │ scaling  │ NEW RS      │ scaling                │
│  │ replicas: ↓ │ DOWN     │ replicas: ↑ │ UP                    │
│  │ [Pod-old-A] │          │ [Pod-new-D] │                        │
│  │ [Pod-old-B] │          │ [Pod-new-E] │                        │
│  └─────────────┘          └─────────────┘                        │
│        ↑                        ↑                                 │
│   Being removed            Being added                           │
│   from Service              to Service                           │
│   endpoints                endpoints                             │
│        ↓                        ↓                                 │
│  ┌──────────────────────────────────────────┐                    │
│  │ Service: routes to BOTH old+new Pods     │                    │
│  │ during transition (based on readiness)    │                    │
│  └──────────────────────────────────────────┘                    │
└──────────────────────────────────────────────────────────────────┘
```

### Interaction Details

| Component | During Rolling Update | When Rolling Update Fails |
|-----------|----------------------|--------------------------|
| **Deployment Controller** | Creates new RS, orchestrates scale up/down respecting maxSurge/maxUnavailable | Stops scaling — rollout frozen. Old Pods may keep serving. |
| **Scheduler** | Places new Pods on suitable nodes | If no capacity → new Pods Pending → rollout stuck |
| **Service** | Routes to all Pods matching selector (mix of old + new) | If new Pods not Ready → Service only routes to old Pods (graceful) |
| **Readiness Probes** | Control when new Pod is added to Service endpoints | No probe → Pod added immediately → possible errors. Failed probe → Pod NOT added → old Pods keep serving |
| **Controller Manager** | Runs Deployment controller and RS controller | If down → no rollout orchestration → update frozen |
| **etcd** | Stores Deployment, RS, Pod state throughout rollout | If down → no state updates → everything frozen |
| **HPA** | May try to scale during rollout → can interact unexpectedly | May thrash: rollout creates Pods → HPA adjusts → chaos. Best to pause HPA during manual rollouts |
| **PDB** | May prevent old Pods from being evicted | Rollout slows or stalls if PDB disallows disruption |

### Failure Cascade During Stuck Rollout
```
Bad image deployed
  → New RS created, new Pods ImagePullBackOff
    → maxUnavailable old Pods may have been terminated
      → Service capacity reduced (but not zero if maxUnavailable < replicas)
        → Rollout frozen (ProgressDeadlineExceeded after 10 min)
          → K8s does NOT auto-rollback!
            → Manual intervention needed: rollout undo
              → Old RS scales back up → Service restored
```

# 1️⃣4️⃣ Self-Evaluation Questions

## ✔ Can I Break This Intentionally?

| Breakage | How |
|----------|-----|
| Stuck rollout (ImagePull) | `kubectl set image deploy/<name> web=fake:999` |
| Stuck rollout (CrashLoop) | Update to image with broken entrypoint |
| Stuck rollout (Pending) | Set `requests.cpu: "100"` in template |
| Downtime during update | Switch `strategy.type: Recreate` |
| Unsafe rollout (errors) | Remove readiness probe from template |
| Full capacity loss | Set `maxUnavailable: 100%` |
| Can't rollback | Set `revisionHistoryLimit: 0` then update |
| Multiple overlapping rollouts | `set image` twice rapidly |

**All 8 → you understand every rolling update failure mode.**

```bash
# 10 sec: Is rollout stuck?
kubectl rollout status deployment/<name>

# 15 sec: Which RS is problematic?

kubectl rollout undo deployment/<name>
```

**60 seconds: detect → diagnose → fix. CKA speed achieved.**
