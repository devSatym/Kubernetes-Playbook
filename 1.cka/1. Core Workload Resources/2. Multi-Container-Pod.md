# 🧠 Multi-Container Pod — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- A Pod that runs **2+ containers** sharing the same network namespace, storage volumes, and lifecycle
- Containers in the same Pod share:
  - `localhost` (same IP address, can communicate via `127.0.0.1`)
  - Mounted volumes (shared filesystem)
  - IPC namespace (inter-process communication)
  - Pod lifecycle (start together, die together)
- Each container runs its own process but they cooperate as a single unit

## Real-World Use Cases
- **Sidecar** — Log shipper alongside an app (e.g., Fluentd + App)
- **Sidecar** — Service mesh proxy (e.g., Envoy/Istio proxy + App)
- **Sidecar** — Config reloader watching for ConfigMap changes
- **Init Container** — Database schema migration before app starts
- **Init Container** — Wait for dependency service to be ready
- **Init Container** — Clone git repo before app boots
- **Ambassador** — Proxy outbound connections (local proxy → external DB)
- **Adapter** — Transform output format (app logs → standardized format)

## Where It Fits in Cluster Architecture
```
┌─────────────────────────────────────────────────────┐
│                      NODE                            │
│  ┌───────────────────────────────────────────────┐  │
│  │                   POD                          │  │
│  │  ┌─────────────┐  ┌─────────────┐             │  │
│  │  │ Container 1 │  │ Container 2 │  ← Share IP │  │
│  │  │   (main)    │  │  (sidecar)  │  ← Share Vol│  │
│  │  └──────┬──────┘  └──────┬──────┘             │  │
│  │         │     localhost    │                    │  │
│  │         └────────┬────────┘                    │  │
│  │            Shared Volume                       │  │
│  │            Shared Network (Pod IP)             │  │
│  └───────────────────────────────────────────────┘  │
│                                                      │
│  kubelet manages Pod → Container Runtime runs each   │
│  container separately but they live/die together     │
└─────────────────────────────────────────────────────┘
```

# 2️⃣ What Depends On It

## Which Components Rely On It
- **kubelet** — Creates, monitors, and restarts all containers in the Pod
- **kube-scheduler** — Schedules the entire Pod (all containers) to a single Node
  - Resource requests/limits are **summed** across all containers for scheduling
- **Container Runtime** (containerd/CRI-O) — Runs each container as separate process
- **Pod networking (CNI)** — Creates ONE network namespace shared by all containers
- **kube-proxy / Services** — Route traffic to the Pod IP; doesn't distinguish containers

## What Resources It Interacts With
- **Volumes** — All containers can mount the same volume (shared filesystem)
- **ConfigMaps / Secrets** — Each container can independently reference them
- **ServiceAccount** — Shared across all containers in the Pod
- **Resource Requests/Limits** — Set per-container, summed for scheduling
- **Probes** — Defined per-container (each container has its own liveness/readiness)
- **SecurityContext** — Can be set at Pod-level (shared) or container-level (override)

## Upstream & Downstream Dependencies
```
Upstream (what creates it):
  Deployment / ReplicaSet / DaemonSet / StatefulSet / Job → creates Pod

Downstream (what Pod depends on):
  Node (scheduling) → kubelet → Container Runtime → Containers
  ConfigMaps / Secrets (config) → must exist before Pod starts
  PVCs (storage) → must be bound before Pod starts
  Init Containers → must ALL succeed before main containers start
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Multi-Container Pod YAML
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-container-pod
  labels:
    app: myapp
spec:
  # --- Init Containers (run FIRST, sequentially, must all succeed) ---
  initContainers:
  - name: init-setup                    # ← Runs before main containers
    image: busybox:1.36
    command: ['sh', '-c', 'echo "Init done" > /shared/status']
    volumeMounts:
    - name: shared-data
      mountPath: /shared

  # --- Main Containers (run TOGETHER after init completes) ---
  containers:
  - name: app                           # ← Primary application container
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: shared-data
      mountPath: /usr/share/nginx/html
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        cpu: 200m
        memory: 256Mi

  - name: sidecar                       # ← Sidecar container
    image: busybox:1.36
    command: ['sh', '-c', 'while true; do date >> /shared/index.html; sleep 5; done']
    volumeMounts:
    - name: shared-data
      mountPath: /shared

  # --- Shared Volumes ---
  volumes:
  - name: shared-data
    emptyDir: {}                         # ← Shared between ALL containers
```

## Key Sections Explained
| Section | Purpose |
|---------|---------|
| `apiVersion: v1` | Pods are core API (`v1`) |
| `kind: Pod` | Resource type |
| `metadata.name` | Pod name (unique in namespace) |
| `spec.initContainers[]` | Run sequentially BEFORE main containers |
| `spec.containers[]` | Main containers that run simultaneously |
| `spec.volumes[]` | Volume definitions shared across containers |
| `volumeMounts[]` | Per-container mount of a volume (can mount to different paths) |

## ⚠️ Common Indentation Traps
```yaml
# ❌ WRONG — sidecar container indented under first container
containers:
- name: app
  image: nginx
    - name: sidecar        # ← WRONG! This is NOT a new container
      image: busybox

# ✅ CORRECT — sidecar at same level as app
containers:
- name: app
  image: nginx
- name: sidecar             # ← CORRECT! Same list level
  image: busybox

# ❌ WRONG — volumeMount under wrong container
containers:
- name: app
  image: nginx
- name: sidecar
  image: busybox
  volumeMounts:              # ← This mounts ONLY on sidecar, not app
  - name: data
    mountPath: /data

# ❌ WRONG — initContainers inside containers
containers:
- name: app
  image: nginx
  initContainers:            # ← WRONG! initContainers is a Pod-level field
  - name: init
    image: busybox

# ✅ CORRECT — initContainers at spec level
spec:
  initContainers:            # ← CORRECT! Same level as containers
  - name: init
    image: busybox
  containers:
  - name: app
    image: nginx
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks If Wrong |
|-------|---------------|---------------------|
| `containers[].name` | Unique ID per container; used in `kubectl logs -c` | Can't target specific container for logs/exec |
| `containers[].image` | Image to pull | `ImagePullBackOff` / `ErrImagePull` |
| `containers[].command` | Overrides Dockerfile `ENTRYPOINT` | Container exits immediately or wrong behavior |
| `containers[].args` | Overrides Dockerfile `CMD` | Wrong arguments, crash |
| `containers[].ports[].containerPort` | Documents the port (informational) | No direct breakage, but confusion |
| `containers[].volumeMounts` | Where to mount shared volume | Container can't access shared data |
| `containers[].resources` | CPU/memory requests & limits | OOMKilled, scheduling failure, Pending |
| `initContainers[]` | Must succeed before main containers start | Pod stuck in `Init:Error` or `Init:CrashLoopBackOff` |
| `initContainers[].command` | Init logic (wait, copy, migrate) | Init never completes → Pod stuck in `Init:0/1` |
| `volumes[]` | Define shared storage | Mount fails → `CreateContainerConfigError` |
| `volumes[].name` | Links `volumes[]` ↔ `volumeMounts[].name` | Name mismatch → container won't start |
| `restartPolicy` | `Always` (default), `OnFailure`, `Never` | Sidecar crashes not restarted if set to `Never` |

## Key Field Relationships
```
volumes[].name ←——must match——→ volumeMounts[].name
  (Pod-level)                     (Container-level)

initContainers runs BEFORE → containers
  (sequential, must succeed)     (parallel, all start together)

containers[].command → overrides → Dockerfile ENTRYPOINT
containers[].args    → overrides → Dockerfile CMD

resources.requests → used by → Scheduler (summed across ALL containers)
resources.limits   → enforced by → kubelet (per container)
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- How many containers in `containers[]`? (Count the `- name:` entries)
- Any `initContainers[]`? How many?
- Do all containers have correct `image`?
- Are `volumeMounts[].name` matching `volumes[].name`?
- Are `command` and `args` correct? (Easy typos here)
- Does each container have unique `name`?

## 🟢 Usually Irrelevant for CKA
- `imagePullPolicy` (defaults work fine unless private registry)
- `terminationGracePeriodSeconds`
- `dnsPolicy`
- `serviceAccountName` (unless question specifically about RBAC)
- `nodeSelector` / `affinity` (unless question is about scheduling)

## 🔴 Red Flags to Detect in 10 Seconds
- `initContainers` nested INSIDE `containers` → **wrong YAML structure**
- `volumeMounts.name` doesn't match any entry in `volumes[].name` → **mount failure**
- Container `command: ["wrong-binary"]` → **CrashLoopBackOff**
- Missing `volumes[]` definition but `volumeMounts` present → **CreateContainerConfigError**
- Two containers with the **same name** → **validation error, Pod won't create**
- Two containers using the **same containerPort** → **port conflict**
- `initContainers` with `command` that never exits → **Pod stuck forever in Init**

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe pod <pod-name>`

**Where to look FIRST:**
```
Events:                               ← ALWAYS check Events section FIRST
  Type     Reason     Message
  ----     ------     -------
  Normal   Pulling    Pulling image "nginx:1.25"
  Normal   Pulled     Successfully pulled image
  Normal   Created    Created container app
  Normal   Started    Started container app
  Warning  BackOff    Back-off restarting failed container sidecar  ← PROBLEM!
```

**Key sections to scan:**
| Section | What to Look For |
|---------|-----------------|
| `Init Containers:` | State (Waiting/Running/Terminated), Exit Code, Reason |
| `Containers:` | State of EACH container, Ready status, Restart Count |
| `Conditions:` | `Initialized` (init done?), `ContainersReady` (all running?), `Ready` |
| `Events:` | Warnings, errors, pulling/pull failures, crash messages |

**Failure signals:**
- `Init Containers: init-setup: State: Waiting, Reason: CrashLoopBackOff` → Init failed
- `Containers: sidecar: State: Waiting, Reason: CrashLoopBackOff` → Sidecar crashing
- `Containers: app: Ready: False` → App not passing readiness probe
- `Restart Count: 5` on any container → That container is crashing repeatedly
- `Reason: CreateContainerConfigError` → Missing ConfigMap/Secret/Volume

## 📝 `kubectl logs <pod-name> -c <container-name>`

**CRITICAL: Multi-container Pods REQUIRE the `-c` flag**
```bash
# ❌ This FAILS if Pod has multiple containers
kubectl logs multi-container-pod

# ✅ Specify the container name
kubectl logs multi-container-pod -c app
kubectl logs multi-container-pod -c sidecar
kubectl logs multi-container-pod -c init-setup    # Init container logs

# Check previous crashed container logs
kubectl logs multi-container-pod -c sidecar --previous
```

**Failure signals in logs:**
- Permission denied errors → SecurityContext or volume mount issue
- Connection refused → Dependency not ready, wrong port
- File not found → Volume not mounted or init didn't write data
- No output at all → Container exiting immediately (check `command`)
- OOMKilled (no log, container just dies) → Check via `describe`

---

## 🔧 `kubectl exec -it <pod> -c <container> -- <command>`

**CRITICAL: Must specify `-c` for multi-container Pods**
```bash
# Shell into specific container
kubectl exec -it multi-container-pod -c app -- /bin/sh
kubectl exec -it multi-container-pod -c sidecar -- /bin/sh

# Check shared volume from each container
kubectl exec multi-container-pod -c app -- ls /usr/share/nginx/html
kubectl exec multi-container-pod -c sidecar -- ls /shared

# Test localhost communication between containers
kubectl exec multi-container-pod -c sidecar -- curl localhost:80

# Check networking (same Pod IP for both containers)
kubectl exec multi-container-pod -c app -- hostname -i
kubectl exec multi-container-pod -c sidecar -- hostname -i   # Same IP!

# Check running processes
kubectl exec multi-container-pod -c app -- ps aux
```

**What to look for:**
- Can containers see each other's files on shared volumes?
- Can containers reach each other via `localhost:<port>`?
- Are both containers reporting the same IP?
- Is the shared volume populated (init container worked)?

# 7️⃣ Failure Patterns (Very Important)

## Common Misconfigurations & Their Symptoms

### Pattern 1: Init Container Fails
```
STATUS:    Init:CrashLoopBackOff    or    Init:Error
CAUSE:     Init container command fails, wrong image, missing dependency
EVENTS:    "Back-off restarting failed container init-setup"
BEHAVIOR:  Main containers NEVER start. Pod stuck in Init state forever.
FIX:       Fix init container image/command/permissions
```

### Pattern 2: Init Container Never Completes
```
STATUS:    Init:0/1  (stuck)
CAUSE:     Init container command runs forever (e.g., sleep infinity, infinite loop)
EVENTS:    Init container shows State: Running (never terminates)
BEHAVIOR:  Main containers NEVER start. Pod shows Init:0/1 indefinitely.
FIX:       Init command must EXIT with code 0. It's a run-to-completion container.
```

### Pattern 3: Sidecar Container Crashes
```
STATUS:    Running but RESTARTS increasing    or    CrashLoopBackOff
CAUSE:     Sidecar has wrong command, missing dependency, OOM
EVENTS:    "Back-off restarting failed container sidecar"
BEHAVIOR:  Main app may still run, but Pod shows NOT Ready.
           Restart count keeps climbing.
FIX:       Check sidecar logs: kubectl logs <pod> -c sidecar --previous
```

### Pattern 4: Volume Name Mismatch
```
STATUS:    CreateContainerConfigError
CAUSE:     volumeMounts[].name doesn't match any volumes[].name
EVENTS:    "Error: failed to create containerd container"
BEHAVIOR:  Container won't start at all.
FIX:       Match volumeMounts.name exactly with volumes.name
```

### Pattern 5: Both Containers Use Same Port
```
STATUS:    CrashLoopBackOff (second container)
CAUSE:     Both containers try to bind the same port (e.g., 80)
EVENTS:    Second container logs: "Address already in use"
BEHAVIOR:  First container works, second crashes.
FIX:       Use different ports for each container, communicate via localhost
```

### Pattern 6: Image Pull Failure
```
STATUS:    Init:ImagePullBackOff    or    ImagePullBackOff
CAUSE:     Wrong image name/tag, private registry without imagePullSecrets
EVENTS:    "Failed to pull image", "ErrImagePull"
BEHAVIOR:  Container never starts. Other containers may be fine.
FIX:       Fix image name/tag, add imagePullSecrets if private registry
```

### Pattern 7: Shared Volume Not Populated
```
STATUS:    Running (but app behaves wrong)
CAUSE:     Init container didn't write expected files, mount path mismatch
SYMPTOMS:  App container starts but serves empty page or errors
BEHAVIOR:  No crash, just wrong behavior. Hard to spot.
FIX:       Verify init ran: kubectl logs <pod> -c <init>
           Verify volume mount paths match between init and main containers
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Labels Mismatch?
- Pod labels DON'T affect multi-container behavior directly
- But if a Deployment selector doesn't match Pod template labels → **ReplicaSet creates 0 Pods**
- The multi-container Pod itself doesn't care about its own labels (only the controller does)

## What Happens If Ports Conflict?
- Two containers binding to the **same port** → Second container gets `Address already in use`
- Second container enters `CrashLoopBackOff`
- First container runs fine
- **Fix:** Use different ports. Sidecar talks to app via `localhost:<app-port>`

## What Happens If Image Is Wrong?
- Only the container with the wrong image fails → `ImagePullBackOff`
- Other containers in the Pod MAY still start (if init containers passed)
- But Pod status will show error, `Ready` condition will be `False`
- For **init containers**: ALL subsequent containers (init + main) are blocked

## What Happens If Resource Limits Are Wrong?
- Per-container enforcement:
  - Container exceeds memory limit → `OOMKilled` (only that container dies)
  - kubelet restarts only the killed container (not other containers)
  - Other containers keep running
- Scheduling impact:
  - Total `resources.requests` across ALL containers is too high → Pod `Pending`
  - `kubectl describe pod` → Event: `Insufficient cpu` or `Insufficient memory`

## What Happens If Dependencies Are Missing?
- Volume references a non-existent ConfigMap → `CreateContainerConfigError`
- Volume references an unbound PVC → Pod `Pending` (waiting for PVC)
- Init container waits for a Service that doesn't exist → Init hangs (if using `nslookup` wait loop)
- Environment variable references missing Secret key → `CreateContainerConfigError`

# 9️⃣ Fast Detection Strategy

## Exact Commands (in order, under 60 seconds)

```bash
# STEP 1: Quick status check (5 sec)
kubectl get pod <pod-name> -o wide
# Look at: STATUS, READY (e.g., 1/2 means one container not ready), RESTARTS

# STEP 2: Identify which container is failing (10 sec)
kubectl describe pod <pod-name> | grep -A5 "State:"
# Each container shows: Waiting/Running/Terminated and Reason

# STEP 3: Check events (10 sec)
kubectl describe pod <pod-name> | tail -20
# Events section: look for Warnings, BackOff, Failed, Error

# STEP 4: Check logs of failing container (15 sec)
kubectl logs <pod-name> -c <failing-container>
kubectl logs <pod-name> -c <failing-container> --previous  # If CrashLoop

# STEP 5: If init container issue (10 sec)
kubectl logs <pod-name> -c <init-container-name>
kubectl describe pod <pod-name> | grep -A5 "Init Containers"

# STEP 6: Verify shared volume (10 sec)
kubectl exec <pod-name> -c <working-container> -- ls /shared/path
```

## Decision Tree
```
Pod STATUS not Running?
├── Init:Error / Init:CrashLoopBackOff
│   └── Fix init container (image, command, permissions)
├── Init:0/1 (stuck)
│   └── Init container command never exits → make it finite
├── CrashLoopBackOff
│   └── Which container? → kubectl logs -c <name> --previous
├── ImagePullBackOff
│   └── Which container? → fix image name/tag/registry
├── Pending
│   └── Resources too high (summed) or scheduling constraint
└── CreateContainerConfigError
    └── Missing ConfigMap/Secret/Volume referenced by a container
```

# 🔟 Fast Fix Strategy

## Fix with kubectl edit
```bash
# Edit Pod directly (works for some fields, not all)
kubectl edit pod <pod-name>
# ⚠️ Most Pod spec fields are IMMUTABLE after creation
# You'll likely need to delete and recreate

# Edit the controlling Deployment (if managed)
kubectl edit deployment <deployment-name>
# This recreates Pods with updated spec
```

## Fix by Delete & Recreate
```bash
# Export current YAML
kubectl get pod <pod-name> -o yaml > fix-pod.yaml

# Edit the file
vim fix-pod.yaml
# Fix the issue (image, command, volume, etc.)
# Remove status section, resourceVersion, uid, creationTimestamp

# Delete and recreate
kubectl delete pod <pod-name>
kubectl apply -f fix-pod.yaml
```

## Fix via Deployment (preferred)
```bash
# If Pod is managed by Deployment, fix the Deployment
kubectl set image deployment/<dep-name> sidecar=busybox:1.36
# Or
kubectl edit deployment <dep-name>
# Deployment automatically recreates Pods
```

## Validate Fix
```bash
# Confirm all containers running
kubectl get pod <pod-name>
# READY should show 2/2 (or n/n for all containers)

# Verify each container
kubectl logs <pod-name> -c app
kubectl logs <pod-name> -c sidecar

# Test functionality
kubectl exec <pod-name> -c app -- curl localhost:80
kubectl exec <pod-name> -c sidecar -- cat /shared/data
```

## Confirm Recovery
```bash
# No restarts increasing
watch kubectl get pod <pod-name>
# RESTARTS count should be stable (not increasing)

# Events clean
kubectl describe pod <pod-name> | tail -10
# No new Warning events
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Minimal Working Multi-Container Pod
```yaml
# File: multi-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-demo
  labels:
    app: multi-demo
spec:
  initContainers:
  - name: init-page
    image: busybox:1.36
    command: ['sh', '-c', 'echo "<h1>Hello from init container!</h1>" > /work/index.html']
    volumeMounts:
    - name: shared
      mountPath: /work

  containers:
  - name: web
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: shared
      mountPath: /usr/share/nginx/html

  - name: updater
    image: busybox:1.36
    command: ['sh', '-c', 'while true; do echo "<p>Updated: $(date)</p>" >> /work/index.html; sleep 10; done']
    volumeMounts:
    - name: shared
      mountPath: /work

  volumes:
  - name: shared
    emptyDir: {}
```

### Validation Steps
```bash
# Create
kubectl apply -f multi-pod.yaml

# Verify all containers running
kubectl get pod multi-demo
# Expected: READY 2/2, STATUS Running

# Check init container completed
kubectl describe pod multi-demo | grep -A3 "Init Containers"
# State: Terminated, Reason: Completed, Exit Code: 0

# Verify shared volume works
kubectl exec multi-demo -c web -- cat /usr/share/nginx/html/index.html
# Should show "Hello from init container!" + "Updated: ..."

# Test web server
kubectl exec multi-demo -c updater -- wget -qO- localhost:80
# Should return HTML content

# Check both containers share same IP
kubectl exec multi-demo -c web -- hostname -i
kubectl exec multi-demo -c updater -- hostname -i
# Both return same IP
```

## Step 2 — Break It Intentionally

### Break 1: Wrong Init Container Image
```yaml
  initContainers:
  - name: init-page
    image: busybox:nonexistent    # ← Bad tag
```
```bash
kubectl apply -f broken.yaml
kubectl get pod multi-demo
# STATUS: Init:ImagePullBackOff
# Main containers never start
```

### Break 2: Init Container Command That Never Exits
```yaml
  initContainers:
  - name: init-page
    image: busybox:1.36
    command: ['sh', '-c', 'sleep infinity']    # ← Never exits!
```
```bash
kubectl get pod multi-demo
# STATUS: Init:0/1  (stuck forever)
# Main containers never start
```

### Break 3: Init Container Exits With Error
```yaml
  initContainers:
  - name: init-page
    image: busybox:1.36
    command: ['sh', '-c', 'exit 1']    # ← Non-zero exit
```
```bash
kubectl get pod multi-demo
# STATUS: Init:CrashLoopBackOff
```

### Break 4: Volume Name Mismatch
```yaml
  containers:
  - name: web
    volumeMounts:
    - name: wrong-name              # ← Doesn't match volumes[].name
      mountPath: /usr/share/nginx/html
  volumes:
  - name: shared                    # ← Name is "shared" not "wrong-name"
    emptyDir: {}
```
```bash
kubectl apply -f broken.yaml
# Error during creation: "volume wrong-name not found"
# Pod won't even be created (validation error)
```

### Break 5: Two Containers Bind Same Port
```yaml
  containers:
  - name: web1
    image: nginx:1.25
    ports:
    - containerPort: 80
  - name: web2
    image: nginx:1.25              # ← Also binds port 80!
    ports:
    - containerPort: 80
```
```bash
kubectl get pod multi-demo
# STATUS: CrashLoopBackOff (web2 keeps dying)
kubectl logs multi-demo -c web2
# "Address already in use"
```

### Break 6: Sidecar OOM Kill
```yaml
  - name: updater
    image: busybox:1.36
    command: ['sh', '-c', 'head -c 1G /dev/urandom > /dev/null']  # ← Consume memory
    resources:
      limits:
        memory: 10Mi              # ← Very low limit
```
```bash
kubectl get pod multi-demo
# RESTARTS incrementing for updater
kubectl describe pod multi-demo | grep -A3 "updater"
# Last State: Terminated, Reason: OOMKilled
```

### Break 7: Missing Volume Mount in One Container
```yaml
  containers:
  - name: web
    image: nginx:1.25
    # ← NO volumeMount — web serves default nginx page, not shared content
  - name: updater
    image: busybox:1.36
    volumeMounts:
    - name: shared
      mountPath: /work            # ← Updater writes here, but web can't see it
```
```bash
# Pod runs fine! But web shows default nginx page, not shared content
# This is a LOGIC error, not a crash — hardest to detect
kubectl exec multi-demo -c web -- ls /usr/share/nginx/html
# Shows default nginx files (no shared data)
```

## Step 3 — Debug Without Looking at Solution

### First Command Always
```bash
kubectl get pod <name> -o wide
# Read: STATUS, READY (x/y), RESTARTS, NODE
```

### Narrow the Issue
```bash
# If STATUS shows Init problem
kubectl describe pod <name> | grep -A10 "Init Containers"
kubectl logs <name> -c <init-container-name>

# If STATUS shows CrashLoopBackOff
kubectl describe pod <name> | grep "Restart Count"   # Which container?
kubectl logs <name> -c <failing-container> --previous

# If STATUS shows Running but READY is wrong (1/2)
kubectl describe pod <name> | grep -A5 "Conditions"
kubectl describe pod <name> | grep -B2 "Ready: False"
```

### Confirm Root Cause
```bash
# Volume issues → exec into container, check mount
kubectl exec <name> -c <container> -- ls /path/to/mount
kubectl exec <name> -c <container> -- df -h

# Network issues → test localhost communication
kubectl exec <name> -c <container> -- curl -s localhost:<other-container-port>

# Init issues → check init logs
kubectl logs <name> -c <init-name>
```

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] Multi-container Pod patterns: Sidecar, Init Container, Ambassador, Adapter
- [ ] Init container lifecycle: runs before main containers, must exit 0
- [ ] Shared resources: `localhost` networking, shared volumes (`emptyDir`)
- [ ] Container-level vs Pod-level configuration
- [ ] `command` and `args` override Dockerfile `ENTRYPOINT` and `CMD`
- [ ] Per-container probes, resources, security context
- [ ] How `kubectl logs -c` and `kubectl exec -c` work

### What to Focus On
- Writing multi-container Pod YAML from scratch
- Debugging which container is failing in a multi-container Pod
- Init container troubleshooting (most common exam scenario)
- Volume sharing between containers (`emptyDir`)
- The `-c <container-name>` flag for logs and exec

### What to Practice Repeatedly
- Create a Pod with init container + 2 main containers
- Break and fix init containers (wrong command, non-zero exit)
- Break and fix sidecar containers (wrong image, port conflict)
- Read `kubectl describe` output to identify failing container
- Use `kubectl logs -c <name> --previous` for crashed containers

### What to Ignore (Not CKA Relevant)
- Sidecar container injection mechanisms (Istio/service mesh internals)
- Pod lifecycle hook implementation details
- Ambassador/Adapter pattern deep theory
- Container runtime internals (how containerd manages multi-container)

### 5 Quiz Questions

**Q1:** A Pod has `initContainers: [{name: init, image: busybox, command: ['sh', '-c', 'sleep infinity']}]` and one main container. What happens?
> **A:** Pod is stuck in `Init:0/1` forever. Init container never exits, so main container never starts.

**Q2:** A Pod shows `READY: 1/2` and `STATUS: Running`. What does this mean?
> **A:** One of the two main containers is not passing its readiness probe, or one container has crashed but the Pod is still Running.

**Q3:** How do you view logs of a crashed init container named `setup`?
> **A:** `kubectl logs <pod-name> -c setup` (init container logs persist even after completion)

**Q4:** Two containers in a Pod both need to read/write to `/data`. How?
> **A:** Define a shared `emptyDir` volume and mount it in both containers at their respective paths.

**Q5:** Container `web` runs on port 80, container `proxy` needs to forward traffic to it. What address does `proxy` use?
> **A:** `localhost:80` — both containers share the same network namespace.

# 1️⃣3️⃣ Component Interaction Map

## How Multi-Container Pod Interacts With Everything

```
┌──────────────────────────────────────────────────────────────────┐
│                    MULTI-CONTAINER POD                            │
│                                                                  │
│  Init Container → Main Container A → Main Container B           │
│       │                │                    │                    │
│       ▼                ▼                    ▼                    │
│  Shared Volumes ←──────┴────────────────────┘                   │
│  Shared Network (localhost)                                      │
│  Shared Service Account                                          │
└──────────────────────────┬───────────────────────────────────────┘
                           │
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
      ┌──────────┐  ┌──────────┐  ┌──────────────┐
      │ kubelet  │  │ Scheduler│  │ Controller   │
      │          │  │          │  │ Manager      │
      └──────────┘  └──────────┘  └──────────────┘
            │              │              │
            ▼              ▼              ▼
     Manages each    Sums resource   Maintains desired
     container       requests of     replica count
     independently   ALL containers  (via Deployment)
     in the Pod      for placement   creates/deletes Pods
```

### Interaction Details

| Component | Normal Behavior | When Things Go Wrong |
|-----------|----------------|---------------------|
| **kubelet** | Starts init containers sequentially, then all main containers | Restarts individual crashed containers, reports container statuses to API server |
| **Scheduler** | Sums `resources.requests` of ALL containers, finds Node with capacity | If sum exceeds any Node capacity → Pod `Pending` forever |
| **Controller Manager** | Deployment controller maintains replica count | If Pod keeps crashing (all containers fail), controller keeps recreating |
| **etcd** | Stores Pod spec and status | Pod status includes per-container state (visible in `kubectl get pod -o yaml`) |
| **Services** | Route traffic to Pod IP; one Service selects the whole Pod | Service `targetPort` must match the correct container's port; can only target one port per Service port entry |
| **Network (CNI)** | Creates one network namespace for all containers | If CNI fails, entire Pod has no networking (all containers affected) |
| **Storage** | Provisions volumes defined in Pod spec | Volume mount failure blocks the specific container from starting |
| **Probes** | Per-container liveness/readiness | If one container's readiness probe fails, Pod removed from Service endpoints (even if other containers are fine) |

# 1️⃣4️⃣ Self-Evaluation Questions

## ✔ Can I Break This Intentionally?

| Breakage | Can You Do It? |
|----------|---------------|
| Make init container fail with CrashLoopBackOff | `command: ['sh', '-c', 'exit 1']` |
| Make init container hang forever | `command: ['sleep', 'infinity']` |
| Make sidecar crash with OOM | Set `limits.memory: 5Mi`, run memory-intensive command |
| Make volume mount fail | Mismatch `volumeMounts.name` with `volumes.name` |
| Make port conflict | Two containers binding same port |
| Make shared data invisible | Mount volume at wrong path in one container |
| Make Pod Pending from resources | Sum of all container requests exceeds node capacity |

## ✔ Can I Detect Failure in Under 60 Seconds?

```bash
# 10 sec: Status + Ready count
kubectl get pod <name>

# 20 sec: Which container is failing
kubectl describe pod <name> | grep -E "State:|Reason:|Exit Code:|Restart Count:"

# 20 sec: Why it's failing
kubectl logs <name> -c <failing-container> --previous

# 10 sec: Confirm
kubectl describe pod <name> | tail -15   # Events
```

**If your answer is YES to all, you can do this in under 60 seconds → You're CKA ready for multi-container Pods.**

## ✔ Can I Fix Without Googling?

| Scenario | Fix |
|----------|-----|
| Init container CrashLoopBackOff | Check logs → fix command/image → delete & recreate Pod |
| Sidecar ImagePullBackOff | Fix image name/tag → `kubectl edit` or delete & recreate |
| Volume mount mismatch | Match `volumeMounts.name` ↔ `volumes.name` → recreate |
| Port conflict between containers | Change one container's port → recreate |
| Pod Pending (resource sum too high) | Reduce per-container requests or add node capacity |
| READY 1/2 (readiness probe failing) | Check probe config, fix app issue, or adjust probe |
| OOMKilled on one container | Increase memory limit for that container |
