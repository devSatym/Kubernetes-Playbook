# 🧠 Rollbacks — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Reverts a Deployment to a previous known-good version** when a rollout goes wrong
- Every Deployment update creates a new **ReplicaSet** — old ReplicaSets are kept as **revisions**
- Rollback = scale the old ReplicaSet back up, scale the failed new ReplicaSet down
- **Instant recovery** — no need to rebuild, rewrite YAML, or find the old image tag
- Kubernetes does **NOT auto-rollback** — you must do it manually with `kubectl rollout undo`

## How Rollback Works Internally
```
BEFORE ROLLBACK:
  Deployment revision history:
    RS-v1 (nginx:1.25) → replicas: 0  (old, kept for rollback)
    RS-v2 (nginx:1.26) → replicas: 0  (old, kept for rollback)
    RS-v3 (nginx:BAD)  → replicas: 3  (current, BROKEN)

  kubectl rollout undo deployment/myapp

AFTER ROLLBACK:
  Deployment revision history:
    RS-v1 (nginx:1.25) → replicas: 0  (old)
    RS-v3 (nginx:BAD)  → replicas: 0  (old, was broken)
    RS-v2 (nginx:1.26) → replicas: 3  (RE-ACTIVATED as v4!)

  NOTE: The undo creates a NEW revision (v4) pointing to the old RS template.
        The revision numbers keep incrementing, never go backwards.
```

## Real-World Use Cases
- **Bad deployment** — New image crashes → rollback to previous version
- **Performance regression** — New release is slow → rollback while investigating
- **Config error** — Env variables or mounts wrong → rollback to last working config
- **Emergency recovery** — Production broken by update → instant rollback in seconds
- **CKA exam** — "The Deployment update is broken, revert to the previous version"

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────┐
│  ROLLBACK = Deployment Controller Using Old ReplicaSets       │
│                                                               │
│  Deployment                                                   │
│    │                                                          │
│    ├── RS-v1 (replicas: 0) ← kept for rollback               │
│    ├── RS-v2 (replicas: 0) ← kept for rollback               │
│    └── RS-v3 (replicas: 3) ← current (maybe broken)          │
│                                                               │
│  kubectl rollout undo → RS-v2 scales to 3, RS-v3 scales to 0│
│                                                               │
│  revisionHistoryLimit controls how many old RS are kept       │
│  (default: 10)                                                │
└──────────────────────────────────────────────────────────────┘
```

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Deployment Controller** — executes the rollback by rescaling ReplicaSets
- **ReplicaSets** — old RS are the rollback targets; must still exist (not deleted)
- **Pods** — new Pods created from old RS spec; broken Pods terminated
- **Service** — traffic shifts back to healthy old Pods
- **Operators / CI/CD** — may use rollback as a safety mechanism

## What Resources It Interacts With
- **revisionHistoryLimit** — determines how many old RS are kept (no old RS = no rollback)
- **ReplicaSet** — each revision IS an old RS stored by the Deployment
- **Pod template** — rollback restores the old template (image, env, volumes, command)
- **Service endpoints** — updated as old Pods come back and new Pods are removed

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User detects broken rollout → decides to rollback
  kubectl rollout undo → API server patches Deployment spec

WHAT UNDO DOES:
  1. Finds the target revision's RS (default: previous revision)
  2. Copies that RS's Pod template INTO the Deployment spec
  3. This looks like a NEW template change → triggers a rolling update
  4. Deployment Controller creates/re-scales RS with old template
  5. Broken RS scales to 0

DOWNSTREAM:
  Old RS scales up → old Pod template → Pods with known-good config
  Broken RS scales down → broken Pods terminated
  Service endpoints updated → traffic goes to healthy Pods
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Rollback Is NOT a YAML Object — It's a kubectl Command

> Rollback has no YAML manifest. It's an **operational action** performed via `kubectl rollout undo`.

### The Commands ARE the Structure
```bash
# =====================================================
#  CORE ROLLBACK COMMANDS (memorize these for CKA)
# =====================================================

# Rollback to PREVIOUS revision (most common)
kubectl rollout undo deployment/<name>

# Rollback to SPECIFIC revision number
kubectl rollout undo deployment/<name> --to-revision=<N>

# Check available revisions before rolling back
kubectl rollout history deployment/<name>

# Inspect a specific revision's Pod template
kubectl rollout history deployment/<name> --revision=<N>

# Check if rollback completed
kubectl rollout status deployment/<name>
```

### Related YAML Field: revisionHistoryLimit
```yaml
# This is in the Deployment spec — controls how many old RS survive
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  revisionHistoryLimit: 10            # ← Default: 10 old RS kept
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0                # ← Safe during rollback too
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: web
        image: nginx:1.25
```

## ⚠️ Common Traps

### Trap 1: revisionHistoryLimit: 0 → No Rollback Possible
```yaml
spec:
  revisionHistoryLimit: 0     # ← Old RS deleted immediately
                               # No rollback targets exist!
                               # kubectl rollout undo → nothing to undo to
```

### Trap 2: Thinking Rollback Goes "Back in Time"
```
# ❌ WRONG mental model:
#   v1 → v2 → v3 → UNDO → now at v2, next undo goes to v1

# ✅ CORRECT mental model:
#   v1 → v2 → v3 → UNDO → v4 (which IS v2's template)
#   Revision numbers always increment
#   kubectl rollout undo again → v5 (which IS v3's template)
#   You're ping-ponging, not going backward!
```

### Trap 3: kubectl rollout undo Without Checking History
```bash
# ❌ RISKY — you don't know what "previous" is
kubectl rollout undo deployment/myapp

# ✅ SAFER — check first
kubectl rollout history deployment/myapp
# See all revisions → decide which one to go to
kubectl rollout undo deployment/myapp --to-revision=2
```

### Trap 4: Expecting Auto-Rollback
```
# ❌ WRONG — K8s does NOT auto-rollback on failure
# A stuck rollout just stays stuck forever
# ProgressDeadlineExceeded is just a status condition, not an action

# ✅ CORRECT — You must manually run:
kubectl rollout undo deployment/<name>
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields & Commands

| Field / Command | Why It Matters | What Breaks | CKA Priority |
|----------------|---------------|-------------|-------------|
| `kubectl rollout undo` | THE rollback command | Without this, no rollback | 🔴 |
| `--to-revision=N` | Target specific revision | Wrong N = wrong version restored | 🔴 |
| `kubectl rollout history` | View available revisions | Can't choose correct revision without this | 🔴 |
| `kubectl rollout status` | Verify rollback completed | Don't know if rollback succeeded | 🔴 |
| `revisionHistoryLimit` | How many old RS kept | 0 = no rollback possible | 🔴 |
| `spec.strategy` | Rolling update config applies to rollback too | maxSurge/maxUnavailable affect rollback speed | 🟡 |
| `kubernetes.io/change-cause` annotation | Gives context to each revision in history | Without it, history shows `<none>` — hard to identify | 🟡 |

## Key Relationships
```
revisionHistoryLimit → controls number of old RS kept → rollback targets
  Default: 10 → last 10 versions available for rollback
  Set to 0 → CAN'T ROLLBACK AT ALL

Deployment.spec.template → each unique template = 1 revision = 1 RS
  Rollback = copy old revision's template → paste into Deployment spec
  This triggers a new rolling update using the old template

Revision numbering:
  v1 → v2 → v3 → undo → v4 (=v2's template)
  Revision numbers NEVER decrease, always increment
  
  kubectl rollout undo = copies (latest - 1) revision into Deployment
  kubectl rollout undo --to-revision=N = copies revision N into Deployment
```

# 5️⃣ How Much Should You Notice?

## ⚡ Scan Immediately (5 seconds)
- How many revisions exist? (`kubectl rollout history`)
- What revision is current?
- What's the image/config in each revision?
- Is `revisionHistoryLimit` set to a reasonable value?
- Is there a `change-cause` annotation for context?

## 🟢 Usually Irrelevant for CKA
- Exact revision number assigned (K8s auto-generates)
- Pod template hash on RS labels
- The internal mapping between RS names and revision numbers
- Deployment status conditions during rollback (just check `rollout status`)
- progressDeadlineSeconds for rollback timing

## 🔴 Red Flags

| Red Flag | Problem |
|----------|---------|
| `kubectl rollout history` shows only 1 revision | Nothing to roll back to |
| `revisionHistoryLimit: 0` in Deployment spec | Old RS deleted → no rollback |
| `kubectl rollout undo` → error: no revision found | Either limit=0 or only 1 revision |
| CHANGE-CAUSE column all `<none>` | Can't identify which revision has which image |
| Running `undo` twice without checking | Ping-pongs between two versions |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Rollback Not Possible — No Revision History
```
SYMPTOM:        kubectl rollout undo → error or unexpected behavior
CAUSE:          revisionHistoryLimit: 0 → all old RS deleted
                OR: only 1 revision exists (first deploy, never updated)
DESCRIBE RS:    Only 1 RS exists; no old RS to rollback to
FIX:            Must fix forward: kubectl set image deployment/<name> <container>=<correct-image>
PREVENTION:     Keep revisionHistoryLimit ≥ 5 (default is 10)
EXAM SCENARIO:  "Cannot rollback. Find another way to restore previous version."
```

## Pattern 2: Rollback to Wrong Revision
```
SYMPTOM:        After undo, Pods run an unexpected version (not the one you wanted)
CAUSE:          Used kubectl rollout undo without --to-revision, and "previous" wasn't what you expected
                OR: revision numbers shifted after previous undo operations
DETECTION:      kubectl get pods -o jsonpath → check image; doesn't match expected
FIX:            Check history: kubectl rollout history deployment/<name>
                Find correct revision: kubectl rollout history deployment/<name> --revision=N
                Undo to correct revision: kubectl rollout undo deployment/<name> --to-revision=N
```

## Pattern 3: Rollback Itself Gets Stuck
```
SYMPTOM:        kubectl rollout status hangs after undo
CAUSE:          The old version also has problems (bad config, resource issues)
                OR: node capacity insufficient for old version's resource requests
DESCRIBE:       Rollback creates new Pods from old template → those Pods fail too
FIX:            Fix forward: kubectl set image deployment/<name> <container>=<known-good-image>
                Or fix underlying issue (add nodes, fix config)
EXAM TIP:       If rollback gets stuck, the old revision is also broken
```

## Pattern 4: Rollback During Active Rollout
```
SYMPTOM:        Rollout in progress, you need to undo immediately
CAUSE:          New version is clearly broken mid-rollout
BEHAVIOR:       kubectl rollout undo works even during active rollout
                K8s cancels current rollout → starts rollback
                Some Pods may be in transitional state briefly
FIX:            Just run: kubectl rollout undo deployment/<name>
                K8s handles the cancellation cleanly
```

## Pattern 5: ProgressDeadlineExceeded But No Auto-Rollback
```
SYMPTOM:        kubectl describe deploy → Progressing: False, ProgressDeadlineExceeded
MISCONCEPTION:  "K8s will rollback automatically"
REALITY:        K8s does NOT auto-rollback. Ever. The stuck rollout stays stuck.
                Existing old Pods may still serve (depends on maxUnavailable)
FIX:            Manual rollback: kubectl rollout undo deployment/<name>
EXAM TIP:       Always remember: Kubernetes NEVER auto-rollbacks. YOU must do it.
```

## Pattern 6: Change-Cause Not Recorded
```
SYMPTOM:        kubectl rollout history shows <none> for all revisions
CAUSE:          Changes made without --record flag or annotation
IMPACT:         Can't identify which revision had which config
FIX TODAY:      Inspect each: kubectl rollout history deployment/<name> --revision=N
PREVENTION:     Always annotate: kubectl annotate deployment/<name> kubernetes.io/change-cause="reason"
```

## Pattern 7: Ping-Pong Effect on Double Undo
```
SYMPTOM:        After second undo, you're back to the broken version
CAUSE:          First undo: v3(bad) → v4(=v2 good). Second undo: v4(good) → v5(=v3 bad!)
BEHAVIOR:       Undo alternates between last two versions
FIX:            Use --to-revision=N to target specific version:
                kubectl rollout undo deployment/<name> --to-revision=2
```

# 8️⃣ Misconfiguration Behavior

## What Happens If revisionHistoryLimit = 0?
- **Behavior:** Old RS deleted immediately after new RS takes over
- **Impact:** `kubectl rollout undo` has no target → can't rollback
- **Detection:** `kubectl get rs -l app=<label>` → only 1 RS exists
- **Fix:** Must fix forward with `kubectl set image` or `kubectl edit`
- **Prevention:** Set `revisionHistoryLimit: 10` (default)

## What Happens If You Undo Without Checking History?
- **Behavior:** Reverts to revision N-1 (whatever that happened to be)
- **Risk:** If you've done multiple undos, N-1 may be the BROKEN version
- **Detection:** `kubectl get pods -o jsonpath` → check image → surprise!
- **Fix:** `kubectl rollout undo --to-revision=<correct-N>`

## What Happens If Old Revision Also Has Issues?
- **Behavior:** Rollback starts rolling update → old template Pods fail too
- Old Pods: CrashLoopBackOff, ImagePullBackOff, Pending
- Rollback itself gets stuck
- **Fix:** Must fix forward: identify a known-good image manually

## What Happens If maxSurge=0 and maxUnavailable=0 During Rollback?
- **Behavior:** K8s rejects this config at validation
- Rollback uses same strategy settings as forward deployment
- Can't rollback if strategy is deadlocked
- **Fix:** Fix strategy to allow at least maxSurge=1 or maxUnavailable=1

## What Happens If You Rollback a Deployment That Uses PVCs?
- **Behavior:** New Pods from old template may need PVCs
- If PVCs still exist → Pods bind to them → OK
- If PVCs were deleted → Pods stay Pending
- Rollback doesn't recreate deleted PVCs
- **Fix:** Recreate PVCs before rollback, or after rollback if Pods are Pending

## What Happens If Image Was Deleted from Registry Between Versions?
- **Behavior:** Rollback tries to pull old image → ImagePullBackOff
- The revision exists but the image doesn't anymore
- **Fix:** Push the old image back to registry, or choose a different revision

# 9️⃣ Fast Detection Strategy

## 60-Second Rollback Decision Flow

```bash
# ⏱️ 0-10 sec: Is the current deployment broken?
kubectl get deploy <name>
# READY not matching desired → broken

kubectl rollout status deployment/<name>
# Hanging → stuck rollout → need rollback

# ⏱️ 10-25 sec: What revisions are available?
kubectl rollout history deployment/<name>
# How many revisions? Which one was last known good?

# ⏱️ 25-40 sec: Identify the good revision
kubectl rollout history deployment/<name> --revision=<N>
# Check image and config of candidate revision

# ⏱️ 40-50 sec: Execute rollback
kubectl rollout undo deployment/<name>
# OR for specific version:
kubectl rollout undo deployment/<name> --to-revision=<N>

# ⏱️ 50-60 sec: Confirm
kubectl rollout status deployment/<name>
# "successfully rolled out" ✅
kubectl get deploy <name>
# READY n/n ✅
```

## Quick Decision Matrix
```
SITUATION                                → ACTION
──────────────────────────────────────────────────────────
Stuck rollout, previous version was fine → kubectl rollout undo
Stuck rollout, need specific old version → kubectl rollout undo --to-revision=N
Stuck rollout, no history (limit=0)      → Fix forward: kubectl set image
Stuck rollout after undo (old also bad)  → Fix forward: kubectl set image with known-good
Multiple undos, getting confused         → kubectl rollout history → pick right --to-revision
Not sure which revision is good          → Inspect each: rollout history --revision=N
```

# 🔟 Fast Fix Strategy

## The Complete Rollback Command Kit

```bash
# === THE ESSENTIAL FOUR ===

# 1. CHECK what's available to roll back to
kubectl rollout history deployment/<name>

# 2. INSPECT a revision before committing
kubectl rollout history deployment/<name> --revision=2

# 3. EXECUTE the rollback
kubectl rollout undo deployment/<name>                    # To previous
kubectl rollout undo deployment/<name> --to-revision=2    # To specific

# 4. VERIFY the rollback succeeded
kubectl rollout status deployment/<name>

# === ADVANCED ===

# Record WHY you're making changes (shows in rollout history)
kubectl annotate deployment/<name> \
  kubernetes.io/change-cause="Rollback to v1.25 due to crash in v1.26"

# If rollback itself fails → fix forward
kubectl set image deployment/<name> web=nginx:1.25

# Restart all Pods with current spec (not a rollback, but a refresh)
kubectl rollout restart deployment/<name>
```

## Step-by-Step CKA Rollback Scenario

### Scenario: "Deployment is broken after update. Fix it."
```bash
# Step 1: Confirm it's broken
kubectl get deploy myapp
# READY: 1/3, UP-TO-DATE: 2 → rollout stuck

# Step 2: Check what went wrong
kubectl get pods -l app=myapp
# 2 new Pods: CrashLoopBackOff
kubectl logs <new-pod> --previous
# "Error: missing config file" → new image needs a ConfigMap

# Step 3: Check rollback options
kubectl rollout history deployment/myapp
# REVISION  CHANGE-CAUSE
# 1         initial deployment
# 2         updated to v2.0
# 3         updated to v3.0 (broken)

# Step 4: Rollback to revision 2
kubectl rollout undo deployment/myapp --to-revision=2

# Step 5: Verify
kubectl rollout status deployment/myapp
# "successfully rolled out"
kubectl get deploy myapp
# READY: 3/3 ✅

# Step 6: Verify Pods running old-good version
kubectl get pods -l app=myapp -o jsonpath='{.items[*].spec.containers[*].image}'
# nginx:1.26 (revision 2's image) ✅
```

## Validate Recovery
```bash
# Deployment healthy
kubectl get deploy <name>
# READY n/n, UP-TO-DATE n, AVAILABLE n ✅

# All Pods running
kubectl get pods -l app=<label>
# All Running, 0 restarts ✅

# Correct image
kubectl get pods -l app=<label> -o jsonpath='{.items[0].spec.containers[0].image}'

# RS state clean
kubectl get rs -l app=<label>
# Active RS has all replicas; broken RS at 0

# No events warnings
kubectl describe deploy <name> | tail -10
# Clean events ✅
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example with Revision History

```bash
# Create initial deployment (Revision 1)
kubectl create deployment roll-back-demo --image=nginx:1.24 --replicas=3
kubectl annotate deployment/roll-back-demo \
  kubernetes.io/change-cause="Initial deploy: nginx:1.24"

# Update to v1.25 (Revision 2)
kubectl set image deployment/roll-back-demo nginx=nginx:1.25
kubectl annotate deployment/roll-back-demo \
  kubernetes.io/change-cause="Update to nginx:1.25"

# Update to v1.26 (Revision 3)
kubectl set image deployment/roll-back-demo nginx=nginx:1.26
kubectl annotate deployment/roll-back-demo \
  kubernetes.io/change-cause="Update to nginx:1.26"

# Wait for each rollout
kubectl rollout status deployment/roll-back-demo
```

### Validation Steps
```bash
# Check revision history
kubectl rollout history deployment/roll-back-demo
# REVISION  CHANGE-CAUSE
# 1         Initial deploy: nginx:1.24
# 2         Update to nginx:1.25
# 3         Update to nginx:1.26

# All Pods running latest
kubectl get deploy roll-back-demo
# READY 3/3

# 3 ReplicaSets (only latest has replicas)
kubectl get rs -l app=roll-back-demo
# RS-v1: 0/0, RS-v2: 0/0, RS-v3: 3/3

# Inspect a specific revision
kubectl rollout history deployment/roll-back-demo --revision=1
# Image: nginx:1.24

# Test rollback to revision 1
kubectl rollout undo deployment/roll-back-demo --to-revision=1
kubectl rollout status deployment/roll-back-demo
kubectl get pods -l app=roll-back-demo -o jsonpath='{.items[0].spec.containers[0].image}'
# nginx:1.24 ✅

# Undo back (to whatever was "previous" before "undo")
kubectl rollout undo deployment/roll-back-demo
```

## Step 2 — Break It Intentionally

### Break 1: Deploy a Bad Image Then Rollback
```bash
# Deploy broken image
kubectl set image deployment/roll-back-demo nginx=nginx:DOES-NOT-EXIST
kubectl annotate deployment/roll-back-demo \
  kubernetes.io/change-cause="BROKEN: bad image tag"

kubectl rollout status deployment/roll-back-demo
# Hangs... ❌

kubectl get pods -l app=roll-back-demo
# New Pods: ImagePullBackOff

# ROLLBACK
kubectl rollout undo deployment/roll-back-demo
kubectl rollout status deployment/roll-back-demo
# "successfully rolled out" ✅
```

### Break 2: Deploy Crashing Image Then Rollback
```bash
kubectl set image deployment/roll-back-demo nginx=busybox:1.36
# busybox exits immediately → CrashLoopBackOff

# ROLLBACK
kubectl rollout undo deployment/roll-back-demo
# Restored ✅
```

### Break 3: Set revisionHistoryLimit to 0 Then Try Rollback
```bash
kubectl patch deployment roll-back-demo \
  -p '{"spec":{"revisionHistoryLimit":0}}'
# Wait a moment...
kubectl get rs -l app=roll-back-demo
# Only 1 RS left! Old RS deleted.

# Try to rollback
kubectl rollout undo deployment/roll-back-demo
# May error or have unexpected behavior — nothing to rollback to

# FIX: Must set image manually
kubectl set image deployment/roll-back-demo nginx=nginx:1.25

# RESTORE: Set revisionHistoryLimit back
kubectl patch deployment roll-back-demo \
  -p '{"spec":{"revisionHistoryLimit":10}}'
```

### Break 4: Double Undo (Ping-Pong Effect)
```bash
# Setup: v1=1.24, v2=1.25, v3=1.26 (current)
# First undo: goes to v2 (1.25), creates revision 4
kubectl rollout undo deployment/roll-back-demo
kubectl get pods -o jsonpath='{.items[0].spec.containers[0].image}' -l app=roll-back-demo
# nginx:1.25 ← revision 4 (= v2 template)

# Second undo: goes back to revision 3 (1.26!)
kubectl rollout undo deployment/roll-back-demo
kubectl get pods -o jsonpath='{.items[0].spec.containers[0].image}' -l app=roll-back-demo
# nginx:1.26 ← ping-pong! Back to v3 template as revision 5

# LESSON: Use --to-revision to avoid confusion
```

### Break 5: Image Deleted from Registry
```bash
# Deploy with a specific tag
kubectl set image deployment/roll-back-demo nginx=nginx:1.20
kubectl rollout status deployment/roll-back-demo
# Suppose nginx:1.20 was later removed from Docker Hub

# Later updates...
kubectl set image deployment/roll-back-demo nginx=nginx:1.26
kubectl rollout status deployment/roll-back-demo

# Try to rollback to 1.20
kubectl rollout undo deployment/roll-back-demo --to-revision=<revision-with-1.20>
# If image was removed from registry:
# Pods → ImagePullBackOff → rollback itself fails!

# FIX: Choose a different revision or set a valid image
kubectl rollout undo deployment/roll-back-demo
kubectl set image deployment/roll-back-demo nginx=nginx:1.25
```

### Break 6: Rollback When Rollback Target Is Also Broken
```bash
# v1: broken image, v2: also broken, v3: broken
# No good revision to rollback to!
kubectl rollout history deployment/roll-back-demo --revision=1
# Image: nginx:broken-v1
kubectl rollout history deployment/roll-back-demo --revision=2
# Image: nginx:broken-v2

# FIX FORWARD (the only option):
kubectl set image deployment/roll-back-demo nginx=nginx:1.25
kubectl rollout status deployment/roll-back-demo
```

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] `kubectl rollout undo deployment/<name>` — the core rollback command
- [ ] `--to-revision=N` — target specific revision
- [ ] `kubectl rollout history` — view all revision list
- [ ] `kubectl rollout history --revision=N` — inspect specific revision details
- [ ] `kubectl rollout status` — verify rollback completed
- [ ] Rollback creates a NEW revision (numbers always increment)
- [ ] Kubernetes does NOT auto-rollback on failure
- [ ] `revisionHistoryLimit` controls how many old RS are kept (default: 10)
- [ ] Double undo = ping-pong between two versions
- [ ] `kubernetes.io/change-cause` annotation for readable history

### What to Focus On
- **The 4-command rollback workflow:** history → inspect → undo → status
- **Recognizing when rollback is needed** (stuck rollout, broken Pods)
- **Using --to-revision** vs bare undo
- **Understanding revision numbering** (always goes up)
- **What to do when rollback itself fails** (fix forward)

### What to Practice Repeatedly
- Create Deployment → update 3 times → rollback to specific revision → verify
- Deploy bad image → detect stuck → rollback → confirm restored → under 2 minutes
- Double undo → observe ping-pong → use --to-revision to fix
- Set revisionHistoryLimit to 0 → observe can't rollback → fix forward
- Record change-cause → read it from history → use it to identify revision

### What to Ignore (Not CKA Relevant)
- How Deployment controller internally maps RS to revision numbers
- Revision pruning algorithm
- How kubectl rollout undo patches the Deployment spec
- Comparing rollback mechanisms across different controllers (StatefulSet, DaemonSet)
- Argo Rollouts or Flagger rollback mechanisms

### 5 Quiz Questions

**Q1:** After `kubectl rollout undo deployment/myapp`, what revision number is the restored version?
> **A:** It gets the NEXT available revision number (increments). If the latest was revision 5, the undo creates revision 6 with the template from revision 4. Revision numbers never go backwards.

**Q2:** You run `kubectl rollout undo` twice in a row. What happens?
> **A:** First undo: reverts to previous version (N-1). Second undo: reverts to the version BEFORE the first undo — which was the broken version! This is the ping-pong effect. Use `--to-revision=N` to avoid this.

**Q3:** A Deployment has `revisionHistoryLimit: 2`. It has been updated 10 times. How many old RS exist?
> **A:** 2 old RS (plus the current active RS). Only the 2 most recent old revisions are kept. Older RS are garbage collected. You can only rollback to these 2 versions.

**Q4:** A rollout is stuck with `ProgressDeadlineExceeded`. Does Kubernetes automatically rollback?
> **A:** NO. Kubernetes never auto-rollbacks. `ProgressDeadlineExceeded` is just a status condition that marks the rollout as failed. You must manually run `kubectl rollout undo`.

**Q5:** During a rollback, does the Service experience downtime?
> **A:** No (if using RollingUpdate strategy). The rollback uses the same rolling update mechanism — old-good Pods scale up while broken Pods scale down. Service routes to available Pods throughout. Only if using `Recreate` strategy would there be downtime.

# 1️⃣3️⃣ Component Interaction Map

```
┌──────────────────────────────────────────────────────────────────┐
│              ROLLBACK IN ACTION                                    │
│                                                                   │
│  kubectl rollout undo                                             │
│       │                                                           │
│       ▼                                                           │
│  API Server patches Deployment spec.template                     │
│  with old revision's template                                    │
│       │                                                           │
│       ▼                                                           │
│  Deployment Controller detects template change                   │
│       │                                                           │
│       ├──→ Scale UP old-good RS (restore)                        │
│       │    [Pod-good-1] [Pod-good-2] [Pod-good-3]                │
│       │                                                           │
│       └──→ Scale DOWN broken RS                                  │
│            [Pod-bad-1 ✗] [Pod-bad-2 ✗]  → terminated             │
│                                                                   │
│  Service endpoints:                                               │
│    Remove broken Pods → Add good Pods                            │
│    Traffic smoothly transitions back                             │
└──────────────────────────────────────────────────────────────────┘
```

### Interaction Details

| Component | During Rollback | When Rollback Fails |
|-----------|----------------|-------------------|
| **Deployment Controller** | Detects template change → manages RS scaling (same as rolling update) | If broken AND old version also broken → both RS have READY=0 |
| **ReplicaSets** | Old-good RS scales up; broken RS scales down | If old-good RS was deleted (limit=0) → nothing to scale up |
| **Scheduler** | Places restored Pods on nodes | If no capacity → restored Pods Pending → rollback stuck |
| **Service** | Updates endpoints: good Pods added, bad Pods removed | During transition, routes to mix; after rollback, only good Pods |
| **etcd** | Stores updated Deployment spec (now with old template) | If etcd is slow → rollback takes longer to propagate |
| **Controller Manager** | Runs Deployment controller that orchestrates rollback | If controller-manager down → rollback doesn't execute |
| **Network** | New IPs assigned to restored Pods | CNI failure → restored Pods can't get IPs |
| **Storage** | If old template uses PVCs → PVCs must exist for restored Pods | PVC deleted between versions → restored Pods Pending |

### Failure Cascade When Rollback Fails
```
Rollback initiated (kubectl rollout undo)
  → Deployment template updated with old revision
    → Old RS scales up → new Pods from old template
      → Case A: Old image still in registry → Pods start → ✅ success
      → Case B: Old image deleted from registry → ImagePullBackOff → rollback stuck
      → Case C: Old config needs ConfigMap that was deleted → CreateContainerConfigError
      → Case D: Nodes don't have capacity → Pending
    
  If rollback fails → must FIX FORWARD:
    kubectl set image deployment/<name> web=<definitely-good-image>
```
