# 1пёҸвғЈ What It Does

## Core Purpose
- **Runs a task to completion** вҖ” creates one or more Pods that execute work and then terminate successfully
- Unlike Deployments/StatefulSets: **Pods are NOT restarted after success** вҖ” they run once (or N times) and stop
- Guarantees that a **specified number of Pods successfully terminate** before marking the Job as complete
- Tracks **completions** вҖ” how many Pods must finish successfully
- Tracks **parallelism** вҖ” how many Pods can run simultaneously
- Provides **automatic retry on failure** вҖ” recreates Pods if they fail (up to `backoffLimit`)
- Records **completion status** вҖ” Job tracks succeeded/failed counts

## Real-World Use Cases
- **Database migrations** вҖ” run schema update once, exit on success
- **Batch processing** вҖ” process 100 items, run 10 Pods in parallel, each handles 10 items
- **Report generation** вҖ” generate daily report, upload, exit
- **Data import/export** вҖ” ETL pipeline step that must complete
- **One-time setup tasks** вҖ” initialize a database, seed data, create indexes
- **Backup operations** вҖ” snapshot a database, upload to S3, exit
- **CI/CD pipeline tasks** вҖ” run tests, build artifacts

### Job Types at a Glance
| Type | completions | parallelism | Behavior |
|------|------------|-------------|----------|
| **Single Pod** (default) | 1 (default) | 1 (default) | Run 1 Pod, wait for success |
| **Fixed completions** | N | 1-M | Run N total successful Pods, M at a time |
| **Work queue** | unset | N | N Pods run; Job done when ANY Pod succeeds |

# 2пёҸвғЈ What Depends On It

## Which Components Rely On It
- **Job Controller** (inside `kube-controller-manager`) вҖ” watches Jobs, creates/monitors Pods
- **CronJob** вҖ” creates Job objects on a schedule (CronJob вҶ’ Job вҶ’ Pod)
- **CI/CD pipelines** вҖ” external tools may create Jobs and poll for completion

## What Resources It Interacts With
- **Pods** вҖ” Job creates Pods to do the work
- **ConfigMaps / Secrets** вҖ” provide configuration to Job Pods
- **PVCs** вҖ” if Job needs to read/write persistent data
- **ServiceAccount** вҖ” Pod identity for RBAC (e.g., API access)
- **ResourceQuota** вҖ” constrains how many Jobs / Pods can be created
- **LimitRange** вҖ” applies default resource requests/limits to Job Pods

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User вҶ’ kubectl apply вҶ’ Job object вҶ’ API Server вҶ’ etcd
  CronJob вҶ’ creates Job object automatically on schedule

DOWNSTREAM:
  Job Controller creates вҶ’ Pods (based on completions/parallelism)
  Scheduler places вҶ’ Pods on Nodes
  Pod runs вҶ’ exits 0 (success) or non-zero (failure)
  Job Controller tracks вҶ’ completions vs failures
    If failure вҶ’ retry (up to backoffLimit)
    If success вҶ’ mark Job Complete

LIFECYCLE:
  Job created вҶ’ Pods scheduled вҶ’ Pods run вҶ’ Pods complete
    вҶ’ Job status: Complete (all completions met)
    вҶ’ Job status: Failed (backoffLimit exceeded)
  Completed Pods are NOT deleted (kept for log inspection)
  Job itself is NOT deleted (unless TTL or manual cleanup)
```

# 3пёҸвғЈ Recognize Structure (YAML Skeleton)

## Minimal Valid Job YAML
```yaml
apiVersion: batch/v1                       # вҶҗ batch/v1 (NOT apps/v1!)
kind: Job
metadata:
  name: my-job
spec:
  completions: 1                           # вҶҗ How many Pods must succeed (default: 1)
  parallelism: 1                           # вҶҗ How many Pods run simultaneously (default: 1)
  backoffLimit: 6                          # вҶҗ Max retries on failure (default: 6)
  activeDeadlineSeconds: 100               # вҶҗ Max total runtime; Job fails after this
  template:                                # вҶҗ Pod template (same as Deployment)
    spec:
      containers:
      - name: worker
        image: busybox:1.36
        command: ["echo", "Hello from Job"]
      restartPolicy: Never                 # вҶҗ REQUIRED: "Never" or "OnFailure"
                                           # вҶҗ CANNOT be "Always" (that's for Deployments)
```

## Imperative Shortcut (Exam Speed)
```bash
# Create Job quickly
kubectl create job my-job --image=busybox:1.36 -- echo "Hello"

# Generate YAML without creating
kubectl create job my-job --image=busybox:1.36 \
  --dry-run=client -o yaml -- echo "Hello" > job.yaml

# Create Job from CronJob (run CronJob immediately)
kubectl create job my-job --from=cronjob/my-cronjob
```

## вҡ пёҸ Common Indentation Traps

### Trap 1: Wrong apiVersion
```yaml
# вқҢ WRONG
apiVersion: apps/v1          # Jobs are NOT in apps group
kind: Job

# вқҢ WRONG  
apiVersion: v1               # Jobs are NOT in core v1
kind: Job

# вң… CORRECT
apiVersion: batch/v1         # Jobs are in batch group
kind: Job
```

### Trap 2: restartPolicy: Always (Invalid for Jobs)
```yaml
# вқҢ WRONG вҖ” Jobs CANNOT use "Always"
spec:
  template:
    spec:
      restartPolicy: Always    # вҶҗ REJECTED вҖ” validation error

# вң… CORRECT вҖ” Must be "Never" or "OnFailure"
spec:
  template:
    spec:
      restartPolicy: Never     # Pod fails вҶ’ Job creates NEW Pod
      # OR
      restartPolicy: OnFailure # Pod fails вҶ’ SAME Pod restarted (container level)
```

### Trap 3: Missing restartPolicy
```yaml
# вқҢ WRONG вҖ” restartPolicy defaults to "Always" if omitted
spec:
  template:
    spec:
      containers:
      - name: worker
        image: busybox
      # restartPolicy not set вҶ’ defaults to "Always" вҶ’ VALIDATION ERROR

# вң… CORRECT
spec:
  template:
    spec:
      containers:
      - name: worker
        image: busybox
      restartPolicy: Never     # вҶҗ MUST specify explicitly
```

### Trap 4: template.spec at wrong level
```yaml
# вқҢ WRONG вҖ” spec is at Job level, not under template
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: worker
  spec:                        # вҶҗ This is Job.spec, NOT template.spec!
    containers:
    - name: worker

# вң… CORRECT
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: worker
    spec:                      # вҶҗ Under template
      containers:
      - name: worker
      restartPolicy: Never
```

### Trap 5: selector confusion
```yaml
# вҡ пёҸ Jobs auto-generate selector вҖ” do NOT set manually unless you know what you're doing
# Unlike Deployment/ReplicaSet, Jobs add a unique label automatically
# If you set selector manually, you MUST set manualSelector: true

# вң… BEST PRACTICE: don't set selector at all for Jobs
spec:
  template:
    spec:
      containers: ...
      restartPolicy: Never
# Job auto-adds: controller-uid label to match Pods
```

# 4пёҸвғЈ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: batch/v1` | Required API group | Job not recognized | рҹ”ҙ |
| `spec.template.spec.restartPolicy` | Must be `Never` or `OnFailure` | `Always` вҶ’ validation error | рҹ”ҙ |
| `spec.template.spec.containers[].image` | Container image | ImagePullBackOff вҶ’ Job stuck | рҹ”ҙ |
| `spec.template.spec.containers[].command` | What the Job runs | Wrong command вҶ’ exit non-zero вҶ’ retries until backoffLimit | рҹ”ҙ |
| `spec.completions` | Total Pods that must succeed | Default: 1; set wrong вҶ’ too many/few executions | рҹ”ҙ |
| `spec.parallelism` | How many Pods run at once | Default: 1; set too high вҶ’ resource pressure | рҹҹЎ |
| `spec.backoffLimit` | Max retries before Job fails | Default: 6; set to 0 вҶ’ no retries; too high вҶ’ spams failing Pods | рҹ”ҙ |
| `spec.activeDeadlineSeconds` | Total Job wall-clock timeout | Once exceeded вҶ’ Job terminated + marked Failed | рҹҹЎ |
| `spec.ttlSecondsAfterFinished` | Auto-cleanup completed Jobs | Not set вҶ’ Jobs accumulate forever | рҹҹЎ |
| `spec.suspend` | Pause/resume a Job | Suspended Job = no Pods running | рҹҹЎ |

## Key Field Relationships
```
spec.completions + spec.parallelism вҶ’ control execution pattern:
  completions: 5, parallelism: 2
    вҶ’ Run 2 Pods at a time until 5 total succeed
    вҶ’ Sequence: [P1,P2] вҶ’ [P3,P4] вҶ’ [P5]
    вҶ’ Each Pod runs independently

spec.backoffLimit вҶ’ controls failure tolerance:
  backoffLimit: 6 (default)
    вҶ’ Job tolerates up to 6 Pod failures before marking Job as Failed
    вҶ’ Each retry has exponential backoff (10s, 20s, 40s... up to 6min)

spec.activeDeadlineSeconds вҶ’ overrides everything:
  activeDeadlineSeconds: 100
    вҶ’ After 100 seconds of TOTAL Job runtime, Job is killed
    вҶ’ Even if Pods are still running вҶ’ terminated
    вҶ’ Takes priority over backoffLimit

restartPolicy interaction:
  Never  вҶ’ Failed Pod stays Failed; Job creates a NEW Pod for retry
  OnFailure вҶ’ Failed CONTAINER restarted in SAME Pod; Pod stays
  
  "Never" = more Pods created (can inspect failed Pods)
  "OnFailure" = fewer Pods but can't inspect failed container state
```

## restartPolicy: Never vs OnFailure
```
restartPolicy: Never
  в”ңв”Җв”Җ Container fails вҶ’ Pod status = Failed
  в”ңв”Җв”Җ Job Controller creates NEW Pod for retry
  в”ңв”Җв”Җ Failed Pods remain вҶ’ can inspect with kubectl logs <failed-pod>
  в”ңв”Җв”Җ DOWNSIDE: accumulates failed Pods
  в””в”Җв”Җ USE WHEN: you need to inspect failed Pod logs

restartPolicy: OnFailure
  в”ңв”Җв”Җ Container fails вҶ’ kubelet restarts container IN SAME Pod
  в”ңв”Җв”Җ Pod stays; container restarts (RESTARTS counter goes up)
  в”ңв”Җв”Җ After backoffLimit container failures вҶ’ Pod is Failed вҶ’ Job is Failed
  в”ңв”Җв”Җ DOWNSIDE: failed container's state is overwritten on restart
  в””в”Җв”Җ USE WHEN: you don't need to preserve failed Pod for inspection
```

# 5пёҸвғЈ How Much Should You Notice in YAML?

## вҡЎ Scan Immediately (5 seconds)
- `apiVersion: batch/v1`?
- `restartPolicy: Never` or `OnFailure` (NOT `Always`)?
- Container `image` and `command` correct?
- `backoffLimit` reasonable?
- `completions` and `parallelism` make sense for the task?
- `activeDeadlineSeconds` вҖ” is there a timeout?

## рҹҹў Usually Irrelevant for CKA
- `metadata.annotations`
- `spec.selector` (auto-generated)
- `spec.manualSelector`
- `spec.completionMode` (Indexed mode вҖ” advanced feature)
- `spec.podFailurePolicy` (v1.26+ feature)
- `spec.suspend`

## рҹ”ҙ Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| `apiVersion: apps/v1` or `apiVersion: v1` | Wrong API group for Job |
| `restartPolicy: Always` | Validation error вҖ” invalid for Jobs |
| Missing `restartPolicy` | Defaults to `Always` вҶ’ validation error |
| `backoffLimit: 0` | No retries вҖ” first failure = Job Failed |
| `command` has shell syntax without `sh -c` | Command won't execute properly |
| `activeDeadlineSeconds: 10` | Very short вҖ” Job may timeout before completing |
| `completions: 0` | No work to do вҖ” Job immediately Complete |
| Image tag `latest` or typo | ImagePullBackOff вҶ’ Job stuck |
| Missing `command` with non-service image | Pod exits immediately вҶ’ CrashLoopBackOff |

| Signal | Meaning |

| `Active: 1, Succeeded: 0, Failed: 5` | Job still trying; 5 failures so far |
| No events | Controller not processing вҖ” check controller-manager |
| `Warning BackoffLimitExceeded` | Job has given up |

# 7пёҸвғЈ Failure Patterns (Very Important)

## Pattern 1: BackoffLimitExceeded (Most Common)
```
SYMPTOM:        Job status: Failed
                Conditions: BackoffLimitExceeded
CAUSE:          Pod keeps failing (wrong command, missing dependency, bad config)
                Retried backoffLimit times вҶ’ Job gives up
PODS:           Multiple Failed Pods (restartPolicy: Never)
                OR: 1 Pod with high RESTARTS (restartPolicy: OnFailure)
FIX:            Inspect failed Pod logs: kubectl logs <failed-pod>
                Fix command/image/config вҶ’ delete Job вҶ’ recreate
EXAM SCENARIO:  "Job keeps failing. Investigate and fix."
                вҶ’ kubectl describe job вҶ’ BackoffLimitExceeded
                вҶ’ kubectl logs <failed-pod> вҶ’ find error
                вҶ’ fix YAML вҶ’ recreate Job
```

## Pattern 2: ImagePullBackOff
```
SYMPTOM:        Job Pod stuck in ImagePullBackOff
                Job is Active but not making progress
CAUSE:          Wrong image name/tag; private registry without pull secret
DESCRIBE POD:   Events: "Failed to pull image..."
FIX:            Delete Job вҶ’ fix image вҶ’ recreate
                OR: kubectl edit job won't help (immutable spec.template)
NOTE:           Job spec.template is IMMUTABLE after creation!
                Must delete and recreate the Job
```

## Pattern 3: ActiveDeadlineExceeded
```
SYMPTOM:        Job was running fine but suddenly marked Failed
                Conditions: DeadlineExceeded
CAUSE:          Total wall-clock time exceeded activeDeadlineSeconds
BEHAVIOR:       ALL running Pods terminated immediately
                Job marked as Failed вҖ” no more retries
FIX:            Increase activeDeadlineSeconds
                OR: optimize the job to run faster
                Delete Job вҶ’ recreate with higher deadline
```

## Pattern 4: Pod Pending (Scheduling Failure)
```
SYMPTOM:        Job shows Active: 1, but Pod is Pending
                Job clock is ticking (activeDeadlineSeconds countdown)
CAUSE:          Resource requests too high, no nodes available, taints
DESCRIBE POD:   Events: FailedScheduling
FIX:            Reduce resource requests
                OR: add nodes, fix taints
                If activeDeadlineSeconds expires while Pending вҶ’ Job Failed
```

## Pattern 5: restartPolicy: Always (Validation Error)
```
SYMPTOM:        Job creation fails immediately
ERROR:          "spec.template.spec.restartPolicy: Unsupported value: 'Always'"
BEHAVIOR:       Job NOT created вҖ” validation error
FIX:            Change restartPolicy to "Never" or "OnFailure"
EXAM TRAP:      This is a very common CKA trick question!
```

## Pattern 6: Command Fails Immediately (Exit Code 1)
```
SYMPTOM:        Job creates Pods, Pods complete with Error status
                Multiple failed Pods accumulating (restartPolicy: Never)
CAUSE:          Command has bug, wrong syntax, missing args
PODS:           Status: Error; Exit Code: 1 (or other non-zero)
FIX:            kubectl logs <pod> вҶ’ identify command error
                Fix command вҶ’ delete Job вҶ’ recreate
```

## Pattern 7: Job Completes but Wrong Number of Times
```
SYMPTOM:        Job ran but did too many or too few iterations
CAUSE:          completions set incorrectly
                OR: parallelism too high/low
BEHAVIOR:       Job is "Complete" but business logic is wrong
FIX:            Adjust completions and parallelism вҶ’ delete Job вҶ’ recreate
```

## Pattern 8: Completed Jobs Accumulating
```
SYMPTOM:        kubectl get jobs shows hundreds of old Completed Jobs
CAUSE:          No ttlSecondsAfterFinished set вҶ’ Jobs never cleaned up
BEHAVIOR:       Completed Pods and Jobs consuming etcd storage
FIX:            Set ttlSecondsAfterFinished: 100 (auto-delete after 100s)
                OR: manually delete old Jobs:
                kubectl delete job --field-selector status.successful>0
```

# 8пёҸвғЈ Misconfiguration Behavior

## What Happens If Labels Mismatch?
- **Job auto-generates selector** using `controller-uid` вҖ” you rarely set labels manually
- If you set `manualSelector: true` with wrong labels вҶ’ Job can't find its own Pods
- **Behavior:** Job thinks 0 Pods exist вҶ’ keeps creating new Pods вҶ’ resource leak
- **Fix:** Don't set manual selector unless required; let Job manage it

## What Happens If Image Is Wrong?
- **Behavior:** Job creates Pod вҶ’ Pod in `ImagePullBackOff`
- Pod counts as `Active` (not Failed yet) вҖ” it's still trying
- Eventually Pod may be counted as Failed вҶ’ retry вҶ’ same error вҶ’ backoffLimit hit
- **Behavior:** Job marked as Failed after backoffLimit + waiting
- **Fix:** Delete Job вҶ’ fix image вҶ’ recreate (can't edit spec.template)

## What Happens If Command Is Wrong?
- **Behavior:** Pod starts вҶ’ command fails вҶ’ exit code non-zero вҶ’ Pod status: Error
- **restartPolicy: Never** вҶ’ Job creates new Pod вҶ’ same failure вҶ’ repeat until backoffLimit
- **restartPolicy: OnFailure** вҶ’ kubelet restarts container вҶ’ same failure вҶ’ repeat
- **Result:** backoffLimit exceeded вҶ’ Job Failed
- **Fix:** Check logs of failed Pod вҶ’ fix command вҶ’ delete Job вҶ’ recreate

## What Happens If Resource Limits Are Wrong?
- **Requests too high** вҶ’ Pod `Pending` вҶ’ Job Active but not progressing
- **Memory limit too low** вҶ’ Pod `OOMKilled` вҶ’ retry вҶ’ OOMKilled again вҶ’ backoffLimit
- **CPU limit too low** вҶ’ Job runs extremely slowly вҶ’ may hit activeDeadlineSeconds
- **Fix:** Fix resources вҶ’ delete Job вҶ’ recreate

## What Happens If Dependencies Are Missing?
- **Missing ConfigMap/Secret** вҶ’ `CreateContainerConfigError` вҶ’ Pod stuck
- **Missing PVC** вҶ’ Pod `Pending`
- **Missing external service** вҶ’ Pod starts but command fails вҶ’ retry loop
- **Fix:** Create missing dependency вҶ’ if Pod hasn't failed, it may auto-recover
- **Otherwise:** Delete Job вҶ’ recreate after fixing dependencies

## What Happens If You Edit a Job?
- **spec.template is IMMUTABLE** вҖ” you CANNOT edit Running/Failed Job's Pod template
- **You CAN edit:** `spec.activeDeadlineSeconds`, `spec.suspend`, `spec.parallelism`
- **For any template change:** Delete Job вҶ’ recreate with fixed YAML
- **CKA Trap:** "Fix the Job" вҶ’ you must delete and recreate, not just edit

# 9пёҸвғЈ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# вҸұпёҸ 0-10 sec: Job overview
```

# рҹ”ҹ Fast Fix Strategy

## Important: Job Limitations
```
вҡ пёҸ Job spec.template is IMMUTABLE after creation!
   You CANNOT kubectl edit the Pod template of an existing Job.
   
   Workflow for fixing Jobs:
   1. kubectl get job <name> -o yaml > job-fix.yaml
   2. Edit job-fix.yaml вҶ’ fix the issue
   3. kubectl delete job <name>
   4. Remove status, resourceVersion, uid, creationTimestamp from YAML
   5. kubectl apply -f job-fix.yaml
```

## Common Fixes

### Fix: Wrong Command
```bash
# Export вҶ’ Edit вҶ’ Delete вҶ’ Recreate
kubectl get job my-job -o yaml > job-fix.yaml
# Edit job-fix.yaml: fix command/args
# Clean up metadata (remove status, resourceVersion, etc.)
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Wrong Image
```bash
kubectl get job my-job -o yaml > job-fix.yaml
# Edit: fix image name/tag
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Increase backoffLimit
```bash
# backoffLimit CAN'T be changed on existing Job
kubectl get job my-job -o yaml > job-fix.yaml
# Edit: increase backoffLimit
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Increase activeDeadlineSeconds
```bash
# activeDeadlineSeconds CAN be patched on running Job
kubectl patch job my-job -p '{"spec":{"activeDeadlineSeconds":300}}'
```

### Fix: Suspend/Resume Job
```bash
# Pause a running Job
kubectl patch job my-job -p '{"spec":{"suspend":true}}'
# Running Pods are terminated; Job paused

# Resume
kubectl patch job my-job -p '{"spec":{"suspend":false}}'
# Job resumes creating Pods
```

### Fix: Scale Parallelism
```bash
# parallelism CAN be patched on running Job
kubectl patch job my-job -p '{"spec":{"parallelism":3}}'
```

### Fix: Clean Up Old Jobs
```bash
# Delete completed Jobs
kubectl delete jobs --field-selector status.successful=1

# Delete all failed Jobs
kubectl delete jobs --field-selector status.successful=0

# Or use ttlSecondsAfterFinished in YAML for auto-cleanup
```

## 5 Quiz Questions to Test Understanding

### Q1: What are the only valid values for `restartPolicy` in a Job's Pod template?
<details>
<summary>Answer</summary>
"Never" or "OnFailure". "Always" is NOT valid for Jobs and causes a validation error.
</details>

### Q2: A Job has `completions: 5`, `parallelism: 2`, `backoffLimit: 6`. How does it execute?
<details>
<summary>Answer</summary>
It runs 2 Pods simultaneously. When a Pod succeeds, another is launched until 5 total Pods have succeeded. If any Pod fails, it's retried. If total failures across all Pods reach 6, the Job is marked as Failed.
</details>

### Q3: You need to fix a Job's container image. Can you use `kubectl edit job`?
<details>
<summary>Answer</summary>
No вҖ” spec.template is immutable. You must: export the Job YAML, edit the file, delete the old Job, and apply the fixed YAML.
</details>

### Q4: What is the difference between `restartPolicy: Never` and `restartPolicy: OnFailure` in a Job?
<details>
<summary>Answer</summary>
- Never: Failed Pod stays in Error state; Job creates a NEW Pod for retry. You can inspect failed Pods' logs. More Pods accumulate.
- OnFailure: kubelet restarts the container in the SAME Pod. Failed container logs are overwritten. Fewer Pods created.
</details>

### Q5: A Job has `backoffLimit: 6` and `activeDeadlineSeconds: 60`. The job takes 90 seconds per attempt. What happens?
<details>
<summary>Answer</summary>
The Job will be terminated after 60 seconds (activeDeadlineSeconds takes priority). Even on the first attempt, the deadline kills the Pod at 60s. Condition: Failed, Reason: DeadlineExceeded. The backoffLimit is never reached.
</details>

# 1пёҸвғЈ3пёҸвғЈ Component Interaction Map

## How Job Interacts with Components

### Job вҶ” Pods
```
NORMAL:    Job creates Pods to do work
           Pod names: <job-name>-<random-suffix>
           Completed Pods: STATUS = Completed, READY = 0/1
           Failed Pods: STATUS = Error (restartPolicy: Never)
           OR: container RESTARTS counter increments (restartPolicy: OnFailure)

FAILURE:   Pod CrashLoopBackOff вҶ’ Job retries (up to backoffLimit)
           Pod Pending вҶ’ Job Active but not progressing
           Pod OOMKilled вҶ’ Job retries
           All retries exhausted вҶ’ Job Failed (BackoffLimitExceeded)
```

### Job вҶ” Services
```
NORMAL:    Jobs typically DON'T use Services (one-time work, not server)
           Exception: Job Pod may CONNECT to a Service (e.g., database)

FAILURE:   Dependency Service down вҶ’ Job command fails вҶ’ retry loop
           DNS resolution failure вҶ’ command fails
           Service is output target (e.g., S3 endpoint) вҶ’ upload fails вҶ’ retry
```

### Job вҶ” Nodes
```
NORMAL:    Scheduler places Job Pods on suitable nodes
           Considers: resources, taints, tolerations, affinity

FAILURE:   No suitable node вҶ’ Pod Pending вҶ’ Job stuck Active
           Node failure during Job вҶ’ Pod rescheduled (if possible)
           Node resource exhaustion вҶ’ OOMKilled вҶ’ retry
```

### Job вҶ” Scheduler
```
NORMAL:    Scheduler places Job Pods like any other Pod
           Job Pods are regular Pods with all scheduling features

FAILURE:   FailedScheduling вҶ’ Pod Pending вҶ’ Job appears stuck
           If activeDeadlineSeconds is set вҶ’ Job may timeout while waiting
           Resource contention вҶ’ Pod Pending вҶ’ delays completions
```

### Job вҶ” Controller Manager
```
NORMAL:    Job Controller (in kube-controller-manager):
             Watches Job objects
             Creates Pods (respecting parallelism)
             Tracks Pod success/failure counters
             Manages backoff timing between retries
             Marks Job Complete or Failed based on conditions

FAILURE:   Controller Manager down вҶ’ no new Pods created for Jobs
           Existing running Pods continue (kubelet manages them)
           Recovery: when CM comes back, reconciles Job state
```

### Job вҶ” etcd
```
NORMAL:    Job spec and status stored in etcd
           Pod creation/completion events tracked
           Completion count persisted

FAILURE:   etcd down вҶ’ API server read-only вҶ’ no new Pods
           Running Job Pods continue (kubelet)
           etcd recovery вҶ’ Job reconcilation resumes
```

### Job вҶ” Network Layer
```
NORMAL:    Job Pods get IPs from CNI (like any Pod)
           Job Pods may access Services, external APIs, etc.

FAILURE:   CNI failure вҶ’ Pod ContainerCreating (no IP)
           DNS failure вҶ’ Job command can't resolve hostnames
           Network policy blocking вҶ’ Job command can't reach dependencies
```

### Job вҶ” Storage Layer
```
NORMAL:    Job Pods can mount PVCs for reading input / writing output
           PVCs persist after Job completes (Pods are garbage but PVC stays)

FAILURE:   PVC Pending вҶ’ Pod Pending вҶ’ Job stuck
           PV full вҶ’ Job command write fails вҶ’ exit non-zero вҶ’ retry
           RWO PV occupied вҶ’ second parallel Pod can't mount вҶ’ Pending
           Tip: For parallel Jobs writing to same storage вҶ’ use ReadWriteMany
```

```bash
kubectl delete job <name> --cascade=orphan

# Clean up completed Jobs
kubectl delete jobs --field-selector status.successful=1
```

### Mental Model: Job = "Run and Done"
```
Deployment: Run FOREVER вҶ’ restart if container stops    (web servers)
Job:        Run ONCE    вҶ’ stop when container succeeds   (batch work)
CronJob:    Run ONCE    вҶ’ on a SCHEDULE                 (cron + batch)

Key rules to remember:
  вң… apiVersion: batch/v1 (NOT apps/v1)
  вң… restartPolicy: Never or OnFailure (NOT Always)
  вң… spec.template is IMMUTABLE вҶ’ must delete + recreate
  вң… backoffLimit controls retry count (default: 6)
  вң… activeDeadlineSeconds is a total timeout safety net
  вң… Completed Pods are kept (for log inspection)
  вң… Jobs are kept after completion (unless ttlSecondsAfterFinished)
  вқҢ Cannot kubectl edit the Pod template
  вқҢ No rolling updates (not applicable to Jobs)
  вқҢ No rollout undo (not applicable to Jobs)
```
