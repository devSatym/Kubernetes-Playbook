# 🧠 Jobs — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Runs a task to completion** — creates one or more Pods that execute work and then terminate successfully
- Unlike Deployments/StatefulSets: **Pods are NOT restarted after success** — they run once (or N times) and stop
- Guarantees that a **specified number of Pods successfully terminate** before marking the Job as complete
- Tracks **completions** — how many Pods must finish successfully
- Tracks **parallelism** — how many Pods can run simultaneously
- Provides **automatic retry on failure** — recreates Pods if they fail (up to `backoffLimit`)
- Records **completion status** — Job tracks succeeded/failed counts

## Real-World Use Cases
- **Database migrations** — run schema update once, exit on success
- **Batch processing** — process 100 items, run 10 Pods in parallel, each handles 10 items
- **Report generation** — generate daily report, upload, exit
- **Data import/export** — ETL pipeline step that must complete
- **One-time setup tasks** — initialize a database, seed data, create indexes
- **Backup operations** — snapshot a database, upload to S3, exit
- **CI/CD pipeline tasks** — run tests, build artifacts

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────────┐
│                      JOB HIERARCHY                               │
│                                                                   │
│  User/CI ──→ kubectl apply ──→ Job (stored in etcd)              │
│                                   │                               │
│                         Job Controller                            │
│                    (in kube-controller-manager)                   │
│                                   │                               │
│              ┌────────────────────┼────────────────┐             │
│              ▼                    ▼                ▼             │
│          Pod-xxxxx            Pod-yyyyy        Pod-zzzzz         │
│         (completed ✅)      (completed ✅)    (running ⏳)       │
│                                                                   │
│  Key difference from Deployment:                                  │
│    Deployment: Pods run FOREVER (restarted if they stop)         │
│    Job:        Pods run ONCE to completion (NOT restarted)       │
│                                                                   │
│  Job                                                              │
│    ├── completions: 3     (need 3 successful Pods total)         │
│    ├── parallelism: 2     (run 2 at a time)                      │
│    └── backoffLimit: 6    (retry up to 6 times on failure)       │
└──────────────────────────────────────────────────────────────────┘
```

### Job Types at a Glance
| Type | completions | parallelism | Behavior |
|------|------------|-------------|----------|
| **Single Pod** (default) | 1 (default) | 1 (default) | Run 1 Pod, wait for success |
| **Fixed completions** | N | 1-M | Run N total successful Pods, M at a time |
| **Work queue** | unset | N | N Pods run; Job done when ANY Pod succeeds |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Job Controller** (inside `kube-controller-manager`) — watches Jobs, creates/monitors Pods
- **CronJob** — creates Job objects on a schedule (CronJob → Job → Pod)
- **CI/CD pipelines** — external tools may create Jobs and poll for completion

## What Resources It Interacts With
- **Pods** — Job creates Pods to do the work
- **ConfigMaps / Secrets** — provide configuration to Job Pods
- **PVCs** — if Job needs to read/write persistent data
- **ServiceAccount** — Pod identity for RBAC (e.g., API access)
- **ResourceQuota** — constrains how many Jobs / Pods can be created
- **LimitRange** — applies default resource requests/limits to Job Pods

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User → kubectl apply → Job object → API Server → etcd
  CronJob → creates Job object automatically on schedule

DOWNSTREAM:
  Job Controller creates → Pods (based on completions/parallelism)
  Scheduler places → Pods on Nodes
  Pod runs → exits 0 (success) or non-zero (failure)
  Job Controller tracks → completions vs failures
    If failure → retry (up to backoffLimit)
    If success → mark Job Complete

LIFECYCLE:
  Job created → Pods scheduled → Pods run → Pods complete
    → Job status: Complete (all completions met)
    → Job status: Failed (backoffLimit exceeded)
  Completed Pods are NOT deleted (kept for log inspection)
  Job itself is NOT deleted (unless TTL or manual cleanup)
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid Job YAML
```yaml
apiVersion: batch/v1                       # ← batch/v1 (NOT apps/v1!)
kind: Job
metadata:
  name: my-job
spec:
  completions: 1                           # ← How many Pods must succeed (default: 1)
  parallelism: 1                           # ← How many Pods run simultaneously (default: 1)
  backoffLimit: 6                          # ← Max retries on failure (default: 6)
  activeDeadlineSeconds: 100               # ← Max total runtime; Job fails after this
  template:                                # ← Pod template (same as Deployment)
    spec:
      containers:
      - name: worker
        image: busybox:1.36
        command: ["echo", "Hello from Job"]
      restartPolicy: Never                 # ← REQUIRED: "Never" or "OnFailure"
                                           # ← CANNOT be "Always" (that's for Deployments)
```

## Imperative Shortcut (Exam Speed)
```bash
# Create Job quickly
kubectl create job my-job --image=busybox:1.36 -- echo "Hello"

# Generate YAML without creating
kubectl create job my-job --image=busybox:1.36 \
  --dry-run=client -o yaml -- echo "Hello" > job.yaml

# Create Job from CronJob (run CronJob immediately)
kubectl create job my-job --from=cronjob/my-cronjob
```

## ⚠️ Common Indentation Traps

### Trap 1: Wrong apiVersion
```yaml
# ❌ WRONG
apiVersion: apps/v1          # Jobs are NOT in apps group
kind: Job

# ❌ WRONG  
apiVersion: v1               # Jobs are NOT in core v1
kind: Job

# ✅ CORRECT
apiVersion: batch/v1         # Jobs are in batch group
kind: Job
```

### Trap 2: restartPolicy: Always (Invalid for Jobs)
```yaml
# ❌ WRONG — Jobs CANNOT use "Always"
spec:
  template:
    spec:
      restartPolicy: Always    # ← REJECTED — validation error

# ✅ CORRECT — Must be "Never" or "OnFailure"
spec:
  template:
    spec:
      restartPolicy: Never     # Pod fails → Job creates NEW Pod
      # OR
      restartPolicy: OnFailure # Pod fails → SAME Pod restarted (container level)
```

### Trap 3: Missing restartPolicy
```yaml
# ❌ WRONG — restartPolicy defaults to "Always" if omitted
spec:
  template:
    spec:
      containers:
      - name: worker
        image: busybox
      # restartPolicy not set → defaults to "Always" → VALIDATION ERROR

# ✅ CORRECT
spec:
  template:
    spec:
      containers:
      - name: worker
        image: busybox
      restartPolicy: Never     # ← MUST specify explicitly
```

### Trap 4: template.spec at wrong level
```yaml
# ❌ WRONG — spec is at Job level, not under template
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: worker
  spec:                        # ← This is Job.spec, NOT template.spec!
    containers:
    - name: worker

# ✅ CORRECT
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: worker
    spec:                      # ← Under template
      containers:
      - name: worker
      restartPolicy: Never
```

### Trap 5: selector confusion
```yaml
# ⚠️ Jobs auto-generate selector — do NOT set manually unless you know what you're doing
# Unlike Deployment/ReplicaSet, Jobs add a unique label automatically
# If you set selector manually, you MUST set manualSelector: true

# ✅ BEST PRACTICE: don't set selector at all for Jobs
spec:
  template:
    spec:
      containers: ...
      restartPolicy: Never
# Job auto-adds: controller-uid label to match Pods
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: batch/v1` | Required API group | Job not recognized | 🔴 |
| `spec.template.spec.restartPolicy` | Must be `Never` or `OnFailure` | `Always` → validation error | 🔴 |
| `spec.template.spec.containers[].image` | Container image | ImagePullBackOff → Job stuck | 🔴 |
| `spec.template.spec.containers[].command` | What the Job runs | Wrong command → exit non-zero → retries until backoffLimit | 🔴 |
| `spec.completions` | Total Pods that must succeed | Default: 1; set wrong → too many/few executions | 🔴 |
| `spec.parallelism` | How many Pods run at once | Default: 1; set too high → resource pressure | 🟡 |
| `spec.backoffLimit` | Max retries before Job fails | Default: 6; set to 0 → no retries; too high → spams failing Pods | 🔴 |
| `spec.activeDeadlineSeconds` | Total Job wall-clock timeout | Once exceeded → Job terminated + marked Failed | 🟡 |
| `spec.ttlSecondsAfterFinished` | Auto-cleanup completed Jobs | Not set → Jobs accumulate forever | 🟡 |
| `spec.suspend` | Pause/resume a Job | Suspended Job = no Pods running | 🟡 |

## Key Field Relationships
```
spec.completions + spec.parallelism → control execution pattern:
  completions: 5, parallelism: 2
    → Run 2 Pods at a time until 5 total succeed
    → Sequence: [P1,P2] → [P3,P4] → [P5]
    → Each Pod runs independently

spec.backoffLimit → controls failure tolerance:
  backoffLimit: 6 (default)
    → Job tolerates up to 6 Pod failures before marking Job as Failed
    → Each retry has exponential backoff (10s, 20s, 40s... up to 6min)

spec.activeDeadlineSeconds → overrides everything:
  activeDeadlineSeconds: 100
    → After 100 seconds of TOTAL Job runtime, Job is killed
    → Even if Pods are still running → terminated
    → Takes priority over backoffLimit

restartPolicy interaction:
  Never  → Failed Pod stays Failed; Job creates a NEW Pod for retry
  OnFailure → Failed CONTAINER restarted in SAME Pod; Pod stays
  
  "Never" = more Pods created (can inspect failed Pods)
  "OnFailure" = fewer Pods but can't inspect failed container state
```

## restartPolicy: Never vs OnFailure
```
restartPolicy: Never
  ├── Container fails → Pod status = Failed
  ├── Job Controller creates NEW Pod for retry
  ├── Failed Pods remain → can inspect with kubectl logs <failed-pod>
  ├── DOWNSIDE: accumulates failed Pods
  └── USE WHEN: you need to inspect failed Pod logs

restartPolicy: OnFailure
  ├── Container fails → kubelet restarts container IN SAME Pod
  ├── Pod stays; container restarts (RESTARTS counter goes up)
  ├── After backoffLimit container failures → Pod is Failed → Job is Failed
  ├── DOWNSIDE: failed container's state is overwritten on restart
  └── USE WHEN: you don't need to preserve failed Pod for inspection
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `apiVersion: batch/v1`?
- `restartPolicy: Never` or `OnFailure` (NOT `Always`)?
- Container `image` and `command` correct?
- `backoffLimit` reasonable?
- `completions` and `parallelism` make sense for the task?
- `activeDeadlineSeconds` — is there a timeout?

## 🟢 Usually Irrelevant for CKA
- `metadata.annotations`
- `spec.selector` (auto-generated)
- `spec.manualSelector`
- `spec.completionMode` (Indexed mode — advanced feature)
- `spec.podFailurePolicy` (v1.26+ feature)
- `spec.suspend`

## 🔴 Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| `apiVersion: apps/v1` or `apiVersion: v1` | Wrong API group for Job |
| `restartPolicy: Always` | Validation error — invalid for Jobs |
| Missing `restartPolicy` | Defaults to `Always` → validation error |
| `backoffLimit: 0` | No retries — first failure = Job Failed |
| `command` has shell syntax without `sh -c` | Command won't execute properly |
| `activeDeadlineSeconds: 10` | Very short — Job may timeout before completing |
| `completions: 0` | No work to do — Job immediately Complete |
| Image tag `latest` or typo | ImagePullBackOff → Job stuck |
| Missing `command` with non-service image | Pod exits immediately → CrashLoopBackOff |

## 📋 `kubectl describe job <name>`

### Where to Look FIRST
```
Name:           my-job
Namespace:      default
Selector:       controller-uid=xxxx-xxxx
Parallelism:    1
Completions:    3
Start Time:     Thu, 26 Feb 2026 15:00:00 +0530
Completed At:   Thu, 26 Feb 2026 15:02:00 +0530
Duration:       2m
Pods Statuses:  0 Active / 3 Succeeded / 0 Failed     ← ✅ Healthy
Conditions:
  Type      Status  Reason
  ----      ------  ------
  Complete  True    ← Job finished successfully
Events:
  Normal  SuccessfulCreate  Created pod: my-job-xxxxx
  Normal  SuccessfulCreate  Created pod: my-job-yyyyy
  Normal  SuccessfulCreate  Created pod: my-job-zzzzz
  Normal  Completed         Job completed
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `Pods Statuses: 0 Active / 0 Succeeded / 6 Failed` | All retries exhausted |
| `Conditions: Failed True, Reason: BackoffLimitExceeded` | backoffLimit hit — Job permanently Failed |
| `Conditions: Failed True, Reason: DeadlineExceeded` | activeDeadlineSeconds expired |
| Events: `SuccessfulCreate` repeated many times | Lots of retries — Pods keep failing |
| `Active: 1, Succeeded: 0, Failed: 5` | Job still trying; 5 failures so far |
| No events | Controller not processing — check controller-manager |
| `Warning BackoffLimitExceeded` | Job has given up |

### Where to Look FIRST
```bash
# Job Pods have auto-generated names: <job-name>-<random>
kubectl get pods --selector=job-name=my-job
# NAME             READY   STATUS      RESTARTS   AGE
# my-job-abc12     0/1     Completed   0          5m    ← succeeded
# my-job-def34     0/1     Error       0          4m    ← failed (if restartPolicy: Never)

# View successful Pod logs
kubectl logs my-job-abc12
# "Hello from Job"

# View failed Pod logs (critical for debugging)
kubectl logs my-job-def34
# Error message from your app → tells you WHY it failed

# If restartPolicy: OnFailure, check previous container
kubectl logs my-job-abc12 --previous
```

### Failure Signals in Logs
| Signal | Meaning |
|--------|---------|
| No output at all | Container exited before producing output; check command |
| Permission denied | RBAC issue or filesystem permissions |
| Connection refused | Dependency not available (DB, API, etc.) |
| Exit code 1 / non-zero | Application-level error — check app logic |
| OOMKilled (not in logs but in Pod events) | Memory limit too low |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: BackoffLimitExceeded (Most Common)
```
SYMPTOM:        Job status: Failed
                Conditions: BackoffLimitExceeded
CAUSE:          Pod keeps failing (wrong command, missing dependency, bad config)
                Retried backoffLimit times → Job gives up
PODS:           Multiple Failed Pods (restartPolicy: Never)
                OR: 1 Pod with high RESTARTS (restartPolicy: OnFailure)
FIX:            Inspect failed Pod logs: kubectl logs <failed-pod>
                Fix command/image/config → delete Job → recreate
EXAM SCENARIO:  "Job keeps failing. Investigate and fix."
                → kubectl describe job → BackoffLimitExceeded
                → kubectl logs <failed-pod> → find error
                → fix YAML → recreate Job
```

## Pattern 2: ImagePullBackOff
```
SYMPTOM:        Job Pod stuck in ImagePullBackOff
                Job is Active but not making progress
CAUSE:          Wrong image name/tag; private registry without pull secret
DESCRIBE POD:   Events: "Failed to pull image..."
FIX:            Delete Job → fix image → recreate
                OR: kubectl edit job won't help (immutable spec.template)
NOTE:           Job spec.template is IMMUTABLE after creation!
                Must delete and recreate the Job
```

## Pattern 3: ActiveDeadlineExceeded
```
SYMPTOM:        Job was running fine but suddenly marked Failed
                Conditions: DeadlineExceeded
CAUSE:          Total wall-clock time exceeded activeDeadlineSeconds
BEHAVIOR:       ALL running Pods terminated immediately
                Job marked as Failed — no more retries
FIX:            Increase activeDeadlineSeconds
                OR: optimize the job to run faster
                Delete Job → recreate with higher deadline
```

## Pattern 4: Pod Pending (Scheduling Failure)
```
SYMPTOM:        Job shows Active: 1, but Pod is Pending
                Job clock is ticking (activeDeadlineSeconds countdown)
CAUSE:          Resource requests too high, no nodes available, taints
DESCRIBE POD:   Events: FailedScheduling
FIX:            Reduce resource requests
                OR: add nodes, fix taints
                If activeDeadlineSeconds expires while Pending → Job Failed
```

## Pattern 5: restartPolicy: Always (Validation Error)
```
SYMPTOM:        Job creation fails immediately
ERROR:          "spec.template.spec.restartPolicy: Unsupported value: 'Always'"
BEHAVIOR:       Job NOT created — validation error
FIX:            Change restartPolicy to "Never" or "OnFailure"
EXAM TRAP:      This is a very common CKA trick question!
```

## Pattern 6: Command Fails Immediately (Exit Code 1)
```
SYMPTOM:        Job creates Pods, Pods complete with Error status
                Multiple failed Pods accumulating (restartPolicy: Never)
CAUSE:          Command has bug, wrong syntax, missing args
PODS:           Status: Error; Exit Code: 1 (or other non-zero)
FIX:            kubectl logs <pod> → identify command error
                Fix command → delete Job → recreate
```

## Pattern 7: Job Completes but Wrong Number of Times
```
SYMPTOM:        Job ran but did too many or too few iterations
CAUSE:          completions set incorrectly
                OR: parallelism too high/low
BEHAVIOR:       Job is "Complete" but business logic is wrong
FIX:            Adjust completions and parallelism → delete Job → recreate
```

## Pattern 8: Completed Jobs Accumulating
```
SYMPTOM:        kubectl get jobs shows hundreds of old Completed Jobs
CAUSE:          No ttlSecondsAfterFinished set → Jobs never cleaned up
BEHAVIOR:       Completed Pods and Jobs consuming etcd storage
FIX:            Set ttlSecondsAfterFinished: 100 (auto-delete after 100s)
                OR: manually delete old Jobs:
                kubectl delete job --field-selector status.successful>0
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Labels Mismatch?
- **Job auto-generates selector** using `controller-uid` — you rarely set labels manually
- If you set `manualSelector: true` with wrong labels → Job can't find its own Pods
- **Behavior:** Job thinks 0 Pods exist → keeps creating new Pods → resource leak
- **Fix:** Don't set manual selector unless required; let Job manage it

## What Happens If Image Is Wrong?
- **Behavior:** Job creates Pod → Pod in `ImagePullBackOff`
- Pod counts as `Active` (not Failed yet) — it's still trying
- Eventually Pod may be counted as Failed → retry → same error → backoffLimit hit
- **Behavior:** Job marked as Failed after backoffLimit + waiting
- **Fix:** Delete Job → fix image → recreate (can't edit spec.template)

## What Happens If Command Is Wrong?
- **Behavior:** Pod starts → command fails → exit code non-zero → Pod status: Error
- **restartPolicy: Never** → Job creates new Pod → same failure → repeat until backoffLimit
- **restartPolicy: OnFailure** → kubelet restarts container → same failure → repeat
- **Result:** backoffLimit exceeded → Job Failed
- **Fix:** Check logs of failed Pod → fix command → delete Job → recreate

## What Happens If Resource Limits Are Wrong?
- **Requests too high** → Pod `Pending` → Job Active but not progressing
- **Memory limit too low** → Pod `OOMKilled` → retry → OOMKilled again → backoffLimit
- **CPU limit too low** → Job runs extremely slowly → may hit activeDeadlineSeconds
- **Fix:** Fix resources → delete Job → recreate

## What Happens If Dependencies Are Missing?
- **Missing ConfigMap/Secret** → `CreateContainerConfigError` → Pod stuck
- **Missing PVC** → Pod `Pending`
- **Missing external service** → Pod starts but command fails → retry loop
- **Fix:** Create missing dependency → if Pod hasn't failed, it may auto-recover
- **Otherwise:** Delete Job → recreate after fixing dependencies

## What Happens If You Edit a Job?
- **spec.template is IMMUTABLE** — you CANNOT edit Running/Failed Job's Pod template
- **You CAN edit:** `spec.activeDeadlineSeconds`, `spec.suspend`, `spec.parallelism`
- **For any template change:** Delete Job → recreate with fixed YAML
- **CKA Trap:** "Fix the Job" → you must delete and recreate, not just edit

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: Job overview
kubectl get jobs
# NAME     COMPLETIONS   DURATION   AGE
# my-job   3/3           2m         5m    ← Complete ✅
# my-job   0/3           5m         5m    ← Problem! 0 completions
# my-job   1/1                      5m    ← Complete (single run) ✅

# Quick check: is it Complete or Failed?
kubectl get job my-job -o jsonpath='{.status.conditions[0].type}'
# "Complete" or "Failed"

# ⏱️ 10-20 sec: Check Pods
kubectl get pods --selector=job-name=my-job
# NAME             READY   STATUS      RESTARTS   AGE
# my-job-abc12     0/1     Completed   0          5m    ← succeeded ✅
# my-job-def34     0/1     Error       0          4m    ← failed ❌
# my-job-ghi56     0/1     Pending     0          1m    ← stuck
# Count: how many Completed? How many Error? How many still trying?

# ⏱️ 20-35 sec: Inspect failed Pods
kubectl logs my-job-def34
# See application error output → root cause

kubectl describe pod my-job-def34 | tail -20
# Events section → scheduling, image, or runtime errors

# ⏱️ 35-50 sec: Check Job details
kubectl describe job my-job
# Read: Pods Statuses, Conditions, Events
# BackoffLimitExceeded? DeadlineExceeded? Still Active?

# ⏱️ 50-60 sec: Decide action
# Wrong image? → delete job → fix → recreate
# Command error? → check logs → fix command → recreate
# Pod Pending? → check resources/scheduling → fix → recreate
# ActiveDeadline hit? → increase or optimize → recreate
```

## Quick Diagnosis Table
```
JOB STATUS                            → LIKELY CAUSE
──────────────────────────────────────────────────────────────
COMPLETIONS 0/1, Active: 1           → Pod still running (or stuck Pending)
COMPLETIONS 0/1, Active: 0           → Job Failed (backoffLimit or deadline)
COMPLETIONS 1/1                      → Job Complete ✅
COMPLETIONS 0/3, many Failed Pods    → Command/image/config bug; retry loop
COMPLETIONS 2/3, 1 Active, 0 Failed  → In progress; healthy so far ✅
No Pods created at all               → Validation error or controller issue
Failed, Reason: BackoffLimitExceeded  → Pod keeps crashing; check logs
Failed, Reason: DeadlineExceeded      → Took too long; increase deadline
Pods in ImagePullBackOff              → Wrong image name/tag
Pods in Pending                       → Resource/scheduling issue
```

# 🔟 Fast Fix Strategy

## Important: Job Limitations
```
⚠️ Job spec.template is IMMUTABLE after creation!
   You CANNOT kubectl edit the Pod template of an existing Job.
   
   Workflow for fixing Jobs:
   1. kubectl get job <name> -o yaml > job-fix.yaml
   2. Edit job-fix.yaml → fix the issue
   3. kubectl delete job <name>
   4. Remove status, resourceVersion, uid, creationTimestamp from YAML
   5. kubectl apply -f job-fix.yaml
```

## Common Fixes

### Fix: Wrong Command
```bash
# Export → Edit → Delete → Recreate
kubectl get job my-job -o yaml > job-fix.yaml
# Edit job-fix.yaml: fix command/args
# Clean up metadata (remove status, resourceVersion, etc.)
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Wrong Image
```bash
kubectl get job my-job -o yaml > job-fix.yaml
# Edit: fix image name/tag
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Increase backoffLimit
```bash
# backoffLimit CAN'T be changed on existing Job
kubectl get job my-job -o yaml > job-fix.yaml
# Edit: increase backoffLimit
kubectl delete job my-job
kubectl apply -f job-fix.yaml
```

### Fix: Increase activeDeadlineSeconds
```bash
# activeDeadlineSeconds CAN be patched on running Job
kubectl patch job my-job -p '{"spec":{"activeDeadlineSeconds":300}}'
```

### Fix: Suspend/Resume Job
```bash
# Pause a running Job
kubectl patch job my-job -p '{"spec":{"suspend":true}}'
# Running Pods are terminated; Job paused

# Resume
kubectl patch job my-job -p '{"spec":{"suspend":false}}'
# Job resumes creating Pods
```

### Fix: Scale Parallelism
```bash
# parallelism CAN be patched on running Job
kubectl patch job my-job -p '{"spec":{"parallelism":3}}'
```

### Fix: Clean Up Old Jobs
```bash
# Delete completed Jobs
kubectl delete jobs --field-selector status.successful=1

# Delete all failed Jobs
kubectl delete jobs --field-selector status.successful=0

# Or use ttlSecondsAfterFinished in YAML for auto-cleanup
```

## Validate Fix
```bash
# Job completed
kubectl get job my-job
# COMPLETIONS: 1/1 (or 3/3) ✅

# Check condition
kubectl get job my-job -o jsonpath='{.status.conditions[0].type}'
```

## Step 2 — Break It Intentionally

### Break 1: Wrong Image
```bash
kubectl create job broken-img --image=busybox:NONEXISTENT -- echo "test"
kubectl get pods --selector=job-name=broken-img
# Pod in ImagePullBackOff → eventually retried → backoffLimit hit
kubectl describe job broken-img
# Conditions: Failed, BackoffLimitExceeded
# FIX: Delete job → recreate with correct image
kubectl delete job broken-img
```

### Break 2: Command That Fails (exit code 1)
```bash
kubectl create job broken-cmd --image=busybox:1.36 -- sh -c "exit 1"
kubectl get pods --selector=job-name=broken-cmd
# Multiple Pods with STATUS: Error (restartPolicy: Never by default)
kubectl describe job broken-cmd
# Multiple SuccessfulCreate events → then BackoffLimitExceeded
# FIX: Fix command → delete → recreate
kubectl delete job broken-cmd
kubectl create job broken-cmd --image=busybox:1.36 -- sh -c "exit 0"
```

### Break 3: restartPolicy: Always
```bash
cat <<'EOF' | kubectl apply -f -
apiVersion: batch/v1
kind: Job
metadata:
  name: broken-restart
spec:
  template:
    spec:
      containers:
      - name: test
        image: busybox:1.36
        command: ["echo", "test"]
      restartPolicy: Always
EOF
# ERROR: "Unsupported value: 'Always': supported values: 'OnFailure', 'Never'"
# Job NOT created
# FIX: Change to restartPolicy: Never
```

### Break 4: Excessive Resources
```bash
cat <<'EOF' | kubectl apply -f -
apiVersion: batch/v1
kind: Job
metadata:
  name: broken-resources
spec:
  activeDeadlineSeconds: 30
  template:
    spec:
      containers:
      - name: test
        image: busybox:1.36
        command: ["echo", "test"]
        resources:
          requests:
            cpu: "100"
            memory: "1000Gi"
      restartPolicy: Never
EOF
kubectl get pods --selector=job-name=broken-resources
# Pod: Pending (FailedScheduling)
# After 30s: Job Failed (DeadlineExceeded)
# FIX: Delete Job, reduce resources, recreate
kubectl delete job broken-resources
```

### Break 5: Very Low activeDeadlineSeconds
```bash
cat <<'EOF' | kubectl apply -f -
```

## Step 3 — Debug Without Looking at Solution

### What Command to Run FIRST
```bash
kubectl get job <name>
# Read COMPLETIONS column — is it done?
# Is the JOB still Active, Complete, or Failed?
```

### How to Narrow the Issue
```bash
# 1. What's the Job status?
kubectl describe job <name> | grep -A5 "Conditions"
# Complete? Failed? BackoffLimitExceeded? DeadlineExceeded?

# 2. What Pods were created?
kubectl get pods --selector=job-name=<name>
# How many Completed? How many Error? How many Pending?

# 3. Why did Pods fail?
kubectl logs <failed-pod>
# Application error → command/config issue

kubectl describe pod <failed-pod> | tail -20
# Events → image pull, scheduling, OOM

# 4. Is it a Pod-level or Job-level issue?
# Pod-level: image, command, resources, dependencies
# Job-level: backoffLimit, activeDeadlineSeconds, restartPolicy
```

### How to Confirm Root Cause
```bash
# Command/config issue confirmed:
kubectl logs <failed-pod> → error message from app

# Image issue confirmed:
kubectl describe pod → Events: ImagePullBackOff

# Resource issue confirmed:
kubectl describe pod → Events: FailedScheduling, OOMKilled

# Deadline issue confirmed:
kubectl describe job → Conditions: DeadlineExceeded

# restartPolicy issue confirmed:
kubectl get job -o yaml | grep restartPolicy → "Always" (invalid)
```

## 5 Quiz Questions to Test Understanding

### Q1: What are the only valid values for `restartPolicy` in a Job's Pod template?
<details>
<summary>Answer</summary>
"Never" or "OnFailure". "Always" is NOT valid for Jobs and causes a validation error.
</details>

### Q2: A Job has `completions: 5`, `parallelism: 2`, `backoffLimit: 6`. How does it execute?
<details>
<summary>Answer</summary>
It runs 2 Pods simultaneously. When a Pod succeeds, another is launched until 5 total Pods have succeeded. If any Pod fails, it's retried. If total failures across all Pods reach 6, the Job is marked as Failed.
</details>

### Q3: You need to fix a Job's container image. Can you use `kubectl edit job`?
<details>
<summary>Answer</summary>
No — spec.template is immutable. You must: export the Job YAML, edit the file, delete the old Job, and apply the fixed YAML.
</details>

### Q4: What is the difference between `restartPolicy: Never` and `restartPolicy: OnFailure` in a Job?
<details>
<summary>Answer</summary>
- Never: Failed Pod stays in Error state; Job creates a NEW Pod for retry. You can inspect failed Pods' logs. More Pods accumulate.
- OnFailure: kubelet restarts the container in the SAME Pod. Failed container logs are overwritten. Fewer Pods created.
</details>

### Q5: A Job has `backoffLimit: 6` and `activeDeadlineSeconds: 60`. The job takes 90 seconds per attempt. What happens?
<details>
<summary>Answer</summary>
The Job will be terminated after 60 seconds (activeDeadlineSeconds takes priority). Even on the first attempt, the deadline kills the Pod at 60s. Condition: Failed, Reason: DeadlineExceeded. The backoffLimit is never reached.
</details>

# 1️⃣3️⃣ Component Interaction Map

## How Job Interacts with Components

### Job ↔ Pods
```
NORMAL:    Job creates Pods to do work
           Pod names: <job-name>-<random-suffix>
           Completed Pods: STATUS = Completed, READY = 0/1
           Failed Pods: STATUS = Error (restartPolicy: Never)
           OR: container RESTARTS counter increments (restartPolicy: OnFailure)

FAILURE:   Pod CrashLoopBackOff → Job retries (up to backoffLimit)
           Pod Pending → Job Active but not progressing
           Pod OOMKilled → Job retries
           All retries exhausted → Job Failed (BackoffLimitExceeded)
```

### Job ↔ Services
```
NORMAL:    Jobs typically DON'T use Services (one-time work, not server)
           Exception: Job Pod may CONNECT to a Service (e.g., database)

FAILURE:   Dependency Service down → Job command fails → retry loop
           DNS resolution failure → command fails
           Service is output target (e.g., S3 endpoint) → upload fails → retry
```

### Job ↔ Nodes
```
NORMAL:    Scheduler places Job Pods on suitable nodes
           Considers: resources, taints, tolerations, affinity

FAILURE:   No suitable node → Pod Pending → Job stuck Active
           Node failure during Job → Pod rescheduled (if possible)
           Node resource exhaustion → OOMKilled → retry
```

### Job ↔ Scheduler
```
NORMAL:    Scheduler places Job Pods like any other Pod
           Job Pods are regular Pods with all scheduling features

FAILURE:   FailedScheduling → Pod Pending → Job appears stuck
           If activeDeadlineSeconds is set → Job may timeout while waiting
           Resource contention → Pod Pending → delays completions
```

### Job ↔ Controller Manager
```
NORMAL:    Job Controller (in kube-controller-manager):
             Watches Job objects
             Creates Pods (respecting parallelism)
             Tracks Pod success/failure counters
             Manages backoff timing between retries
             Marks Job Complete or Failed based on conditions

FAILURE:   Controller Manager down → no new Pods created for Jobs
           Existing running Pods continue (kubelet manages them)
           Recovery: when CM comes back, reconciles Job state
```

### Job ↔ etcd
```
NORMAL:    Job spec and status stored in etcd
           Pod creation/completion events tracked
           Completion count persisted

FAILURE:   etcd down → API server read-only → no new Pods
           Running Job Pods continue (kubelet)
           etcd recovery → Job reconcilation resumes
```

### Job ↔ Network Layer
```
NORMAL:    Job Pods get IPs from CNI (like any Pod)
           Job Pods may access Services, external APIs, etc.

FAILURE:   CNI failure → Pod ContainerCreating (no IP)
           DNS failure → Job command can't resolve hostnames
           Network policy blocking → Job command can't reach dependencies
```

### Job ↔ Storage Layer
```
NORMAL:    Job Pods can mount PVCs for reading input / writing output
           PVCs persist after Job completes (Pods are garbage but PVC stays)

FAILURE:   PVC Pending → Pod Pending → Job stuck
           PV full → Job command write fails → exit non-zero → retry
           RWO PV occupied → second parallel Pod can't mount → Pending
           Tip: For parallel Jobs writing to same storage → use ReadWriteMany
```

```
│                    COMPONENT INTERACTION MAP                   │
│                                                               │
```

## 🎯 CKA Exam Quick Reference

### Imperative Commands for Speed
```bash
# Create simple Job
kubectl create job my-job --image=busybox:1.36 -- echo "Hello"

# Create Job from CronJob (run CronJob now)
kubectl create job test-run --from=cronjob/my-cronjob

# Generate YAML
kubectl create job my-job --image=busybox:1.36 \
  --dry-run=client -o yaml -- sh -c "echo Hi" > job.yaml

# Check Job status
kubectl get jobs
kubectl describe job <name>

# Check Job Pods
kubectl get pods --selector=job-name=<name>
kubectl logs <pod-name>

# Patch (limited fields)
kubectl patch job <name> -p '{"spec":{"activeDeadlineSeconds":300}}'
kubectl patch job <name> -p '{"spec":{"parallelism":5}}'
kubectl patch job <name> -p '{"spec":{"suspend":true}}'

# Delete Job (and its Pods)
kubectl delete job <name>

# Delete Job but keep Pods (orphan)
kubectl delete job <name> --cascade=orphan

# Clean up completed Jobs
kubectl delete jobs --field-selector status.successful=1
```

### Mental Model: Job = "Run and Done"
```
Deployment: Run FOREVER → restart if container stops    (web servers)
Job:        Run ONCE    → stop when container succeeds   (batch work)
CronJob:    Run ONCE    → on a SCHEDULE                 (cron + batch)

Key rules to remember:
  ✅ apiVersion: batch/v1 (NOT apps/v1)
  ✅ restartPolicy: Never or OnFailure (NOT Always)
  ✅ spec.template is IMMUTABLE → must delete + recreate
  ✅ backoffLimit controls retry count (default: 6)
  ✅ activeDeadlineSeconds is a total timeout safety net
  ✅ Completed Pods are kept (for log inspection)
  ✅ Jobs are kept after completion (unless ttlSecondsAfterFinished)
  ❌ Cannot kubectl edit the Pod template
  ❌ No rolling updates (not applicable to Jobs)
  ❌ No rollout undo (not applicable to Jobs)
```
