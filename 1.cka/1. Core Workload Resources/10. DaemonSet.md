# 🧠 DaemonSet — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Ensures exactly ONE Pod runs on EVERY node** (or a subset of nodes) in the cluster
- When a new node is added → DaemonSet automatically places a Pod on it
- When a node is removed → the DaemonSet Pod is garbage collected
- Designed for **node-level infrastructure services** that must run everywhere
- No `replicas` field — DaemonSet manages Pod count based on node count

## Real-World Use Cases
- **Log collection** — Fluentd / Filebeat on every node collecting container logs
- **Node monitoring** — Prometheus Node Exporter on every node for metrics
- **CNI network plugin** — Calico / Flannel / Weave Pods on every node
- **Storage daemons** — GlusterFS / Ceph agent on every node
- **kube-proxy** — Runs as a DaemonSet on every node for Service networking
- **Security agents** — Falco / Sysdig on every node for runtime security

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────────┐
│                    DAEMONSET PLACEMENT                             │
│                                                                   │
│  DaemonSet: fluentd-logging                                      │
│    selector: app=fluentd                                         │
│                                                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐              │
│  │   Node-1    │  │   Node-2    │  │   Node-3    │              │
│  │             │  │             │  │             │              │
│  │ ┌─────────┐ │  │ ┌─────────┐ │  │ ┌─────────┐ │              │
│  │ │ fluentd │ │  │ │ fluentd │ │  │ │ fluentd │ │              │
│  │ └─────────┘ │  │ └─────────┘ │  │ └─────────┘ │              │
│  │  (1 Pod)    │  │  (1 Pod)    │  │  (1 Pod)    │              │
│  └─────────────┘  └─────────────┘  └─────────────┘              │
│                                                                   │
│  New Node-4 added → DaemonSet auto-places fluentd Pod           │
│  Node-2 removed   → DaemonSet Pod garbage collected             │
│                                                                   │
│  TOTAL PODS = NUMBER OF (matching) NODES                         │
└──────────────────────────────────────────────────────────────────┘
```

### DaemonSet vs Deployment vs ReplicaSet
| Aspect | DaemonSet | Deployment | ReplicaSet |
|--------|-----------|-----------|-----------|
| Pod count determined by | Number of nodes | `spec.replicas` | `spec.replicas` |
| Pods per node | Exactly 1 | 0 to many (Scheduler decides) | 0 to many |
| Has `replicas` field? | ❌ No | ✅ Yes | ✅ Yes |
| Uses Scheduler? | ✅ Yes (since K8s 1.12) | ✅ Yes | ✅ Yes |
| Use case | Node agents, log collectors | Stateless apps, APIs | Managed by Deployment |
| Rolling update? | ✅ Yes | ✅ Yes | ❌ No (use Deployment) |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **DaemonSet Controller** (inside kube-controller-manager) — watches nodes and DaemonSet specs
- **kubelet** — runs the DaemonSet Pod on each node
- **Scheduler** — places DaemonSet Pods (since K8s 1.12; uses `nodeAffinity`)
- **Cluster networking** — kube-proxy and CNI plugins run as DaemonSets
- **Monitoring/logging pipelines** — depend on DaemonSet agents being on every node

## What Resources It Interacts With
- **Nodes** — one Pod per node (or per matching node via nodeSelector/affinity)
- **Tolerations** — must tolerate node taints to run on tainted nodes (e.g., control plane)
- **nodeSelector / nodeAffinity** — select which nodes get a DaemonSet Pod
- **hostPath volumes** — DaemonSet Pods often mount node filesystem
- **hostNetwork / hostPID** — often use host namespaces for node-level access
- **ServiceAccount** — Pods may need cluster-level permissions (RBAC)

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User / Operator → creates DaemonSet
  DaemonSet Controller → watches node list + DaemonSet spec
  For each node matching criteria → ensures 1 Pod exists

DOWNSTREAM:
  DaemonSet Pod runs on node → collects logs / metrics / network
  If Pod crashes → kubelet restarts it (container restart policy)
  If node is added → DaemonSet Controller creates Pod on new node
  If node is removed → Pod is garbage collected

NODE MATCHING CHAIN:
  All nodes
    → Filter by nodeSelector (if set)
      → Filter by nodeAffinity (if set)
        → Must tolerate node taints
          → Result: Set of nodes that get DaemonSet Pods
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid DaemonSet YAML
```yaml
apiVersion: apps/v1              # ← Same as Deployment/RS
kind: DaemonSet
metadata:
  name: my-daemonset
  namespace: kube-system          # ← Infra DaemonSets usually in kube-system
  labels:
    app: my-daemon
spec:
  selector:                       # ← Required: how DS finds its Pods
    matchLabels:
      app: my-daemon              # ← MUST match template.metadata.labels
  template:                       # ← Pod template (no replicas field!)
    metadata:
      labels:
        app: my-daemon            # ← MUST match selector.matchLabels
    spec:
      tolerations:                # ← KEY: tolerate taints to run on all nodes
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - name: daemon
        image: fluentd:v1.16
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
          limits:
            cpu: 100m
            memory: 128Mi
        volumeMounts:
        - name: varlog
          mountPath: /var/log
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
```

## Real kube-proxy DaemonSet (What CKA Uses)
```bash
kubectl get daemonset kube-proxy -n kube-system -o yaml
# Key observations:
# - tolerations for control plane taints
# - hostNetwork: true (for network-level access)
# - serviceAccountName: kube-proxy
# - configMap volume for kube-proxy config
# - updateStrategy: RollingUpdate
```

## ⚠️ Common Indentation Traps

### Trap 1: Adding replicas field (doesn't exist)
```yaml
# ❌ WRONG — DaemonSets do NOT have replicas
spec:
  replicas: 3                    # ← INVALID for DaemonSet
  selector:
    matchLabels:
      app: my-daemon

# ✅ CORRECT — no replicas; Pod count = node count
spec:
  selector:
    matchLabels:
      app: my-daemon
```

### Trap 2: Missing tolerations → skips tainted nodes
```yaml
# ❌ WRONG — won't run on control plane nodes
spec:
  template:
    spec:
      containers:
      - name: daemon
        image: fluentd
      # No tolerations → DaemonSet skips nodes with taints!
      # Control plane has: node-role.kubernetes.io/control-plane:NoSchedule

# ✅ CORRECT — tolerate control plane taint
spec:
  template:
    spec:
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - name: daemon
        image: fluentd
```

### Trap 3: Selector ≠ template labels (same as RS/Deployment)
```yaml
# ❌ WRONG — validation error
spec:
  selector:
    matchLabels:
      app: logging
  template:
    metadata:
      labels:
        app: monitoring          # ← MISMATCH

# ✅ CORRECT
spec:
  selector:
    matchLabels:
      app: logging
  template:
    metadata:
      labels:
        app: logging             # ← MATCH
```

### Trap 4: Wrong apiVersion
```yaml
# ❌ WRONG
apiVersion: v1
kind: DaemonSet

# ✅ CORRECT
apiVersion: apps/v1
kind: DaemonSet
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: apps/v1` | Required API version | DaemonSet not recognized | 🔴 |
| `spec.selector.matchLabels` | How DS finds its Pods | Must match template labels — validation error | 🔴 |
| `spec.template.metadata.labels` | Labels on created Pods | Must match selector — validation error | 🔴 |
| `spec.template.spec.tolerations` | Run on tainted nodes | Missing = DaemonSet skips tainted nodes (control plane, dedicated) | 🔴 |
| `spec.template.spec.nodeSelector` | Restrict to specific nodes | Wrong selector = Pods on wrong/no nodes | 🔴 |
| `spec.template.spec.containers[].image` | What runs | ImagePullBackOff on all nodes | 🔴 |
| `spec.updateStrategy.type` | `RollingUpdate` (default) or `OnDelete` | OnDelete = manual Pod deletion needed for updates | 🟡 |
| `spec.template.spec.hostNetwork` | Use node's network stack | Required for network plugins; conflicts if port in use | 🟡 |
| `spec.template.spec.volumes[].hostPath` | Mount node filesystem | Wrong path = no data; mutates host filesystem | 🟡 |

## Key Relationships
```
selector.matchLabels ←—— MUST MATCH ——→ template.metadata.labels
  (exactly the same as Deployment/RS rule)

nodeSelector / nodeAffinity → selects WHICH nodes get Pods
tolerations → allows Pods on TAINTED nodes
  Together: nodeSelector filters nodes IN, tolerations allow tainted nodes

No replicas field → Pod count = matching node count
  3 worker nodes + 1 control plane (tainted)
    Without control-plane toleration → 3 Pods
    With control-plane toleration → 4 Pods

updateStrategy:
  RollingUpdate (default) → new Pod created BEFORE old Pod deleted (per node)
    maxUnavailable: 1 → update 1 node at a time
  OnDelete → old Pod must be manually deleted → then new Pod created
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `kind: DaemonSet` + `apiVersion: apps/v1`?
- No `replicas` field? (DaemonSets don't have one)
- `selector.matchLabels` matches `template.metadata.labels`?
- `tolerations` present? (to run on tainted nodes)
- `nodeSelector` if targeting specific nodes?
- Container `image` correct?

## 🟢 Usually Irrelevant for CKA
- `metadata.annotations` on the DaemonSet
- `spec.minReadySeconds`
- `spec.revisionHistoryLimit`
- `spec.updateStrategy` exact configuration (defaults are fine)
- `status` fields (managed by system)

## 🔴 Red Flags in 10 Seconds

| Red Flag | Problem |
|----------|---------|
| `replicas:` present | DaemonSets don't have replicas — wrong object type? |
| `selector.matchLabels ≠ template.metadata.labels` | Validation error |
| No `tolerations` when needed on tainted nodes | DS skips those nodes |
| Wrong `nodeSelector` | Pods on wrong nodes or no nodes |
| `apiVersion: v1` | Wrong API version |
| Image tag wrong/missing | ImagePullBackOff on all nodes |
| `updateStrategy: OnDelete` when RollingUpdate expected | Manual pod deletion required |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: DaemonSet Pod Missing on Some Nodes
```
SYMPTOM:        DESIRED=4, CURRENT=2 — Pods not on all nodes
CAUSE:          Missing toleration for node taints (e.g., control-plane:NoSchedule)
                OR: nodeSelector excludes some nodes
DETECTION:      kubectl describe ds <name> → "Desired: 4, Current: 2"
                kubectl get nodes --show-labels → check which nodes are missing
                kubectl describe node <missing-node> | grep Taint
FIX:            Add toleration to DaemonSet template OR remove taint from node
EXAM SCENARIO:  "A DaemonSet should run on all nodes but only runs on 2 out of 4"
```

## Pattern 2: All DaemonSet Pods ImagePullBackOff
```
SYMPTOM:        DESIRED=4, CURRENT=4, READY=0
                All Pods in ImagePullBackOff
CAUSE:          Wrong image name or tag in template
BEHAVIOR:       DS creates Pods on all nodes, but image can't be pulled
FIX:            kubectl edit ds <name> -n <ns> → fix image
                RollingUpdate strategy: Pods auto-update
EXAM TIP:       DS with RollingUpdate strategy will auto-replace Pods after edit
```

## Pattern 3: All DaemonSet Pods CrashLoopBackOff
```
SYMPTOM:        DESIRED=4, CURRENT=4, READY=0
                All Pods in CrashLoopBackOff
CAUSE:          Wrong command, missing config, wrong hostPath mount
DETECTION:      kubectl logs <ds-pod> -n <ns> → see crash reason
FIX:            Fix command/config in DS template
```

## Pattern 4: DaemonSet Pods Not Updating
```
SYMPTOM:        kubectl edit ds → updated, but old Pods still running old image
CAUSE:          updateStrategy.type: OnDelete → must manually delete old Pods
DETECTION:      kubectl get ds → UP-TO-DATE column < DESIRED
FIX:            Option A: Delete Pods manually (kubectl delete pod <name>) → new Pods created
                Option B: Change updateStrategy to RollingUpdate
```

## Pattern 5: Selector ≠ Template Labels
```
SYMPTOM:        DaemonSet creation fails
ERROR:          "selector does not match template labels"
BEHAVIOR:       DS not created at all
FIX:            Match selector.matchLabels with template.metadata.labels
```

## Pattern 6: Resource Quota Prevents Pod Creation
```
SYMPTOM:        DESIRED=4, CURRENT=2 — some Pods can't be created
EVENTS:         "FailedCreate: exceeded quota"
CAUSE:          ResourceQuota in namespace limits Pod/resource count
FIX:            Increase quota or reduce DS resource requests
```

## Pattern 7: hostPath Volume Permission Issue
```
SYMPTOM:        Pod Running but application inside errors: "permission denied"
CAUSE:          Container user can't read/write host filesystem
DETECTION:      kubectl logs → permission denied errors
FIX:            securityContext.runAsUser: 0 (run as root) OR fix file permissions on host
EXAM TIP:       Common for log collectors needing /var/log access
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Selector ≠ Template Labels?
- **Behavior:** DaemonSet creation rejected — validation error
- Same error as Deployment/RS: `selector does not match template labels`
- Zero Pods created
- **Fix:** Match the labels exactly

## What Happens If Tolerations Are Missing?
- **Behavior:** DaemonSet skips all tainted nodes
- Pods only created on untainted nodes
- DESIRED shows fewer than total node count
- **Example:** 4 nodes (1 control plane with NoSchedule taint) → DESIRED=3 without toleration
- **Fix:** Add toleration for the specific taint

## What Happens If Image Is Wrong?
- **Behavior:** DS creates Pods on all nodes → all get ImagePullBackOff
- DESIRED=4, CURRENT=4, READY=0
- **Fix:** `kubectl edit ds <name>` → fix image → Pods auto-update (RollingUpdate)

## What Happens If nodeSelector Is Wrong?
- **Behavior:** DS finds 0 matching nodes → DESIRED=0, zero Pods
- **Detection:** `kubectl get nodes --show-labels` → check if any node has the label
- **Fix:** Fix nodeSelector to match actual node labels, or label nodes correctly

## What Happens If Resource Limits Are Too Low?
- **Behavior:** DS Pods get OOMKilled on busy nodes → CrashLoopBackOff
- Some nodes fine (low load), others constantly killed (high load)
- **Fix:** Increase `limits.memory`; DaemonSet Pods shouldn't be resource-starved

## What Happens If hostPath Points to Non-Existent Directory?
- **Behavior depends on `hostPath.type`:**
  - `Directory` → Pod creation fails (dir must exist on host)
  - `DirectoryOrCreate` → kubelet creates the dir → Pod starts
  - No type → permissive, usually creates dir
- **Fix:** Use `DirectoryOrCreate` or ensure dir exists on all nodes

## What Happens If You Delete a DaemonSet?
- **Behavior:** All managed Pods are deleted across all nodes
- To keep Pods alive during DS deletion: `kubectl delete ds <name> --cascade=orphan`
- Orphaned Pods keep running but are unmanaged

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: DaemonSet health
kubectl get ds -n <namespace>
# Read: DESIRED, CURRENT, READY, UP-TO-DATE, AVAILABLE
# DESIRED ≠ CURRENT → missing Pods on some nodes
# CURRENT ≠ READY → Pods exist but failing

# ⏱️ 10-20 sec: Which nodes are missing?
kubectl get pods -n <namespace> -l <selector> -o wide
# Check NODE column → which node has no Pod?
# Or which node's Pod is failing?

# ⏱️ 20-35 sec: Why is a node missing?
kubectl describe node <missing-node> | grep -A5 Taint
# Tainted? → DS needs toleration
kubectl get nodes --show-labels
# Missing label? → nodeSelector issue

# ⏱️ 35-50 sec: Why are Pods failing?
kubectl describe pod <failing-ds-pod> | tail -15
kubectl logs <failing-ds-pod> -n <namespace> --previous

# ⏱️ 50-60 sec: Fix path
# Missing toleration → edit DS → add it
# Wrong image → edit DS → fix
# Wrong nodeSelector → edit DS → fix or label node
```

## Quick Diagnosis Table
```
DS STATUS                              → LIKELY CAUSE
──────────────────────────────────────────────────────────
DESIRED=4, CURRENT=2                   → 2 nodes tainted or nodeSelector excludes them
DESIRED=4, CURRENT=4, READY=0          → All Pods failing (image/command/config)
DESIRED=4, CURRENT=4, READY=4, UP=2    → 2 Pods not updated (OnDelete strategy)
DESIRED=0, CURRENT=0                   → nodeSelector matches no nodes
DESIRED=4, CURRENT=4, READY=2          → 2 Pods failing on specific nodes (node-specific issue)
```

# 🔟 Fast Fix Strategy

## Common Fixes

### Fix: Missing Pods on tainted nodes (add toleration)
```bash
kubectl edit ds <name> -n <namespace>
# Add under spec.template.spec:
#   tolerations:
#   - key: node-role.kubernetes.io/control-plane
#     operator: Exists
#     effect: NoSchedule

# OR: tolerate ALL taints (aggressive but effective):
#   tolerations:
#   - operator: Exists
```

### Fix: Wrong image
```bash
kubectl set image ds/<name> <container>=<correct-image> -n <namespace>
# If RollingUpdate → Pods auto-replaced
# If OnDelete → manually delete Pods:
kubectl delete pods -l <selector> -n <namespace>
```

### Fix: Wrong nodeSelector (target all nodes)
```bash
kubectl edit ds <name> -n <namespace>
# Remove the nodeSelector block to target all nodes
# OR fix the label value to match actual node labels

# Alternatively, label the nodes:
kubectl label node <node-name> <key>=<value>
```

### Fix: OnDelete strategy → force update
```bash
# If DS uses OnDelete and you need Pods updated:
kubectl get pods -l <selector> -n <namespace> -o name | xargs kubectl delete -n <namespace>
# DS recreates Pods with new template

# Or change strategy:
kubectl patch ds <name> -n <namespace> \
  -p '{"spec":{"updateStrategy":{"type":"RollingUpdate"}}}'
```

### Fix: Create DaemonSet quickly (exam)
```bash
# No kubectl create daemonset command exists!
# Strategy: Generate Deployment YAML, convert to DaemonSet

# Step 1: Generate base
kubectl create deployment ds-base --image=nginx --dry-run=client -o yaml > ds.yaml

# Step 2: Edit ds.yaml
# Change kind: Deployment → kind: DaemonSet
# Remove spec.replicas line
# Remove spec.strategy section
# Keep everything else
vim ds.yaml

# Step 3: Apply
kubectl apply -f ds.yaml
```

## Validate Fix
```bash
# DS health
kubectl get ds <name> -n <namespace>
# DESIRED=CURRENT=READY=UP-TO-DATE=AVAILABLE ✅

# All Pods running on expected nodes
kubectl get pods -l <selector> -n <namespace> -o wide
# Pod on every target node, Status: Running

# No warning events
kubectl describe ds <name> -n <namespace> | tail -10
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Minimal DaemonSet
```yaml
# File: ds-demo.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: ds-demo
  namespace: default
  labels:
    app: ds-demo
spec:
  selector:
    matchLabels:
      app: ds-demo
  template:
    metadata:
      labels:
        app: ds-demo
    spec:
      tolerations:
      - operator: Exists         # Tolerate ALL taints → run on EVERY node
      containers:
      - name: agent
        image: nginx:1.25
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
          limits:
            cpu: 100m
            memory: 128Mi
```

### Validation Steps
```bash
kubectl apply -f ds-demo.yaml

# Check DS status
kubectl get ds ds-demo
# DESIRED = total nodes in cluster, READY = all

# Check Pods on every node
kubectl get pods -l app=ds-demo -o wide
# 1 Pod per node ✅

# Verify ownership
kubectl describe pod <ds-pod-name> | grep "Controlled By"
# Controlled By: DaemonSet/ds-demo

# Add a node to cluster → DS auto-creates Pod
# Remove a node → Pod auto-removed
```

```bash
kubectl set image ds/ds-demo agent=nginx:1.26
kubectl get ds ds-demo
# UP-TO-DATE shows 0 → old Pods still running!
# Must manually delete Pods for update
```

### 5 Quiz Questions

**Q1:** A DaemonSet should run on all 4 nodes but only shows DESIRED=3. What's the most likely cause?
> **A:** One node has a taint (likely the control plane with `node-role.kubernetes.io/control-plane:NoSchedule`) and the DaemonSet template doesn't have a matching toleration. Add the toleration to fix it.

**Q2:** How do you create a DaemonSet imperatively (no `kubectl create daemonset`)?
> **A:** Generate a Deployment YAML: `kubectl create deployment tmp --image=nginx --dry-run=client -o yaml > ds.yaml`. Edit the file: change `kind: Deployment` to `kind: DaemonSet`, remove `spec.replicas` and `spec.strategy`. Apply: `kubectl apply -f ds.yaml`.

**Q3:** A DaemonSet shows UP-TO-DATE=0 after you edited it. Why?
> **A:** The DaemonSet uses `updateStrategy.type: OnDelete`. Pods aren't automatically replaced — you must manually delete each Pod for the DS to recreate it with the new template.

**Q4:** How is a DaemonSet different from a Deployment with replica count = node count?
> **A:** DaemonSet guarantees exactly 1 Pod per node and auto-adjusts when nodes are added/removed. Deployment with matching replica count has no node awareness — Scheduler could place all Pods on one node. DaemonSet has no `replicas` field at all.

**Q5:** You need a DaemonSet to run ONLY on nodes labeled `disk=ssd`. How?
> **A:** Add `nodeSelector: {"disk": "ssd"}` to the DaemonSet's Pod template spec. DaemonSet will only create Pods on nodes with that label.

# 1️⃣3️⃣ Component Interaction Map

```
┌──────────────────────────────────────────────────────────────────┐
│                    DAEMONSET INTERACTIONS                          │
│                                                                   │
│  ┌──────────────┐                                                │
│  │  DaemonSet   │                                                │
│  │  Controller  │ watches nodes + DS spec                        │
│  └──────┬───────┘                                                │
│         │ For each matching node, ensure 1 Pod exists            │
│   ┌─────┼─────────────────┬───────────────────┐                 │
│   ▼     ▼                 ▼                   ▼                 │
│ ┌──────┐ ┌──────┐     ┌──────┐          ┌──────┐              │
│ │Node-1│ │Node-2│     │Node-3│          │Node-4│              │
│ │[Pod] │ │[Pod] │     │[Pod] │          │[Pod] │              │
│ └──────┘ └──────┘     └──────┘          └──────┘              │
│                                                                   │
│  Each Pod:                                                       │
│    - Runs same template                                          │
│    - Accesses node's filesystem via hostPath                    │
│    - May use hostNetwork for node-level networking              │
│    - Collects logs/metrics from that specific node              │
└──────────────────────────────────────────────────────────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Things Go Wrong |
|-----------|----------------|---------------------|
| **Controller Manager** | DS controller watches nodes and DS specs. Creates/deletes Pods to maintain 1-per-node. | Controller down → no new DS Pods, no adjustment for new nodes. |
| **Scheduler** | Places DS Pods (uses nodeAffinity internally since K8s 1.12). | If Scheduler down → new DS Pods may not be placed (stuck Pending). |
| **Nodes** | Each node runs 1 DS Pod. New node → new Pod. Removed node → Pod gone. | Node NotReady → DS Pod on that node may be evicted or marked unknown. |
| **Taints/Tolerations** | DS Pod must tolerate node taints to be scheduled there. | Missing toleration → DS skips that node → DESIRED < total nodes. |
| **Service** | Can select DS Pods by labels. Useful for node-level metrics endpoints. | If DS Pods crash → Service has fewer endpoints. |
| **etcd** | Stores DS spec and Pod state. | etcd down → no state changes. |
| **Network** | DS Pods often use `hostNetwork: true` for node-level access. | Port conflict → DS Pod on that node can't start. |
| **Storage** | DS Pods use `hostPath` to access node filesystem (/var/log, /etc). | Dir missing on some nodes → Pod fails on those nodes only. |

### Failure Cascade
```
DaemonSet with wrong image
  → DS creates Pods on all nodes → all get ImagePullBackOff
    → No monitoring/logging agents running on any node
      → Monitoring dashboard shows 0 nodes reporting
        → Alerts fire: "no data from nodes"

Control-plane toleration missing
  → DS skips control-plane node
    → Control-plane node unmonitored
      → No logs collected from API server node
        → Blind spot in observability

Node-specific hostPath missing
  → DS Pod fails only on THAT node
    → DESIRED=4, READY=3
      → One node has no agent running
        → Partial monitoring gap
```

## ✔ Can I Break This Intentionally?

| Breakage | How |
|----------|-----|
| Skip tainted nodes | Remove tolerations from DS template |
| Zero Pods | Set nodeSelector to non-existent label |
| All Pods ImagePullBackOff | Set image to non-existent tag |
| All Pods CrashLoop | Add command: ['exit', '1'] |
| Selector mismatch | Mismatch selector vs template labels |
| Pods not updating | Set updateStrategy to OnDelete |
| Pod fails on specific node | Add hostPath with type: Directory to non-existent dir |
| Port conflict | Use hostNetwork with port already in use |
