# 🧠 Init Containers — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Specialized containers that run BEFORE any main (app) containers start**
- Run **sequentially** — each must complete successfully (exit code 0) before the next begins
- Once ALL init containers complete → main containers start in parallel
- Init containers are **run-to-completion** — they MUST exit (unlike sidecars which run forever)
- Used for **setup, precondition checks, dependency waits, and data preparation**

## Real-World Use Cases
- **Wait for a dependency** — Poll until a database or API is reachable
  ```bash
  until nslookup my-database.default.svc.cluster.local; do echo waiting; sleep 2; done
  ```
- **Clone code/config** — Git clone a repo before the app starts
- **Database migration** — Run schema migrations before app boots
- **Generate config files** — Create configuration files and write to a shared volume
- **Seed data** — Populate shared volume with initial data
- **Permission setup** — Fix file permissions on volumes (`chown`, `chmod`)
- **Register with external service** — Call an API to register before app starts
- **Download certificates/secrets** — Fetch from Vault or external secret manager

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────┐
│                          POD LIFECYCLE                         │
│                                                               │
│  ┌──────────┐   ┌──────────┐          ┌──────────────────┐  │
│  │  Init 1  │──→│  Init 2  │──→ ... ──→│  Main Containers │  │
│  │ (must    │   │ (must    │          │  (start together) │  │
│  │  exit 0) │   │  exit 0) │          │  (run forever)   │  │
│  └──────────┘   └──────────┘          └──────────────────┘  │
│                                                               │
│  ← Sequential, run-to-completion →    ← Parallel, long-lived │
│                                                               │
│  Pod Status: Init:0/2 → Init:1/2 → PodInitializing → Running│
└──────────────────────────────────────────────────────────────┘
```

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Main containers** — They CANNOT start until all init containers exit 0
- **kubelet** — Runs init containers one by one, monitors exit codes
- **Pod status reporting** — API server reflects init container state in Pod status
- **Deployment / ReplicaSet** — If init containers keep failing, rollout stalls

## What Resources It Interacts With
- **Shared Volumes** — Init writes data → main container reads it (`emptyDir`, PVC)
- **ConfigMaps / Secrets** — Can mount and use them (same as main containers)
- **ServiceAccount** — Same SA as the Pod (same RBAC permissions)
- **Network** — Has full Pod networking (can make DNS queries, API calls)
- **SecurityContext** — Can have its own securityContext OR inherit Pod-level

## Upstream & Downstream Dependencies
```
UPSTREAM (what triggers init containers):
  Pod creation → kubelet receives Pod spec → starts init containers

DOWNSTREAM (what depends on init completing):
  Init Container 1 (exit 0) → Init Container 2 (exit 0) → ... → Main Containers start
  
  If ANY init fails:
    → All subsequent inits blocked
    → Main containers NEVER start
    → Pod stuck in Init:Error / Init:CrashLoopBackOff
    → Deployment rollout stalled (if maxUnavailable reached)
```

## Key Difference from Main Containers
| Aspect | Init Container | Main Container |
|--------|---------------|----------------|
| Lifecycle | Run-to-completion (must exit) | Long-running (should NOT exit) |
| Execution | Sequential (one at a time) | Parallel (all at once) |
| Restart | Restarted on failure (per Pod restartPolicy) | Restarted on failure |
| Probes | ❌ No liveness/readiness probes | ✅ Supports all probes |
| Resources | Can have own requests/limits | Has own requests/limits |
| Purpose | Setup, preconditions | Application logic |

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid YAML — Pod with Init Container
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-demo
  labels:
    app: init-demo
spec:
  initContainers:                        # ← Pod-level field (same level as containers)
  - name: init-service-check             # ← First init container
    image: busybox:1.36
    command:                             # ← Must exit 0 to succeed
    - 'sh'
    - '-c'
    - 'until nslookup my-svc.default.svc.cluster.local; do echo waiting; sleep 2; done'

  - name: init-data-setup               # ← Second init (runs after first succeeds)
    image: busybox:1.36
    command: ['sh', '-c', 'echo "Ready" > /work/status']
    volumeMounts:
    - name: workdir
      mountPath: /work

  containers:                            # ← Main containers (start after ALL inits complete)
  - name: app
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: workdir
      mountPath: /usr/share/nginx/html

  volumes:
  - name: workdir
    emptyDir: {}
```

## Key Sections Explained
| Section | Role | CKA Importance |
|---------|------|---------------|
| `spec.initContainers[]` | List of init containers (Pod-level, NOT nested under containers) | 🔴 CRITICAL |
| `initContainers[].name` | Unique name per init container | 🔴 CRITICAL — used with `-c` flag |
| `initContainers[].image` | Container image | 🔴 CRITICAL — wrong image = Init:ImagePullBackOff |
| `initContainers[].command` | Command to run (MUST exit) | 🔴 CRITICAL — infinite command = Pod stuck forever |
| `initContainers[].volumeMounts` | Mount volumes to write data for main containers | 🟡 HIGH |
| `spec.volumes[]` | Shared volumes (both init and main can use) | 🟡 HIGH |

## ⚠️ Common Indentation Traps

### Trap 1: initContainers INSIDE containers (WRONG)
```yaml
# ❌ FATAL — initContainers nested under containers
spec:
  containers:
  - name: app
    image: nginx
    initContainers:           # ← WRONG! This is not a valid container field
    - name: init
      image: busybox

# ✅ CORRECT — initContainers at spec level
spec:
  initContainers:             # ← Same level as containers
  - name: init
    image: busybox
  containers:
  - name: app
    image: nginx
```

### Trap 2: Wrong list indentation
```yaml
# ❌ WRONG — command items not aligned
  initContainers:
  - name: init
    image: busybox
    command:
      - 'sh'                  # ← Indented under command (OK but risky)
        - '-c'                 # ← WRONG — extra indent breaks YAML
        - 'echo hello'

# ✅ CORRECT — consistent indentation
    command:
    - 'sh'
    - '-c'
    - 'echo hello'

# ✅ ALSO CORRECT — inline array
    command: ['sh', '-c', 'echo hello']
```

### Trap 3: Forgetting volume in init AND main container
```yaml
# ❌ PROBLEM — init writes to /work but main doesn't mount it
  initContainers:
  - name: init
    image: busybox
    command: ['sh', '-c', 'echo data > /work/file']
    volumeMounts:
    - name: shared
      mountPath: /work
  containers:
  - name: app
    image: nginx
    # ← MISSING volumeMount! App can't see init's data
  volumes:
  - name: shared
    emptyDir: {}
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields Table

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `initContainers[].name` | Used in `kubectl logs -c <name>` and `kubectl describe` | Can't debug; duplicate names = validation error | 🔴 |
| `initContainers[].image` | What runs | `Init:ImagePullBackOff` — Pod never starts | 🔴 |
| `initContainers[].command` | The actual task; MUST EXIT | Never exits = `Init:0/N` stuck forever; Non-zero exit = `Init:CrashLoopBackOff` | 🔴 |
| `initContainers[].args` | Arguments to command | Wrong behavior, potential crash | 🟡 |
| `initContainers[].volumeMounts` | Write data for main containers | Data not available to main; or wrong path | 🟡 |
| `initContainers[].env` / `envFrom` | Config from ConfigMaps/Secrets | Missing env → command may fail | 🟡 |
| `initContainers[].resources` | CPU/memory for init | OOMKilled during init; or scheduling issue if requests too high | 🟡 |
| `initContainers[].securityContext` | Run as root for permission fixes | Insufficient permissions to write volumes | 🟡 |
| `spec.volumes[]` | Shared storage between init and main | Missing volume → `CreateContainerConfigError` | 🔴 |

## Field Relationships
```
initContainers[].volumeMounts[].name ←——MUST MATCH——→ volumes[].name
  (where init writes data)                              (volume definition)

initContainers[].volumeMounts[].mountPath ←——SHOULD ALIGN——→ containers[].volumeMounts[].mountPath
  (path init uses to write)                                   (path main uses to read)
  NOTE: paths can differ! Only volume NAME must match

initContainers[].command ←——OVERRIDES——→ Dockerfile ENTRYPOINT
initContainers[].args    ←——OVERRIDES——→ Dockerfile CMD

Init 1 (exit 0) ——THEN——→ Init 2 (exit 0) ——THEN——→ Main containers start
  (sequential chain — any failure breaks the chain)
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- Is `initContainers` at `spec` level (not nested under `containers`)?
- How many init containers? (Count `- name:` entries under `initContainers`)
- Does each init have a `command` that **will exit**?
- Are there `volumeMounts` matching `volumes[]`? (name match check)
- Is the image valid and pullable?

## 🟢 Usually Irrelevant for CKA
- `imagePullPolicy` on init containers (defaults work)
- `terminationMessagePath` / `terminationMessagePolicy`
- `stdin` / `stdinOnce` / `tty` on init containers
- Init container lifecycle hooks (not supported)
- Init container probes (not supported — no liveness/readiness on init)

## 🔴 Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| `command: ['sleep', 'infinity']` in init | Init never exits → Pod stuck at `Init:0/N` forever |
| `command: ['sh', '-c', 'exit 1']` | Init always fails → `Init:CrashLoopBackOff` |
| No `command` field + image has no ENTRYPOINT | Init does nothing or exits immediately |
| `initContainers` appears inside `containers[0]` | YAML structure wrong → unknown field error |
| `volumeMounts.name: data` but no `volumes[].name: data` | Volume not found → container creation error |
| Init container tries `nslookup` a service that doesn't exist | Init hangs until timeout, or loops forever |
| Init has `resources.limits.memory: 5Mi` for heavy task | OOMKilled during init |

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe pod <pod>`

### Where to Look FIRST
```
Init Containers:                         ← DEDICATED SECTION — check this!
  init-service-check:
    State:          Terminated           ← SUCCESS if Reason: Completed, Exit Code: 0
      Reason:       Completed
      Exit Code:    0                    ← MUST be 0
    Ready:          True

  init-data-setup:
    State:          Waiting              ← STUCK — this init hasn't started or is failing
      Reason:       CrashLoopBackOff    ← FAILING!
    Last State:     Terminated
      Reason:       Error
      Exit Code:    1                    ← Non-zero = failure
    Restart Count:  4                    ← Keeps retrying and failing

Containers:
  app:
    State:          Waiting
      Reason:       PodInitializing     ← Blocked! Waiting for init to complete
```

### Key Fields in `describe` Output
| Field | What to Check |
|-------|--------------|
| `Init Containers:` section | State of each init container |
| `State: Terminated, Exit Code: 0` | ✅ Init succeeded |
| `State: Terminated, Exit Code: 1` (or non-zero) | ❌ Init command failed |
| `State: Waiting, Reason: CrashLoopBackOff` | ❌ Init keeps crashing |
| `State: Waiting, Reason: ImagePullBackOff` | ❌ Can't pull init image |
| `State: Running` | ⚠️ Init is still running (may be stuck) |
| `Restart Count: N` | Init has failed and been retried N times |
| Main container `State: Waiting, Reason: PodInitializing` | Blocked by init containers |
| `Conditions: Initialized: False` | Init containers haven't all completed |
| `Events:` section | Warnings about pull failures, crashes, OOM |

### Failure Signals in Events
```
Events:
  Warning  BackOff    Back-off restarting failed container init-data-setup
  Warning  Failed     Error: failed to start container "init-service-check"
  Warning  Failed     Error: ErrImagePull
  Normal   Pulling    Pulling image "busybox:wrong-tag"
  Warning  Failed     Failed to pull image "busybox:wrong-tag": not found
```

## 📝 `kubectl logs <pod> -c <init-container-name>`

```bash
# View init container logs (MUST use -c flag)
kubectl logs init-demo -c init-service-check
kubectl logs init-demo -c init-data-setup

# If init crashed and was restarted
kubectl logs init-demo -c init-service-check --previous

# Follow logs of a running init container (if it's stuck)
kubectl logs init-demo -c init-service-check -f
```

### What to Look For in Logs
| Log Pattern | Meaning |
|-------------|---------|
| `nslookup: can't resolve 'my-svc.default.svc.cluster.local'` | Init waiting for a Service that doesn't exist |
| `permission denied` | Init needs `runAsUser: 0` or volume has wrong permissions |
| `command not found` | Wrong `command` field or image doesn't have the binary |
| `Connection refused` | Dependency isn't reachable (wrong port/host) |
| No logs at all | Init exited too fast (check exit code via `describe`) |
| Repeating loop output | Init is stuck in a polling loop (may be expected, may be a problem) |

## 🔧 `kubectl exec` for Init Containers

**⚠️ You CANNOT exec into a completed init container.** Init containers are terminated once done.

```bash
# ❌ This won't work if init has already completed
kubectl exec init-demo -c init-service-check -- ls /

# ✅ You CAN exec if init is currently running (stuck or slow)
kubectl exec init-demo -c init-service-check -- nslookup my-svc

# ✅ After init completes, exec into MAIN container to verify init's work
kubectl exec init-demo -c app -- cat /usr/share/nginx/html/status
kubectl exec init-demo -c app -- ls /usr/share/nginx/html/
```

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Init Container Image Not Found
```
POD STATUS:     Init:ImagePullBackOff
CAUSE:          Wrong image name or tag in initContainers[].image
EVENTS:         "Failed to pull image", "ErrImagePull", "ImagePullBackOff"
BEHAVIOR:       Init never starts → main containers never start
                Pod stuck permanently
FIX:            Correct the image name/tag
EXAM SCENARIO:  "A Pod is stuck in Init:ImagePullBackOff. Fix it."
                → kubectl describe pod → find wrong image → kubectl edit or recreate
```

## Pattern 2: Init Container Command Fails (Non-Zero Exit)
```
POD STATUS:     Init:CrashLoopBackOff
CAUSE:          Command returns non-zero exit code (exit 1, error, crash)
EVENTS:         "Back-off restarting failed container"
DESCRIBE:       Exit Code: 1 (or 2, 126, 127, 137, etc.)
BEHAVIOR:       Init retried with increasing backoff delays
                Main containers never start
FIX:            Fix the command logic, fix permissions, fix dependencies
EXAM SCENARIO:  "A Pod is stuck in Init:CrashLoopBackOff. Debug and fix."
                → kubectl logs <pod> -c <init-name> → see error → fix command
```

## Pattern 3: Init Container Never Exits (Infinite Loop/Sleep)
```
POD STATUS:     Init:0/1  or  Init:0/2  (number stuck, not incrementing)
CAUSE:          Command runs forever (sleep infinity, tail -f, infinite loop with no exit)
DESCRIBE:       Init container State: Running (never transitions to Terminated)
BEHAVIOR:       Pod stuck forever. No crash, no error — just waiting.
                Very tricky because there's no error message!
FIX:            Make the command finite — it MUST exit with code 0
EXAM SCENARIO:  "A Pod shows Init:0/1 for a long time. Why?"
                → kubectl describe → init State: Running → check command → fix to exit
```

## Pattern 4: Init Container Waits for Non-Existent Service
```
POD STATUS:     Init:0/1  (stuck, but init IS running)
CAUSE:          Init does `until nslookup my-svc; do sleep 2; done` but my-svc doesn't exist
LOGS:           "nslookup: can't resolve 'my-svc'" repeated
BEHAVIOR:       Init loops waiting forever. Pod never progresses.
FIX:            Create the missing Service, or fix the service name in init command
EXAM SCENARIO:  "A Pod is stuck in Init:0/1. The init container is waiting for a Service."
                → kubectl logs -c init → see DNS failure → create the Service
```

## Pattern 5: Init Container OOMKilled
```
POD STATUS:     Init:CrashLoopBackOff
CAUSE:          Init exceeds memory limit
DESCRIBE:       Exit Code: 137, Reason: OOMKilled
BEHAVIOR:       Init killed by OOM, retried, killed again — infinite loop
FIX:            Increase memory limit or optimize init command
EXAM SCENARIO:  → kubectl describe → see OOMKilled → increase resources.limits.memory
```

## Pattern 6: Volume Mount Mismatch
```
POD STATUS:     Blocked at container creation
CAUSE:          volumeMounts[].name in init doesn't match any volumes[].name
ERROR:          "Error: failed to create containerd container" or validation error
BEHAVIOR:       Pod won't even start
FIX:            Make volumeMounts.name match volumes.name exactly
```

## Pattern 7: Init Succeeds But Main Container Can't Find Data
```
POD STATUS:     Running (main container up, but behaves wrong)
CAUSE:          Init wrote to /work/data but main container mounts at /app/data
                OR main container doesn't mount the shared volume at all
BEHAVIOR:       No crash! App runs but with missing/empty data.
                Hardest to detect because Pod looks healthy.
FIX:            Verify both init and main mount the same volume
                Verify mount paths are what the app expects
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Image Is Wrong?
```
initContainers:
- name: init
  image: busybox:nonexistent     ← Tag doesn't exist
```
- **Behavior:** Pod status = `Init:ImagePullBackOff`
- **Impact:** ALL downstream (other inits + main) are blocked
- **Detection:** `kubectl describe pod` → Events show "Failed to pull image"
- **Fix:** Correct the image tag

## What Happens If Command Is Wrong?
```
initContainers:
- name: init
  command: ['nonexistent-binary']    ← Binary doesn't exist in image
```
- **Behavior:** `Init:CrashLoopBackOff`, Exit Code: 127 (command not found)
- **Detection:** `kubectl logs <pod> -c init` → "exec: nonexistent-binary: not found"
- **Fix:** Use correct binary path or correct image

## What Happens If Command Never Exits?
```
initContainers:
- name: init
  command: ['tail', '-f', '/dev/null']    ← Runs forever
```
- **Behavior:** `Init:0/1` — stuck, no error, no crash
- **Detection:** `kubectl describe pod` → init State: Running (not Terminated)
- **Fix:** Change command to something that exits: `echo done && exit 0`

## What Happens If Resource Limits Are Too Low?
```
initContainers:
- name: init
  resources:
    limits:
      memory: 5Mi          ← Way too low
  command: ['sh', '-c', 'apt-get update']   ← Needs more memory
```
- **Behavior:** `Init:CrashLoopBackOff`, Exit Code: 137 (OOMKilled)
- **Detection:** `kubectl describe` → Reason: OOMKilled
- **Fix:** Increase `resources.limits.memory`

## What Happens If Dependencies Are Missing?

### Missing ConfigMap/Secret
```
initContainers:
- name: init
  envFrom:
  - configMapRef:
      name: nonexistent-config     ← ConfigMap doesn't exist
```
- **Behavior:** `CreateContainerConfigError` — init container can't start
- **Detection:** `kubectl describe` → "configmap nonexistent-config not found"
- **Fix:** Create the ConfigMap: `kubectl create configmap nonexistent-config --from-literal=key=value`

### Missing Service (DNS dependency)
```
initContainers:
- name: init
  command: ['sh', '-c', 'until nslookup missing-svc; do sleep 2; done']
```
- **Behavior:** Init loops forever, Pod stuck at `Init:0/1`
- **Detection:** `kubectl logs <pod> -c init` → "can't resolve 'missing-svc'"
- **Fix:** Create the Service or fix the service name

### Missing PVC
```
volumes:
- name: data
  persistentVolumeClaim:
    claimName: nonexistent-pvc     ← PVC doesn't exist or not bound
```
- **Behavior:** Pod stuck in `Pending` (not even `Init:0/1`)
- **Detection:** `kubectl describe pod` → "persistentvolumeclaim not found"
- **Fix:** Create the PVC and ensure a PV binds to it

## What Happens If Labels Are Wrong?
- **Init containers themselves don't use labels** — labels are on the Pod
- Wrong Pod labels only matter if a **Deployment selector** doesn't match
- If mismatched: no Pods are created at all (Deployment reports 0 replicas ready)
- Init containers are unaffected by their parent Pod's labels

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: What's the Pod status?
kubectl get pod <pod-name>
# KEY: Read STATUS column
#   Init:0/2         → First init hasn't completed
#   Init:1/2         → First init done, second stuck
#   Init:Error       → An init failed
#   Init:CrashLoop   → An init keeps crashing
#   PodInitializing  → Inits done, main starting
#   Running          → Everything passed ✅

# ⏱️ 10-25 sec: Which init container is the problem?
kubectl describe pod <pod-name> | grep -A8 "Init Containers:"
# Look for: State, Reason, Exit Code, Restart Count

# ⏱️ 25-40 sec: WHY is it failing?
kubectl logs <pod-name> -c <failing-init-name>
kubectl logs <pod-name> -c <failing-init-name> --previous   # If CrashLoop

# ⏱️ 40-50 sec: Check events for more clues
kubectl describe pod <pod-name> | tail -15

# ⏱️ 50-60 sec: Verify the root cause
# If image issue → check image name
# If command issue → check command/args
# If dependency → check if Service/ConfigMap/PVC exists
kubectl get svc,cm,secret,pvc -n <namespace>
```

## Status Decoder
```
STATUS VALUE          → MEANING                        → FIRST ACTION
─────────────────────────────────────────────────────────────────────
Init:0/1              → Init still running or stuck     → Check if init command exits
Init:CrashLoopBackOff → Init command fails repeatedly   → kubectl logs -c <init> --previous
Init:ImagePullBackOff → Can't pull init image           → Fix image name/tag
Init:Error            → Init exited with non-zero       → kubectl logs -c <init>
Init:0/2              → First of 2 inits hasn't done    → Check first init
Init:1/2              → 2nd init stuck                  → Focus on 2nd init
Pending               → Can't even schedule Pod         → Check resources, PVC, taints
PodInitializing       → Inits done, main containers     → Wait, or check main containers
                        starting up
```

# 🔟 Fast Fix Strategy

## Fix Method Decision Tree
```
Is Pod managed by Deployment?
├── YES → kubectl edit deployment <name> → fix initContainers → Pods auto-recreated
└── NO (standalone Pod) → Delete & recreate (most init fields are immutable)
    1. kubectl get pod <name> -o yaml > fix.yaml
    2. vim fix.yaml  (fix the issue)
    3. kubectl delete pod <name>
    4. kubectl apply -f fix.yaml
```

## Common Fixes

### Fix: Wrong init image
```bash
# If Deployment:
kubectl edit deployment <name>
# Change initContainers[].image → save

# If standalone Pod:
kubectl get pod <name> -o yaml > fix.yaml
# Edit image in fix.yaml
kubectl delete pod <name> && kubectl apply -f fix.yaml
```

### Fix: Init command that never exits
```bash
# Change command from infinite to finite:
# BEFORE: command: ['tail', '-f', '/dev/null']
# AFTER:  command: ['sh', '-c', 'echo setup done']
```

### Fix: Init waiting for non-existent Service
```bash
# Option A: Create the Service
kubectl create svc clusterip my-svc --tcp=80:80

# Option B: Fix the service name in init command
kubectl edit deployment <name>
# Change the nslookup target
```

### Fix: Missing ConfigMap referenced by init
```bash
kubectl create configmap <name> --from-literal=key=value
# Pod should auto-recover (kubelet retries)
```

## Validate Fix
```bash
# Confirm init completed
kubectl get pod <name>
# STATUS should progress: Init:0/1 → Init:1/1 → PodInitializing → Running
# READY should show n/n for main containers

# Verify init ran correctly
kubectl logs <name> -c <init-name>
# Should show completion message / expected output

# Verify main container got init's data
kubectl exec <name> -c <main-container> -- cat /path/to/data
```

## Confirm Recovery
```bash
# Watch for stable state
watch kubectl get pod <name>
# RESTARTS should stabilize (not keep increasing)
# STATUS should stay at Running

# No warning events
kubectl describe pod <name> | grep -i warning
# Should return nothing or only old events
```

# File: init-demo.yaml

kind: Pod
metadata:
  name: init-demo
  labels:
    app: init-demo
spec:
  initContainers:
  - name: init-svc-check
    image: busybox:1.36
    command:
    - 'sh'
    - '-c'
    - 'echo "Init 1: Checking DNS..." && nslookup kubernetes.default && echo "DNS works!"'

  - name: init-data-writer
    image: busybox:1.36
    command:
    - 'sh'
    - '-c'
    - 'echo "<h1>Page generated by init container at $(date)</h1>" > /work/index.html'

    - name: web-content
      mountPath: /work

  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:

  - name: web-content

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] Init containers run BEFORE main containers, sequentially, must exit 0
- [ ] `initContainers[]` is a Pod-level field (same level as `containers[]`)
- [ ] Init containers share volumes with main containers (`emptyDir`)
- [ ] Init containers can have different images than main containers
- [ ] `command` overrides ENTRYPOINT, `args` overrides CMD (same as main containers)
- [ ] Init containers do NOT support probes (liveness/readiness/startup)
- [ ] Resource handling: init container resources are considered for scheduling but separately from main containers — **effective Pod resources = max(sum(main containers), max(init containers))** for each resource type

### What to Focus On
- **Creating** Pod with init containers from scratch YAML
- **Debugging** Pod stuck in `Init:0/N`, `Init:CrashLoopBackOff`, `Init:Error`
- **Reading** `kubectl describe pod` Init Containers section
- **Using** `kubectl logs <pod> -c <init-name>` (the `-c` flag!)
- **Fixing** common init failures (wrong image, bad command, missing dependencies)
- **Volume sharing** between init and main containers

### What to Practice Repeatedly
- Write a Pod with 2 init containers + 1 main container from memory
- Make init wait for a service (`until nslookup ...`), then create the service
- Debug `Init:CrashLoopBackOff` by reading logs
- Debug `Init:0/1` stuck by checking if command exits
- Fix init image, fix init command, fix volume mismatch — all under 2 minutes

### What to Ignore (Not CKA Relevant)
- How kubelet internally orchestrates init container sequencing
- Init container restart policy internals (follows Pod's restartPolicy)
- Init container resource calculation algorithm details
- Sidecar container KEP (Kubernetes Enhancement Proposal) internals
- Init container lifecycle hooks (not commonly used)

### 5 Quiz Questions

**Q1:** A Pod has 3 init containers. The second one fails with exit code 1. What happens to the third init and main containers?
> **A:** Third init never runs. Main containers never start. Pod status shows `Init:1/3` with `CrashLoopBackOff`. The failed second init is retried.

**Q2:** You run `kubectl logs mypod` on a Pod with init containers and 2 main containers. What happens?
> **A:** Error — you must specify `-c <container-name>` because there are multiple containers. Use `kubectl logs mypod -c <init-name>` for init or `kubectl logs mypod -c <main-name>` for main.

**Q3:** An init container writes a file to `/setup/config.json`. The main container needs it at `/app/config.json`. How do you achieve this?
> **A:** Both mount the same `emptyDir` volume. Init mounts it at `/setup`, writes `config.json`. Main mounts the same volume at `/app`. Both `volumeMounts[].name` must match the same `volumes[].name`.

**Q4:** Can you add a liveness probe to an init container?
> **A:** No. Init containers do not support liveness, readiness, or startup probes. They are run-to-completion; their success is determined solely by exit code 0.

**Q5:** A Pod shows `Init:0/1` for 10 minutes. `kubectl describe` shows init container State: Running. There are no events or errors. What's likely wrong?
> **A:** The init container's command never exits (e.g., `sleep infinity`, `tail -f`, or an infinite polling loop). The fix is to ensure the command has an exit condition.

# 1️⃣3️⃣ Component Interaction Map

```
┌──────────────────────────────────────────────────────────────────┐
│                    INIT CONTAINER INTERACTIONS                     │
│                                                                   │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                   │
│  │  Init 1  │───→│  Init 2  │───→│  Main    │                   │
│  │ (exit 0) │    │ (exit 0) │    │ Container│                   │
│  └────┬─────┘    └────┬─────┘    └────┬─────┘                   │
│       │               │               │                          │
│       └───────┬───────┘               │                          │
│               ▼                       ▼                          │
│         Shared Volume (emptyDir)                                 │
│         Shared Network Namespace                                 │
│         Shared ServiceAccount                                    │
└──────────────────────────────┬───────────────────────────────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         ▼                     ▼                     ▼
   ┌──────────┐         ┌──────────┐         ┌──────────────┐
   │ kubelet  │         │ Scheduler│         │ Controller   │
   │          │         │          │         │ Manager      │
   └──────────┘         └──────────┘         └──────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Init Fails |
|-----------|----------------|-----------------|
| **kubelet** | Runs init containers sequentially, waits for exit 0, then starts main | Retries failed init per restartPolicy. Reports init container status to API server. Main containers never started. |
| **Scheduler** | Considers max(init resources, sum(main resources)) for node placement | If init resources are very high + node capacity is low → Pod `Pending` |
| **Controller Manager** | Deployment controller tracks Pod readiness | If init keeps failing → Pod not Ready → Deployment rollout stalls. No new revision applied. |
| **etcd** | Stores Pod spec including initContainers and init container status | Init container state (Waiting/Running/Terminated) persisted in Pod status |
| **Services** | Route traffic to Pod IP when Pod is Ready | Pod never becomes Ready if init fails → Service has no endpoints for this Pod |
| **Network (CNI)** | Pod gets network namespace before ANY container (init or main) starts | Init containers CAN make network calls (DNS, HTTP) — same Pod IP as main |
| **Storage** | Volumes provisioned when Pod is scheduled, before init containers run | Init can read/write volumes. If PVC unbound → Pod `Pending` (before any init runs) |
| **Probes** | NOT applicable to init containers | — |

### What Breaks When Init Containers Fail?

```
Init failure cascade:
  Init fails
    → Remaining inits blocked
      → Main containers blocked
        → Pod status: Init:CrashLoopBackOff / Init:Error
          → Pod condition: Initialized=False, Ready=False
            → Service endpoints: this Pod NOT included
              → Deployment: Pod counts as unavailable
                → If this affects min replicas → Deployment rollout stalls
```
