# 🧠 Deployment — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **The primary workload controller in Kubernetes** — declaratively manages Pods at scale
- Creates and manages **ReplicaSets**, which in turn create and manage **Pods**
- Provides **rolling updates** — update Pods to a new version with zero downtime
- Provides **rollback** — revert to any previous revision instantly
- Maintains **desired state** — self-heals if Pods crash, are deleted, or nodes fail
- Records **revision history** — every change creates a new ReplicaSet as a revision

## Real-World Use Cases
- Deploy 5 replicas of a web application → update image → roll back if broken
- Blue/green-style deployments via `maxSurge` and `maxUnavailable`
- Canary deployments by pausing a rollout midway
- Scaling up/down for traffic spikes via `kubectl scale` or HPA
- Running stateless microservices (APIs, frontends, workers)

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────────┐
│                   DEPLOYMENT HIERARCHY                            │
│                                                                   │
│  User/CI ──→ kubectl apply ──→ Deployment (stored in etcd)       │
│                                     │                             │
│                          ┌──────────┴──────────┐                 │
│                          ▼                      ▼                │
│                   ReplicaSet-v1           ReplicaSet-v2          │
│                   (old revision)          (new revision)         │
│                    replicas: 0            replicas: 3            │
│                                          ┌────┼────┐            │
│                                          ▼    ▼    ▼            │
│                                       Pod-1 Pod-2 Pod-3         │
│                                                                   │
│  Controller Manager:                                              │
│    Deployment Controller → manages ReplicaSets                   │
│    ReplicaSet Controller → manages Pods                          │
│    Scheduler → places Pods on Nodes                              │
└──────────────────────────────────────────────────────────────────┘
```

### Why Deployment > ReplicaSet > Pod
| Layer | What It Adds |
|-------|-------------|
| **Pod** | Runs containers |
| **ReplicaSet** | Maintains N replicas of identical Pods (self-healing) |
| **Deployment** | Rolling updates, rollbacks, revision history, update strategy |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **ReplicaSet** — Deployment creates/manages RS; RS creates/manages Pods
- **Deployment Controller** (inside `kube-controller-manager`) — watches Deployments, reconciles RS
- **HPA** — Horizontal Pod Autoscaler targets Deployment for scaling
- **Service** — Routes traffic to Pods created by Deployment (via label selectors)
- **Ingress** — Exposes Service (which targets Deployment's Pods) externally

## What Resources It Interacts With
- **ReplicaSets** — Deployment creates RS objects as revisions
- **Pods** — Created by RS (Deployment → RS → Pods)
- **ConfigMaps / Secrets** — Referenced in Pod template; changes can trigger rollouts
- **PVCs** — If Pod template includes PVC, each Pod gets its own binding
- **ServiceAccount** — Pod template may reference a specific SA
- **ResourceQuota / LimitRange** — Constrains Deployment's Pod creation

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User → kubectl apply / CI pipeline → Deployment object in API server

DOWNSTREAM:
  Deployment controller creates → ReplicaSet(s)
  ReplicaSet controller creates → Pods
  Scheduler places → Pods on Nodes
  Service selects → Pods by labels
  HPA scales → Deployment replicas

ROLLING UPDATE FLOW:
  Deployment spec changed (e.g., new image)
    → Deployment controller creates NEW ReplicaSet
      → New RS scales UP (new Pods created)
      → Old RS scales DOWN (old Pods terminated)
        → Controlled by maxSurge + maxUnavailable
          → Zero-downtime update achieved
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid Deployment YAML
```yaml
apiVersion: apps/v1                    # ← apps/v1 (same as ReplicaSet)
kind: Deployment
metadata:
  name: my-deployment
  labels:
    app: myapp                         # ← Label on Deployment itself
spec:
  replicas: 3                          # ← Desired Pod count
  selector:                            # ← How Deployment finds its Pods
    matchLabels:
      app: myapp                       # ← MUST match template.metadata.labels
  strategy:                            # ← Update strategy
    type: RollingUpdate                # ← Or "Recreate"
    rollingUpdate:
      maxSurge: 1                      # ← Max extra Pods during update
      maxUnavailable: 1                # ← Max Pods down during update
  template:                            # ← Pod template (RS/Pod spec embedded)
    metadata:
      labels:
        app: myapp                     # ← MUST match selector.matchLabels
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
```

## Imperative Shortcut (Exam Speed)
```bash
# Create Deployment quickly (generates the YAML structure for you)
kubectl create deployment my-dep --image=nginx:1.25 --replicas=3

# Generate YAML without creating (for editing)
kubectl create deployment my-dep --image=nginx:1.25 --replicas=3 \
  --dry-run=client -o yaml > dep.yaml

# Quick update image
kubectl set image deployment/my-dep web=nginx:1.26

# Quick scale
kubectl scale deployment my-dep --replicas=5
```

## ⚠️ Common Indentation Traps

### Trap 1: Wrong apiVersion
```yaml
# ❌ WRONG
apiVersion: v1           # Deployment is NOT in v1
kind: Deployment

# ✅ CORRECT
apiVersion: apps/v1
kind: Deployment
```

### Trap 2: selector ≠ template labels (same trap as RS)
```yaml
# ❌ WRONG — selector says "web" but template says "myapp"
spec:
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: myapp         # ← MISMATCH — creation fails

# ✅ CORRECT
spec:
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp         # ← MATCH
```

### Trap 3: template.spec at wrong indent level
```yaml
# ❌ WRONG — spec is at Deployment level, not under template
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
  spec:                    # ← This is Deployment.spec, NOT template.spec!
    containers:
    - name: web

# ✅ CORRECT — spec indented under template
  template:
    metadata:
      labels:
        app: myapp
    spec:                  # ← Under template
      containers:
      - name: web
```

### Trap 4: strategy.type misspelled
```yaml
# ❌ WRONG
strategy:
  type: Rolling            # ← Must be "RollingUpdate" (exact casing)

# ✅ CORRECT
strategy:
  type: RollingUpdate      # or "Recreate"
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: apps/v1` | Required API version | Deployment not recognized | 🔴 |
| `spec.replicas` | Desired Pod count | Wrong scale; 0 = no Pods | 🔴 |
| `spec.selector.matchLabels` | Links Deployment → RS → Pods | Must match template labels — validation error | 🔴 |
| `spec.template.metadata.labels` | Labels on created Pods | Must match selector — validation error | 🔴 |
| `spec.template.spec.containers[].image` | Container image | ImagePullBackOff → rollout stuck | 🔴 |
| `spec.strategy.type` | `RollingUpdate` or `Recreate` | Recreate = downtime; RollingUpdate = zero downtime | 🔴 |
| `spec.strategy.rollingUpdate.maxSurge` | Max extra Pods during update | Too high = resource spike; too low = very slow rollout | 🟡 |
| `spec.strategy.rollingUpdate.maxUnavailable` | Max Pods down during update | Too high = service degradation; 0 = must have surge | 🟡 |
| `spec.revisionHistoryLimit` | How many old RS to keep | Low value = can't rollback to old revisions | 🟡 |
| `spec.template.spec.containers[].resources` | CPU/memory requests & limits | Pending or OOMKilled Pods | 🟡 |

## Key Field Relationships
```
Deployment.spec.selector.matchLabels
    ↕ MUST MATCH
Deployment.spec.template.metadata.labels
    ↕ Applied to
Pod.metadata.labels
    ↕ Matched by
Service.spec.selector
    (Traffic routing: Service → Pods created by Deployment)

spec.replicas → ReplicaSet.spec.replicas → actual Pod count
spec.strategy → controls HOW old Pods → new Pods transition
spec.template change → triggers NEW ReplicaSet creation → rolling update starts
```

## Strategy Types Explained
```
RollingUpdate (default):
  Old Pods gradually replaced by new Pods
  maxSurge: 25%      → Can create 25% extra Pods temporarily
  maxUnavailable: 25% → Up to 25% Pods can be down during update
  Result: Zero-downtime deployments ✅

Recreate:
  ALL old Pods killed FIRST → then ALL new Pods created
  Gap in service! Downtime while new Pods start.
  Use when: app can't run 2 versions simultaneously (DB schema conflict)
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `apiVersion: apps/v1`?
- `replicas` count correct?
- `selector.matchLabels` matches `template.metadata.labels`?
- Container `image` name and tag correct?
- `strategy.type` — RollingUpdate or Recreate?
- Template `spec` properly indented under `template`?

## 🟢 Usually Irrelevant for CKA
- `metadata.annotations` (unless recording rollout cause)
- `spec.progressDeadlineSeconds`
- `spec.minReadySeconds` (rarely tested)
- `metadata.generation` / `status.observedGeneration`
- `spec.paused` (unless pause/resume question)

## 🔴 Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| `apiVersion: v1` | Wrong API group for Deployment |
| `selector.matchLabels ≠ template.metadata.labels` | Validation error |
| `strategy.type: Rolling` | Must be `RollingUpdate` (exact) |
| `replicas: 0` | No Pods (intentional or mistake?) |
| Wrong image tag | Rollout stuck in ImagePullBackOff |
| `maxSurge: 0` AND `maxUnavailable: 0` | Deadlock — can't create or remove any Pod during update |
| Missing `selector` | Validation error |
| `template.spec` at Deployment level instead of template level | Wrong containers spec |

## 📋 `kubectl describe deployment <name>`

### Where to Look FIRST
```
Name:                   my-deployment
Replicas:               3 desired | 3 updated | 3 total | 3 available | 0 unavailable
StrategyType:           RollingUpdate

Conditions:
  Type           Status  Reason
  ----           ------  ------
  Available      True    MinimumReplicasAvailable     ← ✅ Healthy
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `0 available` | No healthy Pods — major problem |
| `Progressing: False, Reason: ProgressDeadlineExceeded` | Rollout stuck — new Pods aren't becoming ready |
| `Available: False` | Not enough Pods available to serve traffic |
| `OldReplicaSets: my-dep-old (2/2)` + `NewReplicaSet: my-dep-new (1/3)` | Rolling update in progress or stuck |
| Events: `ScalingReplicaSet` with errors | RS-level creation problems |
| `unavailable: 3` | All Pods down |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Rollout Stuck — ImagePullBackOff
```
SYMPTOM:        kubectl rollout status → hangs
                kubectl get deploy → 0 available, or old + new RS both partly scaled
CAUSE:          New image name/tag doesn't exist
DESCRIBE:       Progressing: False, ProgressDeadlineExceeded
PODS:           New Pods in ImagePullBackOff; old Pods may still be running
FIX:            kubectl set image deployment/<name> <container>=<correct-image>
                OR: kubectl rollout undo deployment/<name>
EXAM SCENARIO:  "Deployment rollout is stuck. Fix it."
                → kubectl rollout status → stuck → check Pod status → fix image → or undo
```

## Pattern 2: Rollout Stuck — CrashLoopBackOff
```
SYMPTOM:        Rollout hangs; new Pods keep crashing
CAUSE:          New image has a bug, wrong command, missing config
DESCRIBE:       Progressing stuck; new RS has READY=0
PODS:           CrashLoopBackOff on new Pods
FIX:            kubectl rollout undo deployment/<name>   ← instant rollback
                Then fix the issue in YAML → apply again
EXAM TIP:       Undo FIRST to restore service, THEN debug
```

## Pattern 3: Selector Doesn't Match Template Labels
```
SYMPTOM:        Deployment creation fails
ERROR:          "selector does not match template labels"
BEHAVIOR:       Deployment NOT created — validation error
FIX:            Match selector.matchLabels with template.metadata.labels
```

## Pattern 4: All Pods Pending
```
SYMPTOM:        Deployment shows 0 available
                All Pods in Pending state
CAUSE:          Resource requests too high, no available nodes, taints, PVC unbound
DESCRIBE POD:   Events: FailedScheduling
FIX:            Reduce resources, add nodes, fix taints, bind PVCs
```

## Pattern 5: maxSurge=0 AND maxUnavailable=0 (Deadlock)
```
SYMPTOM:        Rollout never progresses
CAUSE:          Can't create extra Pods (maxSurge=0) AND can't remove existing (maxUnavailable=0)
                = Impossible to transition
BEHAVIOR:       Rollout stuck forever; no new Pods, no old Pods removed
FIX:            Set at least one of maxSurge or maxUnavailable to >= 1
```

## Pattern 6: Recreate Strategy Causes Downtime
```
SYMPTOM:        Service briefly returns errors during deployment
CAUSE:          strategy.type: Recreate → all old Pods killed before new ones start
BEHAVIOR:       Gap between old Pods terminating and new Pods becoming Ready
FIX:            Switch to strategy.type: RollingUpdate (if app supports it)
```

## Pattern 7: Rollback Needed But Revision History Cleared
```
SYMPTOM:        kubectl rollout undo → no older revision available
CAUSE:          revisionHistoryLimit: 0 (or very low) → old RS cleaned up
BEHAVIOR:       Can't rollback to old version
FIX:            Manually set the correct image and apply
                Increase revisionHistoryLimit for future
```

## Pattern 8: Wrong Namespace
```
SYMPTOM:        Deployment exists but Pods appear in wrong namespace
                Or: "deployment not found" when looking in expected namespace
CAUSE:          Deployed to wrong namespace (forgot -n flag)
FIX:            kubectl get deploy --all-namespaces | grep <name>
                Delete from wrong namespace, apply to correct one
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Selector ≠ Template Labels?
- **Behavior:** Deployment creation rejected — validation error
- Same as ReplicaSet: `selector does not match template labels`
- Zero Pods, zero RS created
- **Fix:** Match the labels exactly

## What Happens If Image Is Wrong?
- **Behavior:** Deployment creates new RS → new RS creates Pods → Pods get `ImagePullBackOff`
- Old RS may still have running Pods (rolling update pauses partway)
- Service still routes to old Pods (graceful degradation)
- Rollout shows as stuck/progressing
- **Fix:** `kubectl rollout undo` or fix the image

## What Happens If Resource Limits Are Wrong?
- **Behavior depends on what's wrong:**
  - Requests too high → Pods `Pending` → rollout stuck
  - Limits too low → Pods `OOMKilled` → CrashLoopBackOff → rollout stuck
  - Combined requests exceed node → some Pods Pending, some Running
- **Fix:** Edit Deployment → adjust resources → new rollout triggered automatically

## What Happens If Ports Mismatch?
- `containerPort` in Deployment template is **informational** — doesn't cause crash
- But if Service `targetPort` doesn't match the app's actual listening port → traffic fails
- Deployment itself doesn't validate ports against Services
- **Fix:** Ensure Service `targetPort` matches the port your app listens on

## What Happens If Dependencies Are Missing?
- **Missing ConfigMap/Secret** → Pod `CreateContainerConfigError` → rollout stuck
- **Missing PVC** → Pod `Pending` → rollout stuck
- **Missing ServiceAccount** → Pod creation error
- Old Pods from previous RS may still be running (service stays up)
- **Fix:** Create the missing resource → Pods will auto-recover

## What Happens If You Edit the Deployment?
- **Any change to `spec.template`** triggers a new rollout (new RS created)
- Changes to `spec.replicas` do NOT trigger rollout (just scales current RS)
- Changes to `metadata`, `strategy`, or `revisionHistoryLimit` do NOT trigger rollout
- **Rule:** Only Pod template changes → new revision

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: Deployment health
kubectl get deploy <name>
# NAME            READY   UP-TO-DATE   AVAILABLE   AGE
# my-deployment   3/3     3            3           10m    ← healthy ✅
# my-deployment   1/3     2            1           2m     ← problem!
# Read: READY=actual/desired, UP-TO-DATE=using latest template, AVAILABLE=passing probes

# ⏱️ 10-20 sec: Check rollout status
kubectl rollout status deployment/<name>
# Stuck? → rollout in progress or failed

# ⏱️ 20-30 sec: Check ReplicaSets
kubectl get rs -l app=myapp
# Multiple RS? → rolling update in progress
# New RS with READY=0 → new Pods are failing
# Old RS still has Pods → old version still serving

# ⏱️ 30-45 sec: Check Pod status
kubectl get pods -l app=myapp
# What state? Pending? CrashLoop? ImagePull? Running?

# ⏱️ 45-55 sec: Drill into failing Pod
kubectl describe pod <pod> | tail -20
kubectl logs <pod> --previous

# ⏱️ 55-60 sec: Decide action
# Bad image? → rollout undo OR fix image
# Pending? → resource/scheduling issue
# CrashLoop? → app bug, undo to restore service
```

## Quick Diagnosis Table
```
DEPLOYMENT STATUS                      → LIKELY CAUSE
───────────────────────────────────────────────────────────
READY 0/3, UP-TO-DATE 0               → Selector/template mismatch, RS not created
READY 0/3, UP-TO-DATE 3               → All Pods failing (image, command, config)
READY 1/3, UP-TO-DATE 3               → 2 Pods failing (resources, scheduling)
READY 3/3, UP-TO-DATE 1               → Rollout stuck (1 new Pod ok, 2 still old)
READY 3/3, UP-TO-DATE 3, AVAILABLE 3  → Healthy ✅
READY 3/3, UP-TO-DATE 3, AVAILABLE 0  → Readiness probes failing
```

# 🔟 Fast Fix Strategy

## Rollout Commands (CKA Must-Know)

```bash
# Check rollout status
kubectl rollout status deployment/<name>

# View rollout history
kubectl rollout history deployment/<name>
kubectl rollout history deployment/<name> --revision=2

# UNDO (rollback to previous revision) — most important CKA command
kubectl rollout undo deployment/<name>

# Undo to specific revision
kubectl rollout undo deployment/<name> --to-revision=2

# Pause rollout (for canary-style)
kubectl rollout pause deployment/<name>

# Resume rollout
kubectl rollout resume deployment/<name>

# Restart all Pods (trigger fresh rollout with same spec)
kubectl rollout restart deployment/<name>
```

## Common Fixes

### Fix: Wrong image (rollout stuck)
```bash
# Option A: Rollback first, fix later
kubectl rollout undo deployment/<name>
# Service restored → then fix image properly

# Option B: Fix forward
kubectl set image deployment/<name> web=nginx:1.25-correct
# New rollout starts with correct image
```

### Fix: Update Deployment spec
```bash
# Interactive edit
kubectl edit deployment <name>
# Make changes → save → new rollout auto-triggers

# Patch specific field
kubectl patch deployment <name> -p '{"spec":{"replicas":5}}'

# Set image (most common fix)
kubectl set image deployment/<name> <container>=<image>:<tag>
```

### Fix: Pods Pending (scale down or fix resources)
```bash
# Lower resource requests
kubectl edit deployment <name>
# Change resources.requests → save → new rollout

# Or scale down
kubectl scale deployment <name> --replicas=1
```

### Fix: Record change cause (for rollout history)
```bash
kubectl set image deployment/my-dep web=nginx:1.26 --record
# ⚠️ --record is deprecated; use:
kubectl annotate deployment/my-dep \
  kubernetes.io/change-cause="Updated to nginx 1.26"
```

## Validate Fix
```bash
# Rollout complete
kubectl rollout status deployment/<name>
# "deployment successfully rolled out" ✅

# All Pods healthy
kubectl get deploy <name>
# READY=3/3, UP-TO-DATE=3, AVAILABLE=3 ✅

# RS clean
kubectl get rs -l app=myapp
# Only NEW RS has replicas; old RS scaled to 0

# Pods all running
kubectl get pods -l app=myapp
# All Running, 0 restarts
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Create Deployment
```bash
# Imperative (fast for exam)
kubectl create deployment dep-demo --image=nginx:1.25 --replicas=3

# OR declarative:
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: dep-demo
  labels:
    app: dep-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: dep-demo
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: dep-demo
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
EOF
```

### Validation Steps
```bash
# Deployment healthy
kubectl get deploy dep-demo
# READY 3/3, UP-TO-DATE 3, AVAILABLE 3

# RS created
kubectl get rs -l app=dep-demo
# 1 RS with DESIRED=3, CURRENT=3, READY=3

# Pods running
kubectl get pods -l app=dep-demo
# 3 Running Pods

# Ownership chain
kubectl describe pod <pod-name> | grep "Controlled By"
# Controlled By: ReplicaSet/dep-demo-xxxxxxxx

kubectl describe rs <rs-name> | grep "Controlled By"
# Controlled By: Deployment/dep-demo

# Test rolling update
kubectl set image deployment/dep-demo web=nginx:1.26
kubectl rollout status deployment/dep-demo
# "successfully rolled out"

kubectl get rs -l app=dep-demo
# 2 RS now — old (0 replicas) + new (3 replicas)

# Rollback
kubectl rollout undo deployment/dep-demo
kubectl rollout status deployment/dep-demo
# Back to nginx:1.25
```

## Step 2 — Break It Intentionally

### Break 1: Wrong Image (Stuck Rollout)
```bash
kubectl set image deployment/dep-demo web=nginx:NONEXISTENT
kubectl rollout status deployment/dep-demo
# Waiting... (stuck)
kubectl get pods -l app=dep-demo
# New Pods in ImagePullBackOff; some old Pods still running
# FIX:
kubectl rollout undo deployment/dep-demo
```

### Break 2: Command That Crashes
```bash
kubectl edit deployment dep-demo
# Add: command: ['exit', '1']
# Save → new rollout
kubectl get pods -l app=dep-demo
# New Pods: CrashLoopBackOff; old Pods may still serve
# FIX:
kubectl rollout undo deployment/dep-demo
```

### Break 3: Selector ≠ Template Labels
```bash
# Edit dep.yaml to mismatch labels → apply
# Error: selector does not match template labels

```

### Break 4: Excessive Resources

### What to Ignore (Not CKA Relevant)

### 5 Quiz Questions

**Q1:** You run `kubectl set image deployment/myapp web=nginx:1.26`. What happens internally?
> **A:** Deployment controller creates a NEW ReplicaSet with image nginx:1.26. New RS scales up, old RS scales down (per maxSurge/maxUnavailable). Old RS is kept (scaled to 0) for rollback.

**Q2:** A Deployment shows `READY 3/3` and `UP-TO-DATE 1`. What does this mean?
> **A:** 3 Pods are running and ready, but only 1 is using the latest template. A rolling update is in progress — 2 Pods still run the old version. Or the rollout is stuck.

**Q3:** How do you rollback to revision 2 of a Deployment?
> **A:** `kubectl rollout undo deployment/<name> --to-revision=2`

**Q4:** What's the difference between strategy `RollingUpdate` and `Recreate`?
> **A:** RollingUpdate gradually replaces Pods (zero downtime). Recreate kills ALL old Pods first, then creates ALL new Pods (causes downtime). Use Recreate only when the app can't run two versions simultaneously.

**Q5:** You edit a Deployment and only change `spec.replicas` from 3 to 5. Does this create a new ReplicaSet?
> **A:** No. Only changes to `spec.template` trigger a new RS (new revision). Changing replicas just scales the existing RS.

# 1️⃣3️⃣ Component Interaction Map

```
┌──────────────────────────────────────────────────────────────────┐
│                    DEPLOYMENT INTERACTIONS                         │
│                                                                   │
│  User                                                             │
│    │ kubectl apply / set image / scale                           │
│    ▼                                                              │
│  ┌──────────┐   creates/manages   ┌──────────────┐              │
│  │Deployment│ ──────────────────→  │  ReplicaSet  │              │
│  └──────────┘   (multiple RS      └──────┬───────┘              │
│       │          for revisions)           │ creates Pods         │
│       │                          ┌───────┼───────┐              │
│       │                          ▼       ▼       ▼              │
│       │                       Pod-1   Pod-2   Pod-3             │
│       │                                                          │
│       │    ┌───────────────────────────────────────┐             │
│       └───→│ Service (selects Pods by same labels) │             │
│            └───────────────────────────────────────┘             │
└──────────────────────────────────────────────────────────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Things Go Wrong |
|-----------|----------------|---------------------|
| **Controller Manager** | Deployment controller creates/manages RS. RS controller creates/manages Pods. | Controller down → no new RS created, no rollout, no self-healing. |
| **Scheduler** | Places Deployment's Pods on suitable nodes. | Insufficient resources → new Pods Pending → rollout stuck. |
| **etcd** | Stores Deployment, RS, Pod specs and statuses. | etcd down → no state changes possible. |
| **Service** | Selects Pods by labels (same labels as Deployment selector). | During rolling update: Service routes to mix of old + new Pods. If new Pods not Ready → fewer endpoints. |
| **HPA** | Targets Deployment for autoscaling. Adjusts replicas. | HPA misconfigured → constant scaling up/down (thrashing). |
| **Ingress** | Exposes the Service backing Deployment's Pods. | If Service has no endpoints (all Pods failing) → 502/503 errors. |
| **Network** | Each Pod gets its own IP. Service provides stable VIP. | CNI failure → Pods get no IP → can't be added to Service endpoints. |
| **Storage** | If template uses PVC: each Pod binding to PVCs. | PVC unbound → Pods Pending → rollout stuck. |

### Failure Cascade
```
Wrong image deployed
  → New RS created with bad image
    → New Pods: ImagePullBackOff
      → maxUnavailable controls how many old Pods are removed
        → If maxUnavailable > 0: some old Pods terminated during rollout
          → Service loses some backends
            → Partial service degradation
        → If maxUnavailable = 0: old Pods all stay (no degradation)
          → Rollout stuck but service stable ✅

Controller Manager down
  → Deployment controller not running
    → Deployment spec changes ignored
      → No rolling updates, no scale, no self-healing
        → Existing Pods keep running but no reconciliation
```
