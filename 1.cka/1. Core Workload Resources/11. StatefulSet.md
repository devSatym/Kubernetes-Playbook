# 🧠 StatefulSet — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Manages stateful applications** — Pods that need stable identity, persistent storage, and ordered operations
- Guarantees **stable, unique network identifiers** — each Pod gets a predictable hostname (`<sts-name>-0`, `<sts-name>-1`, …)
- Guarantees **stable persistent storage** — each Pod gets its own PVC that survives Pod restarts and rescheduling
- Guarantees **ordered, graceful deployment and scaling** — Pods created in order 0→1→2; terminated in reverse 2→1→0
- Guarantees **ordered, automated rolling updates** — updates proceed in reverse ordinal order by default
- Unlike Deployment: **Pods are NOT interchangeable** — each has a sticky identity

## Real-World Use Cases
- **Databases:** MySQL, PostgreSQL, MongoDB replicas — each needs its own data volume
- **Distributed systems:** Zookeeper, etcd, Kafka — nodes must know each other by stable DNS names
- **Caching clusters:** Redis Sentinel — requires stable identity for leader election
- **Search engines:** Elasticsearch — each node has its own data directory
- **Any app where Pod identity matters** — configuration differs per replica

## Where It Fits in Cluster Architecture
```
┌───────────────────────────────────────────────────────────────────────┐
│                    STATEFULSET HIERARCHY                              │
│                                                                       │
│  User/CI ──→ kubectl apply ──→ StatefulSet (stored in etcd)          │
│                                     │                                 │
│                    ┌────────────────┼────────────────┐               │
│                    ▼                ▼                ▼               │
│               web-0             web-1             web-2              │
│           (ordinal 0)       (ordinal 1)       (ordinal 2)           │
│               │                  │                │                  │
│           PVC-web-0          PVC-web-1         PVC-web-2            │
│             ↓                  ↓                 ↓                  │
│           PV-0               PV-1              PV-2                 │
│                                                                       │
│  Headless Service (clusterIP: None):                                 │
│    web-0.nginx.default.svc.cluster.local                             │
│    web-1.nginx.default.svc.cluster.local                             │
│    web-2.nginx.default.svc.cluster.local                             │
│                                                                       │
│  Controller Manager:                                                  │
│    StatefulSet Controller → manages Pods sequentially                │
│    Scheduler → places Pods on Nodes                                  │
│    PV Controller → binds PVCs to PVs                                │
└───────────────────────────────────────────────────────────────────────┘
```

### Why StatefulSet ≠ Deployment
| Feature | Deployment | StatefulSet |
|---------|-----------|-------------|
| Pod identity | Random hash suffix | Stable ordinal index (0, 1, 2…) |
| Pod hostname | Random | Predictable (`<sts>-<ordinal>`) |
| Storage | Shared or no PVC | Per-Pod PVC via `volumeClaimTemplates` |
| Scaling order | Parallel (any order) | Sequential (ordered) |
| DNS | Via Service selector | Per-Pod DNS via Headless Service |
| Rolling update | Old RS → New RS | In-place, reverse ordinal order |
| Use case | Stateless apps | Stateful apps (DBs, clusters) |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **StatefulSet Controller** (inside `kube-controller-manager`) — watches and reconciles StatefulSets
- **Headless Service** — REQUIRED; provides per-Pod DNS records for stable network identity
- **PersistentVolumeClaims** — created from `volumeClaimTemplates`; one PVC per Pod per template
- **PersistentVolumes** — bound to PVCs (static or dynamic via StorageClass)

## What Resources It Interacts With
- **Headless Service** (`clusterIP: None`) — MANDATORY companion resource
- **PVCs / PVs** — stable storage per Pod
- **StorageClass** — for dynamic PV provisioning
- **ConfigMaps / Secrets** — application configuration
- **ServiceAccount** — Pod identity for RBAC
- **ResourceQuota / LimitRange** — constrains Pod creation

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User → kubectl apply → StatefulSet object → API Server → etcd

DOWNSTREAM:
  StatefulSet controller creates → Pods (sequentially, ordered)
  Each Pod gets → PVC(s) from volumeClaimTemplates
  PVC binds to → PV (static or dynamic via StorageClass)
  Headless Service creates → DNS A records per Pod
  Scheduler places → Pods on Nodes

CRITICAL DEPENDENCY:
  Headless Service MUST exist BEFORE StatefulSet is created
  Without it → Pods have no stable DNS identity
  
  StorageClass MUST exist (if using dynamic provisioning)
  Without it → PVCs stay Pending → Pods stay Pending

DNS RESOLUTION:
  <pod-name>.<headless-svc>.<namespace>.svc.cluster.local
  Example: web-0.nginx.default.svc.cluster.local
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid StatefulSet YAML
```yaml
apiVersion: apps/v1                          # ← Same as Deployment
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: "nginx"                       # ← REQUIRED — must match Headless Service name
  replicas: 3                                # ← Desired Pod count
  selector:                                  # ← How STS finds its Pods
    matchLabels:
      app: nginx                             # ← MUST match template.metadata.labels
  template:                                  # ← Pod template
    metadata:
      labels:
        app: nginx                           # ← MUST match selector.matchLabels
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
          name: web
        volumeMounts:
        - name: www                          # ← Must match volumeClaimTemplates name
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:                      # ← Per-Pod PVC template
  - metadata:
      name: www                              # ← Referenced by volumeMounts
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: "standard"           # ← Must exist in cluster
      resources:
        requests:
          storage: 1Gi
---
# REQUIRED Headless Service
apiVersion: v1
kind: Service
metadata:
  name: nginx                               # ← Must match spec.serviceName above
  labels:
    app: nginx
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None                            # ← Makes it a Headless Service
  selector:
    app: nginx                               # ← Matches StatefulSet Pod labels
```

## Imperative Shortcut (Exam Speed)
```bash
# ⚠️ No direct imperative command for StatefulSet!
# Must use declarative YAML — generate from docs or exam reference

# Quick way: generate Deployment YAML, then convert
kubectl create deployment web --image=nginx:1.25 --replicas=3 \
  --dry-run=client -o yaml > sts.yaml
# Then manually change:
#   kind: Deployment → kind: StatefulSet
#   Add spec.serviceName
#   Add volumeClaimTemplates
#   Remove strategy (STS uses updateStrategy instead)
#   Create Headless Service separately
```

## ⚠️ Common Indentation Traps

### Trap 1: Missing serviceName
```yaml
# ❌ WRONG — serviceName is REQUIRED for StatefulSet
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template: ...

# ✅ CORRECT
spec:
  serviceName: "nginx"        # ← MANDATORY
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template: ...
```

### Trap 2: volumeClaimTemplates at wrong indent level
```yaml
# ❌ WRONG — volumeClaimTemplates inside template
spec:
  template:
    spec:
      containers: ...
      volumeClaimTemplates:     # ← WRONG — this is not a Pod field!
      - metadata:
          name: www

# ✅ CORRECT — volumeClaimTemplates at spec level (sibling of template)
spec:
  template:
    spec:
      containers: ...
  volumeClaimTemplates:          # ← At spec level, NOT under template
  - metadata:
      name: www
```

### Trap 3: volumeMount name ≠ volumeClaimTemplates name
```yaml
# ❌ WRONG — names don't match
spec:
  template:
    spec:
      containers:
      - volumeMounts:
        - name: data              # ← "data"
  volumeClaimTemplates:
  - metadata:
      name: www                  # ← "www" — MISMATCH

# ✅ CORRECT
spec:
  template:
    spec:
      containers:
      - volumeMounts:
        - name: www              # ← Same name
  volumeClaimTemplates:
  - metadata:
      name: www                  # ← Same name
```

### Trap 4: selector ≠ template labels (same as Deployment)
```yaml
# ❌ WRONG
spec:
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: nginx               # ← MISMATCH

# ✅ CORRECT
spec:
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx               # ← MATCH
```

### Trap 5: Headless Service name ≠ serviceName
```yaml
# ❌ WRONG — Service name is "web-svc" but serviceName says "nginx"
# StatefulSet:
spec:
  serviceName: "nginx"

# Service:
metadata:
  name: web-svc                  # ← MISMATCH — DNS won't resolve

# ✅ CORRECT
spec:
  serviceName: "nginx"

metadata:
  name: nginx                    # ← MUST match
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: apps/v1` | Required API version | StatefulSet not recognized | 🔴 |
| `spec.serviceName` | Links to Headless Service for DNS | REQUIRED — creation fails without it | 🔴 |
| `spec.replicas` | Desired Pod count | Wrong scale; ordinals depend on this | 🔴 |
| `spec.selector.matchLabels` | Links STS → Pods | Must match template labels — validation error | 🔴 |
| `spec.template.metadata.labels` | Labels on created Pods | Must match selector — validation error | 🔴 |
| `spec.template.spec.containers[].image` | Container image | ImagePullBackOff → Pod stuck | 🔴 |
| `spec.volumeClaimTemplates` | Per-Pod PVC creation | No persistent storage; Pods may fail if app needs disk | 🔴 |
| `spec.volumeClaimTemplates[].spec.storageClassName` | Dynamic provisioning | Wrong/missing SC → PVC Pending → Pod Pending | 🔴 |
| `spec.volumeClaimTemplates[].spec.accessModes` | Storage access type | Wrong mode → PVC can't bind | 🟡 |
| `spec.updateStrategy.type` | `RollingUpdate` or `OnDelete` | OnDelete = manual Pod deletion; RollingUpdate = auto | 🟡 |
| `spec.updateStrategy.rollingUpdate.partition` | Canary: only ordinals ≥ partition update | Misunderstanding leads to partial updates | 🟡 |
| `spec.podManagementPolicy` | `OrderedReady` or `Parallel` | OrderedReady = strict order; Parallel = faster but unordered | 🟡 |
| `spec.persistentVolumeClaimRetentionPolicy` | What happens to PVCs on scale-down/delete | Default: PVCs retained (orphaned); may surprise you | 🟡 |

## Key Field Relationships
```
StatefulSet.spec.serviceName
    ↕ MUST MATCH
Headless Service.metadata.name (with clusterIP: None)
    ↕ Produces DNS
<pod-name>.<service-name>.<namespace>.svc.cluster.local

StatefulSet.spec.selector.matchLabels
    ↕ MUST MATCH
StatefulSet.spec.template.metadata.labels
    ↕ Applied to
Pod.metadata.labels
    ↕ Matched by
Headless Service.spec.selector

volumeClaimTemplates[].metadata.name
    ↕ MUST MATCH
template.spec.containers[].volumeMounts[].name

volumeClaimTemplates[].spec.storageClassName
    ↕ MUST EXIST
StorageClass object in cluster
    ↕ Provisions
PersistentVolume → bound to PVC → mounted in Pod

PVC naming convention:
  <volumeClaimTemplate-name>-<statefulset-name>-<ordinal>
  Example: www-web-0, www-web-1, www-web-2
```

## Update Strategy Types Explained
```
RollingUpdate (default):
  Pods updated in REVERSE ordinal order (2 → 1 → 0)
  Each Pod must be Running + Ready before next one is updated
  partition: N → only Pods with ordinal >= N are updated
    Example: partition: 2 → only Pod-2 gets updated (canary!)
  Result: Controlled stateful rollout ✅

OnDelete:
  Pods NOT updated automatically
  You must MANUALLY delete each Pod → STS recreates it with new spec
  Use when: app requires manual intervention between updates
  CKA tip: rarely tested, but know it exists
```

## Pod Management Policy Explained
```
OrderedReady (default):
  Pods created one at a time: 0 → 1 → 2
  Each must be Running + Ready before next is created
  Deletion: reverse order 2 → 1 → 0
  Use for: databases, anything needing ordered bootstrap

Parallel:
  All Pods launched simultaneously
  No ordering guarantees
  Use for: stateful apps that don't need ordered startup
  Faster scaling, but breaks apps expecting ordered init
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `apiVersion: apps/v1`?
- `kind: StatefulSet` (not Deployment)?
- `spec.serviceName` present and matches a Headless Service?
- `selector.matchLabels` matches `template.metadata.labels`?
- Container `image` name and tag correct?
- `volumeClaimTemplates` present if storage needed?
- `volumeMounts` name matches `volumeClaimTemplates` name?
- `storageClassName` correct?

## 🟢 Usually Irrelevant for CKA
- `metadata.annotations`
- `spec.revisionHistoryLimit`
- `spec.minReadySeconds`
- `spec.persistentVolumeClaimRetentionPolicy` (v1.27+ feature)
- `metadata.generation` / `status.observedGeneration`
- `spec.ordinals.start` (v1.27+ — non-zero ordinal start)

## 🔴 Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| Missing `serviceName` | StatefulSet creation fails |
| `serviceName` doesn't match any Headless Service | Pods have no stable DNS |
| Headless Service missing `clusterIP: None` | Not a Headless Service — no per-Pod DNS |
| `selector.matchLabels ≠ template.metadata.labels` | Validation error |
| Missing `volumeClaimTemplates` when app needs storage | Pods may crash (no disk) |
| `volumeMounts.name ≠ volumeClaimTemplates.name` | Volume not mounted — Pod fails |
| Wrong `storageClassName` | PVC Pending → Pod Pending |
| `accessModes` incompatible with storage backend | PVC can't bind |
| `updateStrategy.type: Recreate` | Invalid — STS uses `RollingUpdate` or `OnDelete` |
| `volumeClaimTemplates` indented under `template` | Wrong YAML structure |

## 📋 `kubectl describe statefulset <name>`

### Where to Look FIRST
```
Name:               web
Namespace:          default
Selector:           app=nginx
Replicas:           3 desired | 3 total
Update Strategy:    RollingUpdate
  Partition:        0
Pods Status:        3 Running / 0 Waiting / 0 Succeeded / 0 Failed
Pod Template:
  Labels:           app=nginx
  Containers:
   nginx:
    Image:     nginx:1.25
    Ports:     80/TCP
    Mounts:    /usr/share/nginx/html from www (rw)
Volume Claims:
  Name:        www
  StorageClass: standard
  Labels:      <none>
  Annotations: <none>
  AccessModes: [ReadWriteOnce]
  Resources:
    Requests: 1Gi
Events:
  Normal  SuccessfulCreate  Created pod: web-0
  Normal  SuccessfulCreate  Created pod: web-1
  Normal  SuccessfulCreate  Created pod: web-2
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `Pods Status: 0 Running / 3 Waiting` | All Pods stuck — PVC or scheduling issue |
| `Pods Status: 1 Running / 2 Waiting` | Pod-1 waiting for Pod-0 (OrderedReady) or PVC issue |
| Events: `FailedCreate` | Pod creation failed — check PVC binding, resources |
| No events after creation | Controller not processing — check controller-manager |
| `Partition: 2` during update | Only ordinal ≥ 2 will update (intentional canary or mistake) |
| Volume Claims show unknown StorageClass | PVCs won't bind — Pods Pending |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: PVC Pending → Pod Pending
```
SYMPTOM:        Pod stuck in Pending
                kubectl get pvc → PVC is Pending
CAUSE:          StorageClass doesn't exist / no PV available / wrong accessModes
DESCRIBE PVC:   Events: "no persistent volumes available for this claim"
                Events: "storageclass.storage.k8s.io 'wrong-class' not found"
FIX:            Create correct StorageClass
                OR: create PV that matches PVC requirements
                OR: fix storageClassName in volumeClaimTemplates
EXAM SCENARIO:  "StatefulSet Pods are Pending. Fix it."
                → check PVC → check StorageClass → fix provisioning
```

## Pattern 2: Headless Service Missing or Misconfigured
```
SYMPTOM:        StatefulSet creates Pods, but Pods can't discover each other
                Application reports "can't connect to peer nodes"
CAUSE:          Headless Service doesn't exist, or name doesn't match serviceName
                OR: Service has clusterIP set (not None)
                OR: Service selector doesn't match Pod labels
DNS BEHAVIOR:   No DNS records created → nslookup fails within Pods
FIX:            Create Headless Service with:
                  name: matching spec.serviceName
                  clusterIP: None
                  selector: matching Pod labels
EXAM SCENARIO:  "StatefulSet Pods can't communicate. Fix it."
```

## Pattern 3: Ordered Startup Stuck at Pod-0
```
SYMPTOM:        Only web-0 exists; web-1 and web-2 never created
CAUSE:          Pod-0 is not Running + Ready (OrderedReady policy)
                Pod-0 might be CrashLoopBackOff, Pending, or not ready
BEHAVIOR:       STS controller waits for Pod N to be Ready before creating Pod N+1
FIX:            Fix whatever is wrong with Pod-0 first
                OR: if ordering doesn't matter, set podManagementPolicy: Parallel
EXAM SCENARIO:  "Only 1 out of 3 StatefulSet Pods is running."
```

## Pattern 4: ImagePullBackOff
```
SYMPTOM:        Pods in ImagePullBackOff; scaling / update stuck
CAUSE:          Wrong image name or tag; private registry without pull secret
DESCRIBE:       Events: "Failed to pull image..."
FIX:            kubectl edit statefulset <name> → fix image
                OR: add imagePullSecrets
NOTE:           Unlike Deployment, no "rollout undo" — must manually fix spec
```

## Pattern 5: PVC Orphaned After Scale-Down
```
SYMPTOM:        After scaling from 3→1, PVCs www-web-1 and www-web-2 still exist
CAUSE:          PVCs are NOT deleted on scale-down (by design for data safety)
BEHAVIOR:       Scale back up → old PVCs are re-attached to new Pods with same ordinal
                Data preserved ✅ but storage is consumed 💰
FIX:            Manually delete orphaned PVCs if data is no longer needed:
                kubectl delete pvc www-web-1 www-web-2
EXAM TIP:       Know that PVCs survive scale-down (often tested conceptually)
```

## Pattern 6: Rolling Update Stuck (Partition Misconfigured)
```
SYMPTOM:        Only some Pods updated; rest still on old version
CAUSE:          updateStrategy.rollingUpdate.partition is set > 0
                Only Pods with ordinal >= partition get updated
BEHAVIOR:       Intentional for canary — but accidental if partition left behind
FIX:            Set partition: 0 to update ALL Pods
                kubectl edit sts <name> → change partition to 0
```

## Pattern 7: Volume Mount Failure
```
SYMPTOM:        Pod stuck in ContainerCreating
                Events: "unable to attach or mount volumes"
CAUSE:          PV already attached to another node (RWO)
                OR: PV doesn't exist
                OR: filesystem corruption
FIX:            Check node affinity / PV node constraints
                Delete stuck Pod → STS recreates on correct node
                Check PV events and node attachment status
```

## Pattern 8: Wrong Namespace
```
SYMPTOM:        StatefulSet exists but Headless Service is in a different namespace
                DNS resolution fails across namespaces
CAUSE:          STS and Headless Service must be in the SAME namespace
FIX:            kubectl get sts,svc --all-namespaces | grep <name>
                Ensure both are in the same namespace
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Labels Mismatch?
- **selector ≠ template labels** → StatefulSet creation fails — validation error
- **Headless Service selector ≠ Pod labels** → Service exists but no endpoints; no DNS records for Pods
- **Behavior:** Pods are running but invisible to DNS — apps can't find each other
- **Fix:** Ensure `STS selector` = `template labels` = `Headless Service selector`

## What Happens If serviceName Is Wrong?
- **serviceName doesn't match any Service** → StatefulSet still creates Pods
- **But:** No DNS records for Pod hostnames — per-Pod DNS resolution fails
- **Application impact:** Clustered apps (DBs) can't discover peers
- **Fix:** Create/rename Headless Service to match `spec.serviceName`

## What Happens If Image Is Wrong?
- **Behavior:** STS creates Pods → Pods get `ImagePullBackOff`
- **OrderedReady:** If Pod-0 can't pull image → Pod-1 never created (stuck at scale 0)
- **Unlike Deployment:** No `rollout undo` command for StatefulSet!
- **Fix:** `kubectl edit sts <name>` → fix image → old Pods deleted and recreated with new image
- **Alternative:** `kubectl rollout undo sts/<name>` (works since k8s 1.24+)

## What Happens If StorageClass Is Wrong?
- **Behavior:** PVC created but stays `Pending` → Pod stays `Pending`
- **OrderedReady:** If Pod-0's PVC is Pending → no other Pods are created
- **Entire StatefulSet stuck at 0 running Pods**
- **Fix:** Create correct StorageClass, or edit PVC (may need to delete and recreate)
- **⚠️ Critical:** You CANNOT change `volumeClaimTemplates` on existing StatefulSet — must delete and recreate STS

## What Happens If Resource Limits Are Wrong?
- **Requests too high** → Pods `Pending` → FailedScheduling
- **Limits too low** → Pods `OOMKilled` → CrashLoopBackOff
- **OrderedReady impact:** Even one failing Pod blocks all subsequent Pods
- **Fix:** Edit StatefulSet → adjust resources → Pods recreated with new limits

## What Happens If Dependencies Are Missing?
- **Missing ConfigMap/Secret** → `CreateContainerConfigError` → Pod stuck
- **Missing PVC/PV** → Pod `Pending`
- **Missing Headless Service** → Pods created but NO DNS identity
- **OrderedReady amplification:** One blocked Pod blocks ALL subsequent ordinals
- **Fix:** Create missing dependency → Pod auto-recovers

## What Happens If You Delete a StatefulSet?
- **`kubectl delete sts <name>`** → All Pods deleted; PVCs are RETAINED
- **`kubectl delete sts <name> --cascade=orphan`** → StatefulSet deleted; Pods LEFT running (orphaned)
- **PVCs always survive StatefulSet deletion by default** — manual cleanup required
- **Re-create STS with same name** → Pods reconnect to existing PVCs (data preserved)

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: StatefulSet health
kubectl get sts <name>
# NAME   READY   AGE
# web    3/3     10m    ← healthy ✅
# web    1/3     2m     ← problem! Only 1 Pod running
# web    0/3     5m     ← all Pods failing

# ⏱️ 10-20 sec: Check Pods (ordered — check ordinals)
kubectl get pods -l app=nginx
# NAME    READY   STATUS              RESTARTS   AGE
# web-0   1/1     Running             0          10m   ← OK
# web-1   0/1     Pending             0          5m    ← Stuck
# web-2                                                ← Never created (OrderedReady)
# Read: If web-1 is stuck, web-2 won't even be attempted

# ⏱️ 20-30 sec: Check PVCs
kubectl get pvc -l app=nginx
# NAME        STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# www-web-0   Bound     pv-0     1Gi        RWO            standard       10m  ← OK
# www-web-1   Pending                                                     5m   ← PROBLEM

# ⏱️ 30-40 sec: Check Headless Service
kubectl get svc <service-name>
# NAME    TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
# nginx   ClusterIP   None         <none>        80/TCP    10m
# Confirm: CLUSTER-IP = None ← It IS headless ✅

# ⏱️ 40-50 sec: Drill into failing Pod/PVC
kubectl describe pod web-1 | tail -20
kubectl describe pvc www-web-1 | tail -10

# ⏱️ 50-60 sec: Decide action
# PVC Pending? → StorageClass / PV issue → fix provisioning
# Pod CrashLoop? → Image or config issue → fix spec
# Pod Pending? → Scheduling issue → check resources/taints
# Only Pod-0 exists? → Pod-0 not Ready → fix Pod-0 first
```

## Quick Diagnosis Table
```
STATEFULSET STATUS                    → LIKELY CAUSE
────────────────────────────────────────────────────────────
READY 0/3                             → Pod-0 failing (blocks all); PVC/Image/Config issue
READY 1/3                             → Pod-0 OK but Pod-1 stuck; PVC or scheduling issue
READY 2/3                             → Pod-2 failing; check Pod-2 PVC/resources
READY 3/3                             → Healthy ✅
Pods Running but app can't connect    → Headless Service missing / wrong serviceName
All PVCs Pending                      → StorageClass missing/wrong
Pods in ContainerCreating             → Volume mount failure
Pod-0 CrashLoopBackOff               → Image/config bug; blocks entire STS
```

# 🔟 Fast Fix Strategy

## Common Fixes

### Fix: Wrong Image
```bash
# Option A: Edit directly
kubectl edit sts <name>
# Fix image → save → Pods updated in reverse order (2→1→0)

# Option B: Patch
kubectl patch sts <name> --type='json' \
  -p='[{"op": "replace", "path": "/spec/template/spec/containers/0/image", "value":"nginx:1.25"}]'

# Option C: Rollout undo (K8s 1.24+)
kubectl rollout undo sts/<name>

# Option D: Set image
kubectl set image sts/<name> nginx=nginx:1.25
```

### Fix: PVC Pending (StorageClass issue)
```bash
# Check what StorageClasses exist
kubectl get sc

# Check what the PVC wants
kubectl describe pvc www-web-0

# If StorageClass doesn't exist → create it
# If using wrong name → must delete STS + PVCs and recreate
# ⚠️ Cannot edit volumeClaimTemplates on existing STS!

# Nuclear option: delete STS (keep PVCs if salvageable)
kubectl delete sts <name> --cascade=orphan
# Fix YAML → reapply
kubectl apply -f fixed-sts.yaml
```

### Fix: Headless Service Missing
```bash
# Create quickly
kubectl create service clusterip nginx --clusterip="None" --tcp=80:80
# Then verify
kubectl get svc nginx
# CLUSTER-IP should be "None"

# OR apply YAML:
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  clusterIP: None
  selector:
    app: nginx
  ports:
  - port: 80
    name: web
EOF
```

### Fix: Pod Stuck — Delete for Recreation
```bash
# StatefulSet will auto-recreate deleted Pods with same identity
kubectl delete pod web-1
# STS controller creates new web-1 → attaches to existing PVC www-web-1 → data preserved

# Force delete (if Pod stuck in Terminating)
kubectl delete pod web-1 --force --grace-period=0
```

### Fix: Update Partition Issues
```bash
# Check current partition
kubectl get sts <name> -o jsonpath='{.spec.updateStrategy.rollingUpdate.partition}'

# Reset partition to update ALL Pods
kubectl patch sts <name> -p '{"spec":{"updateStrategy":{"rollingUpdate":{"partition":0}}}}'
```

## Validate Fix
```bash
# All Pods running with correct ordinals
kubectl get sts <name>
# READY 3/3 ✅

kubectl get pods -l app=nginx -o wide
# web-0, web-1, web-2 all Running

# PVCs all bound
kubectl get pvc -l app=nginx
# All STATUS: Bound ✅

# DNS working
kubectl exec web-0 -- nslookup web-1.nginx.default.svc.cluster.local
# Should resolve ✅

# Data persists
kubectl exec web-0 -- cat /usr/share/nginx/html/index.html
# Data intact ✅
```

## Step 1 — Create Working Example

### Create Headless Service + StatefulSet
```bash
cat <<'EOF' | kubectl apply -f -
---
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None
  selector:
    app: nginx
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: "nginx"
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
          name: web

        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
  - metadata:
      name: www

      accessModes: ["ReadWriteOnce"]
      storageClassName: "standard"
      resources:
        requests:
          storage: 1Gi
EOF
```

## 5 Quiz Questions to Test Understanding

### Q1: What is the DNS hostname for the second Pod of a StatefulSet named `db` with serviceName `db-svc` in namespace `prod`?
<details>
<summary>Answer</summary>
db-1.db-svc.prod.svc.cluster.local
</details>

### Q2: You scale a StatefulSet from 3 to 1 replicas. What happens to PVCs www-db-1 and www-db-2?
<details>
<summary>Answer</summary>
They are RETAINED (not deleted). You must manually delete them if not needed. If you scale back to 3, the old PVCs are re-attached.
</details>

### Q3: StatefulSet has 3 replicas but only web-0 is Running; web-1 is Pending and web-2 doesn't exist. Why?
<details>
<summary>Answer</summary>
Default podManagementPolicy is OrderedReady — each Pod must be Running+Ready before the next is created. web-1 is Pending (likely PVC issue), so web-2 is never attempted. Fix web-1 first.
</details>

### Q4: You update a StatefulSet image but only web-2 gets the new version. web-0 and web-1 remain on old. Why?
<details>
<summary>Answer</summary>
updateStrategy.rollingUpdate.partition is likely set to 2 — only ordinals >= partition are updated. Set partition: 0 to update all.
</details>

### Q5: Can you change volumeClaimTemplates on an existing StatefulSet?
<details>
<summary>Answer</summary>
No — volumeClaimTemplates is immutable. You must delete the StatefulSet (optionally with --cascade=orphan to preserve Pods) and recreate it with the new template. Existing PVCs are not affected.
</details>

# 1️⃣3️⃣ Component Interaction Map

## How StatefulSet Interacts with Components

### StatefulSet ↔ Pods
```
NORMAL:    STS creates Pods with ordinal names: web-0, web-1, web-2
           Pods created sequentially (OrderedReady) or in parallel
           Each Pod has stable hostname = Pod name

FAILURE:   Pod-0 crash → Pod-1 never created (OrderedReady)
           Pod deleted → STS recreates with SAME name + SAME PVC
           Pod rescheduled to different node → same identity, same PVC
```

### StatefulSet ↔ Headless Service
```
NORMAL:    Headless Service (clusterIP: None) creates per-Pod DNS records
           DNS: <pod-name>.<service-name>.<namespace>.svc.cluster.local
           Pods use DNS to discover each other

FAILURE:   Headless Service missing → Pods have no DNS identity
           Service selector wrong → no endpoints → DNS doesn't resolve
           Service name ≠ serviceName → STS works but DNS is broken
           Fix: create/fix Headless Service → DNS auto-recovers
```

### StatefulSet ↔ Storage Layer (PVC/PV/StorageClass)
```
NORMAL:    volumeClaimTemplates → PVC per Pod (named <template>-<sts>-<ordinal>)
           PVC binds to PV (dynamic via StorageClass or static)
           Pod mounts PVC → data persists

FAILURE:   StorageClass missing → PVC Pending → Pod Pending → STS stuck
           PV already bound to another PVC → PVC Pending
           RWO PV on different node → Pod ContainerCreating (can't attach)
           PV full → app write errors (Pod runs but app fails)
           Scale-down: PVCs RETAINED → re-attach on scale-up
           STS delete: PVCs RETAINED → manual cleanup needed
```

### StatefulSet ↔ Scheduler
```
NORMAL:    Scheduler places each Pod on suitable node
           Considers: resources, taints, affinity, PV node constraints

FAILURE:   No node has enough resources → Pod Pending (FailedScheduling)
           PV has node affinity → Pod must go to specific node
           Node full → Pod stuck → blocks subsequent ordinals (OrderedReady)
           Taint prevents scheduling → Pod Pending
```

### StatefulSet ↔ Controller Manager
```
NORMAL:    StatefulSet Controller (in kube-controller-manager):
             watches StatefulSet objects
             creates/deletes Pods sequentially
             manages rolling updates (reverse ordinal)
             creates PVCs from volumeClaimTemplates

FAILURE:   Controller Manager down → no reconciliation
           STS not progressing → check controller-manager Pod health
           kubectl describe sts → no events → controller not processing
```

### StatefulSet ↔ etcd
```
NORMAL:    StatefulSet spec stored in etcd via API server
           Pod identity and PVC references tracked in etcd
           All state changes persisted

FAILURE:   etcd down → API server read-only → no new Pods created
           etcd data loss → StatefulSet definition lost
           Running Pods continue (kubelet) but no reconciliation
```

### StatefulSet ↔ Network Layer
```
NORMAL:    Each Pod gets unique IP from CNI
           Headless Service creates DNS A records per Pod
           Pods can reach each other via DNS names
           SRV records available for port discovery

FAILURE:   CNI misconfigured → Pods can't get IPs → stuck in ContainerCreating
           CoreDNS down → DNS resolution fails → apps can't find peers
           Network policy blocks inter-Pod traffic → app connection failures
           Pod IP changes after reschedule → DNS still works (Headless Service updates)
```

### StatefulSet ↔ Nodes
```
NORMAL:    Pods distributed across nodes (scheduler decides)
           PV node affinity may constrain Pod placement

FAILURE:   Node failure → Pod marked for eviction
           StatefulSet waits for node timeout before rescheduling
           RWO PV on failed node → Pod can't be scheduled elsewhere until PV released
           Manual intervention may be needed: delete Pod → force PV detach → reschedule
```

## Interaction Diagram
```
┌───────────────────────────────────────────────────────────────┐
│                    COMPONENT INTERACTION MAP                   │
│                                                               │
│  etcd ←──→ API Server ←──→ Controller Manager                │
```

### Mental Model: StatefulSet = Deployment + Identity + Storage
```
Deployment:  Pods are cattle 🐄 — interchangeable, random names, shared storage
StatefulSet: Pods are pets 🐕 — unique names, unique storage, ordered lifecycle

Key differences to remember:
  ✅ Stable pod name (web-0, not web-5d7fb8c4-xk2nf)
  ✅ Stable DNS (web-0.svc-name.ns.svc.cluster.local)
  ✅ Stable storage (PVC survives Pod deletion)
  ✅ Ordered operations (create 0→1→2, delete 2→1→0)
  ✅ Requires Headless Service (no equivalent in Deployment)
  ❌ No "rollout undo" in older K8s (fixed in 1.24+)
  ❌ volumeClaimTemplates is immutable after creation
```
