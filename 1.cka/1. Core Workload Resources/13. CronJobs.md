# 🧠 CronJobs — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Runs Jobs on a recurring schedule** — the Kubernetes equivalent of Linux `cron`
- Creates **Job objects automatically** at specified intervals using cron syntax
- Each trigger creates a **new Job** → each Job creates **Pod(s)** → Pod runs to completion
- Manages **Job lifecycle** — controls concurrency, history retention, and missed schedules
- Provides **declarative scheduling** — schedule is part of the spec, not external config

## Real-World Use Cases
- **Database backups** — snapshot every night at 2 AM
- **Log rotation / cleanup** — delete old logs every 6 hours
- **Report generation** — weekly sales report every Monday 8 AM
- **Certificate renewal** — check and renew certificates daily
- **Health checks** — ping external services every 5 minutes
- **Data synchronization** — sync data from external API every hour
- **Cleanup tasks** — delete orphaned resources, expired tokens, stale PVCs

## Where It Fits in Cluster Architecture
```
┌───────────────────────────────────────────────────────────────────────┐
│                      CRONJOB HIERARCHY                               │
│                                                                       │
│  User ──→ kubectl apply ──→ CronJob (stored in etcd)                │
│                                 │                                     │
│                       CronJob Controller                              │
│                  (in kube-controller-manager)                        │
│                                 │                                     │
│           Checks schedule every ~10 seconds                          │
│                                 │                                     │
│              ┌──────────────────┼──────────────────┐                 │
│              ▼                  ▼                  ▼                 │
│          Job (Mon 2AM)      Job (Tue 2AM)     Job (Wed 2AM)         │
│              │                  │                  │                 │
│          Pod-xxxxx          Pod-yyyyy          Pod-zzzzz            │
│         (Completed)        (Completed)        (Running)             │
│                                                                       │
│  Three-level hierarchy:                                               │
│    CronJob → creates → Job → creates → Pod(s)                       │
│                                                                       │
│  CronJob manages:                                                     │
│    ├── Schedule: "0 2 * * *" (when to trigger)                       │
│    ├── ConcurrencyPolicy: Allow/Forbid/Replace                      │
│    ├── History: successfulJobsHistoryLimit / failedJobsHistoryLimit  │
│    └── Missed schedules: startingDeadlineSeconds                     │
└───────────────────────────────────────────────────────────────────────┘
```

### CronJob vs Job vs Deployment
| Feature | Deployment | Job | CronJob |
|---------|-----------|-----|---------|
| Runs | Forever | Once to completion | Repeatedly on schedule |
| Creates | ReplicaSet → Pods | Pods | Job → Pods |
| restartPolicy | Always | Never / OnFailure | Never / OnFailure |
| apiVersion | apps/v1 | batch/v1 | batch/v1 |
| Use case | Web servers | Batch tasks | Scheduled tasks |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **CronJob Controller** (inside `kube-controller-manager`) — watches CronJobs, creates Jobs on schedule
- **Job Controller** — manages Jobs created by CronJob (creates Pods, tracks completions)
- **Monitoring systems** — may alert if CronJob hasn't run or keeps failing

## What Resources It Interacts With
- **Jobs** — CronJob creates Job objects on each trigger
- **Pods** — Jobs create Pods (CronJob → Job → Pod)
- **ConfigMaps / Secrets** — referenced in Job Pod template
- **PVCs** — if scheduled task needs persistent data
- **ServiceAccount** — Pod identity for RBAC
- **ResourceQuota** — constrains Job/Pod creation counts

## Upstream & Downstream Dependencies
```
UPSTREAM:
  User → kubectl apply → CronJob object → API Server → etcd

DOWNSTREAM:
  CronJob Controller (checks every ~10s):
    Schedule matches current time?
      YES → Create new Job object
        → Job Controller creates Pod(s)
          → Pod runs → exits 0 (success) or non-zero (failure)
            → Job tracks completions/failures
              → CronJob tracks Job history

LIFECYCLE OF ONE TRIGGER:
  CronJob schedule fires
    → CronJob Controller creates Job (with owner reference to CronJob)
      → Job Controller creates Pod(s)
        → Pod completes (success/failure)
          → Job marked Complete/Failed
            → CronJob retains Job based on history limits
              → Old Jobs auto-deleted when history limit exceeded

CLEANUP CHAIN:
  CronJob deletes excess Jobs → Job deletion cascades → Pod deletion
  successfulJobsHistoryLimit: 3 → keep last 3 successful Jobs
  failedJobsHistoryLimit: 1    → keep last 1 failed Job
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid CronJob YAML
```yaml
apiVersion: batch/v1                           # ← batch/v1 (same as Job)
kind: CronJob
metadata:
  name: my-cronjob
spec:
  schedule: "*/5 * * * *"                      # ← REQUIRED — cron expression (every 5 min)
  concurrencyPolicy: Allow                     # ← Allow / Forbid / Replace
  successfulJobsHistoryLimit: 3                # ← Keep last 3 successful Jobs (default: 3)
  failedJobsHistoryLimit: 1                    # ← Keep last 1 failed Job (default: 1)
  startingDeadlineSeconds: 200                 # ← Max seconds late a Job can start
  suspend: false                               # ← true = pause scheduling
  jobTemplate:                                 # ← Template for the Job it creates
    spec:                                      # ← This is Job.spec
      completions: 1
      backoffLimit: 3
      activeDeadlineSeconds: 100
      template:                                # ← This is Pod template (Job.spec.template)
        spec:
          containers:
          - name: worker
            image: busybox:1.36
            command:
            - sh
            - -c
            - echo "Running scheduled task at $(date)"
          restartPolicy: OnFailure             # ← REQUIRED: Never or OnFailure
```

## Imperative Shortcut (Exam Speed)
```bash
# Create CronJob quickly
kubectl create cronjob my-cj --image=busybox:1.36 \
  --schedule="*/5 * * * *" -- echo "Hello from CronJob"

# Generate YAML without creating
kubectl create cronjob my-cj --image=busybox:1.36 \
  --schedule="*/5 * * * *" \
  --dry-run=client -o yaml -- echo "Hello" > cj.yaml

# Trigger CronJob immediately (create Job from CronJob)
kubectl create job manual-run --from=cronjob/my-cj
```

## Cron Schedule Syntax Quick Reference
```
┌───────────── minute (0 - 59)
│ ┌───────────── hour (0 - 23)
│ │ ┌───────────── day of month (1 - 31)
│ │ │ ┌───────────── month (1 - 12)
│ │ │ │ ┌───────────── day of week (0 - 6, Sun=0)
│ │ │ │ │
* * * * *

EXAMPLES (CKA must-know):
  "*/5 * * * *"    → every 5 minutes
  "0 * * * *"      → every hour (at minute 0)
  "0 2 * * *"      → daily at 2:00 AM
  "0 0 * * 0"      → weekly on Sunday at midnight
  "0 0 1 * *"      → monthly on the 1st at midnight
  "30 8 * * 1-5"   → weekdays at 8:30 AM
  "0 */6 * * *"    → every 6 hours
```

## ⚠️ Common Indentation Traps

### Trap 1: Wrong apiVersion
```yaml
# ❌ WRONG
apiVersion: apps/v1          # CronJobs are NOT in apps
kind: CronJob

# ❌ WRONG
apiVersion: batch/v1beta1    # Deprecated since K8s 1.21, removed in 1.25

# ✅ CORRECT
apiVersion: batch/v1
kind: CronJob
```

### Trap 2: schedule not quoted
```yaml
# ❌ RISKY — some YAML parsers misinterpret unquoted cron expressions
spec:
  schedule: */5 * * * *       # ← May fail parsing (the * is YAML special char)

# ✅ CORRECT — always quote the schedule
spec:
  schedule: "*/5 * * * *"     # ← Quoted string
```

### Trap 3: Missing jobTemplate nesting (most common!)
```yaml
# ❌ WRONG — Pod template directly under CronJob spec
spec:
  schedule: "*/5 * * * *"
  template:                    # ← WRONG — CronJob needs jobTemplate, not template
    spec:
      containers: ...

# ✅ CORRECT — jobTemplate → spec → template → spec
spec:
  schedule: "*/5 * * * *"
  jobTemplate:                 # ← CronJob level
    spec:                      # ← Job level
      template:                # ← Job's Pod template level
        spec:                  # ← Pod spec level
          containers: ...
          restartPolicy: Never
```

### Trap 4: restartPolicy missing or Always
```yaml
# ❌ WRONG — same trap as Job
spec:
  jobTemplate:
    spec:
      template:
        spec:
          containers: ...
          restartPolicy: Always   # ← INVALID for Jobs (and CronJobs)

# ✅ CORRECT
spec:
  jobTemplate:
    spec:
      template:
        spec:
          containers: ...
          restartPolicy: Never    # ← or OnFailure
```

### Trap 5: Indentation depth — 4 spec levels!
```yaml
# CronJob has the DEEPEST nesting of any Kubernetes object
# Understanding the 4 levels is critical:

apiVersion: batch/v1
kind: CronJob
metadata:
  name: my-cj
spec:                            # ← Level 1: CronJob.spec (schedule, concurrency)
  schedule: "0 * * * *"
  jobTemplate:
    spec:                        # ← Level 2: Job.spec (completions, backoffLimit)
      backoffLimit: 3
      template:
        metadata:
          labels:
            app: worker
        spec:                    # ← Level 3: Pod.spec (containers, restartPolicy)
          containers:
          - name: worker
            image: busybox
            command: ["echo", "hi"]
          restartPolicy: Never

# Mnemonic: CronJob.spec → jobTemplate.spec → template.spec
#           (CronJob)       (Job)              (Pod)
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: batch/v1` | Required API group | CronJob not recognized | 🔴 |
| `spec.schedule` | Cron expression — WHEN to run | Invalid syntax → never triggers / validation error | 🔴 |
| `spec.jobTemplate.spec.template.spec.restartPolicy` | Must be `Never` or `OnFailure` | `Always` → validation error | 🔴 |
| `spec.jobTemplate.spec.template.spec.containers[].image` | Container image | ImagePullBackOff → Job fails | 🔴 |
| `spec.jobTemplate.spec.template.spec.containers[].command` | What the task runs | Wrong command → Job fails repeatedly | 🔴 |
| `spec.concurrencyPolicy` | `Allow` / `Forbid` / `Replace` | Allow = overlap Jobs; Forbid = skip; Replace = kill old | 🔴 |
| `spec.suspend` | `true` pauses scheduling | Accidentally suspended = no Jobs created | 🟡 |
| `spec.startingDeadlineSeconds` | Grace period for missed schedules | Too low → missed runs; not set → might create many Jobs at once | 🟡 |
| `spec.successfulJobsHistoryLimit` | How many completed Jobs to keep | Default: 3; set to 0 → no history; too high → clutter | 🟡 |
| `spec.failedJobsHistoryLimit` | How many failed Jobs to keep | Default: 1; set to 0 → can't debug failures | 🟡 |
| `spec.jobTemplate.spec.backoffLimit` | Max retries per Job | Default: 6; too high → spams Pods; 0 → no retries | 🟡 |
| `spec.jobTemplate.spec.activeDeadlineSeconds` | Per-Job timeout | Kills Job if it runs too long | 🟡 |

## Key Field Relationships
```
spec.schedule → triggers Job creation at matching times
  ↓
spec.concurrencyPolicy → controls overlap behavior:
  Allow:   New Job created even if previous is still running
  Forbid:  New Job SKIPPED if previous is still running
  Replace: Previous running Job KILLED → new Job created
  ↓
spec.jobTemplate.spec → defines the Job that gets created
  ↓
spec.jobTemplate.spec.template.spec → defines the Pod(s) in each Job
  ↓
spec.successfulJobsHistoryLimit + spec.failedJobsHistoryLimit
  → controls how many old Jobs are kept (cleanup)

FULL CHAIN:
  CronJob (schedule) → Job (completions, backoff) → Pod (image, command, restartPolicy)
  
  Each layer adds config:
    CronJob: WHEN and HOW OFTEN
    Job: HOW MANY and HOW TO RETRY
    Pod: WHAT TO RUN
```

## Concurrency Policy Explained
```
Allow (default):
  ├── Multiple Jobs can run simultaneously
  ├── No protection against overlap
  ├── Use when: tasks are independent and idempotent
  └── Risk: resource exhaustion if Jobs pile up

Forbid:
  ├── If previous Job still running → skip this trigger
  ├── No new Job created (silently skipped)
  ├── Use when: tasks MUST NOT overlap (e.g., DB migration)
  └── Risk: missed executions if Job takes too long

Replace:
  ├── If previous Job still running → DELETE it → create new one
  ├── Old Job terminated immediately
  ├── Use when: only latest run matters (e.g., data sync)
  └── Risk: interrupted work from killed Jobs
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `apiVersion: batch/v1`?
- `schedule` — valid cron expression? Quoted?
- `restartPolicy: Never` or `OnFailure` (NOT `Always`)?
- Container `image` and `command` correct?
- `concurrencyPolicy` — appropriate for the task?
- `suspend: false` (not accidentally suspended)?
- Proper nesting: `jobTemplate → spec → template → spec`?

## 🟢 Usually Irrelevant for CKA
- `metadata.annotations`
- `spec.timeZone` (v1.27+ feature)
- `status.lastScheduleTime`
- `status.lastSuccessfulTime`
- `spec.jobTemplate.spec.selector` (auto-generated)
- `spec.jobTemplate.spec.manualSelector`

## 🔴 Red Flags in 10 Seconds
| Red Flag | Problem |
|----------|---------|
| `apiVersion: batch/v1beta1` | Deprecated/removed — use `batch/v1` |
| `schedule` unquoted with `*` | YAML parsing may fail |
| Invalid cron syntax (e.g., `"60 * * * *"`) | Minute must be 0-59 |
| `restartPolicy: Always` | Validation error — invalid for Jobs |
| Missing `jobTemplate` (using `template` directly) | Wrong structure |
| `suspend: true` | CronJob paused — no Jobs will be created |
| `concurrencyPolicy: Allow` with slow Jobs | Jobs will pile up |
| `successfulJobsHistoryLimit: 0` + `failedJobsHistoryLimit: 0` | Can't inspect any history |
| Pod `spec` at wrong indentation level | Wrong container configuration |
| `startingDeadlineSeconds` very small | Easily misses schedules |

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe cronjob <name>`

### Where to Look FIRST
```
Name:                          my-cronjob
Namespace:                     default
Schedule:                      */5 * * * *
Concurrency Policy:            Allow
Suspend:                       False
Successful Job History Limit:  3
Failed Job History Limit:      1
Starting Deadline Seconds:     200
Active Jobs:                   <none>                    ← No Job currently running
Last Schedule Time:            Thu, 26 Feb 2026 15:05:00 +0530
Events:
  Normal  SuccessfulCreate  Created job my-cronjob-28123456
  Normal  SawCompletedJob   Saw completed job: my-cronjob-28123456, status: Complete
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `Suspend: True` | CronJob paused — no new Jobs created |
| `Active Jobs: my-cronjob-xxx, my-cronjob-yyy` | Multiple Jobs running (concurrency issue) |
| `Last Schedule Time` is very old | CronJob not triggering (schedule issue or suspended) |
| No `Last Schedule Time` at all | Never triggered — check schedule syntax |
| Events: `MissedSchedule` | Schedule times were missed |
| Events: `TooManyMissedTimes` | Too many schedules missed → skipping catch-up |
| Events: `FailedCreate` | Couldn't create Job |
| Events: `ForbidConcurrent` | Skipped run because previous Job still active |
| No events at all | Controller not processing — check controller-manager |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: CronJob Never Triggers
```
SYMPTOM:        kubectl get cronjob → LAST SCHEDULE is empty or very old
                No Jobs being created
CAUSE:          Invalid schedule syntax
                OR: suspend: true
                OR: controller-manager not running
CHECK:          kubectl describe cronjob → read schedule + suspend
                kubectl get pods -n kube-system | grep controller-manager
FIX:            Fix schedule syntax or set suspend: false
                kubectl edit cronjob <name>
EXAM SCENARIO:  "CronJob exists but no Jobs are created. Fix it."
```

## Pattern 2: Jobs Piling Up (Concurrency Issue)
```
SYMPTOM:        kubectl get cronjob → ACTIVE: 3, 4, 5...
                Multiple Jobs running simultaneously
CAUSE:          concurrencyPolicy: Allow (default) + Jobs take longer than schedule interval
BEHAVIOR:       New Job every 5 min, but each takes 10 min → overlap grows
FIX:            Change concurrencyPolicy to Forbid or Replace
                kubectl edit cronjob <name>
                OR: fix Job to complete faster
```

## Pattern 3: Jobs Failing Repeatedly
```
SYMPTOM:        kubectl get jobs | grep <cronjob> → all showing COMPLETIONS: 0/1
                failedJobsHistoryLimit shows multiple failed Jobs
CAUSE:          Wrong image, wrong command, missing dependencies
BEHAVIOR:       CronJob keeps creating new Jobs on schedule → each fails → pile up
FIX:            Check Pod logs from a failed Job:
                kubectl get pods --selector=job-name=<job>
                kubectl logs <failed-pod>
                Fix CronJob YAML → kubectl edit cronjob <name>
```

## Pattern 4: Suspended CronJob
```
SYMPTOM:        kubectl get cronjob → SUSPEND: True
                No new Jobs being created
CAUSE:          spec.suspend: true (intentional or accidental)
BEHAVIOR:       Schedule is ignored; no Jobs created; existing Jobs unaffected
FIX:            kubectl patch cronjob <name> -p '{"spec":{"suspend":false}}'
EXAM TRAP:      "CronJob is not running" — always check suspend first!
```

## Pattern 5: Missed Schedules
```
SYMPTOM:        Some scheduled runs were skipped
                Events: MissedSchedule or TooManyMissedTimes
CAUSE:          Controller was down during scheduled time
                OR: startingDeadlineSeconds too low
                OR: concurrencyPolicy: Forbid + previous Job still running
BEHAVIOR:       If > 100 schedules missed → CronJob stops trying catch-up
                Logs error: "Cannot determine if job needs to be started"
FIX:            Set reasonable startingDeadlineSeconds (e.g., 200)
                Fix controller-manager if it was down
```

## Pattern 6: Job Created but Pod Fails (Image/Command)
```
SYMPTOM:        Job exists with COMPLETIONS: 0/1
                Pod in ImagePullBackOff or CrashLoopBackOff
CAUSE:          Wrong image or bad command in jobTemplate
BEHAVIOR:       Job retries up to backoffLimit → Job Failed
                Next schedule trigger → new Job → same error → same failure
FIX:            kubectl edit cronjob <name> → fix image/command in jobTemplate
                Delete failed Jobs: kubectl delete job <name>
```

## Pattern 7: restartPolicy: Always
```
SYMPTOM:        CronJob creation fails or can't create Jobs
ERROR:          "Unsupported value: 'Always'"
BEHAVIOR:       If on CronJob itself → CronJob creation fails
                If on existing CronJob somehow → Jobs fail to create
FIX:            Set restartPolicy to Never or OnFailure
```

## Pattern 8: History Limits Too Low
```
SYMPTOM:        Can't find old Jobs to debug
                kubectl get jobs | grep <cronjob> → only 1 or 0 results
CAUSE:          successfulJobsHistoryLimit: 0 and/or failedJobsHistoryLimit: 0
BEHAVIOR:       Jobs (and their Pods) deleted immediately after completion
                Can't inspect logs or status of past runs
FIX:            Increase history limits:
                kubectl edit cronjob <name>
                Set successfulJobsHistoryLimit: 3, failedJobsHistoryLimit: 3
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Schedule Is Invalid?
- **Syntax error in cron expression** → CronJob may be created but never triggers
- **Empty schedule** → validation error → CronJob not created
- **Unquoted special characters** → YAML parsing error
- **Fix:** Correct the cron syntax; always quote the schedule string

## What Happens If Image Is Wrong?
- **Behavior:** CronJob creates Job on schedule → Job creates Pod → `ImagePullBackOff`
- **Job retries** up to `backoffLimit` → all fail → Job Failed
- **Next schedule trigger** → new Job → same error → same failure
- **Accumulation:** Failed Jobs pile up (up to `failedJobsHistoryLimit`)
- **Fix:** `kubectl edit cronjob <name>` → fix image in `jobTemplate` → future Jobs use correct image
- **Note:** Existing running/failed Jobs are NOT updated — only future runs

## What Happens If Command Is Wrong?
- **Behavior:** Pod starts → command fails → exit non-zero → Job retries → fails again
- **Each schedule trigger** creates new Job → same broken command
- **Result:** Continuous stream of failed Jobs
- **Fix:** `kubectl edit cronjob <name>` → fix command → future runs fixed

## What Happens If Resource Limits Are Wrong?
- **Requests too high** → Pod Pending → Job Active but not progressing
- **Memory limit too low** → OOMKilled → Job retries → fails again
- **Combined with concurrencyPolicy: Allow** → Pending Jobs accumulate
- **Fix:** `kubectl edit cronjob <name>` → fix resources in jobTemplate

## What Happens If Dependencies Are Missing?
- **Missing ConfigMap/Secret** → `CreateContainerConfigError` → Job fails
- **Missing PVC** → Pod Pending → Job stuck
- **Missing external service** → Pod starts → command fails → retry
- **CronJob keeps creating new Jobs** → each one fails the same way
- **Fix:** Create missing dependency; future CronJob triggers will succeed

## What Happens If ConcurrencyPolicy Is Wrong?
- **Allow + slow Jobs** → Multiple Jobs running simultaneously → resource exhaustion
- **Forbid + slow Jobs** → Scheduled runs silently skipped → work not done
- **Replace + important Jobs** → Running Job killed mid-work → data corruption risk
- **Fix:** Choose policy based on job characteristics:
  - Idempotent + fast → Allow
  - Must not overlap → Forbid
  - Only latest matters → Replace

## What Happens If You Edit a CronJob?
- **CronJob `spec` IS mutable** (unlike Job's template!)
- `kubectl edit cronjob <name>` → change schedule, image, command, concurrency → takes effect on NEXT trigger
- **Existing running Jobs are NOT affected** — only future Jobs use the new spec
- **Schedule change** → next trigger time shifts to new schedule

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: CronJob overview
kubectl get cronjob <name>
# NAME         SCHEDULE      SUSPEND   ACTIVE   LAST SCHEDULE   AGE
# my-cj        */5 * * * *   False     0        3m              2h     ← Healthy ✅
# my-cj        */5 * * * *   True      0        1h              2h     ← SUSPENDED ❌
# my-cj        */5 * * * *   False     5        1m              2h     ← OVERLAPPING ❌
# my-cj        */5 * * * *   False     0        <none>          2h     ← NEVER RAN ❌

# ⏱️ 10-20 sec: Check created Jobs
kubectl get jobs | grep <cronjob-name>
# How many? Complete? Failed? Active?
# NAME                    COMPLETIONS   DURATION   AGE
# my-cj-28456789          1/1           5s         15m   ← OK
# my-cj-28456790          0/1                      10m   ← PROBLEM
# my-cj-28456791          0/1                      5m    ← PROBLEM

# ⏱️ 20-35 sec: Check Pods from failing Job
kubectl get pods --selector=job-name=my-cj-28456790
# STATUS: ImagePullBackOff? Error? Pending?

kubectl logs <failed-pod>
# Read error output

# ⏱️ 35-50 sec: Describe CronJob for details
kubectl describe cronjob <name>
# Check: Schedule, Suspend, ConcurrencyPolicy, Events
# Events tell you: SuccessfulCreate? MissedSchedule? ForbidConcurrent?

# ⏱️ 50-60 sec: Decide action
# Suspended? → unsuspend: kubectl patch cronjob <name> -p '{"spec":{"suspend":false}}'
# Never triggered? → fix schedule syntax
# Jobs failing? → fix image/command: kubectl edit cronjob <name>
# Jobs piling up? → change concurrencyPolicy
# Need immediate test? → kubectl create job test --from=cronjob/<name>
```

## Quick Diagnosis Table
```
CRONJOB STATUS                          → LIKELY CAUSE
───────────────────────────────────────────────────────────────
SUSPEND: True                           → Accidentally suspended
LAST SCHEDULE: <none>                   → Invalid schedule or never triggered yet
ACTIVE: 0, Jobs all Complete            → Healthy ✅
ACTIVE: 3+                              → Jobs overlapping; check concurrencyPolicy
Jobs exist but all Failed               → Image/command/config bug in jobTemplate
No Jobs at all, CronJob exists          → Suspended / bad schedule / controller-manager down
Jobs Complete but wrong output          → Command logic error (check logs)
Events: MissedSchedule                  → Controller was down or deadline too tight
Events: ForbidConcurrent                → Previous Job still running; current run skipped
```

# 🔟 Fast Fix Strategy

## Common Fixes

### Fix: Suspended CronJob
```bash
kubectl patch cronjob <name> -p '{"spec":{"suspend":false}}'
# CronJob resumes on next schedule match

# Verify
kubectl get cronjob <name>
# SUSPEND: False ✅
```

### Fix: Wrong Image or Command
```bash
# CronJob spec IS mutable — can edit directly!
kubectl edit cronjob <name>
# Fix image or command in jobTemplate.spec.template.spec.containers
# Save → applies to NEXT triggered Job (existing Jobs unaffected)

# Test immediately without waiting for schedule:
kubectl create job test-run --from=cronjob/<name>
kubectl get pods --selector=job-name=test-run
kubectl logs <pod>
```

### Fix: Wrong Schedule
```bash
kubectl edit cronjob <name>
# Fix schedule field → save
# OR:
kubectl patch cronjob <name> -p '{"spec":{"schedule":"0 */2 * * *"}}'
# Verify:
kubectl get cronjob <name>
# New schedule shown ✅
```

### Fix: Concurrency Policy
```bash
kubectl patch cronjob <name> -p '{"spec":{"concurrencyPolicy":"Forbid"}}'
# No more overlapping Jobs

# Clean up accumulated running Jobs:
kubectl delete job <old-job-name-1> <old-job-name-2>
```

### Fix: History Limits
```bash
kubectl edit cronjob <name>
# Set: successfulJobsHistoryLimit: 3
# Set: failedJobsHistoryLimit: 3
# Old excess Jobs will be cleaned up by controller
```

### Fix: Trigger a Manual Run (Test Fix)
```bash
# Most useful CKA trick for CronJobs!
kubectl create job verify-fix --from=cronjob/<name>
# Creates a Job using CronJob's current jobTemplate
# Check if it works:
kubectl get pods --selector=job-name=verify-fix
kubectl logs <pod-name>
# If Pod Completed → fix is good ✅
# Clean up test:
kubectl delete job verify-fix
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Create CronJob
```bash
# Imperative (fast for exam)
kubectl create cronjob cj-demo --image=busybox:1.36 \
  --schedule="*/1 * * * *" \
  -- sh -c "echo 'Tick at $(date)' && sleep 2 && echo 'Done'"

# OR declarative:
cat <<'EOF' | kubectl apply -f -
apiVersion: batch/v1
kind: CronJob
metadata:
  name: cj-demo
spec:
  schedule: "*/1 * * * *"
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 2
  jobTemplate:
    spec:
      backoffLimit: 2
      template:
        spec:
          containers:
          - name: worker
            image: busybox:1.36
            command:
            - sh
            - -c
            - echo "CronJob running at $(date)" && sleep 2 && echo "Complete"
          restartPolicy: OnFailure
EOF
```

### Validation Steps
```bash
# CronJob created
kubectl get cronjob cj-demo
# SCHEDULE: */1 * * * *, SUSPEND: False
```

## Step 2 — Break It Intentionally

### Break 1: Suspend the CronJob
```bash
kubectl patch cronjob cj-demo -p '{"spec":{"suspend":true}}'
kubectl get cronjob cj-demo
# SUSPEND: True → no new Jobs created
# Wait a few minutes → no new Jobs appear
# FIX:
kubectl patch cronjob cj-demo -p '{"spec":{"suspend":false}}'
```

### Break 2: Wrong Image
```bash
kubectl edit cronjob cj-demo
# Change image to busybox:NONEXISTENT → save
# Wait for next trigger:
kubectl get jobs | grep cj-demo
# New Job with COMPLETIONS: 0/1
kubectl get pods --selector=job-name=<new-job>
# ImagePullBackOff
# FIX:
kubectl edit cronjob cj-demo
# Fix image back → next trigger creates working Job
```

### Break 3: Command That Always Fails
```bash
kubectl edit cronjob cj-demo
# Change command to: ["sh", "-c", "exit 1"]
# Wait for trigger → Job fails → backoffLimit retries → Job Failed
kubectl get jobs | grep cj-demo
# COMPLETIONS: 0/1 (failed)
# FIX:
kubectl edit cronjob cj-demo
# Fix command → delete failed Jobs → next trigger works
```

### Break 4: Concurrency Overlap (Allow + Slow Job)
```bash
kubectl edit cronjob cj-demo
# Change command to: ["sh", "-c", "sleep 120"] (runs 2 min)
# Change concurrencyPolicy to: Allow
# Schedule: */1 * * * * (every minute)
# After 3 minutes:
kubectl get cronjob cj-demo
# ACTIVE: 3 → three Jobs running simultaneously!
# FIX:
kubectl patch cronjob cj-demo -p '{"spec":{"concurrencyPolicy":"Forbid"}}'
# Delete extra Jobs:
kubectl delete jobs -l job-name  # or delete specific ones
```

### Break 5: Invalid Schedule Syntax
```bash
kubectl edit cronjob cj-demo
# Change schedule to: "99 * * * *" (minute 99 — invalid)
# Behavior: CronJob accepted BUT never triggers (no validation on semantic correctness)
# OR: "* * * *" (only 4 fields — requires 5)
# FIX: correct the schedule to valid cron syntax
kubectl patch cronjob cj-demo -p '{"spec":{"schedule":"*/2 * * * *"}}'
```

### Break 6: Excessive Resources
```bash
kubectl edit cronjob cj-demo
# Add to container: resources.requests.cpu: "100"
# Next trigger → Job creates Pod → Pending (FailedScheduling)
# Job stays Active → if concurrencyPolicy: Forbid → future runs skipped too
# FIX:
kubectl edit cronjob cj-demo
# Remove excessive resources → delete stuck Job → next trigger works
```

## Step 3 — Debug Without Looking at Solution

### What Command to Run FIRST
```bash
kubectl get cronjob <name>
# Read: SUSPEND? ACTIVE? LAST SCHEDULE?
```

### How to Narrow the Issue
```bash
# 1. Is it suspended?
kubectl get cronjob <name> -o jsonpath='{.spec.suspend}'
# "true" → that's the problem → unsuspend

# 2. When did it last run?
kubectl get cronjob <name> -o jsonpath='{.status.lastScheduleTime}'
# Empty or very old → schedule issue or controller-manager down

# 3. Are Jobs being created but failing?
kubectl get jobs | grep <cronjob-name>
# Check COMPLETIONS column

# 4. Why are Jobs failing?
kubectl get pods --selector=job-name=<failing-job>
kubectl logs <failed-pod>
# Read error → command, image, or dependency issue

# 5. Are Jobs overlapping?
kubectl get cronjob <name> -o jsonpath='{.spec.concurrencyPolicy}'
kubectl get cronjob <name> -o jsonpath='{.status.active}'
```

### How to Confirm Root Cause
```bash
# Suspended confirmed:
kubectl get cronjob -o yaml | grep "suspend: true"

# Schedule issue confirmed:
kubectl describe cronjob → no SuccessfulCreate events

# Image issue confirmed:
kubectl describe pod <pod> → Events: ImagePullBackOff

# Command issue confirmed:
kubectl logs <failed-pod> → error output

# Concurrency confirmed:
kubectl get cronjob → ACTIVE: 3+ with concurrencyPolicy: Allow

# Quick test after fix:
kubectl create job test --from=cronjob/<name>
kubectl logs <test-pod>
```

## 5 Quiz Questions to Test Understanding

### Q1: What apiVersion is used for CronJobs in current Kubernetes versions?
<details>
<summary>Answer</summary>
batch/v1. The old batch/v1beta1 was deprecated in K8s 1.21 and removed in 1.25.
</details>

### Q2: A CronJob runs every 5 minutes but each Job takes 8 minutes. With concurrencyPolicy: Allow, what happens?
<details>
<summary>Answer</summary>
Jobs pile up. After 15 minutes, 3 Jobs are running simultaneously. After 40 minutes, you could have 8 active Jobs. This causes resource exhaustion. Fix: concurrencyPolicy: Forbid (skip overlapping) or Replace (kill old, start new).
</details>

### Q3: How do you immediately test a CronJob without waiting for the next schedule?
<details>
<summary>Answer</summary>
kubectl create job test-run --from=cronjob/my-cronjob. This creates a one-time Job using the CronJob's current jobTemplate.
</details>

### Q4: A CronJob shows SUSPEND: False, LAST SCHEDULE: <none>, ACTIVE: 0. What's likely wrong?
<details>
<summary>Answer</summary>
The schedule is likely invalid (bad syntax), or the CronJob was just created and hasn't reached its first trigger time yet. Check the schedule field for correctness. Also verify controller-manager is running.
</details>

### Q5: What is the correct nesting order in a CronJob YAML?
<details>
<summary>Answer</summary>
CronJob.spec → jobTemplate.spec → template.spec. Three spec levels:
1. CronJob.spec (schedule, concurrency)
2. jobTemplate.spec (Job config: backoffLimit, completions)
3. template.spec (Pod config: containers, restartPolicy)
</details>

# 1️⃣3️⃣ Component Interaction Map

## How CronJob Interacts with Components

### CronJob ↔ Jobs ↔ Pods
```
NORMAL:    CronJob creates Job on schedule → Job creates Pod(s)
           3-level hierarchy: CronJob → Job → Pod
           Each schedule trigger = new Job with unique name
           Owner references: Pod → Job → CronJob (cascade deletion)

FAILURE:   Job fails → CronJob tracks in failedJobsHistoryLimit
           Job succeeds → CronJob tracks in successfulJobsHistoryLimit
           All Jobs failing → CronJob keeps creating new failing Jobs
           Fix: edit CronJob → future Jobs use corrected template
```

### CronJob ↔ Services
```
NORMAL:    CronJobs typically DON'T expose via Services
           CronJob Pods may connect TO Services (e.g., database, API)

FAILURE:   Dependency Service down → CronJob Pod fails → Job retries
           DNS resolution failure → command fails
           Service in different namespace → must use FQDN
```

### CronJob ↔ Nodes
```
NORMAL:    Scheduler places Job Pods on suitable nodes
           Same scheduling rules as any Pod (affinity, taints, resources)

FAILURE:   No suitable node → Pod Pending → Job Active but stuck
           If concurrencyPolicy: Forbid → future triggers also skipped
           Resource contention → Pending Pods accumulate
```

### CronJob ↔ Scheduler
```
NORMAL:    Scheduler places CronJob's Pods like any other Pod

FAILURE:   FailedScheduling → Pod Pending → Job stuck
           Combined with concurrencyPolicy:
             Allow:   old + new Jobs both have Pending Pods → pile up
             Forbid:  new trigger skipped → missed work
             Replace: old Job killed → new Job also Pending → data loss risk
```

### CronJob ↔ Controller Manager
```
NORMAL:    TWO controllers involved:
           1. CronJob Controller → checks schedule, creates Jobs
           2. Job Controller → manages Jobs, creates/tracks Pods
           
           CronJob Controller checks schedule every ~10 seconds
           If match → creates new Job with owner reference to CronJob

FAILURE:   Controller Manager down → no schedule checking!
           → No new Jobs created → schedules missed
           → When CM recovers → depends on startingDeadlineSeconds:
               Set: if within deadline → catch-up Job created
               Not set: if > 100 schedules missed → gives up
           → Existing running Job Pods continue (kubelet manages them)
```

### CronJob ↔ etcd
```
NORMAL:    CronJob spec, Job objects, and Pod objects stored in etcd
           lastScheduleTime tracked in CronJob status
           History of Jobs persisted

FAILURE:   etcd down → API server read-only
           No new Jobs created
           Running Pods continue (kubelet)
           etcd recovery → CronJob Controller resumes scheduling
```

### CronJob ↔ Network Layer
```
NORMAL:    Job Pods get IPs from CNI
           Pods may access external services, APIs, databases

FAILURE:   CNI failure → Pods stuck in ContainerCreating
           DNS failure → commands that resolve hostnames fail
           Network policy blocking egress → can't reach external dependencies
```

### CronJob ↔ Storage Layer
```
NORMAL:    CronJob Pods can mount PVCs for read/write
           Each Job creates NEW Pods → may need shared storage (RWX)
           Or each run writes to different location

FAILURE:   PVC Pending → Pod Pending → Job stuck
           RWO PV → only 1 Pod can mount at a time
             If concurrencyPolicy: Allow → second Job's Pod can't mount → Pending
           PV full → command writes fail → exit non-zero → Job retry
```

### Common Cron Expressions Cheat Sheet
```
"* * * * *"        → Every minute
"*/5 * * * *"      → Every 5 minutes
"0 * * * *"        → Every hour (at :00)
"0 */2 * * *"      → Every 2 hours
"0 0 * * *"        → Daily at midnight
"0 2 * * *"        → Daily at 2:00 AM
"30 8 * * 1-5"     → Weekdays at 8:30 AM
"0 0 * * 0"        → Weekly on Sunday midnight
"0 0 1 * *"        → Monthly on 1st at midnight
"0 0 1 1 *"        → Yearly on Jan 1st at midnight
```

### Mental Model: CronJob = "Scheduled Job Factory"
```
Deployment: Run FOREVER                       (web servers)
Job:        Run ONCE to completion             (batch work)
CronJob:    Run ONCE, REPEATEDLY on schedule   (scheduled batch)

CronJob is a FACTORY that creates Jobs:
  CronJob ──creates──→ Job ──creates──→ Pod
  (WHEN)               (HOW)            (WHAT)

Key rules to remember:
  ✅ apiVersion: batch/v1 (same as Job)
  ✅ schedule MUST be quoted: "*/5 * * * *"
  ✅ restartPolicy: Never or OnFailure (NOT Always)
  ✅ CronJob spec IS mutable (unlike Job template)
  ✅ Manual trigger: kubectl create job <name> --from=cronjob/<cj>
  ✅ concurrencyPolicy controls overlap behavior
  ✅ suspend: true pauses all future triggers
  ✅ History limits control old Job cleanup
  ❌ Editing CronJob does NOT affect existing Jobs
  ❌ CronJob doesn't have logs — navigate to Pod level
```
