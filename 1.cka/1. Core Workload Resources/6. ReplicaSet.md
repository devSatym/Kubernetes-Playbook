# 🧠 ReplicaSet — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- **Ensures a specified number of identical Pod replicas are running at all times**
- If a Pod dies → ReplicaSet creates a new one to maintain the desired count
- If too many Pods exist → ReplicaSet terminates the excess
- Uses **label selectors** to identify which Pods belong to it
- **Self-healing mechanism** — the foundation of Kubernetes resilience

## Real-World Use Cases
- Maintaining 3 replicas of a web frontend across nodes
- Ensuring a minimum number of worker Pods are always running
- **In practice: you almost NEVER create ReplicaSets directly**
  - Deployments create and manage ReplicaSets automatically
  - CKA tests your understanding of how RS works under a Deployment

## Where It Fits in Cluster Architecture
```
┌────────────────────────────────────────────────────────────────┐
│                     CONTROLLER HIERARCHY                        │
│                                                                 │
│  Deployment                                                     │
│    └── ReplicaSet  (Deployment creates this)                   │
│          ├── Pod-1  (ReplicaSet creates these)                 │
│          ├── Pod-2                                              │
│          └── Pod-3                                              │
│                                                                 │
│  During Rolling Update:                                        │
│  Deployment                                                     │
│    ├── ReplicaSet-v1 (old) ← scaling DOWN                     │
│    │     ├── Pod-old-1                                          │
│    │     └── Pod-old-2  (being terminated)                     │
│    └── ReplicaSet-v2 (new) ← scaling UP                       │
│          ├── Pod-new-1                                          │
│          └── Pod-new-2  (being created)                        │
│                                                                 │
│  Controller Manager (ReplicaSet Controller):                   │
│    Watches RS objects → counts matching Pods → creates/deletes │
│    to match spec.replicas                                      │
└────────────────────────────────────────────────────────────────┘
```

### ReplicaSet vs ReplicationController (Legacy)
| Aspect | ReplicaSet | ReplicationController |
|--------|-----------|----------------------|
| API version | `apps/v1` | `v1` |
| Selector type | Set-based (`matchLabels` + `matchExpressions`) | Equality-based only |
| Used by | Deployments (modern) | Legacy, deprecated |
| CKA focus | ✅ Yes | ❌ Not tested |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Deployment** — Creates and manages ReplicaSets; RS is the execution layer of Deployments
- **Pods** — RS creates and owns Pods (Pods have `ownerReferences` pointing to RS)
- **HPA** — Horizontal Pod Autoscaler scales the Deployment → which changes the RS replicas
- **Controller Manager** — RS controller (inside kube-controller-manager) watches RS objects and reconciles

## What Resources It Interacts With
- **Pods** — Creates/deletes Pods to match `spec.replicas`
- **Labels / Selectors** — `spec.selector` MUST match `spec.template.metadata.labels`
- **Node** — Scheduler places RS-created Pods on nodes
- **Service** — Service selects Pods by labels (same labels RS uses)
- **PDB** — PodDisruptionBudget protects RS-managed Pods

## Upstream & Downstream Dependencies
```
UPSTREAM:
  Deployment → creates ReplicaSet
  User (rarely) → creates ReplicaSet directly

DOWNSTREAM:
  ReplicaSet → creates Pods
  Pods → scheduled by Scheduler to Nodes
  Pods → selected by Services for traffic routing
  Pods → monitored by HPA for autoscaling

RECONCILIATION LOOP:
  Controller Manager watches:
    Desired replicas (spec.replicas) vs Actual Pods (matching selector)
    Too few? → Create new Pods
    Too many? → Delete excess Pods
    Pod crashes? → New Pod auto-created
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid ReplicaSet YAML
```yaml
apiVersion: apps/v1              # ← NOT v1! Must be apps/v1
kind: ReplicaSet
metadata:
  name: my-replicaset
  labels:
    app: myapp                   # ← Labels on the RS itself (not the Pods)
spec:
  replicas: 3                    # ← Desired number of Pods
  selector:                      # ← How RS finds its Pods
    matchLabels:
      app: myapp                 # ← MUST match template.metadata.labels
  template:                      # ← Pod template (same structure as Pod spec)
    metadata:
      labels:
        app: myapp               # ← MUST match selector.matchLabels
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
```

## Key Structural Rules
```
spec.selector.matchLabels  ←—— MUST EXACTLY MATCH ——→  spec.template.metadata.labels
    (how RS finds Pods)                                  (labels on created Pods)

If these don't match:
  ❌ RS creation fails with validation error
  "selector does not match template labels"
```

## ⚠️ Common Indentation Traps

### Trap 1: Wrong apiVersion
```yaml
# ❌ WRONG — ReplicaSet is NOT in core v1
apiVersion: v1
kind: ReplicaSet

# ✅ CORRECT
apiVersion: apps/v1
kind: ReplicaSet
```

### Trap 2: selector not matching template labels
```yaml
# ❌ WRONG — selector says "app: web" but template says "app: myapp"
spec:
  selector:
    matchLabels:
      app: web              # ← Looking for app=web
  template:
    metadata:
      labels:
        app: myapp          # ← But Pods get app=myapp
# Error: selector does not match template labels

# ✅ CORRECT — both must match
spec:
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp          # ← Same label!
```

### Trap 3: template.metadata.labels missing entirely
```yaml
# ❌ WRONG — no labels on template
spec:
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:               # ← labels missing!
    spec:
      containers:
      - name: web
        image: nginx
# Error: selector does not match template labels
```

### Trap 4: template indentation wrong
```yaml
# ❌ WRONG — template.spec not indented under template
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
  spec:                     # ← WRONG! This is at RS spec level, not template level
    containers:
    - name: web

# ✅ CORRECT — spec indented under template
  template:
    metadata:
      labels:
        app: myapp
    spec:                   # ← Under template
      containers:
      - name: web
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `apiVersion: apps/v1` | Required; `v1` alone won't work | RS not created — API error | 🔴 |
| `spec.replicas` | Desired Pod count | Wrong count; 0 = no Pods at all | 🔴 |
| `spec.selector.matchLabels` | How RS identifies its Pods | Must match template labels — validation error if not | 🔴 |
| `spec.template.metadata.labels` | Labels assigned to created Pods | Must match selector — validation error if not | 🔴 |
| `spec.template.spec.containers[].image` | What image Pods run | `ImagePullBackOff` on all Pods | 🔴 |
| `spec.template.spec.containers[].name` | Container name | Missing = validation error | 🟡 |
| `spec.template.spec.containers[].resources` | CPU/memory requests & limits | Pending Pods or OOMKilled | 🟡 |
| `metadata.name` | RS name (used in `kubectl get rs`) | Can't reference RS | 🟡 |

## The CRITICAL Relationship: selector ↔ labels
```
                    ┌──────────────────────────┐
                    │       ReplicaSet          │
                    │                           │
                    │  spec.selector:           │
                    │    matchLabels:           │
                    │      app: myapp ──────────┼──── MUST
                    │                           │     MATCH
                    │  spec.template:           │      ↓
                    │    metadata:              │
                    │      labels:             │
                    │        app: myapp ────────┼──── These labels go on Pods
                    └──────────────────────────┘
                                                      ↓
                    ┌──────────────────────────┐
                    │         Pod               │
                    │  metadata:               │
                    │    labels:               │
                    │      app: myapp          │ ← RS finds this Pod via selector
                    │    ownerReferences:       │
                    │      kind: ReplicaSet     │ ← Pod knows its owner
                    └──────────────────────────┘

  If a Pod with matching label exists (even manually created):
    RS adopts it → may count it toward replicas → may NOT create a new one
    
  If Pod label is changed (removed "app: myapp"):
    RS no longer sees it → creates a replacement → you now have an "orphaned" Pod
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- `apiVersion: apps/v1`? (NOT `v1`)
- `spec.replicas` — how many Pods expected?
- `selector.matchLabels` matches `template.metadata.labels`? **(THE #1 ERROR)**
- Container `image` valid?
- Template `spec` properly indented under `template`?

## 🟢 Usually Irrelevant for CKA
- `metadata.labels` on RS itself (only template labels matter for Pod selection)
- `spec.minReadySeconds`
- `metadata.annotations`
- `spec.selector.matchExpressions` (rarely tested; `matchLabels` is standard)
- RS `status` fields (managed by system)

## 🔴 Red Flags in 10 Seconds

| Red Flag | Problem |
|----------|---------|
| `apiVersion: v1` | Wrong API version for ReplicaSet |
| `selector.matchLabels` ≠ `template.metadata.labels` | Validation error — RS won't create |
| `replicas: 0` | No Pods created (may be intentional but suspect) |
| `template.spec` at wrong indentation level | Pods won't have correct container spec |
| Missing `selector` entirely | Validation error (required field) |
| Missing `template.metadata.labels` | Selector can't match — validation error |
| `kind: ReplicationController` | Legacy object — should be ReplicaSet |

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe rs <replicaset-name>`

### Where to Look FIRST
```
Name:           my-replicaset
Namespace:      default
Selector:       app=myapp                    ← What labels RS looks for
Labels:         app=myapp
Replicas:       3 current / 3 desired        ← Current vs desired count
Pods Status:    3 Running / 0 Waiting / 0 Succeeded / 0 Failed
Pod Template:
  Labels:       app=myapp                    ← Labels on Pods (must match Selector)
  Containers:
   web:
    Image:      nginx:1.25
Events:
  Normal  SuccessfulCreate  Created pod: my-replicaset-abc12
  Normal  SuccessfulCreate  Created pod: my-replicaset-def34
  Normal  SuccessfulCreate  Created pod: my-replicaset-ghi56
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `Replicas: 0 current / 3 desired` | RS can't create Pods — check events |
| `Pods Status: 0 Running / 3 Waiting` | Pods exist but aren't running |
| Events: `FailedCreate` | Pod creation error (quota, resource, etc.) |
| Events: `SuccessfulCreate` then Pods crash | Pods created but app is broken |
| `Selector: app=myapp` but no matching Pods | Label mismatch or Pods on wrong namespace |

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Selector Doesn't Match Template Labels
```
SYMPTOM:        RS creation fails immediately
ERROR:          "The ReplicaSet is invalid: spec.template.metadata.labels:
                Invalid value: selector does not match template labels"
CAUSE:          spec.selector.matchLabels ≠ spec.template.metadata.labels
BEHAVIOR:       RS object NOT created at all — instant validation error
FIX:            Make selector and template labels identical
EXAM SCENARIO:  "Create a ReplicaSet. Your YAML is rejected. Fix it."
                → Check selector vs template labels
```

## Pattern 2: Wrong apiVersion
```
SYMPTOM:        "error: resource mapping not found for kind 'ReplicaSet' in version 'v1'"
CAUSE:          apiVersion: v1 instead of apps/v1
BEHAVIOR:       kubectl apply rejects the YAML
FIX:            Change to apiVersion: apps/v1
EXAM TIP:       ReplicaSet → apps/v1, ALWAYS
```

## Pattern 3: All Pods in ImagePullBackOff
```
SYMPTOM:        RS shows DESIRED=3, CURRENT=3, READY=0
                All Pods show ImagePullBackOff
CAUSE:          Wrong image name/tag in template spec
BEHAVIOR:       RS creates Pods successfully, but containers fail to pull image
FIX:            Fix image → kubectl edit rs <name> or delete & recreate RS
⚠️ NOTE:        Editing RS template does NOT update existing Pods!
                Must delete old Pods manually for RS to recreate with new image
```

## Pattern 4: All Pods Pending
```
SYMPTOM:        RS shows DESIRED=3, CURRENT=3, READY=0
                All Pods in Pending state
CAUSE:          Resources too high, no available nodes, taints blocking, PVC unbound
DESCRIBE POD:   Events show "FailedScheduling: Insufficient cpu/memory"
                Or "no nodes available to schedule pods"
FIX:            Reduce resource requests, add nodes, fix taints/tolerations
```

## Pattern 5: RS Creates Pods But They CrashLoop
```
SYMPTOM:        RS shows DESIRED=3, CURRENT=3, READY=0
                Pods in CrashLoopBackOff
CAUSE:          App crashes (wrong command, missing config, dependency unavailable)
BEHAVIOR:       RS keeps restarting Pods (they crash → restart → crash → backoff)
FIX:            Fix app issue in Pod template (image, command, env, config)
                Delete old Pods after fixing RS template
```

## Pattern 6: RS Has Fewer Pods Than Expected
```
SYMPTOM:        DESIRED=5, CURRENT=2, READY=2
CAUSE:          ResourceQuota preventing more Pods, or node capacity exhausted
EVENTS:         "FailedCreate: forbidden: exceeded quota"
                Or "FailedCreate: pods is forbidden: exceeded the maximum..."
FIX:            Increase quota or reduce resource requests
```

## Pattern 7: Orphaned Pods (Label Mismatch at Runtime)
```
SYMPTOM:        RS shows correct count, but extra Pods with old labels exist
CAUSE:          Someone changed Pod labels → RS lost track → created replacement
BEHAVIOR:       RS creates new Pod to maintain count, old Pod is "orphaned" (still running)
FIX:            Delete orphaned Pod manually, or restore its labels
EXAM TIP:       If you remove a Pod's label, RS thinks it's missing a Pod and creates another
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Selector ≠ Template Labels?
- **Behavior:** RS YAML rejected at creation — validation error
- **Error:** `selector does not match template labels`
- RS is NOT created, 0 Pods created
- **Fix:** Make `selector.matchLabels` identical to `template.metadata.labels`

## What Happens If Image Is Wrong?
- **Behavior:** RS creates Pods, but all Pods show `ImagePullBackOff`
- DESIRED=3, CURRENT=3, READY=0
- Pods exist but containers can't start
- **Fix:** Fix image in RS → `kubectl edit rs` → then delete existing Pods
- **⚠️ CRITICAL:** Editing RS template does NOT auto-update existing Pods!

## What Happens If Resource Requests Are Too High?
- **Behavior:** Pods created but stay `Pending`
- RS shows DESIRED=3, CURRENT=3, READY=0
- `kubectl describe pod` → "FailedScheduling: Insufficient cpu"
- **Fix:** Lower requests, or add more node capacity

## What Happens If You Change a Pod's Labels?
```bash
# Remove label from a Pod managed by RS
kubectl label pod my-replicaset-abc12 app-
```
- **Behavior:**
  - RS no longer sees this Pod (label doesn't match selector)
  - RS thinks it's missing a Pod → creates a new one
  - Original Pod is now "orphaned" — still running but not managed
  - You now have 4 Pods instead of 3
- **Fix:** Delete the orphaned Pod, or restore its label

## What Happens If You Manually Create a Pod with Matching Labels?
```bash
# Create a Pod with same labels as RS selector
kubectl run stray --image=nginx --labels="app=myapp"
```
- **Behavior:**
  - RS detects it has N+1 Pods matching its selector
  - RS deletes one Pod to maintain desired count
  - It might delete YOUR manually-created Pod
- **Lesson:** Don't create Pods with labels that match an existing RS selector

## What Happens If Ports Mismatch?
- `containerPort` in RS template is informational — no direct breakage
- But if a Service targets the wrong port → traffic won't reach the app
- RS itself doesn't validate ports — this is a Pod/Service layer issue

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: Check RS status
kubectl get rs
# Read: DESIRED, CURRENT, READY
# DESIRED=3, CURRENT=3, READY=3 → healthy ✅
# DESIRED=3, CURRENT=3, READY=0 → Pods exist but failing
# DESIRED=3, CURRENT=0, READY=0 → RS can't create Pods at all

# ⏱️ 10-20 sec: Check RS events
kubectl describe rs <rs-name> | tail -15
# Look for: FailedCreate, quota exceeded, etc.

# ⏱️ 20-35 sec: Check Pod status
kubectl get pods -l <selector-labels>
# Look for: Pending, CrashLoopBackOff, ImagePullBackOff, Error

# ⏱️ 35-50 sec: Dig into failing Pod
kubectl describe pod <pod-name> | tail -20
# Events: scheduling issues, image pull failures, config errors

# ⏱️ 50-60 sec: Check logs if CrashLoop
kubectl logs <pod-name> --previous
```

## Quick Diagnosis Table
```
RS STATUS                              → LIKELY CAUSE
──────────────────────────────────────────────────────────
DESIRED=3, CURRENT=0, READY=0         → RS can't create Pods (quota, validation)
DESIRED=3, CURRENT=3, READY=0         → Pods exist but failing (image, command, scheduling)
DESIRED=3, CURRENT=3, READY=2         → 1 Pod failing (node issue, probe failure)
DESIRED=3, CURRENT=5, READY=5         → Orphaned Pods or manual Pods with matching labels
DESIRED=0, CURRENT=0, READY=0         → replicas: 0 (intentional) or Deployment scaled down
```

# 🔟 Fast Fix Strategy

## Fixing RS vs Fixing Deployment

> **CKA Tip:** In the real exam, you'll usually fix the **Deployment**, not the RS directly. Deployments manage RS for you.

```bash
# If you need to fix the RS directly:
kubectl edit rs <rs-name>
# Edit template (image, command, env, etc.)
# ⚠️ Changes WON'T apply to existing Pods — only new Pods!

# Force Pods to use new template:
kubectl delete pods -l <selector-labels>
# RS sees missing Pods → creates new ones with updated template

# If managed by Deployment (preferred):
kubectl edit deployment <deployment-name>
# OR
kubectl set image deployment/<dep> <container>=<new-image>
# Deployment creates new RS → old RS scales down → new Pods roll out
```

## Common Fixes

### Fix: Selector/label mismatch (creation error)
```bash
# Fix YAML: make selector.matchLabels = template.metadata.labels
vim rs.yaml
kubectl apply -f rs.yaml
```

### Fix: Wrong image
```bash
# Edit RS template
kubectl edit rs <rs-name>
# Change image → save

# Delete existing Pods to trigger recreation
kubectl delete pods -l app=myapp
# RS creates new Pods with corrected image
```

### Fix: Pods Pending (resources too high)
```bash
# Option A: Lower resource requests
kubectl edit rs <rs-name>
# Reduce resources.requests → delete existing Pods

# Option B: Add more nodes (if cluster autoscaler available)
```

### Fix: Scale replicas
```bash
kubectl scale rs <rs-name> --replicas=5
# RS immediately creates/deletes Pods to match
```

## Validate Fix
```bash
# Check RS is healthy
kubectl get rs <rs-name>
# DESIRED=CURRENT=READY ✅

# Check all Pods running
kubectl get pods -l app=myapp
# All Running, 0 restarts

# Describe RS — no warning events
kubectl describe rs <rs-name> | tail -10
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Minimal Working ReplicaSet
```yaml
# File: rs-demo.yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: rs-demo
  labels:
    app: rs-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: rs-demo
  template:
    metadata:
      labels:
        app: rs-demo
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
```

### Validation Steps
```bash
# Create
kubectl apply -f rs-demo.yaml

# Verify RS
kubectl get rs rs-demo
# DESIRED=3, CURRENT=3, READY=3 ✅

# Verify Pods
kubectl get pods -l app=rs-demo
# 3 Pods, all Running

# Check ownership
kubectl describe pod <pod-name> | grep "Controlled By"
# Controlled By: ReplicaSet/rs-demo

# Test self-healing: delete a Pod
kubectl delete pod <pod-name>
kubectl get pods -l app=rs-demo
# New Pod auto-created! Still 3 total ✅

# Scale
kubectl scale rs rs-demo --replicas=5
kubectl get pods -l app=rs-demo
# Now 5 Pods ✅

kubectl scale rs rs-demo --replicas=3
kubectl get pods -l app=rs-demo
# Back to 3 Pods (2 terminated) ✅
```

## Step 2 — Break It Intentionally

### Break 1: Selector ≠ Template Labels
```yaml
spec:
  selector:
    matchLabels:
      app: rs-demo
  template:
    metadata:
      labels:
        app: WRONG-LABEL           # ← Doesn't match selector
```
```bash
kubectl apply -f broken.yaml
# Error: selector does not match template labels
# RS not created at all
```

### Break 2: Wrong apiVersion
```yaml
apiVersion: v1                       # ← Should be apps/v1
kind: ReplicaSet
```
```bash
kubectl apply -f broken.yaml
# Error: no matches for kind "ReplicaSet" in version "v1"
```

### Break 3: Wrong Image
```yaml
  template:
    spec:
      containers:
      - name: web
        image: nginx:this-tag-does-not-exist
```
```bash
kubectl get rs rs-demo
# DESIRED=3, CURRENT=3, READY=0
kubectl get pods -l app=rs-demo
# All 3 Pods in ImagePullBackOff
```

### Break 4: Excessive Resource Requests
```yaml
  template:
    spec:
      containers:
      - name: web
        resources:
          requests:
            cpu: "100"              # ← 100 full CPUs! Way too high
            memory: 500Gi
```
```bash
kubectl get rs rs-demo
# DESIRED=3, CURRENT=3, READY=0
kubectl get pods -l app=rs-demo
# All Pods Pending
kubectl describe pod <pod> | grep -A3 Events
# FailedScheduling: Insufficient cpu, Insufficient memory
```

### Break 5: Remove Label from Running Pod (Orphan It)
```bash
# Remove the label RS uses to select
kubectl label pod rs-demo-xxxxx app-
kubectl get pods -l app=rs-demo
# Only 2 Pods now (RS sees 2)

kubectl get pods
# 4 total Pods! The "orphaned" one still runs but lost its label
# RS created a replacement to maintain 3

kubectl get rs rs-demo
# DESIRED=3, CURRENT=3, READY=3 (RS happy with its 3)
# But orphaned Pod is still running unmanaged!
```

### Break 6: Create Stray Pod with Matching Labels
```bash
kubectl run stray-pod --image=nginx --labels="app=rs-demo"
kubectl get pods -l app=rs-demo
# 4 Pods briefly → RS deletes one to stay at 3
# RS might delete YOUR stray pod or one of its own!
kubectl get rs rs-demo
# DESIRED=3, CURRENT=3 — RS enforces its count
```

### Break 7: Wrong Command
```yaml
  template:
    spec:
      containers:
      - name: web
        image: nginx:1.25
        command: ['fake-command']     # ← Doesn't exist
```
```bash
kubectl get rs rs-demo
# DESIRED=3, CURRENT=3, READY=0
kubectl get pods -l app=rs-demo
```

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] ReplicaSet ensures desired Pod count using label selectors
- [ ] `selector.matchLabels` MUST match `template.metadata.labels` (validation enforced)
- [ ] RS is created/managed by Deployments — rarely created directly
- [ ] apiVersion: `apps/v1` (not `v1`)
- [ ] Editing RS template does NOT update existing Pods — must delete Pods to propagate
- [ ] RS uses ownerReferences — Pods "belong to" the RS
- [ ] Removing a Pod's label orphans it — RS creates replacement
- [ ] Creating a Pod with matching labels → RS may adopt and then delete it

### What to Focus On
- **selector ↔ template labels relationship** (exam tests this constantly)
- **Understanding DESIRED / CURRENT / READY** columns
- **Debugging RS-managed Pods** (ImagePull, CrashLoop, Pending)
- **Scaling** with `kubectl scale rs`
- **Rolling updates happen through Deployment** (which manages RS)

### What to Practice Repeatedly
- Create RS from scratch with correct selector/labels in under 2 minutes
- Break selector/label relationship → recognize validation error → fix
- Debug READY=0 on RS → identify Pod-level issue → fix
- Scale RS up and down
- Orphan a Pod by removing its label → observe RS behavior

### What to Ignore (Not CKA Relevant)
- ReplicationController (legacy, not tested)
- RS controller reconciliation internals
- `matchExpressions` advanced selector syntax (rarely needed)
- RS-level `minReadySeconds` (Deployment-level is more common)
- How ownerReferences garbage collection works internally

### 5 Quiz Questions

**Q1:** A ReplicaSet has `selector.matchLabels: {app: web}` and `template.metadata.labels: {app: api}`. What happens?
> **A:** RS creation fails with validation error: "selector does not match template labels." The selector MUST match the template labels.

**Q2:** You fix a wrong image in a ReplicaSet using `kubectl edit rs`. The existing 3 Pods still show the old image. Why?
> **A:** Editing RS template only affects NEW Pods. Existing Pods keep their original spec. You must delete existing Pods: `kubectl delete pods -l app=myapp`, and RS will recreate them with the updated template.

**Q3:** A ReplicaSet shows `DESIRED=3, CURRENT=3, READY=0`. What's your first debug step?
> **A:** `kubectl get pods -l <selector-labels>` to see Pod statuses. Then `kubectl describe pod <pod>` for the specific Pod issue (ImagePull, Pending, CrashLoop).

**Q4:** You remove the label `app=web` from one Pod managed by an RS with 3 replicas. What happens?
> **A:** RS detects only 2 matching Pods → creates a 3rd. The orphaned Pod keeps running but is unmanaged. You now have 4 running Pods total.

**Q5:** Why should you use Deployments instead of ReplicaSets directly?
> **A:** Deployments provide rolling updates, rollbacks, revision history, and declarative update strategy — all by managing ReplicaSets automatically. RS alone can't do rolling updates.

# 1️⃣3️⃣ Component Interaction Map

```
┌────────────────────────────────────────────────────────────────┐
│                    REPLICASET INTERACTIONS                       │
│                                                                 │
│  ┌──────────┐     creates      ┌──────────────┐                │
│  │Deployment│ ────────────────→ │  ReplicaSet  │                │
│  └──────────┘                   └──────┬───────┘                │
│                                        │ creates (via selector) │
│                              ┌─────────┼─────────┐             │
│                              ▼         ▼         ▼             │
│                          ┌──────┐  ┌──────┐  ┌──────┐         │
│                          │Pod-1 │  │Pod-2 │  │Pod-3 │         │
│                          └──┬───┘  └──┬───┘  └──┬───┘         │
│                             │         │         │              │
│                        Labels: app=myapp (same on all)         │
└────────────────────────────────┬───────────────────────────────┘
                                 │
          ┌──────────────────────┼──────────────────────┐
          ▼                      ▼                      ▼
    ┌──────────┐          ┌──────────┐          ┌──────────────┐
    │ Service  │          │ Scheduler│          │ Controller   │
    │          │          │          │          │ Manager      │
    └──────────┘          └──────────┘          └──────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Things Go Wrong |
|-----------|----------------|---------------------|
| **Deployment** | Creates RS with desired replicas and Pod template. During rolling update, creates new RS and scales old one down. | If Deployment spec is wrong → RS inherits the bad spec → Pods fail. Fix the Deployment. |
| **Controller Manager** | RS controller watches RS objects. Counts Pods matching selector. Creates/deletes to match `spec.replicas`. | Controller crashed (static Pod issue) → RS stops reconciling → dead Pods not replaced. |
| **Scheduler** | Schedules RS-created Pods to nodes based on resources, affinity, taints. | Insufficient resources → Pods stay Pending. Taints → Pods rejected. Affinity → no matching node. |
| **etcd** | Stores RS spec and status, Pod specs and status. | etcd down → no state → everything broken. |
| **Service** | Selects Pods by labels (same labels RS uses). Routes traffic to matching Pods. | If Pod labels change → Service loses endpoints too. If RS Pods not Ready → Service has fewer backends. |
| **Network** | Each RS-created Pod gets its own IP from CNI. | CNI failure → Pods have no IP → can't communicate. |
| **Storage** | If RS template includes PVCs → each Pod gets its own PVC binding. | PVC unbound → Pods Pending. For shared storage, all Pods mount same PVC (RWX needed). |
| **HPA** | Scales Deployment replica count → which adjusts RS replicas. | HPA misconfigured → RS replicas fluctuate wildly (thrashing). |

### Failure Cascade
```
RS template has bad image
  → RS creates 3 Pods with bad image
    → All 3 Pods: ImagePullBackOff
      → RS shows READY=0
        → Service has 0 endpoints
          → External traffic gets no response
            → Deployment shows unavailable replicas
              → HPA reports unhealthy

Controller Manager down
  → RS controller not running
    → If Pod dies, no replacement created
      → Gradually losing Pods
        → Service gradually loses backends
```
