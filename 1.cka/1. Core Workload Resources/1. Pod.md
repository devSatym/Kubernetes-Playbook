# 🧠 Kubernetes Pod — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Smallest deployable unit in Kubernetes
  - Wraps one or more containers that share network namespace, storage, and lifecycle
  - Provides a logical host — containers inside a Pod share `localhost` and can communicate via IPC
  - Kubernetes never schedules bare containers; everything runs inside a Pod

- **Real-World Use Case**
  - Single-container Pod → most common (one app per Pod)
  - Multi-container Pod → sidecar patterns (log shipper, proxy, adapter)
  - Init containers → run setup tasks before main container starts (DB migration, config fetch)
  - Ephemeral debug containers → `kubectl debug` attaches a temporary container

- **Where It Fits in Cluster Architecture**
  - Scheduler assigns Pod to a Node
  - Kubelet on the Node creates and monitors the Pod via container runtime (containerd / CRI-O)
  - Pod gets a unique IP from the CNI plugin
  - Controllers (Deployment, ReplicaSet, DaemonSet, Job) manage Pods — you rarely create Pods directly
  - etcd stores Pod spec and status
  - kube-apiserver validates and persists Pod objects

## 2️⃣ What Depends On It

- **Components That Rely on Pods**
  - **ReplicaSet / Deployment** → ensures desired count of Pods
  - **Service** → routes traffic to Pods matching label selector
  - **Endpoints / EndpointSlice** → auto-populated from Pod IPs
  - **HPA (Horizontal Pod Autoscaler)** → scales Pod count based on metrics
  - **PodDisruptionBudget** → controls voluntary disruptions
  - **NetworkPolicy** → selects Pods by label to enforce ingress/egress rules

- **Resources a Pod Interacts With**
  - **ConfigMap / Secret** → injected as env vars or volume mounts
  - **PersistentVolumeClaim (PVC)** → storage attachment
  - **ServiceAccount** → identity for API access and RBAC
  - **Node** → Pod is bound to exactly one Node after scheduling
  - **DNS (CoreDNS)** → resolves Service names for Pod networking

- **Upstream Dependencies (What Must Exist Before Pod Runs)**
  - Node must be Ready
  - Container image must be pullable (registry access, imagePullSecrets)
  - ConfigMap / Secret / PVC must exist if referenced
  - ServiceAccount must exist (default is auto-created)
  - ResourceQuota / LimitRange must allow the Pod's resource requests

- **Downstream Dependencies (What Breaks When Pod Dies)**
  - Service loses a backend endpoint
  - Dependent Pods (if using Pod DNS or IP directly) lose connectivity
  - Jobs / CronJobs report failure

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: v1              # Always v1 for Pods
kind: Pod
metadata:
  name: my-pod              # Required — unique within namespace
  namespace: default        # Optional — defaults to 'default'
  labels:                   # Critical for Service selectors, NetworkPolicy
    app: myapp
    tier: frontend
  annotations:              # Optional — metadata, not used for selection
    description: "demo pod"
spec:
  restartPolicy: Always     # Always | OnFailure | Never
  serviceAccountName: default
  containers:               # Required — at least one container
    - name: app             # Required — unique within Pod
      image: nginx:1.25     # Required — fully qualified recommended
      ports:
        - containerPort: 80 # Informational — does NOT open the port
      env:
        - name: ENV_VAR
          value: "hello"
      envFrom:
        - configMapRef:
            name: my-config
      volumeMounts:
        - name: data
          mountPath: /data
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
      livenessProbe:
        httpGet:
          path: /healthz
          port: 80
        initialDelaySeconds: 5
        periodSeconds: 10
      readinessProbe:
        httpGet:
          path: /ready
          port: 80
        initialDelaySeconds: 3
        periodSeconds: 5
  initContainers:
    - name: init-db
      image: busybox
      command: ['sh', '-c', 'until nslookup mydb; do sleep 2; done']
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: my-pvc
  nodeSelector:
    disktype: ssd
  tolerations:
    - key: "key1"
      operator: "Equal"
      value: "value1"
      effect: "NoSchedule"
```

- **Key Sections Explained**
  - `apiVersion: v1` → Pods are core API, always `v1`
  - `kind: Pod` → object type
  - `metadata` → name, namespace, labels, annotations
  - `spec.containers[]` → the workload definition
  - `spec.volumes[]` → storage sources
  - `spec.initContainers[]` → run-to-completion before main containers

- **Common Indentation Traps**
  - `containers` is a **list** (starts with `-`), not a map
  - `volumeMounts` is under a **container**, not under `spec` directly
  - `volumes` is under `spec`, not under a container
  - `env` entries each need `name` + either `value` or `valueFrom` — mixing causes errors
  - `resources.requests` and `resources.limits` are siblings, not nested

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `metadata.name` | Unique identifier in namespace | Pod creation fails on duplicate |
| `metadata.labels` | Service/NetworkPolicy/ReplicaSet selection | Service has no endpoints → no traffic |
| `spec.containers[].image` | Defines workload | `ErrImagePull` / `ImagePullBackOff` |
| `spec.containers[].command` / `args` | Overrides ENTRYPOINT/CMD | `CrashLoopBackOff` if wrong |
| `spec.containers[].ports.containerPort` | Informational but must match app | Misleading — doesn't block traffic but confuses debugging |
| `spec.containers[].resources.requests` | Scheduler uses this for placement | Pod stays `Pending` if cluster lacks resources |
| `spec.containers[].resources.limits` | OOM kill boundary | Container killed with `OOMKilled` exit code 137 |
| `spec.containers[].volumeMounts` | Mounts volume into container filesystem | App can't read/write data |
| `spec.containers[].env` / `envFrom` | Config injection | App misconfiguration, crash |
| `spec.containers[].livenessProbe` | Kubelet restarts container if probe fails | Restart loops if probe misconfigured |
| `spec.containers[].readinessProbe` | Removes Pod from Service endpoints if failing | Pod exists but gets no traffic |
| `spec.volumes[]` | Defines storage sources | Mount fails → Pod stuck in `ContainerCreating` |
| `spec.restartPolicy` | Controls container restart behavior | `Never` means no auto-recovery |
| `spec.serviceAccountName` | RBAC identity | API calls from Pod get 403 Forbidden |
| `spec.nodeSelector` | Constrains scheduling to labeled nodes | Pod stays `Pending` |
| `spec.tolerations` | Allows scheduling on tainted nodes | Pod stays `Pending` |
| `spec.nodeName` | Bypasses scheduler, pins to a node | Pod unschedulable if node is wrong/missing |

- **Field Relationships**
  - `spec.volumes[].name` ↔ `spec.containers[].volumeMounts[].name` — must match exactly
  - `metadata.labels` ↔ Service `spec.selector` — must match for traffic routing
  - `containerPort` ↔ Service `targetPort` — must match for correct port forwarding
  - `resources.requests` ↔ Node allocatable — scheduler compares these
  - `envFrom.configMapRef.name` ↔ actual ConfigMap name — must exist

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - ✅ `image` — is it correct? Is the tag right?
  - ✅ `labels` — do they match what Services/Deployments expect?
  - ✅ `resources.requests/limits` — are they set?
  - ✅ `volumeMounts` + `volumes` — do names match?
  - ✅ `command` / `args` — any overrides?
  - ✅ `restartPolicy` — appropriate for workload type?

- **Usually Irrelevant for CKA**
  - `metadata.uid`, `metadata.resourceVersion`, `metadata.creationTimestamp`
  - `status.podIP` (auto-assigned)
  - `metadata.managedFields`
  - Most annotations (unless specific like `scheduler.alpha.kubernetes.io`)
  - `terminationGracePeriodSeconds` (unless specifically tested)

- **Red Flags to Detect in 10 Seconds**
  - 🚩 Image tag `:latest` with `imagePullPolicy: Always` in air-gapped exam
  - 🚩 No `resources.requests` → may not schedule or may get evicted
  - 🚩 `volumeMounts.name` doesn't match any `volumes[].name`
  - 🚩 `command` overriding expected entrypoint
  - 🚩 `restartPolicy: Never` on a long-running workload
  - 🚩 Missing `envFrom` or `env` that the application needs

## 6️⃣ What to Notice In Commands

### `kubectl describe pod <name>`

- **Where to Look First**
  - `Status:` → Running / Pending / Failed / CrashLoopBackOff
  - `Conditions:` → PodScheduled, Initialized, ContainersReady, Ready
  - `Events:` section at the bottom → chronological history of what happened
  - `Containers: → State:` → Waiting / Running / Terminated + reason
  - `Last State:` → shows previous crash info (exit code, signal)

- **Failure Signals**
  - `Warning  FailedScheduling` → resource shortage or nodeSelector/taint mismatch
  - `Warning  Failed  ... ErrImagePull` → wrong image name, tag, or missing imagePullSecret
  - `Warning  BackOff  ... CrashLoopBackOff` → container starts and dies repeatedly
  - `Warning  FailedMount` → volume/PVC/Secret/ConfigMap issue
  - `Warning  Unhealthy` → liveness/readiness probe failing
  - Exit code `137` → OOMKilled (memory limit exceeded)
  - Exit code `1` → Application error
  - Exit code `127` → Command not found (bad entrypoint)
  - Exit code `126` → Permission denied on entrypoint

### `kubectl logs <pod> [-c container]`

- **Where to Look First**
  - Application startup errors (connection refused to DB, missing env var)
  - Stack traces or panic messages
  - Permission denied errors
  - "address already in use" → port conflict

- **Failure Signals**
  - Empty logs + CrashLoopBackOff → container exits before writing logs → check `command`/`args`
  - Repeated "connection refused" → dependency not ready
  - "file not found" → wrong volumeMount path or missing configMap

- **Useful Flags**
  - `--previous` → logs from the last crashed container (essential for CrashLoopBackOff)
  - `-f` → follow / stream logs
  - `--tail=50` → last 50 lines
  - `-c <container>` → specific container in multi-container Pod

### `kubectl exec -it <pod> -- <command>`

- **Where to Look First**
  - `exec -it <pod> -- sh` or `-- /bin/bash` → get a shell
  - Check if process is running: `ps aux`
  - Check networking: `curl localhost:<port>`, `nslookup <service>`
  - Check mounted volumes: `ls /data`, `cat /etc/config`
  - Check env vars: `env | grep MY_VAR`
  - Check DNS resolution: `nslookup kubernetes.default`

- **Failure Signals**
  - `OCI runtime exec failed` → container crashed or not running
  - Can't resolve DNS → CoreDNS issue or NetworkPolicy blocking
  - `curl` connection refused → app not listening on expected port
  - File missing in mounted path → volume mount or ConfigMap issue

### `kubectl explain pod.spec.containers`

- **Usage for CKA**
  - Quick field reference during exam — faster than docs
  - `kubectl explain pod.spec --recursive` → shows full tree
  - `kubectl explain pod.spec.containers.livenessProbe` → drill into specific fields
  - Shows `FIELD`, `TYPE`, and `DESCRIPTION` for each

## 7️⃣ Failure Patterns (Very Important)

### Common Misconfigurations

| Issue | Status | Event / Clue |
|---|---|---|
| Wrong image name/tag | `ErrImagePull` → `ImagePullBackOff` | `Failed to pull image` |
| Missing imagePullSecret | `ErrImagePull` → `ImagePullBackOff` | `unauthorized: authentication required` |
| Bad command/args | `CrashLoopBackOff` | Exit code 1, 126, or 127 |
| Missing ConfigMap/Secret ref | `CreateContainerConfigError` | `configmaps "x" not found` |
| Missing PVC | `Pending` | `persistentvolumeclaim "x" not found` |
| Volume name mismatch | Pod creation fails | `volumes[0].name not found` in volumeMounts |
| Insufficient resources | `Pending` | `Insufficient cpu` / `Insufficient memory` |
| Node selector mismatch | `Pending` | `FailedScheduling: 0/N nodes available` |
| Taint with no toleration | `Pending` | `node(s) had taints that the pod didn't tolerate` |
| Liveness probe wrong path/port | `CrashLoopBackOff` | `Liveness probe failed: HTTP probe failed with statuscode: 404` |
| Readiness probe failing | Running but 0/1 Ready | `Readiness probe failed` |
| OOM (memory limit too low) | `OOMKilled` | Exit code 137, `Last State: Terminated, Reason: OOMKilled` |
| Security context violation | `CrashLoopBackOff` | Permission denied in logs |
| Wrong port in container spec | Running but unreachable | No direct error — must debug via `exec` + `curl` |

### Status Values — What They Mean

| Status | Meaning | Action |
|---|---|---|
| `Pending` | Not yet scheduled or waiting for resources | Check events: resource shortage, nodeSelector, taints |
| `ContainerCreating` | Image pull or volume mount in progress | Check events: pull errors, mount errors |
| `Running` | At least one container running | Check readiness, liveness probes |
| `CrashLoopBackOff` | Container keeps crashing, kubelet backing off restart | Check `logs --previous`, describe events, exit code |
| `ErrImagePull` | First image pull failure | Check image name, tag, registry, imagePullSecret |
| `ImagePullBackOff` | Repeated image pull failures | Same as ErrImagePull but backoff timer increasing |
| `CreateContainerConfigError` | Missing ConfigMap/Secret | Check referenced ConfigMaps/Secrets exist |
| `Completed` | Container exited successfully (exit 0) | Normal for Jobs, unexpected for long-running Pods |
| `Terminating` | Pod is shutting down | Check if stuck — may need `--force --grace-period=0` |
| `OOMKilled` | Container exceeded memory limit | Increase `resources.limits.memory` |
| `Evicted` | Node under resource pressure | Check node conditions, resource requests |
| `Unknown` | Node lost contact | Check node status, kubelet health |

### Real Exam-Style Failure Scenarios

1. **Pod stuck Pending** → Cluster has 3 nodes, all with taint `env=prod:NoSchedule`, Pod has no toleration
2. **CrashLoopBackOff** → Pod command is `["slepe", "3600"]` — typo in command
3. **ImagePullBackOff** → Image is `ngnix:1.25` instead of `nginx:1.25`
4. **Pod Running but Service has no endpoints** → Pod labels are `app: myapp` but Service selector is `app: my-app`
5. **CreateContainerConfigError** → Pod references `configMapRef: name: app-config` but ConfigMap is named `app-cfg`
6. **Pod runs but app fails** → `envFrom` references a Secret that exists but has wrong key names
7. **OOMKilled after load** → Memory limit is 64Mi for a Java app that needs 256Mi

## 8️⃣ Misconfiguration Behavior

- **Labels Mismatch**
  - Pod runs fine — labels don't affect Pod lifecycle
  - BUT: Service selector won't match → `kubectl get endpoints <svc>` shows empty ENDPOINTS
  - NetworkPolicy won't apply → Pod is either unrestricted or fully blocked depending on default policy
  - ReplicaSet won't manage the Pod → orphaned Pod, controller creates new ones

- **Ports Mismatch**
  - `containerPort` is **informational only** — doesn't actually open or close ports
  - If Service `targetPort` doesn't match the port the app actually listens on → connection refused
  - If `readinessProbe` port doesn't match app port → probe fails → Pod not Ready → removed from Service endpoints
  - The container still listens on whatever port the application code specifies

- **Image Is Wrong**
  - Pod status: `ErrImagePull` → `ImagePullBackOff`
  - Container never starts — no logs available
  - Events show: `Failed to pull image "wrong-image:tag": rpc error: code = NotFound`
  - Fix: `kubectl set image pod/my-pod app=correct-image:tag` (only works for managed Pods)

- **Resource Limits Are Wrong**
  - **Requests too high** → Pod stays `Pending` (scheduler can't find a node)
  - **Limits too low (memory)** → `OOMKilled`, exit code 137, container restarts
  - **Limits too low (CPU)** → CPU throttled, app runs slow but doesn't crash
  - **No limits set** → Pod can consume all node resources, may get evicted under pressure

- **Dependencies Missing**
  - **ConfigMap missing** → `CreateContainerConfigError`, Pod never starts
  - **Secret missing** → `CreateContainerConfigError`, Pod never starts
  - **PVC missing** → Pod stays `Pending`, event: `persistentvolumeclaim "x" not found`
  - **ServiceAccount missing** → Pod creation fails with admission error
  - **Init container dependency unavailable** → Init container keeps restarting, main container never starts, Pod shows `Init:0/1`

## 9️⃣ Fast Detection Strategy

### Exact Commands (in order)

```bash
# 1. Overview — spot status anomalies (10 sec)
kubectl get pods -o wide

# 2. Deep dive into failing Pod (15 sec)
kubectl describe pod <pod-name>
# → Scroll to Events section FIRST
# → Check Status, Conditions, Container State

# 3. Check logs (10 sec)
kubectl logs <pod-name>
kubectl logs <pod-name> --previous    # if CrashLoopBackOff
kubectl logs <pod-name> -c <container>  # multi-container

# 4. Check related resources (10 sec)
kubectl get events --sort-by=.metadata.creationTimestamp
kubectl get svc,ep                     # if networking issue

# 5. Interactive debug (15 sec if needed)
kubectl exec -it <pod-name> -- sh
# Inside: curl localhost:<port>, env, ls /mount-path, nslookup <svc>
```

### Debugging Decision Tree

```
Pod not running?
├── Status: Pending
│   ├── Check: kubectl describe → Events
│   ├── Cause: Insufficient resources? → Scale cluster or reduce requests
│   ├── Cause: NodeSelector mismatch? → Fix selector or label node
│   └── Cause: Taint not tolerated? → Add toleration
├── Status: CrashLoopBackOff
│   ├── Check: kubectl logs --previous
│   ├── Cause: Exit 127? → Command not found → Fix command/image
│   ├── Cause: Exit 137? → OOMKilled → Increase memory limit
│   ├── Cause: Exit 1? → App error → Check logs for details
│   └── Cause: Liveness probe? → Check probe config
├── Status: ImagePullBackOff
│   ├── Check: kubectl describe → Events
│   ├── Cause: Image name typo? → Fix image
│   └── Cause: Private registry? → Add imagePullSecret
├── Status: ContainerCreating (stuck)
│   ├── Check: kubectl describe → Events
│   ├── Cause: Volume mount failed? → Check PVC/ConfigMap/Secret
│   └── Cause: Image pull slow? → Wait or check network
└── Pod Running but not working?
    ├── Check: kubectl logs
    ├── Check: kubectl exec → curl, env, dns
    ├── Cause: Readiness probe failing? → Fix probe
    └── Cause: Label mismatch with Service? → Fix labels
```

## 🔟 Fast Fix Strategy

### Fix Methods

```bash
# Edit Pod spec live (limited — most fields are immutable)
kubectl edit pod <pod-name>
# Can change: image, activeDeadlineSeconds, tolerations (with some limits)

# Replace Pod (for immutable fields)
kubectl get pod <pod-name> -o yaml > pod-fix.yaml
# Edit pod-fix.yaml
kubectl delete pod <pod-name>
kubectl apply -f pod-fix.yaml

# Quick image fix
kubectl set image pod/<pod-name> <container>=<new-image>

# Fix via Deployment (if Pod is managed)
kubectl edit deployment <deployment-name>
# OR
kubectl set image deployment/<deployment-name> <container>=<new-image>

# Patch specific fields
kubectl patch pod <pod-name> -p '{"metadata":{"labels":{"app":"fixed-label"}}}'

# Fix ConfigMap/Secret (Pod restart needed to pick up changes)
kubectl edit configmap <name>
kubectl delete pod <pod-name>   # controller recreates it

# Force delete stuck Pod
kubectl delete pod <pod-name> --force --grace-period=0
```

### Validate Fix

```bash
# Confirm Pod is running
kubectl get pod <pod-name> -w          # watch status transition

# Confirm all containers ready
kubectl get pod <pod-name>             # should show 1/1 or N/N READY

# Confirm endpoints populated (if Service involved)
kubectl get endpoints <service-name>

# Confirm application responding
kubectl exec -it <pod-name> -- curl localhost:<port>

# Confirm events are clean
kubectl describe pod <pod-name>        # Events should show Normal only
```

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

```yaml
# working-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-pod
  labels:
    app: test
    env: dev
spec:
  containers:
    - name: web
      image: nginx:1.25
      ports:
        - containerPort: 80
      resources:
        requests:
          cpu: "100m"
          memory: "64Mi"
        limits:
          cpu: "200m"
          memory: "128Mi"
      readinessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 3
        periodSeconds: 5
      livenessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 5
        periodSeconds: 10
```

```bash
kubectl apply -f working-pod.yaml
kubectl get pod test-pod -w         # wait for Running + 1/1 Ready
kubectl describe pod test-pod       # verify clean events
kubectl logs test-pod               # verify no errors
kubectl exec test-pod -- curl -s localhost:80  # verify response
```

### Step 2 — Break It Intentionally

| # | Breakage | Change to Make | Expected Behavior |
|---|----------|---------------|-------------------|
| 1 | Wrong image | `image: ngnix:1.25` (typo) | `ErrImagePull` → `ImagePullBackOff` |
| 2 | Bad command | Add `command: ["slepe", "infinity"]` | `CrashLoopBackOff`, exit 127 |
| 3 | Remove labels | Remove `app: test` label | Pod runs, but Service (if any) loses endpoint |
| 4 | Wrong liveness port | Change liveness `port: 8080` | Probe fails → container restarts → `CrashLoopBackOff` |
| 5 | Missing ConfigMap ref | Add `envFrom: [{configMapRef: {name: noexist}}]` | `CreateContainerConfigError` |
| 6 | Memory limit too low | Set `limits.memory: "1Mi"` | `OOMKilled`, exit 137 |
| 7 | Wrong nodeSelector | Add `nodeSelector: {gpu: "true"}` | `Pending` — no matching node |
| 8 | Missing volume ref | Add volumeMount referencing non-existent volume name | Pod creation fails — validation error |
| 9 | Invalid restart policy | `restartPolicy: Never` + crashing command | Pod goes to `Failed`, no auto-restart |
| 10 | Wrong readiness path | Change readiness `path: /nonexistent` | Pod Running but 0/1 Ready — removed from endpoints |

### Step 3 — Debug Without Looking at Solution

- **First command**: `kubectl get pods` — identify status anomaly
- **Second command**: `kubectl describe pod <name>` — read Events section bottom-up
- **Narrow the issue**:
  - `Pending` → scheduling issue → check nodeSelector, taints, resources
  - `CrashLoopBackOff` → `kubectl logs --previous` → look at exit code
  - `ImagePullBackOff` → read the event message for exact image name
  - `CreateContainerConfigError` → missing dependency, read event
  - Running but not Ready → `kubectl describe` → readiness probe status
- **Confirm root cause**: Match the event message to the intentional breakage
- **Fix**: Edit YAML, reapply, watch Pod recover

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

### What to Learn
- Pod lifecycle: Pending → ContainerCreating → Running → Succeeded/Failed
- Multi-container patterns: sidecar, init container, ephemeral debug container
- Resource management: requests, limits, QoS classes (Guaranteed, Burstable, BestEffort)
- Probes: liveness, readiness, startup — types (httpGet, tcpSocket, exec)
- Volumes and volume mounts in Pod context
- Pod scheduling: nodeSelector, nodeName, taints/tolerations, affinity
- Static Pods (managed by kubelet directly, manifest in `/etc/kubernetes/manifests`)
- ServiceAccount association and Pod-level RBAC
- Pod security: securityContext (runAsUser, runAsNonRoot, capabilities)
- Environment variable injection from ConfigMap, Secret, field refs

### What to Focus On
- Creating Pods from scratch via YAML in under 2 minutes
- `kubectl run` imperative command for quick Pod creation
  - `kubectl run mypod --image=nginx --port=80 --dry-run=client -o yaml > pod.yaml`
- Debugging Pods: describe → logs → exec → events flow
- Fixing Pods: edit, delete+apply, set image
- Static Pod manipulation (exam tested frequently)
- Init container troubleshooting

### What to Practice Repeatedly
- Write Pod YAML from memory
- Break and fix Pods 10 different ways
- `kubectl run` → `kubectl expose` → validate end-to-end
- Multi-container Pod YAML with shared volume
- Static Pod creation in `/etc/kubernetes/manifests`

### What to Ignore (Not CKA Required)
- Pod internal implementation in kubelet source code
- CNI plugin internals (just know Pods get IPs)
- Container runtime details (containerd vs CRI-O internals)
- PodPresets (deprecated)
- Pod Topology Spread Constraints (advanced, rarely tested in CKA)

### 5 Quiz Questions

1. **Q: A Pod is in `Pending` status with the event `0/3 nodes are available: 3 node(s) had taints that the pod didn't tolerate`. What is the fix?**
   - A: Add a matching `toleration` to the Pod spec, or remove the taint from a node using `kubectl taint node <node> <key>-`

2. **Q: A Pod shows `CrashLoopBackOff` with exit code 127. What does this mean?**
   - A: Exit code 127 = "command not found". The `command` field has a typo or the binary doesn't exist in the image.

3. **Q: You create a Service with selector `app: web`, but `kubectl get endpoints <svc>` shows no endpoints. The Pod is Running. What's wrong?**
   - A: The Pod labels don't match the Service selector. Check Pod labels with `kubectl get pod --show-labels`.

4. **Q: How do you create a static Pod?**
   - A: Place a Pod manifest YAML file in the kubelet's static pod directory (default `/etc/kubernetes/manifests/`). The kubelet automatically creates and manages it.

5. **Q: A Pod has `readinessProbe` on port 8080, but the app listens on port 80. What happens?**
   - A: The readiness probe fails continuously. The Pod shows `Running` but `0/1 Ready`. It is removed from all Service endpoints and receives no traffic.

## 1️⃣3️⃣ Component Interaction Map

```
                          ┌──────────────┐
                          │     etcd     │
                          │ (stores Pod  │
                          │  spec+status)│
                          └──────┬───────┘
                                 │
                          ┌──────┴───────┐
                          │  API Server  │
```

### Interaction Behavior When Things Go Wrong

| Component | Normal Behavior | Failure Behavior |
|---|---|---|
| **etcd** | Stores Pod object persistently | If etcd is down → API server can't persist Pod → creation fails |
| **API Server** | Validates and stores Pod spec | If API server is down → no Pod CRUD possible |
| **Scheduler** | Binds Pod to a Node | If scheduler is down → Pods stay `Pending` (no `nodeName` assigned) |
| **Controller Manager** | Ensures desired Pod count via ReplicaSet | If down → no auto-healing, no scaling, no rollouts |
| **Kubelet** | Creates containers, runs probes, reports status | If kubelet is down → Pods on that node go `Unknown`, no restarts |
| **Node** | Provides compute resources | Node `NotReady` → Pods evicted after timeout (5 min default) |
| **Service** | Routes traffic to Pod IPs | If Pod labels don't match → Service has no endpoints → traffic blackhole |
| **CNI / Network** | Assigns Pod IP, enables inter-Pod communication | CNI failure → Pod stuck in `ContainerCreating`, no IP assigned |
| **Storage (PVC)** | Provides persistent volume to Pod | PVC not bound → Pod stuck `Pending`; mount failure → `ContainerCreating` |
| **ConfigMap/Secret** | Provides config/credentials | Missing → `CreateContainerConfigError`; wrong content → app misconfiguration |
| **CoreDNS** | Resolves Service names | If CoreDNS Pods are down → DNS resolution fails → Pods can't reach Services by name |

## 📌 Quick Reference Commands

```bash
# Create Pod imperatively
kubectl run mypod --image=nginx --port=80

# Generate YAML without creating
kubectl run mypod --image=nginx --dry-run=client -o yaml > pod.yaml

# Get all Pods with labels and node info
kubectl get pods -o wide --show-labels

# Watch Pod status
kubectl get pods -w

# Describe Pod (most important debug command)
kubectl describe pod <name>

# Logs
kubectl logs <pod>
kubectl logs <pod> --previous
kubectl logs <pod> -c <container>

# Exec into Pod
kubectl exec -it <pod> -- /bin/sh

# Delete Pod
kubectl delete pod <name>
kubectl delete pod <name> --force --grace-period=0

# Quick field lookup during exam
kubectl explain pod.spec.containers
kubectl explain pod.spec.containers.livenessProbe --recursive

# Set image (managed Pod)
kubectl set image pod/<name> <container>=<image>

# Patch labels
kubectl label pod <name> app=newlabel --overwrite

# Create static Pod
sudo cp pod.yaml /etc/kubernetes/manifests/

# Check events cluster-wide
kubectl get events --sort-by=.metadata.creationTimestamp -A
```
