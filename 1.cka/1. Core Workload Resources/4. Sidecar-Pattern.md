# 🧠 Sidecar Pattern — CKA Deep Dive Mind Map
## Senior CKA Trainer & DevOps Architect Perspective

---

# 1️⃣ What It Does

## Core Purpose
- A **helper container** that runs alongside the main application container inside the same Pod
- **Enhances, extends, or supports** the main container without modifying its code
- Both containers share the same:
  - Network namespace → communicate via `localhost`
  - Storage volumes → share files via `emptyDir` or PVC
  - Lifecycle → start together, die together
- The sidecar is a **long-running** container (unlike init containers which run-to-completion)
- Follows the **separation of concerns** principle — main app does business logic, sidecar handles cross-cutting tasks

## Real-World Use Cases

### Logging Sidecar (Most Common CKA Scenario)
- Main app writes logs to a file → sidecar reads file and streams to central logging
- Example: App writes to `/var/log/app.log` → Fluentd/Fluent Bit sidecar tails and ships logs
```
┌─────────────────────────────────────────────┐
│  POD                                         │
│  ┌──────────┐   shared vol   ┌───────────┐  │
│  │   App    │──→ /var/log ──→│  Logging   │  │
│  │ (writes  │                │  Sidecar   │──→ Central Log System
│  │  logs)   │                │ (tails log)│  │
│  └──────────┘                └───────────┘  │
└─────────────────────────────────────────────┘
```

### Proxy Sidecar
- Sidecar handles TLS termination, authentication, or rate-limiting
- Example: Envoy sidecar in service mesh (Istio)
- Main app talks to `localhost:8080` → sidecar forwards encrypted to remote service

### Config/Sync Sidecar
- Sidecar watches for ConfigMap changes and updates config files
- Sidecar pulls latest code/config from git repository on a schedule

### Adapter Sidecar
- Transforms data format (app produces format A → sidecar converts to format B)
- Standardizes metrics/logs across heterogeneous apps

### Monitoring Sidecar
- Exports metrics from main app for Prometheus scraping
- Example: Custom exporter sidecar exposing `/metrics` endpoint

## Where It Fits in Cluster Architecture
```
┌────────────────────────────────────────────────────────────┐
│                         NODE                                │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                      POD                              │  │
│  │                                                       │  │
│  │   ┌─────────────┐     ┌──────────────┐               │  │
│  │   │    Main      │     │   Sidecar    │               │  │
│  │   │  Container   │←──→│  Container   │               │  │
│  │   │  (app logic) │     │  (helper)    │               │  │
│  │   └──────┬───────┘     └──────┬───────┘               │  │
│  │          │    localhost:PORT    │                       │  │
│  │          └────────┬────────────┘                       │  │
│  │           Shared Volume (emptyDir)                     │  │
│  │           Shared Pod IP                                │  │
│  │           Same ServiceAccount                          │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  Service selects Pod by labels → traffic goes to Pod IP    │
│  Both containers handle different aspects of the workload  │
└────────────────────────────────────────────────────────────┘
```

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Main container** — May depend on sidecar for logging, proxying, config sync
- **kubelet** — Manages both containers independently within the Pod
- **Service** — Selects the Pod; traffic may target main container's port or sidecar's port
- **Monitoring systems** — May scrape sidecar's metrics endpoint

## What Resources It Interacts With
- **Shared volumes** — `emptyDir` (most common), `configMap`, `secret`, PVC
- **Pod network** — Both containers share Pod IP; communicate via `localhost`
- **ConfigMaps / Secrets** — Sidecar may read config, translate, write to shared volume
- **ServiceAccount** — Same SA as the Pod (same API permissions)
- **Probes** — Sidecar has its OWN liveness/readiness probes (separate from main)
- **Resources** — Sidecar has its OWN `resources.requests/limits`

## Upstream & Downstream Dependencies
```
UPSTREAM:
  Deployment / DaemonSet / StatefulSet
    → creates Pod with main + sidecar containers

DOWNSTREAM:
  Sidecar depends on:
    → Shared volume being writable
    → Main container writing data (if log-shipping pattern)
    → Network access (if proxy pattern)
    → ConfigMap/Secret (if config-sync pattern)

  Main container may depend on:
    → Sidecar being ready (if proxy pattern)
    → Sidecar writing config files (if config-sync pattern)

LIFECYCLE:
  Init containers (exit) → Main + Sidecar start SIMULTANEOUSLY
  If sidecar crashes → kubelet restarts ONLY the sidecar
  If main crashes → kubelet restarts ONLY the main container
  Pod is Ready only when ALL containers' readiness probes pass
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Valid Sidecar Pattern YAML

### Pattern A: Logging Sidecar (CKA Favorite)
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-logging
  labels:
    app: sidecar-demo
spec:
  containers:
  # --- Main Application Container ---
  - name: app
    image: busybox:1.36
    command: ['sh', '-c',
      'while true; do echo "$(date) INFO App running" >> /var/log/app.log; sleep 5; done']
    volumeMounts:
    - name: log-volume
      mountPath: /var/log              # App writes logs here

  # --- Sidecar Container (Log Shipper) ---
  - name: log-sidecar
    image: busybox:1.36
    command: ['sh', '-c', 'tail -f /logs/app.log']    # Streams logs to stdout
    volumeMounts:
    - name: log-volume
      mountPath: /logs                 # Reads logs from here (same volume!)

  # --- Shared Volume ---
  volumes:
  - name: log-volume
    emptyDir: {}                       # Ephemeral storage shared between containers
```

### Pattern B: Proxy/Ambassador Sidecar
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-proxy
spec:
  containers:
  - name: app
    image: myapp:1.0
    ports:
    - containerPort: 8080
    env:
    - name: DB_HOST
      value: "localhost"               # App talks to sidecar, NOT remote DB directly
    - name: DB_PORT
      value: "3306"

  - name: db-proxy
    image: cloud-sql-proxy:2.0         # Sidecar handles auth & TLS to remote DB
    ports:
    - containerPort: 3306
    command: ['./cloud-sql-proxy', '--port=3306', 'my-project:us-central1:my-db']
```

### Pattern C: Config Sync Sidecar
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-sync
spec:
  containers:
  - name: app
    image: nginx:1.25
    volumeMounts:
    - name: web-content
      mountPath: /usr/share/nginx/html

  - name: git-sync
    image: registry.k8s.io/git-sync:v4.0
    env:
    - name: GITSYNC_REPO
      value: "https://github.com/example/static-site.git"
    - name: GITSYNC_ROOT
      value: "/data"
    volumeMounts:
    - name: web-content
      mountPath: /data

  volumes:
  - name: web-content
    emptyDir: {}
```

## ⚠️ Common Indentation Traps

### Trap 1: Sidecar nested inside main container (WRONG)
```yaml
# ❌ FATAL — sidecar is NOT a sub-element of app container
containers:
- name: app
  image: nginx
  - name: log-sidecar        # ← WRONG! Not a new container, YAML parse error
    image: busybox

# ✅ CORRECT — both at same list level
containers:
- name: app
  image: nginx
- name: log-sidecar           # ← Same level as app
  image: busybox
```

### Trap 2: Volume mounted in sidecar but not defined in Pod
```yaml
# ❌ WRONG — volume "logs" referenced but not defined
containers:
- name: log-sidecar
  volumeMounts:
  - name: logs              # ← References non-existent volume
    mountPath: /logs
# Missing:
# volumes:
# - name: logs
#   emptyDir: {}
```

### Trap 3: Same mountPath in both containers (may be intentional)
```yaml
# ⚠️ Both containers mount at same path — works but can cause confusion
containers:
- name: app
  volumeMounts:
  - name: shared
    mountPath: /data          # ← Both containers see /data
- name: sidecar
  volumeMounts:
  - name: shared
    mountPath: /data          # ← Same path — both read/write same files at /data
# This is VALID but could cause race conditions
# Different mountPaths for different containers is often cleaner
```

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| `containers[].name` | Identifies sidecar for `logs -c`, `exec -c` | Can't target sidecar; duplicate names = error | 🔴 |
| `containers[].image` | Wrong image = container crash or wrong behavior | `ImagePullBackOff` for sidecar | 🔴 |
| `containers[].command` | Sidecar's main task (tail logs, run proxy, sync) | Wrong command = CrashLoopBackOff or no-op | 🔴 |
| `containers[].volumeMounts` | How sidecar accesses shared data from main | Can't read main container's logs/data | 🔴 |
| `containers[].volumeMounts[].name` | Must match `volumes[].name` | Volume not found error | 🔴 |
| `containers[].volumeMounts[].mountPath` | Where in sidecar filesystem the volume appears | Sidecar looks in wrong path | 🟡 |
| `containers[].ports[].containerPort` | Sidecar's listening port (if proxy/metrics) | Port conflict if same as main container | 🟡 |
| `containers[].resources` | CPU/memory for sidecar | OOMKilled, or total requests too high → Pending | 🟡 |
| `volumes[]` | Shared storage definition between main and sidecar | Missing volume → creation error | 🔴 |

## Field Relationships
```
Main container writes to: volumeMounts[].mountPath (e.g., /var/log)
  ↕ SAME volume name
Sidecar reads from: volumeMounts[].mountPath (e.g., /logs)
  ↕ Same volumes[].name connects them
  ↕ mountPath can DIFFER — only the volume name must match

Main container port (e.g., 8080) ←— must NOT conflict ——→ Sidecar port (e.g., 3306)
  Both accessible via localhost within the Pod

Sidecar readiness probe failure → Pod NOT Ready → Pod removed from Service endpoints
  Even if main container is healthy!
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- How many containers? Which one is main, which is sidecar?
- Do both containers mount the same volume? (check `volumeMounts[].name`)
- Is the sidecar `command` correct and will it run indefinitely? (sidecar should NOT exit)
- Are container names unique?
- Are ports different between main and sidecar?

## 🟢 Usually Irrelevant for CKA
- `imagePullPolicy` (defaults are fine)
- `terminationGracePeriodSeconds`
- `hostNetwork`, `dnsPolicy`
- Sidecar-specific environment variables (unless the question is about them)
- `lifecycle` hooks (preStop, postStart)

## 🔴 Red Flags in 10 Seconds

| Red Flag | Problem |
|----------|---------|
| Sidecar `command` that exits immediately (e.g., `echo done`) | Sidecar enters `CrashLoopBackOff` (restartPolicy: Always restarts it) |
| Sidecar `command: tail -f /logs/app.log` but main writes to `/var/log/app.log` | Mount path mismatch → sidecar tails nonexistent file |
| Sidecar and main both use `containerPort: 80` | Port conflict → one container crashes with "address in use" |
| Sidecar has `volumeMounts.name: logs` but `volumes` has `name: log-vol` | Name mismatch → volume not found → container creation fails |
| Sidecar missing `volumeMounts` entirely when it needs shared data | Sidecar can't access main container's output |
| Sidecar with `resources.limits.memory: 10Mi` tailing large log files | OOMKilled when log file gets large |

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe pod <pod>`

### Where to Look FIRST
```
Containers:
  app:
    State:          Running ✅
    Ready:          True
    Restart Count:  0

  log-sidecar:
    State:          Waiting
      Reason:       CrashLoopBackOff    ← PROBLEM — sidecar crashing
    Last State:     Terminated
      Reason:       Error
      Exit Code:    1                   ← Check logs for why
    Restart Count:  7                   ← Keeps restarting
    Ready:          False               ← Pod not fully Ready

Conditions:
  Initialized:     True                 ← Init containers done (if any)
  ContainersReady: False                ← NOT all containers ready
  Ready:           False                ← Pod pulled from Service endpoints!
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| Sidecar `State: CrashLoopBackOff` | Sidecar keeps crashing; check logs |
| Sidecar `Restart Count: N` (growing) | Sidecar unstable; check command & image |
| Sidecar `Ready: False`, Main `Ready: True` | Sidecar readiness probe failing; Pod removed from Service |
| Main `Ready: False`, Sidecar `Ready: True` | Main app issue, not sidecar |
| Both containers `Running` but Pod `Ready: False` | Readiness probe(s) failing |
| Events: `Back-off restarting failed container log-sidecar` | Sidecar crash loop |

## 📝 `kubectl logs <pod> -c <container>`

```bash
# MUST specify -c for multi-container pods
kubectl logs sidecar-logging -c app            # Main container logs
kubectl logs sidecar-logging -c log-sidecar    # Sidecar logs

# Previous crashed sidecar logs
kubectl logs sidecar-logging -c log-sidecar --previous

# Follow sidecar logs live
kubectl logs sidecar-logging -c log-sidecar -f

# All containers in one shot (useful for quick overview)
kubectl logs sidecar-logging --all-containers=true
```

### What to Look For
| Log Pattern | Meaning |
|-------------|---------|
| `tail: can't open '/logs/app.log': No such file` | Mount path mismatch — sidecar looking at wrong path |
| `Address already in use` | Port conflict with main container |
| `Permission denied` | SecurityContext issue or volume permissions |
| `connection refused` on `localhost:PORT` | Main container not ready yet, or wrong port |
| No output at all | Command exiting immediately (check exit code) |
| Repeating sidecar output | Working correctly ✅ (sidecar tails/streams data) |

## 🔧 `kubectl exec -it <pod> -c <container> -- <cmd>`

```bash
# Shell into sidecar
kubectl exec -it sidecar-logging -c log-sidecar -- /bin/sh

# Shell into main container
kubectl exec -it sidecar-logging -c app -- /bin/sh

# Verify shared volume from SIDECAR side
kubectl exec sidecar-logging -c log-sidecar -- ls -la /logs/
kubectl exec sidecar-logging -c log-sidecar -- cat /logs/app.log

# Verify shared volume from MAIN side
kubectl exec sidecar-logging -c app -- ls -la /var/log/
kubectl exec sidecar-logging -c app -- cat /var/log/app.log

# Test localhost communication (proxy pattern)
kubectl exec sidecar-logging -c app -- curl -s localhost:3306
kubectl exec sidecar-logging -c log-sidecar -- wget -qO- localhost:8080

# Verify both containers share same IP
kubectl exec sidecar-logging -c app -- hostname -i
kubectl exec sidecar-logging -c log-sidecar -- hostname -i   # Same IP!
```

### What to Verify via exec
- Both containers see the same files on the shared volume
- `localhost` communication works between containers
- Both containers have the same Pod IP
- File permissions on shared volume are correct
- Sidecar's expected input file/data exists

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Sidecar Image Not Found
```
POD STATUS:     Running (partial) or CrashLoopBackOff
READY:          1/2 (main running, sidecar failing)
CAUSE:          Wrong sidecar image name or tag
EVENTS:         "Failed to pull image", "ErrImagePull"
BEHAVIOR:       Main container may start (if sidecar image pull happens in parallel)
                Pod Ready = False → removed from Service endpoints
FIX:            Fix sidecar image name/tag
EXAM TIP:       kubectl describe pod → check which container has ImagePullBackOff
```

## Pattern 2: Sidecar Command Exits Immediately
```
POD STATUS:     CrashLoopBackOff
READY:          1/2
CAUSE:          Sidecar command runs and exits (e.g., 'echo done')
                restartPolicy: Always → kubelet keeps restarting it
EVENTS:         "Back-off restarting failed container log-sidecar"
DESCRIBE:       Exit Code: 0 (succeeded but exited!) or non-zero
BEHAVIOR:       Restart count climbs rapidly. Main container fine.
FIX:            Make sidecar command long-running (tail -f, while loop, daemon)
EXAM TIP:       Exit Code 0 with CrashLoopBackOff = command completed but shouldn't have
```

## Pattern 3: Sidecar Reads Wrong File Path
```
POD STATUS:     Running (both containers up)
READY:          2/2
CAUSE:          Main writes to /var/log/app.log, sidecar tails /logs/output.log
BEHAVIOR:       NO crash! Sidecar runs but tails an empty/nonexistent file
                Sidecar logs show nothing or "No such file"
                Hardest to detect — Pod looks healthy
FIX:            Align mountPath + file paths between main and sidecar
EXAM TIP:       When "sidecar not shipping logs" → verify paths with exec + ls
```

## Pattern 4: Port Conflict Between Main and Sidecar
```
POD STATUS:     CrashLoopBackOff (second container to bind fails)
READY:          1/2
CAUSE:          Both containers try to bind the same port (e.g., 80)
LOGS:           "bind: address already in use"
BEHAVIOR:       One container works, the other keeps crashing
FIX:            Use different ports for each container
EXAM TIP:       Check containerPort values for conflicts
```

## Pattern 5: Sidecar OOMKilled
```
POD STATUS:     Running but RESTARTS increasing
READY:          1/2 (intermittent)
CAUSE:          Sidecar memory limit too low for its workload
DESCRIBE:       Last State: Terminated, Reason: OOMKilled, Exit Code: 137
BEHAVIOR:       Sidecar killed → restarted → killed again → backoff increases
FIX:            Increase resources.limits.memory for sidecar
EXAM TIP:       Exit Code 137 = OOMKilled (128 + signal 9)
```

## Pattern 6: Sidecar Volume Not Mounted
```
POD STATUS:     Running (2/2) but sidecar does nothing useful
CAUSE:          Sidecar missing volumeMounts — can't access shared data
BEHAVIOR:       Sidecar runs but can't read main container's output
                No crash — LOGIC error
FIX:            Add volumeMounts to sidecar with correct volume name and path
EXAM TIP:       exec into sidecar, `ls` the expected path, verify it's empty
```

## Pattern 7: Sidecar Readiness Probe Fails → Pod Removed from Service
```
POD STATUS:     Running
READY:          1/2 (sidecar not ready)
CAUSE:          Sidecar readiness probe fails (wrong port, path, or startup delay)
BEHAVIOR:       Pod is Running but NOT Ready
                Removed from Service endpoints → no traffic routed
                Main container is perfectly fine but unreachable
FIX:            Fix sidecar readiness probe or add initialDelaySeconds
EXAM TIP:       describe → check Conditions: ContainersReady: False
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Labels Are Wrong?
- Labels on the Pod itself affect Service selector matching
- **Wrong labels → Service has no endpoints → no traffic reaches Pod**
- The sidecar pattern itself is unaffected by labels
- But: if you're asked to expose a sidecar-enhanced Pod via Service, labels MUST match selector

## What Happens If Ports Conflict?
- **Both bind port 80:** Second container to start gets `Address already in use`
- **Only one crashes;** the other runs fine
- `kubectl describe` → only the crashing container shows Error
- **Fix:** Give each container a unique port. Main on 8080, sidecar on 9090. Use `localhost:PORT` between them

## What Happens If Sidecar Image Is Wrong?
- **Sidecar:** `ImagePullBackOff` — sidecar never starts
- **Main container:** MAY still run (depends on Pod startup order)
- **Pod status:** `READY: 1/2` — one container running, one failing
- **Service impact:** Pod removed from endpoints (not fully Ready)
- **Fix:** Correct the sidecar image name/tag

## What Happens If Resource Limits Are Wrong?
- **Sidecar OOMKilled:** Only sidecar dies, main keeps running
- **Sidecar restart → Exit Code 137 → CrashLoopBackOff**
- **Total resources too high:** Pod `Pending` — scheduler can't place it
  - Scheduler adds `resources.requests` of ALL containers
  - Example: Main requests 500m CPU + Sidecar requests 500m CPU = Node needs 1000m free
- **Fix:** Increase sidecar limits or reduce requests

## What Happens If Shared Volume Is Missing?
```yaml
# Scenario: volumeMounts references "data" but no volume "data" defined
containers:
- name: sidecar
  volumeMounts:
  - name: data           # ← No matching volumes[] entry
    mountPath: /data
```
- **Behavior:** Pod creation fails with validation error
- **Error:** `volume "data" not found`
- **Fix:** Add `volumes: [{name: data, emptyDir: {}}]`

## What Happens If Dependencies Are Missing (ConfigMap/Secret)?
- If sidecar references nonexistent ConfigMap → `CreateContainerConfigError`
- Sidecar never starts → Pod `READY: 1/2`
- Main container may still run
- **Fix:** Create the missing ConfigMap/Secret

# 9️⃣ Fast Detection Strategy

## 60-Second Debug Flow

```bash
# ⏱️ 0-10 sec: Quick status
kubectl get pod <pod> -o wide
# Read: READY (1/2 = one container failing), RESTARTS, STATUS

# ⏱️ 10-20 sec: Which container is the problem?
kubectl describe pod <pod> | grep -E "Name:|State:|Reason:|Exit Code:|Ready:|Restart"
# Scan both container blocks — find the failing one

# ⏱️ 20-35 sec: Why is it failing?
kubectl logs <pod> -c <failing-container>
kubectl logs <pod> -c <failing-container> --previous    # If CrashLoop

# ⏱️ 35-45 sec: Check events
kubectl describe pod <pod> | tail -15
# Look for: ImagePull errors, BackOff, OOMKilled

# ⏱️ 45-55 sec: Verify shared volume (if applicable)
kubectl exec <pod> -c <working-container> -- ls -la /shared/path

# ⏱️ 55-60 sec: Verify localhost communication (if proxy pattern)
kubectl exec <pod> -c app -- curl -s localhost:<sidecar-port>
```

## Quick Diagnosis Table
```
SYMPTOM                               → LIKELY CAUSE
───────────────────────────────────────────────────────────
READY 1/2, sidecar CrashLoopBackOff  → Bad command/image, sidecar exits
READY 1/2, sidecar ImagePullBackOff  → Wrong sidecar image name
READY 2/2 but sidecar logs empty     → Mount path mismatch (logic error)
READY 1/2, Exit Code 137             → Sidecar OOMKilled
READY 1/2, Exit Code 0 + restarts    → Sidecar command exits (should be long-running)
READY 1/2, Exit Code 127             → Command not found in sidecar image
READY 2/2, Service returns error     → Wrong port; Service targets wrong container
Pod Pending                           → Combined resource requests too high
```

# 🔟 Fast Fix Strategy

## Fix Decision Tree
```
Is Pod managed by Deployment?
├── YES → kubectl edit deployment <name>
│         Edit sidecar under spec.template.spec.containers[]
│         Deployment auto-recreates Pods
└── NO (standalone Pod) → Most spec fields IMMUTABLE
    1. kubectl get pod <name> -o yaml > fix.yaml
    2. vim fix.yaml (fix sidecar image/command/volume/resources)
    3. kubectl delete pod <name>
    4. kubectl apply -f fix.yaml
```

## Common Fixes

### Fix: Sidecar wrong image
```bash
# Deployment:
kubectl set image deployment/<dep> log-sidecar=busybox:1.36

# Standalone Pod:
kubectl get pod <pod> -o yaml > fix.yaml
# Edit image → delete pod → apply fix.yaml
```

### Fix: Sidecar command exits immediately
```bash
# BEFORE (exits):
command: ['echo', 'done']

# AFTER (long-running):
command: ['sh', '-c', 'tail -f /logs/app.log']
# OR
command: ['sh', '-c', 'while true; do process-data; sleep 5; done']
```

### Fix: Volume mount path mismatch
```bash
# Verify what main writes:
kubectl exec <pod> -c app -- ls /var/log/

# Fix sidecar to read from correct path:
# BEFORE: mountPath: /wrong/path
# AFTER:  mountPath: /var/log
```

### Fix: Port conflict
```bash
# BEFORE: Both containers use port 80
# AFTER: Main on 80, sidecar on 9090
# Update sidecar containerPort and its command to listen on new port
```

### Fix: Sidecar OOMKilled
```bash
# In Deployment:
kubectl edit deployment <dep>
# Find sidecar container → increase resources.limits.memory
# e.g., from 64Mi to 256Mi
```

## Validate Fix
```bash
# All containers running
kubectl get pod <pod>
# READY: 2/2, STATUS: Running, RESTARTS stable

# Sidecar functioning
kubectl logs <pod> -c <sidecar>
# Should show expected output (log lines, proxy activity, etc.)

# Shared volume working
kubectl exec <pod> -c <sidecar> -- ls /shared/path
kubectl exec <pod> -c <sidecar> -- cat /shared/path/file

# Localhost communication working (proxy pattern)
kubectl exec <pod> -c app -- curl -s localhost:<sidecar-port>
```

## Confirm Recovery
```bash
# Watch stability
watch kubectl get pod <pod>
# RESTARTS should NOT increase
# READY should stay 2/2

# Service endpoints restored
```

# 1️⃣1️⃣ Hands-On Learning Cycle

## Step 1 — Create Working Example

### Logging Sidecar (CKA Exam Staple)
```yaml
# File: sidecar-demo.yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-demo
  labels:
    app: sidecar-demo
spec:
  containers:
  # Main application — writes logs to file
  - name: app
    image: busybox:1.36
    command:
    - 'sh'
    - '-c'
    - |
      i=0
      while true; do
        echo "$(date) LOG_ENTRY_$i: Application running normally" >> /var/log/app.log
        i=$((i+1))
        sleep 3
      done
    volumeMounts:
    - name: log-volume
      mountPath: /var/log

  # Sidecar — reads log file and streams to stdout (for kubectl logs)
  - name: log-agent
    image: busybox:1.36
    command: ['sh', '-c', 'tail -n+1 -F /logs/app.log']
    volumeMounts:
    - name: log-volume
      mountPath: /logs

  volumes:
  - name: log-volume
    emptyDir: {}
```

### Validation Steps
```bash
# Create
kubectl apply -f sidecar-demo.yaml

# Verify both containers running
kubectl get pod sidecar-demo
# Expected: READY 2/2, STATUS Running

# Check main app is writing logs
kubectl exec sidecar-demo -c app -- cat /var/log/app.log
# Should show timestamped log entries

# Check sidecar is streaming logs
kubectl logs sidecar-demo -c log-agent
# Should show the SAME log entries (read from shared volume)

# Verify shared volume
kubectl exec sidecar-demo -c app -- ls /var/log/
kubectl exec sidecar-demo -c log-agent -- ls /logs/
# Both should show app.log

# Verify same Pod IP
kubectl exec sidecar-demo -c app -- hostname -i
kubectl exec sidecar-demo -c log-agent -- hostname -i
# Same IP ✅

# Follow sidecar logs live
kubectl logs sidecar-demo -c log-agent -f
# New entries every 3 seconds ✅
```

## Step 2 — Break It Intentionally

### Break 1: Wrong Sidecar Image
```yaml
  - name: log-agent
    image: busybox:this-does-not-exist    # ← Bad tag
```
```bash
kubectl get pod sidecar-demo
# READY: 1/2, one container ImagePullBackOff
# Main app still running, sidecar can't start
```

### Break 2: Sidecar Command Exits Immediately
```yaml
  - name: log-agent
    command: ['echo', 'sidecar started']   # ← Exits after printing
```
```bash
kubectl get pod sidecar-demo
# READY: 1/2, CrashLoopBackOff (sidecar keeps exiting)
# Exit Code: 0 — succeeded but exited (restartPolicy: Always restarts it)
kubectl logs sidecar-demo -c log-agent --previous
# "sidecar started" — then it exited
```

### Break 3: Volume Mount Path Mismatch
```yaml
  - name: app
    volumeMounts:
    - name: log-volume
      mountPath: /var/log        # App writes to /var/log/app.log

  - name: log-agent
    command: ['sh', '-c', 'tail -f /output/app.log']   # ← WRONG path
    volumeMounts:
    - name: log-volume
      mountPath: /logs           # Volume mounted at /logs, NOT /output
```
```bash
kubectl get pod sidecar-demo
# READY: 1/2 or 2/2 depending on tail behavior
kubectl logs sidecar-demo -c log-agent
# "tail: can't open '/output/app.log': No such file or directory"
# Sidecar crashes → CrashLoopBackOff
```

### Break 4: Volume Name Mismatch
```yaml
  - name: log-agent
    volumeMounts:
    - name: wrong-volume         # ← Doesn't match volumes[].name
      mountPath: /logs
  volumes:
  - name: log-volume             # ← Name is log-volume, not wrong-volume
    emptyDir: {}
```
```bash
kubectl apply -f broken.yaml
# Error: volume "wrong-volume" not found
# Pod creation rejected entirely
```

### Break 5: Port Conflict
```yaml
  containers:
  - name: web
    image: nginx:1.25
    ports:
    - containerPort: 80
  - name: another-web
    image: nginx:1.25             # ← Also listens on port 80!
    ports:
    - containerPort: 80
```
```bash
kubectl get pod sidecar-demo
# READY: 1/2, CrashLoopBackOff for second nginx
kubectl logs sidecar-demo -c another-web
# "nginx: [emerg] bind() to 0.0.0.0:80 failed (98: Address already in use)"
```

### Break 6: Sidecar OOM
```yaml
  - name: log-agent
    image: busybox:1.36
    command: ['sh', '-c', 'cat /dev/urandom > /dev/null']   # Memory hungry
    resources:
      limits:
        memory: 10Mi              # ← Too low
```
```bash
kubectl get pod sidecar-demo
# RESTARTS increasing for log-agent
kubectl describe pod sidecar-demo | grep -A3 "log-agent"
# Reason: OOMKilled, Exit Code: 137
```

### Break 7: Missing Volume Definition Entirely
```yaml
  containers:
  - name: app
    command: ['sh', '-c', 'echo data >> /var/log/app.log']
    # ← No volumeMount for app

  - name: log-agent
    command: ['sh', '-c', 'tail -f /var/log/app.log']
    # ← No volumeMount for sidecar

  # ← No volumes[] at all
```
```bash
# Pod runs! Both containers start. BUT:
# App writes to container-local /var/log/app.log
# Sidecar has its OWN /var/log/ (different filesystem!)
# Sidecar sees nothing because there's no shared volume
kubectl exec sidecar-demo -c log-agent -- cat /var/log/app.log
# Empty or "No such file" — data never shared
```

## Step 3 — Debug Without Looking at Solution

### First Command Always
```bash
kubectl get pod <name>
# Read: READY (2/2 or 1/2?), STATUS, RESTARTS
```

### Narrow the Issue
```bash
# Which container is failing?
kubectl describe pod <name> | grep -B1 -A5 "State:"
# Find container with State: Waiting or Terminated

# What's the error?
kubectl logs <name> -c <failing-container>
kubectl logs <name> -c <failing-container> --previous

# If both Running but sidecar not working:
kubectl exec <name> -c <sidecar> -- ls /expected/path
kubectl exec <name> -c <sidecar> -- cat /expected/path/file
```

### Confirm Root Cause
```bash
# Image issue → describe shows ImagePullBackOff
# Command exits → Exit Code 0 + CrashLoopBackOff
# Command fails → Exit Code non-zero + logs show error
# Path mismatch → exec shows missing file at expected path
# Port conflict → logs show "Address already in use"
# OOM → Exit Code 137, Reason: OOMKilled
# Volume missing → Pod creation error

# After fix:
kubectl get pod <name> -w
# Watch READY go from 1/2 → 2/2
```

# 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

## CKA Domain: Workloads & Scheduling (15%)

### What to Learn
- [ ] Sidecar is just another entry in `spec.containers[]` (no special K8s API)
- [ ] Sidecars run long-running processes (unlike init containers)
- [ ] Volume sharing via `emptyDir` between main and sidecar
- [ ] `localhost` communication between containers in same Pod
- [ ] Each container has independent: probes, resources, security context
- [ ] Pod is Ready only when ALL containers pass readiness probes
- [ ] `kubectl logs -c` and `kubectl exec -c` to target specific container
- [ ] Common patterns: logging sidecar, proxy sidecar, config sync sidecar

### What to Focus On
- **Logging sidecar pattern** — most likely CKA scenario
- Main writes to file → sidecar tails file → output visible in `kubectl logs -c sidecar`
- Volume sharing mechanics (same volume name, different mount paths is OK)
- Debugging which container is failing in a multi-container Pod
- Understanding READY x/y → which of the y containers are not ready

### What to Practice Repeatedly
- Create a logging sidecar Pod from scratch in under 3 minutes
- Break sidecar (wrong path, wrong image, port conflict) → debug → fix → under 2 minutes
- Use `kubectl logs -c`, `kubectl exec -c`, `kubectl describe` for multi-container debugging
- Add a sidecar to an existing Pod/Deployment YAML
- Explain READY 1/2 vs 2/2 and what each means

### What to Ignore (Not CKA Relevant)
- Service mesh sidecar injection internals (Istio, Linkerd)
- Sidecar container KEP (Kubernetes Enhancement Proposal) details
- Ambassador pattern deep theory
- Adapter pattern deep theory
- Container lifecycle hooks (postStart/preStop) for sidecars
- Sidecar ordering guarantees (not firmly specified)

### 5 Quiz Questions

**Q1:** A Pod has a main container writing to `/var/log/app.log` and a sidecar that should stream those logs. The sidecar mounts the shared volume at `/data`. What command should the sidecar use?
> **A:** `tail -f /data/app.log` — the volume is mounted at `/data` in the sidecar, so the file path is `/data/app.log`, not `/var/log/app.log`.

**Q2:** A Pod shows `READY: 1/2`. Both containers are in `Running` state. What's the most likely cause?
> **A:** One container's readiness probe is failing. The container is running but not considered ready. Check `kubectl describe pod` for readiness probe details.

**Q3:** A sidecar has `command: ['echo', 'hello']` and the Pod shows `CrashLoopBackOff` with Exit Code 0. Why?
> **A:** The sidecar command ran successfully (exit 0) but then exited. Since `restartPolicy: Always` (default), kubelet keeps restarting it, and it keeps exiting. Fix: make the command long-running (e.g., `tail -f`, `while true; do ... done`).

**Q4:** Can a sidecar container use a different image than the main container?
> **A:** Yes. Each container in a Pod can use a completely different image. This is common — e.g., main uses `nginx`, sidecar uses `fluentd` for log shipping.

**Q5:** How does a sidecar proxy communicate with the main container?
> **A:** Via `localhost:<port>`. Both containers share the same network namespace (same Pod IP), so they can reach each other on `localhost`. Main app connects to `localhost:3306`, sidecar listens on `3306` and forwards to remote database.

# 1️⃣3️⃣ Component Interaction Map

```
┌─────────────────────────────────────────────────────────────────┐
│                    SIDECAR PATTERN POD                            │
│                                                                  │
│  ┌─────────────┐   emptyDir   ┌─────────────┐                  │
│  │    Main      │←────────────→│   Sidecar   │                  │
│  │  Container   │  (shared)    │  Container  │                  │
│  │  Port: 8080  │  localhost   │ Port: 9090  │                  │
│  └──────────────┘              └─────────────┘                  │
│         ↑                            ↑                           │
│    Pod IP:10.244.0.5           Same Pod IP                      │
└─────────────────────────┬───────────────────────────────────────┘
                          │
         ┌────────────────┼────────────────┐
         ▼                ▼                ▼
   ┌──────────┐    ┌──────────┐    ┌──────────────┐
   │ Service  │    │ kubelet  │    │ Scheduler    │
   │          │    │          │    │              │
   └──────────┘    └──────────┘    └──────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Sidecar Fails |
|-----------|----------------|-------------------|
| **kubelet** | Starts both containers in parallel after inits. Manages each independently. | Restarts only the sidecar (not main). Reports container-level status to API server. |
| **Scheduler** | Sums `resources.requests` of ALL containers for placement. | If combined requests too high → Pod `Pending`. Sidecar's resources add to scheduling burden. |
| **Controller Manager** | Deployment controller maintains replica count. | If sidecar keeps crashing → Pod not Ready → Deployment reports unavailable replicas. Rollout may stall. |
| **etcd** | Stores Pod spec with all container definitions and per-container status. | Each container's state (Running/Waiting/Terminated) stored separately. |
| **Service** | Selects Pod by labels. Routes traffic to Pod IP + target port. | Sidecar readiness fails → Pod Ready=False → Pod removed from Service endpoints → zero traffic. Even if main is healthy! |
| **Network (CNI)** | One network namespace for entire Pod. Both containers share it. | If CNI fails → entire Pod has no network → both containers affected. |
| **Storage** | Provisions volumes at Pod schedule time. Available to all containers. | If PVC unbound → Pod `Pending`. If emptyDir → always available. |
| **Probes** | Each container has independent probes. Pod Ready = ALL containers ready. | Sidecar readiness probe fail → Pod not Ready even if main is fine. This is the #1 subtle failure. |

### Failure Cascade
```
Sidecar crashes
  → kubelet restarts sidecar container only
    → If keeps crashing: CrashLoopBackOff (sidecar only)
      → Pod READY: 1/2
        → Condition: ContainersReady=False, Ready=False
          → Service removes Pod from endpoints
            → External traffic stops reaching this Pod
              → If Deployment: unavailable replica count increases
                → If hits maxUnavailable: rollout stalls

Main container keeps running the entire time!
But no traffic reaches it because Pod is not "Ready"
```

# 1️⃣4️⃣ Self-Evaluation Questions

## ✔ Can I Break This Intentionally?

| Breakage | How to Do It |
|----------|-------------|
| Sidecar exits immediately | `command: ['echo', 'done']` |
| Sidecar crashes with error | `command: ['sh', '-c', 'exit 1']` |
| Sidecar image pull failure | `image: fake-image:v999` |
| Sidecar OOMKilled | `limits.memory: 5Mi` + memory-heavy command |
| Port conflict | Both containers bind same port (e.g., 80) |
| Sidecar can't read shared data | Mount volume at wrong path in sidecar |
| No shared volume at all | Remove volumes + volumeMounts from both |
| Sidecar kills Service traffic | Add readiness probe to sidecar that always fails |

**If you can do ALL 8 → you fully understand sidecar failure modes.**

## ✔ Can I Detect Failure in Under 60 Seconds?

```bash
# 10 sec: Check Pod
kubectl get pod <name>

# READY 2/2 but wrong behavior? → logic error (path mismatch)

# 15 sec: Which container?
kubectl describe pod <name> | grep -E "Name:|State:|Reason:|Exit Code:|Restart"

# 15 sec: Why?
kubectl logs <name> -c <sidecar> [--previous]

# 10 sec: Events
kubectl describe pod <name> | tail -15

# 10 sec: Verify shared volume
kubectl exec <name> -c <sidecar> -- ls /expected/path
```

## ✔ Can I Fix Without Googling?

| Problem | Fix From Memory |
|---------|----------------|
| Sidecar `ImagePullBackOff` | Fix image → recreate Pod or edit Deployment |
| Sidecar `CrashLoopBackOff` (Exit 0) | Command exits — make it long-running (`tail -f`, `while true`) |
| Sidecar `CrashLoopBackOff` (Exit 1) | Logs show error → fix command/dependencies |
| Sidecar `OOMKilled` (Exit 137) | Increase `resources.limits.memory` |
| Port conflict | Change one container's port → update command |
| Path mismatch (no crash but no data) | Align mountPath between containers, verify with `exec` |
| Volume name mismatch | Match `volumeMounts.name` with `volumes.name` |
| Sidecar readiness kills Service | Fix probe config or remove readiness probe from sidecar |
