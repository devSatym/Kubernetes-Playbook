# 1️⃣ What It Does

## Core Purpose
- Pods managed **directly by the kubelet** on a specific node — **NOT through the API server**
- kubelet watches a local directory for YAML manifest files and automatically creates/manages Pods
- No Deployment, ReplicaSet, or any controller involved — kubelet IS the controller
- If the manifest file exists → Pod runs. Delete the file → Pod dies. Edit the file → Pod is recreated.
- kubelet creates a **mirror Pod** in the API server (read-only representation visible via `kubectl`)

## Real-World Use Cases
- **Control plane components** — This is THE primary use case:
  - `kube-apiserver` — runs as static Pod
  - `kube-controller-manager` — runs as static Pod
  - `kube-scheduler` — runs as static Pod
  - `etcd` — runs as static Pod
  - All defined in `/etc/kubernetes/manifests/` on the control plane node
- **Node-level services** — Custom static Pods for node monitoring or logging
- **Bootstrap scenarios** — Running essential services before API server is available

## Where It Fits in Cluster Architecture
```
┌──────────────────────────────────────────────────────────────────┐
│                    CONTROL PLANE NODE                              │
│                                                                   │
│  /etc/kubernetes/manifests/          kubelet watches this dir     │
│  ├── kube-apiserver.yaml      ──→  kubelet creates Pod           │
│  ├── kube-controller-manager.yaml ──→  kubelet creates Pod       │
│  ├── kube-scheduler.yaml      ──→  kubelet creates Pod           │
│  └── etcd.yaml                ──→  kubelet creates Pod           │
│                                                                   │
│  kubelet ──→ Creates Pod directly                                │
│           ──→ Creates "mirror Pod" in API server (read-only)     │
│           ──→ Monitors manifest file for changes                 │
│           ──→ Restarts Pod if container crashes                   │
│           ──→ Deletes Pod if manifest file is removed            │
│                                                                   │
│  API Server:                                                      │
│    kubectl get pods -n kube-system                                │
│    Shows mirror Pods: kube-apiserver-<node-name>                  │
│    These are READ-ONLY — cannot delete via kubectl!              │
└──────────────────────────────────────────────────────────────────┘
```

### Static Pod vs Regular Pod
| Aspect | Static Pod | Regular Pod |
|--------|-----------|-------------|
| Created by | kubelet (from manifest file) | API server (via controller) |
| Managed by | kubelet directly | Controller (Deployment, RS, etc.) |
| Scheduler involved? | ❌ No | ✅ Yes |
| Visible in `kubectl`? | ✅ Yes (mirror Pod, read-only) | ✅ Yes (full control) |
| Can delete via `kubectl`? | ❌ No (mirror Pod comes back) | ✅ Yes |
| Naming convention | `<name>-<node-name>` (e.g., `kube-apiserver-master`) | `<name>-<random-hash>` |
| Manifest location | Node filesystem (`/etc/kubernetes/manifests/`) | etcd (via API server) |
| Node-bound? | ✅ Yes (always on that node) | ❌ No (scheduler decides) |

# 2️⃣ What Depends On It

## Which Components Rely On It
- **Entire Kubernetes control plane** depends on static Pods:
  - Without `kube-apiserver` static Pod → no API server → cluster is dead
  - Without `etcd` static Pod → no data store → cluster loses all state
  - Without `kube-scheduler` static Pod → new Pods can't be scheduled
  - Without `kube-controller-manager` static Pod → controllers don't run
- **kubelet** — directly manages static Pods (kubelet IS the only controller)
- **Mirror Pod mechanism** — API server stores read-only representations

## What Resources It Interacts With
- **Node filesystem** — manifest directory (typically `/etc/kubernetes/manifests/`)
- **kubelet configuration** — `staticPodPath` in kubelet config defines the directory
- **Container runtime** — containerd/CRI-O runs the actual containers
- **Host network** — control plane static Pods often use `hostNetwork: true`
- **Host volumes** — static Pods mount host paths for certs, data, config
  - `/etc/kubernetes/pki/` — certificates
  - `/var/lib/etcd/` — etcd data
  - `/etc/kubernetes/` — kubeconfig files

## Upstream & Downstream Dependencies
```
UPSTREAM:
  Manifest file on disk → kubelet reads it → kubelet creates Pod
  No Deployment, No ReplicaSet, No Scheduler — just kubelet + file

DOWNSTREAM:
  Static Pod running → API server available
  API server available → kubectl works
  kubectl works → Regular Pods can be created
  Regular Pods created → Scheduler assigns them
  Controllers run → Deployments, Services, etc. work

CRITICAL CHAIN:
  kubelet service running
    → reads /etc/kubernetes/manifests/
      → starts static Pods (apiserver, etcd, scheduler, controller-manager)
        → API server becomes available
          → cluster is functional
```

# 3️⃣ Recognize Structure (YAML Skeleton)

## Minimal Static Pod Manifest
```yaml
# File: /etc/kubernetes/manifests/my-static-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-static-pod                    # Pod name (node name gets appended)
  namespace: kube-system                  # Typically kube-system for control plane
  labels:
    component: my-static-pod
spec:
  containers:
  - name: my-container
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: data
      mountPath: /data
  hostNetwork: true                       # Common for control plane static Pods
  volumes:
  - name: data
    hostPath:
      path: /var/lib/my-data
      type: DirectoryOrCreate
```

## Real Control Plane Static Pod Example (kube-apiserver)
```yaml
# File: /etc/kubernetes/manifests/kube-apiserver.yaml
apiVersion: v1
kind: Pod
metadata:
  name: kube-apiserver
  namespace: kube-system
  labels:
    component: kube-apiserver
    tier: control-plane
spec:
  hostNetwork: true                       # Uses host network (port 6443 on node IP)
  containers:
  - name: kube-apiserver
    image: registry.k8s.io/kube-apiserver:v1.31.0
    command:
    - kube-apiserver
    - --advertise-address=192.168.1.10
    - --etcd-servers=https://127.0.0.1:2379
    - --service-cluster-ip-range=10.96.0.0/12
    - --client-ca-file=/etc/kubernetes/pki/ca.crt
    - --tls-cert-file=/etc/kubernetes/pki/apiserver.crt
    - --tls-private-key-file=/etc/kubernetes/pki/apiserver.key
    # ... many more flags
    livenessProbe:
      httpGet:
        host: 192.168.1.10
        path: /livez
        port: 6443
        scheme: HTTPS
    volumeMounts:
    - name: k8s-certs
      mountPath: /etc/kubernetes/pki
      readOnly: true
    - name: etcd-certs
      mountPath: /etc/kubernetes/pki/etcd
      readOnly: true
  volumes:
  - name: k8s-certs
    hostPath:
      path: /etc/kubernetes/pki
      type: DirectoryOrCreate
  - name: etcd-certs
    hostPath:
      path: /etc/kubernetes/pki/etcd
      type: DirectoryOrCreate
```

## How to Find the Static Pod Manifest Path
```bash
# Method 1: Check kubelet config
cat /var/lib/kubelet/config.yaml | grep staticPodPath
# Output: staticPodPath: /etc/kubernetes/manifests

# Method 2: Check kubelet process flags
ps aux | grep kubelet | grep -- --pod-manifest-path
# Or check systemd drop-in:
cat /etc/systemd/system/kubelet.service.d/10-kubeadm.conf

# Method 3: Check kubelet config file referenced by systemd
systemctl cat kubelet
# Find --config flag → read that config file → find staticPodPath
```

## ⚠️ Common Traps

### Trap 1: Putting manifest in wrong directory
```bash
# ❌ WRONG — file in home directory (kubelet doesn't watch this)
/root/my-static-pod.yaml

# ❌ WRONG — file in /etc/kubernetes/ but NOT in manifests/
/etc/kubernetes/my-static-pod.yaml

# ✅ CORRECT — must be in the staticPodPath directory
/etc/kubernetes/manifests/my-static-pod.yaml
```

### Trap 2: Trying to delete via kubectl
```bash
# ❌ This will NOT work — mirror Pod is recreated instantly
kubectl delete pod my-static-pod-master -n kube-system
# Pod disappears briefly then reappears!

# ✅ CORRECT — remove the manifest file from the node
ssh node-master
rm /etc/kubernetes/manifests/my-static-pod.yaml
# kubelet detects removal → deletes the Pod
```

### Trap 3: YAML syntax error in manifest file
```yaml
# ❌ Invalid YAML — kubelet silently ignores the file
apiVersion: v1
kind: Pod
metadata:
  name: broken
spec:
  containers:
  - name: app
    image: nginx
     ports:              # ← Extra space! Invalid YAML
    - containerPort: 80
```
- kubelet logs an error but may not create the Pod
- No kubectl event to see (Pod never reaches API server)
- Must check `journalctl -u kubelet` for YAML parse errors

# 4️⃣ Critical Fields Only (CKA Focus)

## High-Impact Fields

| Field | Why It Matters | What Breaks | CKA Priority |
|-------|---------------|-------------|-------------|
| Manifest file location | Must be in `staticPodPath` directory | Pod never created if file in wrong dir | 🔴 |
| `metadata.name` | Pod name; node name appended for mirror Pod | Wrong name → can't find Pod via kubectl | 🔴 |
| `metadata.namespace` | Where mirror Pod appears | Default `default`; control plane uses `kube-system` | 🟡 |
| `spec.containers[].image` | What runs | `ImagePullBackOff` — kubelet retries; control plane may crash | 🔴 |
| `spec.containers[].command` | Container startup | Wrong flags → component won't start → cluster broken | 🔴 |
| `spec.hostNetwork` | Use node's network stack | Control plane needs this; wrong setting → port binding issues | 🔴 |
| `spec.volumes[].hostPath` | Mount node filesystem | Wrong cert/data paths → component can't read certs or data | 🔴 |
| Liveness probe | Restarts container if health fails | Wrong probe → healthy Pod killed; or missing probe → no health check | 🟡 |

## Key Relationships
```
Manifest file on disk ←——watched by——→ kubelet (staticPodPath)
  kubelet creates Pod ←——mirrors to——→ API server (read-only mirror Pod)

Pod name: "kube-apiserver" ←+——→ Mirror Pod: "kube-apiserver-master"
                              └── Node name appended: "-<node-name>"

hostPath volumes ←——point to——→ Node filesystem paths
  /etc/kubernetes/pki/ — certificates for TLS
  /var/lib/etcd/ — etcd data directory
  /etc/kubernetes/ — kubeconfig files

hostNetwork: true ←——means——→ Pod uses node's IP and ports directly
  API server on port 6443 → accessible at <node-IP>:6443
```

# 5️⃣ How Much Should You Notice in YAML?

## ⚡ Scan Immediately (5 seconds)
- Is the file in `/etc/kubernetes/manifests/` (or whatever `staticPodPath` is)?
- Is `kind: Pod`? (Static pods are always Pods, never Deployments)
- Is the `image` correct and pullable?
- For control plane: are `command` flags correct (cert paths, etcd endpoints)?
- Is `hostNetwork: true` set (for control plane components)?
- Are `hostPath` volume paths correct (certs, data directories)?

## 🟢 Usually Irrelevant for CKA
- `dnsPolicy` (hostNetwork Pods default to `ClusterFirstWithHostNet`)
- `priorityClassName` (control plane uses `system-node-critical`)
- `tolerations` (control plane tolerates `NoSchedule`)
- `ownerReferences` (kubelet manages these)
- Detailed resource limits on control plane static Pods

## 🔴 Red Flags in 10 Seconds

| Red Flag | Problem |
|----------|---------|
| File not in `/etc/kubernetes/manifests/` | kubelet won't see it → Pod never created |
| File has `.yml` or `.json` extension on some kubelet configs | May need `.yaml` — check consistency |
| `kind: Deployment` in manifest file | Static Pods MUST be `kind: Pod` only |
| Wrong image version (1.30 vs 1.31) | Version mismatch → component may not start |
| Typo in `command` flags (`--etcd-servrs` instead of `--etcd-servers`) | Component crashes at startup |
| Wrong cert path (`/etc/kubernetes/pki/apiserver.cet` vs `.crt`) | TLS handshake fails → component crashes |
| Missing `hostNetwork: true` for API server | API server binds to Pod IP, not node IP → unreachable |
| `hostPath.path: /var/lib/etdc/` (typo) | etcd can't find data directory → data loss |

# 6️⃣ What to Notice in Key Commands

## 📋 `kubectl describe pod <static-pod-name> -n kube-system`

### This Works on the MIRROR Pod
```bash
# Static Pods have node name appended
kubectl describe pod kube-apiserver-master -n kube-system
kubectl describe pod etcd-master -n kube-system
kubectl describe pod kube-scheduler-master -n kube-system
kubectl describe pod kube-controller-manager-master -n kube-system
```

### Where to Look
```
Name:         kube-apiserver-master          ← "<name>-<node>" = static Pod indicator
Namespace:    kube-system
Node:         master/192.168.1.10            ← Always bound to this node
Controlled By: Node/master                    ← "Node" = static Pod (not Deployment/RS)

Containers:
  kube-apiserver:
    State:       Running ✅
    Image:       registry.k8s.io/kube-apiserver:v1.31.0
    Port:        6443/TCP
    Restart Count: 0                          ← High restarts = component instability

Events:
  Normal   Pulled    kubelet  Successfully pulled image
  Normal   Created   kubelet  Created container kube-apiserver
  Normal   Started   kubelet  Started container kube-apiserver
```

### Key Failure Signals
| Signal | Meaning |
|--------|---------|
| `Controlled By: Node/<node-name>` | ✅ Confirms this is a static Pod |
| `State: CrashLoopBackOff` | Component crashing — check command flags, certs |
| `State: Waiting, Reason: ImagePullBackOff` | Can't pull image — check image name |
| `Restart Count: 15` | Component unstable — likely config error |
| Events show `Liveness probe failed` | Probe is killing healthy container, or component unhealthy |
| Events show `Warning BackOff` | Container keeps crashing |

## 📝 `kubectl logs <static-pod> -n kube-system`

```bash
kubectl logs kube-apiserver-master -n kube-system
kubectl logs etcd-master -n kube-system
kubectl logs kube-scheduler-master -n kube-system
kubectl logs kube-controller-manager-master -n kube-system

# Previous crashed run
kubectl logs kube-apiserver-master -n kube-system --previous

# Follow logs
kubectl logs kube-apiserver-master -n kube-system -f
```

### What to Look For
| Log Pattern | Meaning |
|-------------|---------|
| `Unable to read certificate` | Wrong cert path in command flags |
| `connection refused 127.0.0.1:2379` | etcd not running or wrong etcd endpoint |
| `unable to load client certificate` | Cert file missing or wrong permissions |
| `bind: address already in use` | Another process using the port |
| `error unmarshalling` | Corrupted config or wrong flag format |
| `context deadline exceeded` | etcd timeout — etcd slow or down |
| Healthy startup messages | ✅ Component running fine |

### ⚠️ When kubectl logs WON'T Work
- If API server static Pod is broken → **kubectl doesn't work at all**
- Must SSH to node and use:
  ```bash
  crictl ps                    # List running containers
  crictl logs <container-id>   # View container logs
  journalctl -u kubelet        # kubelet logs (manages static Pods)
  or 
  journalctl -xeu kubelet
  ```

# 7️⃣ Failure Patterns (Very Important)

## Pattern 1: Manifest in Wrong Directory
```
SYMPTOM:        kubectl get pods shows nothing; Pod doesn't exist
CAUSE:          YAML file placed in wrong directory (not staticPodPath)
DETECTION:      ls /etc/kubernetes/manifests/ → file not there
                grep staticPodPath /var/lib/kubelet/config.yaml → check actual path
BEHAVIOR:       kubelet never sees the file → Pod never created → no mirror Pod
FIX:            Move file to correct directory
EXAM SCENARIO:  "Create a static Pod. Where should you place the manifest?"
```

## Pattern 2: YAML Syntax Error in Manifest
```
SYMPTOM:        Pod not created, no mirror Pod visible
CAUSE:          Invalid YAML (indentation, typo, missing field)
DETECTION:      journalctl -u kubelet | grep -i error → YAML parse error
BEHAVIOR:       kubelet logs error, skips the file, no Pod created
                No kubectl visibility (no mirror Pod)
FIX:            Fix YAML syntax, kubelet auto-detects the fix
EXAM TIP:       If pod doesn't appear, don't just check kubectl — check kubelet logs!
```

## Pattern 3: Wrong Image in Control Plane Static Pod
```
SYMPTOM:        Control plane component in CrashLoopBackOff or ImagePullBackOff
CAUSE:          Wrong image name/tag in /etc/kubernetes/manifests/*.yaml
DETECTION:      kubectl describe pod <component>-<node> -n kube-system → image pull error
                OR if apiserver is down: crictl ps → crictl logs
BEHAVIOR:       Component doesn't start → cluster may be partially/fully broken
FIX:            Edit manifest file directly on node → kubelet auto-restarts
EXAM SCENARIO:  "kube-apiserver is down. Investigate and fix."
```

## Pattern 4: Wrong Command Flags
```
SYMPTOM:        CrashLoopBackOff for control plane component
CAUSE:          Typo in command flags (e.g., --etcd-servrs, --wrong-flag)
DETECTION:      kubectl logs <pod> -n kube-system → startup error
                If apiserver: crictl logs <container-id>
BEHAVIOR:       Component exits immediately with error
FIX:            Edit manifest file → fix the flag → kubelet recreates Pod
EXAM SCENARIO:  "After a change, kube-scheduler is crashing. Fix it."
                → cat /etc/kubernetes/manifests/kube-scheduler.yaml → find typo → fix
```

## Pattern 5: Wrong Certificate Path
```
SYMPTOM:        CrashLoopBackOff, TLS errors in logs
CAUSE:          volumeMount path or command flag points to wrong cert file
LOGS:           "unable to load server certificate" or "x509: certificate"
DETECTION:      kubectl logs or crictl logs → TLS error
BEHAVIOR:       Component can't establish TLS → crashes
FIX:            Verify cert files exist: ls /etc/kubernetes/pki/
                Fix path in manifest → kubelet auto-recreates
EXAM TIP:       Check actual files on disk vs what manifest references
```

## Pattern 6: Trying to Delete Static Pod via kubectl
```
SYMPTOM:        Pod keeps coming back after kubectl delete
CAUSE:          Static Pod — kubelet recreates it from manifest file
DETECTION:      kubectl get pod → "Controlled By: Node/<name>" in describe
BEHAVIOR:       Mirror Pod deleted → kubelet immediately recreates it
FIX:            Remove the manifest file from node:
                rm /etc/kubernetes/manifests/<pod>.yaml
EXAM SCENARIO:  "Remove the static pod named my-static-pod from node01"
                → SSH to node01 → find and remove the manifest file
```

## Pattern 7: kubelet Not Running
```
SYMPTOM:        ALL static Pods down, entire control plane dead
CAUSE:          kubelet service stopped or crashed
DETECTION:      systemctl status kubelet → inactive/failed
                journalctl -u kubelet → see crash reason
BEHAVIOR:       No static Pods managed, nothing runs on the node
FIX:            systemctl restart kubelet → fix underlying issue
EXAM SCENARIO:  "A node is NotReady. All control plane components are down."
                → SSH → systemctl status kubelet → fix → restart
```

# 8️⃣ Misconfiguration Behavior

## What Happens If Manifest Is in Wrong Directory?
- **Behavior:** kubelet never sees it → Pod is never created → no mirror Pod
- **Detection:** `kubectl get pods` shows nothing; `ls /etc/kubernetes/manifests/` — file not there
- **Fix:** Move file to correct `staticPodPath`

## What Happens If Image Is Wrong?
- **Behavior:** kubelet creates the Pod, Container enters `ImagePullBackOff`
- **Impact on control plane:** If kube-apiserver image is wrong → API server down → `kubectl` stops working
- **Detection when kubectl works:** `kubectl describe pod` → image pull error
- **Detection when kubectl doesn't work:** SSH to node → `crictl ps -a` → `crictl logs <id>`
- **Fix:** Edit manifest file directly → kubelet detects change → recreates Pod

## What Happens If Command Flags Are Wrong?
- **Behavior:** Container starts but exits immediately → `CrashLoopBackOff`
- **Logs:** "unknown flag" or "invalid value" or specific component error
- **Impact:** That control plane component is down
- **Fix:** Edit manifest → fix flag → kubelet recreates

## What Happens If Certificate Paths Are Wrong?
- **Behavior:** `CrashLoopBackOff` with TLS errors in logs
- **Logs show:** "unable to load certificate", "x509: certificate signed by unknown authority"
- **Impact:** Component can't do TLS → can't talk to other components → crashes
- **Fix:** Verify cert files on disk (`ls /etc/kubernetes/pki/`), fix paths in manifest

## What Happens If Resource Limits Are Too Low?
- **Behavior:** Component gets `OOMKilled` when under load
- **Control plane impact:** API server OOM → cluster API down temporarily
- **Fix:** Increase `resources.limits.memory` in manifest file

## What Happens If hostPath Volume Points to Non-Existent Directory?
- **Behavior:** Depends on `hostPath.type`:
  - `Directory` → Container creation fails (dir must exist)
  - `DirectoryOrCreate` → kubelet creates directory → container starts (usually OK)
  - Missing type → permissive behavior, dir may be created
- **Impact:** If cert directory doesn't exist → component can't load certs → crash

## What Happens If You Edit a Static Pod Manifest?
- **Behavior:** kubelet detects file change → deletes old Pod → creates new Pod from updated manifest
- **Timing:** Usually within seconds (kubelet polls the directory periodically)
- **No `kubectl apply` needed** — file change is enough

# 9️⃣ Fast Detection Strategy

## Identifying Static Pods (30 Seconds)

```bash
# Method 1: Check "Controlled By" field
kubectl describe pod <pod-name> -n kube-system | grep "Controlled By"
# "Controlled By: Node/master" ← STATIC POD
# "Controlled By: ReplicaSet/xxx" ← Regular Pod

# Method 2: Pod name ends with node name
kubectl get pods -n kube-system
# kube-apiserver-master              ← Static (ends with "-master")
# coredns-5dd5756b68-xxxxx           ← Regular (random hash suffix)

# Method 3: Check ownerReferences
kubectl get pod <pod> -n kube-system -o yaml | grep -A3 ownerReferences
# kind: Node ← Static Pod
# kind: ReplicaSet ← Regular Pod

# Method 4: Find static pod path
cat /var/lib/kubelet/config.yaml | grep staticPodPath
# staticPodPath: /etc/kubernetes/manifests
ls /etc/kubernetes/manifests/
```

## 60-Second Debug Flow for Static Pod Issues

```bash
# ⏱️ 0-10 sec: Is the control plane static Pod running?
kubectl get pods -n kube-system
# If kubectl doesn't work → API server static Pod is likely broken
# SSH to master node

# ⏱️ 10-20 sec: Check kubelet status (if kubectl broken)
systemctl status kubelet
# If stopped → systemctl restart kubelet

# ⏱️ 20-35 sec: Check which static Pod is failing
# If kubectl works:
kubectl get pods -n kube-system | grep -E "apiserver|etcd|scheduler|controller"
kubectl describe pod <failing-pod> -n kube-system | tail -15

# If kubectl doesn't work:
crictl ps -a
crictl logs <container-id>

# ⏱️ 35-50 sec: Check the manifest file
cat /etc/kubernetes/manifests/<component>.yaml
# Look for: wrong image, wrong flags, wrong cert paths

# ⏱️ 50-60 sec: Check kubelet logs for parse errors
journalctl -u kubelet --no-pager | tail -30 | grep -i error
```

## Decision Tree
```
Static Pod not running?
├── kubectl works? (API server is up)
│   ├── YES → kubectl describe pod <name> -n kube-system
│   │         Look at: Events, Container State, Image, Restart Count
│   └── NO → SSH to node
│            ├── systemctl status kubelet → Is kubelet running?
│            │   ├── NO → systemctl restart kubelet → check why it stopped
│            │   └── YES → Check manifests + container runtime
```

# 🔟 Fast Fix Strategy

## ⚠️ Critical: Static Pods Are NOT Fixed via kubectl
```bash
# ❌ These DON'T work for static Pods:
kubectl edit pod kube-apiserver-master -n kube-system    # Read-only mirror Pod
kubectl delete pod kube-apiserver-master -n kube-system  # Comes right back
kubectl apply -f fixed.yaml                              # Won't override static Pod

# ✅ FIX STATIC PODS BY EDITING THE MANIFEST FILE ON THE NODE:
vim /etc/kubernetes/manifests/kube-apiserver.yaml
# Make changes → save → kubelet auto-detects → recreates Pod
```

## Common Fixes

### Fix: Wrong image version
```bash
# SSH to node
ssh master-node
vim /etc/kubernetes/manifests/kube-apiserver.yaml
# Change: image: registry.k8s.io/kube-apiserver:v1.30.0

```

### Fix: Wrong command flag
```bash

# Fix it → save
# Watch: kubectl get pods -n kube-system -w
```

### Fix: Wrong cert path
```bash
# First verify actual cert files
ls -la /etc/kubernetes/pki/
# Then fix the manifest to reference correct files
vim /etc/kubernetes/manifests/kube-apiserver.yaml
# Fix --tls-cert-file, --tls-private-key-file, etc.
```

### Fix: Create a new static Pod
```bash
# Find the static pod path
cat /var/lib/kubelet/config.yaml | grep staticPodPath
# staticPodPath: /etc/kubernetes/manifests

# Create manifest file
cat <<EOF > /etc/kubernetes/manifests/my-static-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-static-pod
  namespace: default
spec:
  containers:
  - name: web
    image: nginx:1.25
    ports:
    - containerPort: 80
EOF

# kubelet creates Pod automatically — no kubectl apply needed
kubectl get pods   # Should see: my-static-pod-<node-name>
```

### Fix: Remove a static Pod
```bash
# Find and remove the manifest
rm /etc/kubernetes/manifests/my-static-pod.yaml
```

### 5 Quiz Questions

**Q1:** How do you identify a static Pod from `kubectl get pods` output?
> **A:** Static Pod names end with the node name (e.g., `kube-apiserver-master`). Confirm with `kubectl describe pod <name> | grep "Controlled By"` — shows `Node/<node-name>` for static Pods.

**Q2:** You run `kubectl delete pod static-web-node01`. What happens?
> **A:** The mirror Pod is deleted briefly, but kubelet immediately recreates it from the manifest file. The Pod comes back within seconds. To truly delete it, remove the manifest file from the node.

**Q3:** The kube-apiserver is down and `kubectl` doesn't work. How do you debug?
> **A:** SSH to the master node → `systemctl status kubelet` → `crictl ps -a` to see containers → `crictl logs <container-id>` for apiserver container → `cat /etc/kubernetes/manifests/kube-apiserver.yaml` to check manifest → fix → kubelet recreates it.

**Q4:** How do you find the static Pod manifest directory?
> **A:** `cat /var/lib/kubelet/config.yaml | grep staticPodPath` or check the kubelet process flags with `ps aux | grep kubelet`.

**Q5:** You place a Pod manifest at `/etc/kubernetes/my-pod.yaml` but the Pod doesn't appear. Why?
> **A:** The file is not in the `staticPodPath` directory. It must be in `/etc/kubernetes/manifests/` (or whatever path `staticPodPath` specifies). Move the file there.

# 1️⃣3️⃣ Component Interaction Map

```
┌────────────────────────────────────────────────────────────────┐
│                    STATIC POD INTERACTIONS                       │
│                                                                 │
│  /etc/kubernetes/manifests/                                     │
│  └── pod.yaml ──────┐                                          │
│                      ▼                                          │
│                 ┌──────────┐                                    │
│                 │ kubelet  │ ← Watches staticPodPath            │
│                 └────┬─────┘                                    │
│                      │                                          │
│               ┌──────┼──────┐                                   │
│               ▼      ▼      ▼                                   │
│         Creates   Creates   Restarts                            │
│          Pod     Mirror Pod  on crash                           │
│               │      │                                          │
│               ▼      ▼                                          │
│         Container  API Server                                   │
│         Runtime    (read-only                                   │
│         (runs it)   copy)                                       │
└────────────────────────────────────────────────────────────────┘
```

### Interaction Details

| Component | Normal Behavior | When Static Pod Fails |
|-----------|----------------|-----------------------|
| **kubelet** | Watches directory, creates Pod from YAML, creates mirror Pod, manages lifecycle | Restarts crashed container. If YAML invalid → logs error, no Pod created. If kubelet dies → ALL static Pods die (control plane down). |
| **Scheduler** | **NOT involved.** Static Pods bypass the scheduler entirely. | N/A — scheduler has no role. Static Pod always runs on the node where manifest exists. |
| **Controller Manager** | **NOT involved.** No Deployment/RS manages static Pods. | N/A — if controller-manager itself is a static Pod and fails, other controllers stop (Deployments stop reconciling, etc.) |
| **API Server** | Stores mirror Pod (read-only). Allows `kubectl get/describe/logs`. | If apiserver static Pod fails → kubectl stops working. Must use SSH + crictl. Mirror Pod disappears. |
| **etcd** | **NOT involved in static Pod creation** (kubelet reads from disk, not etcd). | If etcd static Pod fails → all cluster state lost → API server can't function. |
| **Services** | CAN select mirror Pods by labels. | If static Pod crashes → Service has no backend → traffic fails. |
| **Network** | `hostNetwork: true` for control plane — uses node's IP. | If network breaks → control plane components can't reach each other (apiserver ↔ etcd). |
| **Storage** | `hostPath` volumes for certs and data. | If cert files missing/wrong → component TLS fails → crash. If etcd data dir missing → etcd starts fresh (data loss!). |

### Failure Cascade When Control Plane Static Pod Dies
```
kube-apiserver static Pod crashes
  → API server unreachable
    → kubectl stops working
      → No new Pods/Services can be created
        → Existing Pods keep running (kubelet manages them locally)
          → But no new scheduling, no controller reconciliation
            → Cluster is in "read-only" mode (running but can't change)

etcd static Pod crashes  
  → etcd data unreachable
    → API server can't read/write state
      → Same cascade as apiserver failure

kube-scheduler static Pod crashes
  → New Pods stay Pending (no scheduling)
    → Existing Pods keep running fine
      → Impact: can't deploy new workloads

kube-controller-manager static Pod crashes
  → Deployments/ReplicaSets stop reconciling
    → If a Pod dies, no replacement created
      → Self-healing broken
        → Existing healthy Pods keep running
```
