## 1пёҸвғЈ What It Does

- **Core Purpose**
  - Next-generation API for managing traffic routing into and within a Kubernetes cluster
  - Replaces Ingress as the standard way to handle HTTP/HTTPS (and TCP/UDP/gRPC) traffic
  - Provides **role-oriented design** вҖ” separates infrastructure concerns (platform admin) from application routing (developer)
  - Three core resources: **GatewayClass** вҶ’ **Gateway** вҶ’ **HTTPRoute** (plus TCPRoute, TLSRoute, GRPCRoute)
  - Offers expressive routing: header-based, weighted traffic splitting, request mirroring, URL rewrites вҖ” built into the API spec (no annotations)

- **Real-World Use Case**
  - Platform team creates a `Gateway` (shared load balancer entry point) вҶ’ developers attach `HTTPRoute` rules to route their apps
  - Route `api.example.com/v1` вҶ’ api-v1-svc, `api.example.com/v2` вҶ’ api-v2-svc using HTTPRoute
  - Canary deployments with weighted backends: 90% вҶ’ stable, 10% вҶ’ canary
  - Header-based routing: `X-Version: beta` вҶ’ beta Service
  - Multi-team cluster: each namespace gets its own HTTPRoute attached to a shared Gateway
  - TLS termination at the Gateway level, plain HTTP to backend Pods

- **Where It Fits in Cluster Architecture**
  - Lives at the **edge** of the cluster вҖ” same position as Ingress, but more powerful
  - **GatewayClass** вҖ” defines the controller implementation (like IngressClass, but richer)
  - **Gateway** вҖ” the actual listener (like a load balancer instance with ports/protocols/TLS)
  - **HTTPRoute** вҖ” routing rules that attach to a Gateway (like Ingress rules, but separate resource)
  - Flow: `Client вҶ’ Gateway (Controller Pod) вҶ’ HTTPRoute rules вҶ’ Service вҶ’ Pods`
  - Requires a Gateway API controller (e.g., NGINX Gateway Fabric, Envoy Gateway, Istio, Cilium, Contour)
  - Gateway API CRDs must be installed in the cluster (`gateway.networking.k8s.io`)
  - **Graduated to GA** in Kubernetes 1.29 вҖ” CKA 2025/2026 now includes it

## 2пёҸвғЈ What Depends On It

- **Which Components Rely On It**
  - External users/clients accessing HTTP/HTTPS/gRPC applications
  - Development teams needing self-service routing without touching infrastructure
  - Platform teams managing shared ingress infrastructure
  - Traffic splitting / canary release workflows
  - Service mesh integrations (Istio uses Gateway API as its standard routing API)

- **What Resources It Interacts With**
  - **GatewayClass** вҖ” selects which controller implementation handles Gateways
  - **Gateway** вҖ” defines listeners (ports, protocols, TLS, hostname filters)
  - **HTTPRoute** вҖ” routing rules (paths, headers, backends, filters)
  - **Services (ClusterIP)** вҖ” backends that HTTPRoute points traffic to
  - **Pods** вҖ” ultimate traffic destination behind Services
  - **Secrets (TLS)** вҖ” TLS certificates referenced by Gateway listeners
  - **Namespaces** вҖ” cross-namespace routing is a first-class concept
  - **ReferenceGrant** вҖ” allows cross-namespace references (e.g., HTTPRoute in ns-a вҶ’ Service in ns-b)

- **Upstream Dependencies**
  - Gateway API **CRDs must be installed** вҖ” they are NOT built into core Kubernetes
  - A Gateway API controller must be deployed (NGINX Gateway Fabric, Envoy Gateway, etc.)
  - GatewayClass resource must exist and reference a valid controller
  - TLS Secrets must exist before Gateway references them
  - Backend Services must exist in the target namespace

- **Downstream Dependencies**
  - ClusterIP Services (backends referenced by HTTPRoute)
  - Pods serving actual application traffic
  - Endpoints / EndpointSlices connecting Services to Pods
  - ReferenceGrant resources for cross-namespace routing

## 3пёҸвғЈ Recognize Structure (YAML Skeleton)

### GatewayClass

```yaml
apiVersion: gateway.networking.k8s.io/v1      # GA API version
kind: GatewayClass
metadata:
  name: my-gateway-class                       # Cluster-scoped (no namespace)
spec:
  controllerName: example.com/gateway-controller  # Which controller handles this class
```

### Gateway

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: default                           # Namespace-scoped
spec:
  gatewayClassName: my-gateway-class           # Must match a GatewayClass name
  listeners:                                   # One or more listeners (ports/protocols)
    - name: http                               # Listener name (must be unique within Gateway)
      protocol: HTTP                           # HTTP | HTTPS | TLS | TCP | UDP
      port: 80                                 # Port to listen on
      hostname: "*.example.com"                # Optional hostname filter
      allowedRoutes:                           # Which routes can attach
        namespaces:
          from: All                            # All | Same | Selector
        kinds:
          - kind: HTTPRoute                    # Which Route types
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "app.example.com"
      tls:                                     # TLS config for HTTPS listeners
        mode: Terminate                        # Terminate | Passthrough
        certificateRefs:
          - name: app-tls-secret               # Secret in same namespace as Gateway
            kind: Secret
      allowedRoutes:
        namespaces:
          from: Same
```

### HTTPRoute

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-route
  namespace: default
spec:
  parentRefs:                                  # Which Gateway(s) this route attaches to
    - name: my-gateway
      namespace: default                       # Gateway's namespace (if different)
      sectionName: http                        # Specific listener name (optional)
  hostnames:                                   # Hostnames this route matches
    - "app.example.com"
  rules:                                       # Routing rules
    - matches:                                 # Match conditions
        - path:
            type: PathPrefix                   # PathPrefix | Exact
            value: /api
          headers:                             # Optional header-based matching
            - name: X-Version
              value: v2
          method: GET                          # Optional HTTP method matching
      filters:                                 # Request/response transformations
        - type: RequestHeaderModifier
          requestHeaderModifier:
            add:
              - name: X-Custom
                value: "added-by-gateway"
      backendRefs:                             # Backend Services
        - name: api-svc
          port: 80
          weight: 90                           # Traffic weight (for splitting)
        - name: api-canary-svc
          port: 80
          weight: 10
```

- **Key Sections Explained**
  - `apiVersion: gateway.networking.k8s.io/v1` вҖ” the GA API version (also v1beta1 for older setups)
  - `gatewayClassName` вҖ” links Gateway to its controller implementation (like ingressClassName)
  - `listeners` вҖ” define entry points (port, protocol, TLS, hostname filtering)
  - `parentRefs` вҖ” how HTTPRoute attaches to a Gateway (replaces the old annotation approach)
  - `matches` вҖ” path, header, method, query matching (far more expressive than Ingress)
  - `backendRefs` вҖ” target Services with optional weight for traffic splitting
  - `filters` вҖ” request/response transformations (rewrites, header mods, redirects, mirrors)
  - `allowedRoutes` вҖ” Gateway-side control over which namespaces/route types can attach

- **Common Indentation Traps**
  - `parentRefs` is an array under `spec` вҖ” each entry has `name`, `namespace`, `sectionName`
  - `matches` is nested under `rules[]` вҖ” `path`, `headers`, `method` are siblings under each match
  - `backendRefs` is at the same level as `matches` under `rules[]` (NOT nested inside matches)
  - `filters` is at the same level as `matches` and `backendRefs` under `rules[]`
  - `certificateRefs` under `tls` is an array вҖ” each item has `name` and `kind`
  - `listeners` is an array вҖ” each listener has its own `name`, `port`, `protocol` вҖ” easy to misplace
  - `allowedRoutes.namespaces.from` takes `All`, `Same`, or `Selector` вҖ” easy to use wrong value

## 4пёҸвғЈ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `GatewayClass.spec.controllerName` | Links to controller implementation | GatewayClass created but no controller acts on it |
| `Gateway.spec.gatewayClassName` | Selects which GatewayClass/controller | Gateway is ignored вҖ” Programmed=False |
| `Gateway.spec.listeners[].port` | Defines which port accepts traffic | Traffic on that port won't be accepted |
| `Gateway.spec.listeners[].protocol` | HTTP/HTTPS/TLS/TCP | Protocol mismatch вҖ” connections fail |
| `Gateway.spec.listeners[].hostname` | Filters which hostnames this listener handles | Hostname doesn't match вҶ’ Route not accepted |
| `Gateway.spec.listeners[].tls` | TLS termination config | HTTPS fails, falls back to default cert or connection refused |
| `Gateway.spec.listeners[].allowedRoutes` | Controls which Routes can attach | HTTPRoute rejected вҖ” not allowed by listener |
| `HTTPRoute.spec.parentRefs` | Attaches route to Gateway | Route is orphaned вҖ” no traffic flows |
| `HTTPRoute.spec.parentRefs[].sectionName` | Targets specific listener | Route attaches to wrong or no listener |
| `HTTPRoute.spec.hostnames` | Hostname matching | Hostname must intersect with listener's hostname or route is rejected |
| `HTTPRoute.spec.rules[].matches[].path` | Path-based routing | Requests to that path don't match вҶ’ fall through |
| `HTTPRoute.spec.rules[].backendRefs` | Target Service(s) | Traffic has no destination вҶ’ 500 error |
| `HTTPRoute.spec.rules[].backendRefs[].port` | Service port | Wrong port вҶ’ connection refused or 503 |
| `HTTPRoute.spec.rules[].backendRefs[].weight` | Traffic splitting ratio | Unexpected traffic distribution |
| `ReferenceGrant` | Cross-namespace references | HTTPRoute in ns-a can't reach Service in ns-b вҶ’ RefNotPermitted |

- **Field Relationships**
  - `Gateway.spec.gatewayClassName` вҶ” must match an existing `GatewayClass.metadata.name`
  - `HTTPRoute.spec.parentRefs[].name` вҶ” must match an existing `Gateway.metadata.name`
  - `HTTPRoute.spec.parentRefs[].sectionName` вҶ” must match a `Gateway.spec.listeners[].name`
  - `HTTPRoute.spec.hostnames` вҶ” must intersect with `Gateway.spec.listeners[].hostname` (if set)
  - `HTTPRoute.spec.rules[].backendRefs[].name` вҶ” must match a Service in the same namespace (or cross-ns with ReferenceGrant)
  - `HTTPRoute.spec.rules[].backendRefs[].port` вҶ” must match the Service's `.spec.ports[].port`
  - `Gateway.spec.listeners[].tls.certificateRefs[].name` вҶ” must match a Secret of type `kubernetes.io/tls`

## 5пёҸвғЈ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - `apiVersion` вҖ” is it `gateway.networking.k8s.io/v1`?
  - `kind` вҖ” is it GatewayClass, Gateway, or HTTPRoute?
  - `gatewayClassName` (on Gateway) вҖ” does the GatewayClass exist?
  - `parentRefs` (on HTTPRoute) вҖ” does it reference the correct Gateway?
  - `listeners` вҖ” correct port, protocol, hostname?
  - `backendRefs` вҖ” correct Service name and port?
  - `allowedRoutes` вҖ” does the Gateway allow routes from this namespace?

- **Usually Irrelevant for CKA**
  - Advanced filter types (RequestMirror, ExtensionRef, custom filters)
  - Controller-specific parametersRefs on GatewayClass
  - Infrastructure-level configuration (IP allocation, load balancer annotations)
  - Advanced TLS modes (Passthrough details, mTLS configuration)
  - Status conditions beyond basic Accepted/Programmed

- **Red Flags to Detect in 10 Seconds**
  - вқҢ Gateway API CRDs not installed вҶ’ `error: the server doesn't have a resource type "gateway"`
  - вқҢ No Gateway controller deployed вҶ’ GatewayClass exists but Gateway status shows `Accepted: False`
  - вқҢ `parentRefs` pointing to non-existent Gateway вҶ’ HTTPRoute status: `Accepted: False`
  - вқҢ `sectionName` typo вҶ’ Route doesn't attach to intended listener
  - вқҢ `hostnames` on HTTPRoute don't intersect with Gateway listener hostname вҶ’ Route rejected
  - вқҢ `allowedRoutes.namespaces.from: Same` but HTTPRoute is in different namespace вҶ’ Route rejected
  - вқҢ Missing `ReferenceGrant` for cross-namespace backend references вҶ’ `RefNotPermitted`
  - вқҢ `backendRefs` Service name or port wrong вҶ’ traffic drops, 500/503
  - вқҢ Using `v1beta1` API on a cluster that only has `v1` CRDs installed (or vice versa)

## 6пёҸвғЈ What to Notice In

### kubectl describe gateway \<name\>

- **What Signals Failure**
  - `Accepted: False` вҖ” GatewayClass doesn't exist or controller rejects it
  - `Programmed: False` вҖ” controller couldn't provision the Gateway (resource issue, port conflict)
  - Listener status: `Conflicted: True` вҖ” two listeners sharing the same port/hostname
  - Listener status: `ResolvedRefs: False` вҖ” TLS Secret referenced doesn't exist
  - `AttachedRoutes: 0` on a listener that should have routes вҖ” route attachment failing
  - Empty `Addresses` вҖ” controller hasn't assigned an IP/hostname

- **Where to Look First**
  - `Status.Conditions` вҖ” check Accepted and Programmed conditions
  - `Status.Listeners` вҖ” per-listener conditions (Accepted, Programmed, ResolvedRefs, Conflicted)
  - `Status.Addresses` вҖ” the external IP/hostname assigned by the controller
  - `Events` section вҖ” controller-generated events

- **Fields That Matter Most**
  - `Accepted: True/False` вҖ” is the Gateway accepted by the controller?
  - `Programmed: True/False` вҖ” is the Gateway actually provisioned and working?
  - Per-listener `AttachedRoutes` count вҖ” how many routes are attached
  - `Addresses` вҖ” the entry point IP/hostname

### kubectl describe httproute \<name\>

- **What Signals Failure**
  - `Accepted: False` вҖ” parentRef Gateway doesn't exist, wrong namespace, or listener doesn't allow this route
  - `ResolvedRefs: False` вҖ” backend Service doesn't exist or port doesn't match
  - `RefNotPermitted` reason вҖ” cross-namespace reference without ReferenceGrant
  - Parent status showing `False` for any condition

- **Where to Look First**
  - `Status.Parents` вҖ” per-parent (Gateway) condition status
  - `Status.Parents[].Conditions` вҖ” Accepted, ResolvedRefs conditions
  - Events section

- **Fields That Matter Most**
  - Whether each parentRef is `Accepted: True`
  - Whether `ResolvedRefs: True` вҖ” all backends resolvable
  - Condition reasons вҖ” `NoMatchingListenerHostname`, `NotAllowedByListeners`, `RefNotPermitted`

### kubectl logs \<gateway-controller-pod\>

- **What Signals Failure**
  - `upstream connect error` вҖ” backend Service/Pod unreachable
  - `no endpoints found` вҖ” Service has no ready Pods
  - `TLS handshake error` вҖ” certificate issue
  - `connection refused` вҖ” backend Pod not listening on expected port
  - Configuration reload errors after Gateway/HTTPRoute changes

- **Where to Look First**
  - Error-level logs related to upstream/backend connectivity
  - Reconciliation loop errors for specific Gateway or HTTPRoute resources
  - TLS certificate loading failures

- **Fields That Matter Most**
  - HTTP status codes in access logs (502/503/504)
  - Backend endpoint addresses
  - Reconciliation success/failure messages

### kubectl exec (Testing from inside the cluster)

- **Useful Commands**
  ```bash
  # Test backend Service directly
  kubectl exec -it test-pod -- curl http://api-svc:80

  # Test via Gateway Controller's ClusterIP
  kubectl exec -it test-pod -- curl -H "Host: app.example.com" http://<gateway-svc-ip>

  # Verify DNS resolution
  kubectl exec -it test-pod -- nslookup api-svc.default.svc.cluster.local
  ```

- **What Signals Failure**
  - `Connection refused` вҖ” Service exists but no healthy Pods
  - `Could not resolve host` вҖ” Service doesn't exist
  - HTTP 502/503 вҖ” Service exists but Pods not ready
  - Empty response вҖ” Pod running but application not responding

### kubectl explain

- **Key Usage**
  ```bash
  kubectl explain gateway.spec.listeners
  kubectl explain gateway.spec.listeners.allowedRoutes
  kubectl explain httproute.spec.rules.matches
  kubectl explain httproute.spec.rules.backendRefs
  kubectl explain httproute.spec.parentRefs
  kubectl explain gatewayclass.spec
  ```
- During CKA вҖ” quickly verify field names and nesting
- Confirm which fields are required vs optional
- Check allowed values for enums (protocol, pathType, namespace from)

## 7пёҸвғЈ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - вқҢ Gateway API CRDs not installed вҶ’ `error: the server doesn't have a resource type "gateways"`
  - вқҢ No Gateway controller deployed вҶ’ GatewayClass exists, but Gateway status: `Accepted: False`
  - вқҢ Wrong `gatewayClassName` on Gateway вҶ’ Gateway ignored by controller
  - вқҢ Wrong `parentRefs` on HTTPRoute вҶ’ route orphaned, no traffic flows
  - вқҢ `sectionName` typo in parentRefs вҶ’ route doesn't attach to listener
  - вқҢ Hostname mismatch between Gateway listener and HTTPRoute вҶ’ route rejected
  - вқҢ `allowedRoutes.namespaces.from: Same` but HTTPRoute in different namespace вҶ’ rejected
  - вқҢ Backend Service name typo вҶ’ `ResolvedRefs: False`
  - вқҢ Backend port wrong вҶ’ traffic drops, 503 errors
  - вқҢ Cross-namespace backendRef without ReferenceGrant вҶ’ `RefNotPermitted`
  - вқҢ TLS Secret missing вҶ’ listener `ResolvedRefs: False`, HTTPS connections fail
  - вқҢ Port conflict between listeners вҶ’ `Conflicted: True`

- **Status Values to Recognize**
  - **Gateway Conditions:**
    - `Accepted: True` вҖ” controller acknowledges the Gateway вң…
    - `Accepted: False` вҖ” controller rejects the Gateway вқҢ
    - `Programmed: True` вҖ” Gateway is provisioned and serving traffic вң…
    - `Programmed: False` вҖ” Gateway exists but isn't functional вқҢ
  - **Listener Conditions:**
    - `ResolvedRefs: False` вҖ” TLS Secret or other reference not found
    - `Conflicted: True` вҖ” port/hostname conflict with another listener
  - **HTTPRoute Parent Conditions:**
    - `Accepted: True` вҖ” route attached to Gateway вң…
    - `Accepted: False` вҖ” route rejected by Gateway вқҢ
    - `ResolvedRefs: False` вҖ” backend Service not found вқҢ

- **Event Messages / Condition Reasons to Recognize**
  - `NoMatchingParent` вҖ” parentRef doesn't match any Gateway
  - `NoMatchingListenerHostname` вҖ” HTTPRoute hostname doesn't intersect with listener
  - `NotAllowedByListeners` вҖ” route kind or namespace not permitted by allowedRoutes
  - `RefNotPermitted` вҖ” cross-namespace reference without ReferenceGrant
  - `InvalidCertificateRef` вҖ” TLS Secret not found or wrong type
  - `BackendNotFound` вҖ” Service referenced in backendRefs doesn't exist
  - `PortNotFound` вҖ” port in backendRefs doesn't match any Service port

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: `kubectl get gateway` shows Gateway but `Programmed: False` вҶ’ no controller installed, or gatewayClassName is wrong
  - Scenario 2: HTTPRoute created but traffic not flowing вҶ’ check `kubectl describe httproute` вҖ” `Accepted: False` because `allowedRoutes` doesn't permit the HTTPRoute's namespace
  - Scenario 3: HTTPS not working on Gateway вҶ’ listener `ResolvedRefs: False` вҖ” TLS Secret name typo or Secret doesn't exist
  - Scenario 4: Path `/api` works but `/api/v1` doesn't вҶ’ `path.type: Exact` instead of `PathPrefix`
  - Scenario 5: HTTPRoute with cross-namespace backendRef returns 500 вҶ’ missing ReferenceGrant in target namespace
  - Scenario 6: Two HTTPRoutes with conflicting rules вҶ’ traffic goes to the one with more specific match (longest path prefix wins)
  - Scenario 7: Canary split not working вҶ’ weight fields missing or both set to 0

## 8пёҸвғЈ Misconfiguration Behavior

- **What Happens If Labels Mismatch (Service вҶ’ Pods)?**
  - Service gets 0 Endpoints/EndpointSlices
  - Gateway controller sees no upstreams for that backend
  - HTTPRoute status may still show `ResolvedRefs: True` (Service exists, controller can't see endpoint count)
  - Result: **503 Service Unavailable** вҖ” traffic routed to empty backend
  - The issue is NOT in Gateway/HTTPRoute вҖ” it's downstream at Service level

- **What Happens If Ports Mismatch?**
  - `backendRefs[].port` doesn't match Service `.spec.ports[].port`
  - HTTPRoute condition: `ResolvedRefs: False` with reason `PortNotFound` (controller-dependent)
  - Or traffic routes to wrong port вҶ’ connection refused from Pod
  - Result: **503 or connection reset**

- **What Happens If Gateway Controller Image Is Wrong?**
  - Controller Pod enters `ImagePullBackOff` or `CrashLoopBackOff`
  - ALL Gateways and HTTPRoutes managed by that controller stop working
  - Existing connections drop, Gateway status stops updating
  - No new route changes are processed

- **What Happens If GatewayClass Is Wrong / Missing?**
  - `gatewayClassName` on Gateway references non-existent GatewayClass
  - Gateway is stored in etcd but never processed
  - Gateway status: `Accepted: False` (or no status at all)
  - All attached HTTPRoutes are effectively orphaned

- **What Happens If CRDs Are Not Installed?**
  - `kubectl apply -f gateway.yaml` вҶ’ `error: the server doesn't have a resource type "gateways"`
  - Can't create Gateway, HTTPRoute, GatewayClass, or ReferenceGrant resources
  - This is the **most fundamental failure** вҖ” must install CRDs first

- **What Happens If ReferenceGrant Is Missing?**
  - HTTPRoute in namespace-a references Service in namespace-b
  - Without ReferenceGrant in namespace-b, the cross-namespace reference is denied
  - HTTPRoute condition: `ResolvedRefs: False`, reason: `RefNotPermitted`
  - Traffic to that backendRef drops, other backendRefs in the same rule may still work

- **What Happens If Dependencies Are Missing (Service/Pods)?**
  - Missing Service вҶ’ `ResolvedRefs: False`, reason: `BackendNotFound`
  - Service exists, no Pods вҶ’ 503 from controller (no upstream endpoints)
  - Pods exist but not Ready вҶ’ 503 (endpoints not populated)

## 9пёҸвғЈ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Verify Gateway API CRDs are installed
  kubectl get crd | grep gateway.networking.k8s.io

  # 2. Check GatewayClass exists and has a valid controller
  kubectl get gatewayclass

  # 3. Check Gateway exists and status
  kubectl get gateway -A
  kubectl describe gateway <name> -n <namespace>

  # 4. Check HTTPRoute exists and status
  kubectl get httproute -A
  kubectl describe httproute <name> -n <namespace>

  # 5. Verify Gateway controller is running
  kubectl get pods -A | grep -i gateway    # (controller namespace varies)

  # 6. Check backend Service exists
  kubectl get svc <backend-svc> -n <namespace>

  # 7. Check Service has Endpoints
  kubectl get endpoints <backend-svc> -n <namespace>

  # 8. Check backend Pods are running and Ready
  kubectl get pods -l <selector-labels> -n <namespace>

  # 9. Check controller logs
  kubectl logs -n <controller-ns> <controller-pod> --tail=50

  # 10. Check ReferenceGrant (if cross-namespace)
  kubectl get referencegrant -A

  # 11. Test from inside the cluster
  kubectl run tmp --image=busybox --rm -it -- wget -qO- http://<svc-name>:<port>
  ```

- **Order of Debugging Steps**
  1. Are Gateway API CRDs installed? вҶ’ `kubectl get crd | grep gateway`
  2. Is a Gateway controller running? вҶ’ `kubectl get pods -A | grep gateway`
  3. Does GatewayClass exist? вҶ’ `kubectl get gatewayclass`
  4. Does Gateway exist and is it `Programmed: True`? вҶ’ `kubectl get gateway`
  5. Does HTTPRoute exist and is it `Accepted: True`? вҶ’ `kubectl get httproute`
  6. Does the backend Service exist in the correct namespace?
  7. Does the Service have Endpoints?
  8. Are backend Pods Running and Ready?
  9. If cross-namespace: does ReferenceGrant exist? вҶ’ `kubectl get referencegrant -n <target-ns>`
  10. Check controller logs for specific errors

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get gateway` вҖ” `Programmed`? `Accepted`? вҶ’ If False, it's a Gateway/controller issue
  - `kubectl describe httproute` вҶ’ check `Status.Parents[].Conditions` вҖ” `Accepted: False`? Read the reason
  - `kubectl get endpoints <backend>` вҶ’ empty? вҶ’ label/selector mismatch on Service
  - `kubectl logs <controller-pod> --tail=20` вҶ’ 502/503? вҶ’ backend issue

## рҹ”ҹ Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix wrong gatewayClassName on Gateway
  kubectl edit gateway <name> -n <namespace>
  # вҶ’ Correct spec.gatewayClassName

  # Fix wrong parentRefs on HTTPRoute
  kubectl edit httproute <name> -n <namespace>
  # вҶ’ Correct parentRefs[].name, namespace, sectionName

  # Fix wrong backend Service name or port
  kubectl edit httproute <name> -n <namespace>
  # вҶ’ Correct rules[].backendRefs[].name and port

  # Fix allowedRoutes on Gateway listener
  kubectl edit gateway <name> -n <namespace>
  # вҶ’ Change allowedRoutes.namespaces.from to "All" or "Selector"

  # Install missing Gateway API CRDs
  kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/standard-install.yaml

  # Create missing TLS Secret
  kubectl create secret tls app-tls --cert=tls.crt --key=tls.key -n <namespace>

  # Fix or create ReferenceGrant for cross-namespace routing
  kubectl apply -f - <<EOF
  apiVersion: gateway.networking.k8s.io/v1beta1
  kind: ReferenceGrant
  metadata:
    name: allow-from-other-ns
    namespace: target-ns
  spec:
    from:
      - group: gateway.networking.k8s.io
        kind: HTTPRoute
        namespace: source-ns
    to:
      - group: ""
        kind: Service
  EOF

  # Fix path type
  kubectl edit httproute <name>
  # вҶ’ Change path.type from Exact to PathPrefix (or vice versa)

  # Fix Service selector (if no Endpoints)
  kubectl edit svc <backend-svc>
  # вҶ’ Correct selector labels to match Pod labels
  ```

- **Validate Fix**
  ```bash
  # Check Gateway is now Programmed
  kubectl get gateway <name> -n <namespace>
  # вҶ’ PROGRAMMED=True

  # Check HTTPRoute is Accepted
  kubectl describe httproute <name> -n <namespace>
  # вҶ’ Status.Parents[].Conditions: Accepted=True, ResolvedRefs=True

  # Check Endpoints populated
  kubectl get endpoints <backend-svc>

  # Test from inside cluster
  kubectl run tmp --image=busybox --rm -it -- wget -qO- http://<svc-name>:<port>

  # Test via Gateway
  curl -H "Host: app.example.com" http://<gateway-ip>/
  ```

- **Confirm Recovery**
  - Gateway shows `Programmed: True` and `Accepted: True` вң…
  - All listeners show `ResolvedRefs: True` вң…
  - HTTPRoute shows `Accepted: True` for all parentRefs вң…
  - `curl` via Gateway returns expected HTTP response вң…
  - Controller logs show no upstream errors вң…
  - Endpoints are populated for backend Services вң…

### Step 2 вҖ” Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Wrong gatewayClassName | Change `example-gc` вҶ’ `wrong-gc` | Gateway `Accepted: False` |
| Wrong parentRef | Change HTTPRoute parentRef name вҶ’ `nonexistent-gateway` | HTTPRoute `Accepted: False` |
| Wrong sectionName | Add `sectionName: wrong-listener` to parentRef | HTTPRoute not attached to any listener |
| Wrong backend Service | Change `web-svc` вҶ’ `web-svc-wrong` | HTTPRoute `ResolvedRefs: False` |
| Wrong backend port | Change `port: 80` вҶ’ `port: 9999` | 503 errors, possible `ResolvedRefs: False` |
| Hostname mismatch | Gateway listener hostname `*.example.com`, HTTPRoute hostname `other.dev` | HTTPRoute rejected, `NoMatchingListenerHostname` |
| Namespace restriction | Set `allowedRoutes.namespaces.from: Same`, move HTTPRoute to other ns | HTTPRoute `Accepted: False`, `NotAllowedByListeners` |
| Missing TLS Secret | Add HTTPS listener referencing non-existent secret | Listener `ResolvedRefs: False` |
| Break Service selector | Edit Service selector to mismatch Pods | 0 Endpoints вҶ’ 503 |
| Delete CRDs | `kubectl delete -f standard-install.yaml` | ALL Gateway resources vanish |
| Cross-ns without ReferenceGrant | HTTPRoute in ns-a, backendRef to Service in ns-b, no ReferenceGrant | `RefNotPermitted` |
| Weight = 0 on all backends | Set all backendRef weights to 0 | No traffic routed, 500 errors |
| Remove parentRefs | Delete parentRefs from HTTPRoute | Route orphaned, no traffic |
| Port conflict | Two listeners with same port and conflicting hostnames | Listener `Conflicted: True` |

### Step 3 вҖ” Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get gateway -A
  kubectl get httproute -A
  ```

- **How to Narrow Issue**
  1. Gateway `Programmed: True`? вҶ’ If no: GatewayClass / controller issue
  2. HTTPRoute `Accepted: True`? вҶ’ If no: read the condition reason
  3. Describe HTTPRoute вҶ’ check each parent condition and `ResolvedRefs`
  4. Backend Service exists? вҶ’ `kubectl get svc`
  5. Endpoints populated? вҶ’ `kubectl get endpoints <svc>`
  6. Controller logs? вҶ’ `kubectl logs <controller-pod> --tail=30`

- **How to Confirm Root Cause**
  - Trace: `GatewayClass вҶ’ Gateway вҶ’ HTTPRoute вҶ’ Service вҶ’ Endpoints вҶ’ Pods`
  - Read `kubectl describe` status conditions вҖ” the **reason** field tells you exactly what's wrong
  - Gateway API has much better status reporting than Ingress вҖ” trust the status conditions
  - Compare YAML field values against actual resource names/namespaces

## 1пёҸвғЈ2пёҸвғЈ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - Gateway API resource model: GatewayClass вҶ’ Gateway вҶ’ HTTPRoute
  - Role-oriented design: platform admin (Gateway) vs developer (HTTPRoute)
  - Difference from Ingress вҖ” why Gateway API exists
  - YAML structure for GatewayClass, Gateway, and HTTPRoute

  - HTTPRoute matching (path, headers, method)
  - Backend references (backendRefs, ports, weights)

  - Installing Gateway API CRDs in a cluster

- **What to Focus On**
  - Writing Gateway + HTTPRoute YAML under time pressure
  - Understanding parentRefs and how routes attach to Gateways
  - Debugging using status conditions (much richer than Ingress status)
  - Setting up `allowedRoutes` correctly for multi-namespace scenarios
  - Traffic splitting with weights in backendRefs
  - Creating ReferenceGrant for cross-namespace routing
  - TLS termination on Gateway listeners

- **What to Practice Repeatedly**
  - Create full Gateway + HTTPRoute вҶ’ verify вҶ’ break вҶ’ fix cycle
  - Write HTTPRoute YAML from memory using `kubectl explain`
  - Debug "HTTPRoute not Accepted" scenarios
  - Debug cross-namespace routing failures (ReferenceGrant)
  - Switch between PathPrefix and Exact matching
  - Set up weighted traffic splitting (canary pattern)
  - Multi-listener Gateway with HTTP + HTTPS

- **What to Ignore (Deep Internals)**
  - Gateway controller source code or architecture
  - Advanced filter types (RequestMirror, ExtensionRef)
  - Controller-specific parametersRef configurations
  - TCPRoute, UDPRoute, TLSRoute (less likely on CKA)
  - GRPCRoute specifics (unless explicitly asked)
  - Policy attachment (BackendTLSPolicy, etc.)
  - Custom Gateway controller development

- **5 Quiz Questions**
  1. What are the three core resources in the Gateway API and what does each do?
     вҶ’ *GatewayClass (defines controller implementation), Gateway (defines listeners вҖ” ports/protocols/TLS), HTTPRoute (defines routing rules вҖ” paths/backends/weights)*
  2. An HTTPRoute shows `Accepted: False` with reason `NotAllowedByListeners`. What's the cause?
     вҶ’ *The Gateway listener's `allowedRoutes` doesn't permit routes from the HTTPRoute's namespace or kind. Fix: change `allowedRoutes.namespaces.from` to `All` or add a namespace selector*
  3. How does cross-namespace routing work in Gateway API?
     вҶ’ *An HTTPRoute in namespace-a can reference a Service in namespace-b ONLY if a ReferenceGrant in namespace-b explicitly allows it. Without ReferenceGrant, the reference is denied with `RefNotPermitted`*
  4. How do you implement canary traffic splitting with Gateway API?
     вҶ’ *Use multiple `backendRefs` in an HTTPRoute rule with `weight` fields, e.g., stable weight: 90, canary weight: 10*
  5. What is the key difference between Gateway API and Ingress?
     вҶ’ *Gateway API separates infrastructure (Gateway вҖ” managed by platform admin) from routing (HTTPRoute вҖ” managed by developer). It's more expressive (header matching, weighted routing, filters built into the API spec) and supports multi-namespace routing with explicit delegation via allowedRoutes and ReferenceGrant*

## 1пёҸвғЈ3пёҸвғЈ Component Interaction Map

- **Gateway API вҶ” Pods**
  - Gateway API does NOT directly interact with Pods
  - Traffic path: Gateway Controller вҶ’ HTTPRoute rules вҶ’ Service вҶ’ Endpoints вҶ’ Pods
  - If Pods not Ready вҶ’ removed from Endpoints вҶ’ Gateway returns 503
  - **When things go wrong:** Pod crashes вҶ’ endpoint removed вҶ’ traffic to that backend fails; if all Pods down вҶ’ 503 for the route

- **Gateway API вҶ” Services**
  - HTTPRoute references Services in `backendRefs[].name`
  - Service must be in the same namespace as HTTPRoute (or cross-ns with ReferenceGrant)
  - `backendRefs[].port` must match `.spec.ports[].port` of the Service
  - **When things go wrong:** wrong Service name вҶ’ `ResolvedRefs: False`; wrong port вҶ’ 503; Service deleted вҶ’ route becomes unhealthy

- **Gateway API вҶ” Nodes**
  - Gateway controller typically runs as a Deployment on Nodes
  - Controller Pods may be exposed via NodePort or LoadBalancer Service
  - **When things go wrong:** controller Pod unschedulable вҶ’ Gateway stops processing; all routes go stale

- **Gateway API вҶ” Scheduler**
  - Scheduler places Gateway controller Pods on Nodes
  - Scheduler does NOT interact with Gateway/HTTPRoute resources directly
  - **When things go wrong:** controller Pod Pending вҶ’ insufficient resources or taints blocking scheduling вҶ’ all Gateways down

- **Gateway API вҶ” Controller Manager**
  - Gateway API controllers are **separate** from kube-controller-manager
  - They run as their own Deployments (e.g., envoy-gateway, nginx-gateway-fabric)
  - They watch Gateway, HTTPRoute, Service resources via API Server
  - **When things go wrong:** controller crashed вҶ’ no reconciliation вҶ’ stale routing; new routes ignored; status conditions stop updating

- **Gateway API вҶ” etcd**
  - All Gateway API resources (GatewayClass, Gateway, HTTPRoute, ReferenceGrant) are stored in etcd
  - Controllers read them via API Server (never directly from etcd)
  - **When things go wrong:** etcd unavailable вҶ’ can't create/update Gateway resources; existing running configuration may continue working until controller restarts

- **Gateway API вҶ” Network Layer**
  - Gateway operates at **Layer 7** (HTTP/HTTPS) for HTTPRoute, Layer 4 for TCPRoute
  - Relies on kube-proxy and CNI for underlying network connectivity
  - Gateway controller Pods must reach backend Pod IPs via the CNI network
  - **When things go wrong:** CNI broken вҶ’ controller can't reach Pods вҶ’ 502/504; NetworkPolicy blocking controller вҶ’ traffic dropped silently

- **Gateway API вҶ” Storage Layer**
  - No direct interaction with storage
  - Gateway controllers may use ConfigMaps for configuration
  - TLS Secrets are referenced for certificate data (not PV/PVC)

## Gateway API vs Ingress вҖ” Quick Comparison (CKA Reference)

| Aspect | Ingress | Gateway API |
|---|---|---|
| API Group | `networking.k8s.io/v1` | `gateway.networking.k8s.io/v1` |
| Resources | 1 (Ingress) | 3+ (GatewayClass, Gateway, HTTPRoute, ReferenceGrant) |
| Role separation | None вҖ” single resource | Platform admin (Gateway) vs Developer (HTTPRoute) |
| Routing features | Path + Host only | Path, Host, Headers, Method, Query params, Weights |
| Annotations | Required for advanced features | Built-in via filters and matches |
| Cross-namespace | Not supported | First-class with ReferenceGrant |
| Traffic splitting | Requires controller-specific annotations | Built-in with backendRefs weights |
| Status reporting | Minimal (ADDRESS only) | Rich conditions (Accepted, Programmed, ResolvedRefs) |
| TLS config | On the Ingress resource | On the Gateway listener |
| CRD installation | Built into Kubernetes | Separate CRD install required |

Use `kubectl explain gateway.spec.listeners` and `kubectl explain httproute.spec.rules` to quickly check field names during the exam.
