## 1️⃣ What It Does

- **Core Purpose**
  - Next-generation API for managing traffic routing into and within a Kubernetes cluster
  - Replaces Ingress as the standard way to handle HTTP/HTTPS (and TCP/UDP/gRPC) traffic
  - Provides **role-oriented design** — separates infrastructure concerns (platform admin) from application routing (developer)
  - Three core resources: **GatewayClass** → **Gateway** → **HTTPRoute** (plus TCPRoute, TLSRoute, GRPCRoute)
  - Offers expressive routing: header-based, weighted traffic splitting, request mirroring, URL rewrites — built into the API spec (no annotations)

- **Real-World Use Case**
  - Platform team creates a `Gateway` (shared load balancer entry point) → developers attach `HTTPRoute` rules to route their apps
  - Route `api.example.com/v1` → api-v1-svc, `api.example.com/v2` → api-v2-svc using HTTPRoute
  - Canary deployments with weighted backends: 90% → stable, 10% → canary
  - Header-based routing: `X-Version: beta` → beta Service
  - Multi-team cluster: each namespace gets its own HTTPRoute attached to a shared Gateway
  - TLS termination at the Gateway level, plain HTTP to backend Pods

- **Where It Fits in Cluster Architecture**
  - Lives at the **edge** of the cluster — same position as Ingress, but more powerful
  - **GatewayClass** — defines the controller implementation (like IngressClass, but richer)
  - **Gateway** — the actual listener (like a load balancer instance with ports/protocols/TLS)
  - **HTTPRoute** — routing rules that attach to a Gateway (like Ingress rules, but separate resource)
  - Flow: `Client → Gateway (Controller Pod) → HTTPRoute rules → Service → Pods`
  - Requires a Gateway API controller (e.g., NGINX Gateway Fabric, Envoy Gateway, Istio, Cilium, Contour)
  - Gateway API CRDs must be installed in the cluster (`gateway.networking.k8s.io`)
  - **Graduated to GA** in Kubernetes 1.29 — CKA 2025/2026 now includes it

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - External users/clients accessing HTTP/HTTPS/gRPC applications
  - Development teams needing self-service routing without touching infrastructure
  - Platform teams managing shared ingress infrastructure
  - Traffic splitting / canary release workflows
  - Service mesh integrations (Istio uses Gateway API as its standard routing API)

- **What Resources It Interacts With**
  - **GatewayClass** — selects which controller implementation handles Gateways
  - **Gateway** — defines listeners (ports, protocols, TLS, hostname filters)
  - **HTTPRoute** — routing rules (paths, headers, backends, filters)
  - **Services (ClusterIP)** — backends that HTTPRoute points traffic to
  - **Pods** — ultimate traffic destination behind Services
  - **Secrets (TLS)** — TLS certificates referenced by Gateway listeners
  - **Namespaces** — cross-namespace routing is a first-class concept
  - **ReferenceGrant** — allows cross-namespace references (e.g., HTTPRoute in ns-a → Service in ns-b)

- **Upstream Dependencies**
  - Gateway API **CRDs must be installed** — they are NOT built into core Kubernetes
  - A Gateway API controller must be deployed (NGINX Gateway Fabric, Envoy Gateway, etc.)
  - GatewayClass resource must exist and reference a valid controller
  - TLS Secrets must exist before Gateway references them
  - Backend Services must exist in the target namespace

- **Downstream Dependencies**
  - ClusterIP Services (backends referenced by HTTPRoute)
  - Pods serving actual application traffic
  - Endpoints / EndpointSlices connecting Services to Pods
  - ReferenceGrant resources for cross-namespace routing

## 3️⃣ Recognize Structure (YAML Skeleton)

### GatewayClass

```yaml
apiVersion: gateway.networking.k8s.io/v1      # GA API version
kind: GatewayClass
metadata:
  name: my-gateway-class                       # Cluster-scoped (no namespace)
spec:
  controllerName: example.com/gateway-controller  # Which controller handles this class
```

### Gateway

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: default                           # Namespace-scoped
spec:
  gatewayClassName: my-gateway-class           # Must match a GatewayClass name
  listeners:                                   # One or more listeners (ports/protocols)
    - name: http                               # Listener name (must be unique within Gateway)
      protocol: HTTP                           # HTTP | HTTPS | TLS | TCP | UDP
      port: 80                                 # Port to listen on
      hostname: "*.example.com"                # Optional hostname filter
      allowedRoutes:                           # Which routes can attach
        namespaces:
          from: All                            # All | Same | Selector
        kinds:
          - kind: HTTPRoute                    # Which Route types
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "app.example.com"
      tls:                                     # TLS config for HTTPS listeners
        mode: Terminate                        # Terminate | Passthrough
        certificateRefs:
          - name: app-tls-secret               # Secret in same namespace as Gateway
            kind: Secret
      allowedRoutes:
        namespaces:
          from: Same
```

### HTTPRoute

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-route
  namespace: default
spec:
  parentRefs:                                  # Which Gateway(s) this route attaches to
    - name: my-gateway
      namespace: default                       # Gateway's namespace (if different)
      sectionName: http                        # Specific listener name (optional)
  hostnames:                                   # Hostnames this route matches
    - "app.example.com"
  rules:                                       # Routing rules
    - matches:                                 # Match conditions
        - path:
            type: PathPrefix                   # PathPrefix | Exact
            value: /api
          headers:                             # Optional header-based matching
            - name: X-Version
              value: v2
          method: GET                          # Optional HTTP method matching
      filters:                                 # Request/response transformations
        - type: RequestHeaderModifier
          requestHeaderModifier:
            add:
              - name: X-Custom
                value: "added-by-gateway"
      backendRefs:                             # Backend Services
        - name: api-svc
          port: 80
          weight: 90                           # Traffic weight (for splitting)
        - name: api-canary-svc
          port: 80
          weight: 10
```

- **Key Sections Explained**
  - `apiVersion: gateway.networking.k8s.io/v1` — the GA API version (also v1beta1 for older setups)
  - `gatewayClassName` — links Gateway to its controller implementation (like ingressClassName)
  - `listeners` — define entry points (port, protocol, TLS, hostname filtering)
  - `parentRefs` — how HTTPRoute attaches to a Gateway (replaces the old annotation approach)
  - `matches` — path, header, method, query matching (far more expressive than Ingress)
  - `backendRefs` — target Services with optional weight for traffic splitting
  - `filters` — request/response transformations (rewrites, header mods, redirects, mirrors)
  - `allowedRoutes` — Gateway-side control over which namespaces/route types can attach

- **Common Indentation Traps**
  - `parentRefs` is an array under `spec` — each entry has `name`, `namespace`, `sectionName`
  - `matches` is nested under `rules[]` — `path`, `headers`, `method` are siblings under each match
  - `backendRefs` is at the same level as `matches` under `rules[]` (NOT nested inside matches)
  - `filters` is at the same level as `matches` and `backendRefs` under `rules[]`
  - `certificateRefs` under `tls` is an array — each item has `name` and `kind`
  - `listeners` is an array — each listener has its own `name`, `port`, `protocol` — easy to misplace
  - `allowedRoutes.namespaces.from` takes `All`, `Same`, or `Selector` — easy to use wrong value

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `GatewayClass.spec.controllerName` | Links to controller implementation | GatewayClass created but no controller acts on it |
| `Gateway.spec.gatewayClassName` | Selects which GatewayClass/controller | Gateway is ignored — Programmed=False |
| `Gateway.spec.listeners[].port` | Defines which port accepts traffic | Traffic on that port won't be accepted |
| `Gateway.spec.listeners[].protocol` | HTTP/HTTPS/TLS/TCP | Protocol mismatch — connections fail |
| `Gateway.spec.listeners[].hostname` | Filters which hostnames this listener handles | Hostname doesn't match → Route not accepted |
| `Gateway.spec.listeners[].tls` | TLS termination config | HTTPS fails, falls back to default cert or connection refused |
| `Gateway.spec.listeners[].allowedRoutes` | Controls which Routes can attach | HTTPRoute rejected — not allowed by listener |
| `HTTPRoute.spec.parentRefs` | Attaches route to Gateway | Route is orphaned — no traffic flows |
| `HTTPRoute.spec.parentRefs[].sectionName` | Targets specific listener | Route attaches to wrong or no listener |
| `HTTPRoute.spec.hostnames` | Hostname matching | Hostname must intersect with listener's hostname or route is rejected |
| `HTTPRoute.spec.rules[].matches[].path` | Path-based routing | Requests to that path don't match → fall through |
| `HTTPRoute.spec.rules[].backendRefs` | Target Service(s) | Traffic has no destination → 500 error |
| `HTTPRoute.spec.rules[].backendRefs[].port` | Service port | Wrong port → connection refused or 503 |
| `HTTPRoute.spec.rules[].backendRefs[].weight` | Traffic splitting ratio | Unexpected traffic distribution |
| `ReferenceGrant` | Cross-namespace references | HTTPRoute in ns-a can't reach Service in ns-b → RefNotPermitted |

- **Field Relationships**
  - `Gateway.spec.gatewayClassName` ↔ must match an existing `GatewayClass.metadata.name`
  - `HTTPRoute.spec.parentRefs[].name` ↔ must match an existing `Gateway.metadata.name`
  - `HTTPRoute.spec.parentRefs[].sectionName` ↔ must match a `Gateway.spec.listeners[].name`
  - `HTTPRoute.spec.hostnames` ↔ must intersect with `Gateway.spec.listeners[].hostname` (if set)
  - `HTTPRoute.spec.rules[].backendRefs[].name` ↔ must match a Service in the same namespace (or cross-ns with ReferenceGrant)
  - `HTTPRoute.spec.rules[].backendRefs[].port` ↔ must match the Service's `.spec.ports[].port`
  - `Gateway.spec.listeners[].tls.certificateRefs[].name` ↔ must match a Secret of type `kubernetes.io/tls`

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - `apiVersion` — is it `gateway.networking.k8s.io/v1`?
  - `kind` — is it GatewayClass, Gateway, or HTTPRoute?
  - `gatewayClassName` (on Gateway) — does the GatewayClass exist?
  - `parentRefs` (on HTTPRoute) — does it reference the correct Gateway?
  - `listeners` — correct port, protocol, hostname?
  - `backendRefs` — correct Service name and port?
  - `allowedRoutes` — does the Gateway allow routes from this namespace?

- **Usually Irrelevant for CKA**
  - Advanced filter types (RequestMirror, ExtensionRef, custom filters)
  - Controller-specific parametersRefs on GatewayClass
  - Infrastructure-level configuration (IP allocation, load balancer annotations)
  - Advanced TLS modes (Passthrough details, mTLS configuration)
  - Status conditions beyond basic Accepted/Programmed

- **Red Flags to Detect in 10 Seconds**
  - ❌ Gateway API CRDs not installed → `error: the server doesn't have a resource type "gateway"`
  - ❌ No Gateway controller deployed → GatewayClass exists but Gateway status shows `Accepted: False`
  - ❌ `parentRefs` pointing to non-existent Gateway → HTTPRoute status: `Accepted: False`
  - ❌ `sectionName` typo → Route doesn't attach to intended listener
  - ❌ `hostnames` on HTTPRoute don't intersect with Gateway listener hostname → Route rejected
  - ❌ `allowedRoutes.namespaces.from: Same` but HTTPRoute is in different namespace → Route rejected
  - ❌ Missing `ReferenceGrant` for cross-namespace backend references → `RefNotPermitted`
  - ❌ `backendRefs` Service name or port wrong → traffic drops, 500/503
  - ❌ Using `v1beta1` API on a cluster that only has `v1` CRDs installed (or vice versa)

## 6️⃣ What to Notice In

### kubectl describe gateway \<name\>

- **What Signals Failure**
  - `Accepted: False` — GatewayClass doesn't exist or controller rejects it
  - `Programmed: False` — controller couldn't provision the Gateway (resource issue, port conflict)
  - Listener status: `Conflicted: True` — two listeners sharing the same port/hostname
  - Listener status: `ResolvedRefs: False` — TLS Secret referenced doesn't exist
  - `AttachedRoutes: 0` on a listener that should have routes — route attachment failing
  - Empty `Addresses` — controller hasn't assigned an IP/hostname

- **Where to Look First**
  - `Status.Conditions` — check Accepted and Programmed conditions
  - `Status.Listeners` — per-listener conditions (Accepted, Programmed, ResolvedRefs, Conflicted)
  - `Status.Addresses` — the external IP/hostname assigned by the controller
  - `Events` section — controller-generated events

- **Fields That Matter Most**
  - `Accepted: True/False` — is the Gateway accepted by the controller?
  - `Programmed: True/False` — is the Gateway actually provisioned and working?
  - Per-listener `AttachedRoutes` count — how many routes are attached
  - `Addresses` — the entry point IP/hostname

### kubectl describe httproute \<name\>

- **What Signals Failure**
  - `Accepted: False` — parentRef Gateway doesn't exist, wrong namespace, or listener doesn't allow this route
  - `ResolvedRefs: False` — backend Service doesn't exist or port doesn't match
  - `RefNotPermitted` reason — cross-namespace reference without ReferenceGrant
  - Parent status showing `False` for any condition

- **Where to Look First**
  - `Status.Parents` — per-parent (Gateway) condition status
  - `Status.Parents[].Conditions` — Accepted, ResolvedRefs conditions
  - Events section

- **Fields That Matter Most**
  - Whether each parentRef is `Accepted: True`
  - Whether `ResolvedRefs: True` — all backends resolvable
  - Condition reasons — `NoMatchingListenerHostname`, `NotAllowedByListeners`, `RefNotPermitted`

### kubectl logs \<gateway-controller-pod\>

- **What Signals Failure**
  - `upstream connect error` — backend Service/Pod unreachable
  - `no endpoints found` — Service has no ready Pods
  - `TLS handshake error` — certificate issue
  - `connection refused` — backend Pod not listening on expected port
  - Configuration reload errors after Gateway/HTTPRoute changes

- **Where to Look First**
  - Error-level logs related to upstream/backend connectivity
  - Reconciliation loop errors for specific Gateway or HTTPRoute resources
  - TLS certificate loading failures

- **Fields That Matter Most**
  - HTTP status codes in access logs (502/503/504)
  - Backend endpoint addresses
  - Reconciliation success/failure messages

### kubectl exec (Testing from inside the cluster)

- **Useful Commands**
  ```bash
  # Test backend Service directly
  kubectl exec -it test-pod -- curl http://api-svc:80

  # Test via Gateway Controller's ClusterIP
  kubectl exec -it test-pod -- curl -H "Host: app.example.com" http://<gateway-svc-ip>

  # Verify DNS resolution
  kubectl exec -it test-pod -- nslookup api-svc.default.svc.cluster.local
  ```

- **What Signals Failure**
  - `Connection refused` — Service exists but no healthy Pods
  - `Could not resolve host` — Service doesn't exist
  - HTTP 502/503 — Service exists but Pods not ready
  - Empty response — Pod running but application not responding

### kubectl explain

- **Key Usage**
  ```bash
  kubectl explain gateway.spec.listeners
  kubectl explain gateway.spec.listeners.allowedRoutes
  kubectl explain httproute.spec.rules.matches
  kubectl explain httproute.spec.rules.backendRefs
  kubectl explain httproute.spec.parentRefs
  kubectl explain gatewayclass.spec
  ```
- During CKA — quickly verify field names and nesting
- Confirm which fields are required vs optional
- Check allowed values for enums (protocol, pathType, namespace from)

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ Gateway API CRDs not installed → `error: the server doesn't have a resource type "gateways"`
  - ❌ No Gateway controller deployed → GatewayClass exists, but Gateway status: `Accepted: False`
  - ❌ Wrong `gatewayClassName` on Gateway → Gateway ignored by controller
  - ❌ Wrong `parentRefs` on HTTPRoute → route orphaned, no traffic flows
  - ❌ `sectionName` typo in parentRefs → route doesn't attach to listener
  - ❌ Hostname mismatch between Gateway listener and HTTPRoute → route rejected
  - ❌ `allowedRoutes.namespaces.from: Same` but HTTPRoute in different namespace → rejected
  - ❌ Backend Service name typo → `ResolvedRefs: False`
  - ❌ Backend port wrong → traffic drops, 503 errors
  - ❌ Cross-namespace backendRef without ReferenceGrant → `RefNotPermitted`
  - ❌ TLS Secret missing → listener `ResolvedRefs: False`, HTTPS connections fail
  - ❌ Port conflict between listeners → `Conflicted: True`

- **Status Values to Recognize**
  - **Gateway Conditions:**
    - `Accepted: True` — controller acknowledges the Gateway ✅
    - `Accepted: False` — controller rejects the Gateway ❌
    - `Programmed: True` — Gateway is provisioned and serving traffic ✅
    - `Programmed: False` — Gateway exists but isn't functional ❌
  - **Listener Conditions:**
    - `ResolvedRefs: False` — TLS Secret or other reference not found
    - `Conflicted: True` — port/hostname conflict with another listener
  - **HTTPRoute Parent Conditions:**
    - `Accepted: True` — route attached to Gateway ✅
    - `Accepted: False` — route rejected by Gateway ❌
    - `ResolvedRefs: False` — backend Service not found ❌

- **Event Messages / Condition Reasons to Recognize**
  - `NoMatchingParent` — parentRef doesn't match any Gateway
  - `NoMatchingListenerHostname` — HTTPRoute hostname doesn't intersect with listener
  - `NotAllowedByListeners` — route kind or namespace not permitted by allowedRoutes
  - `RefNotPermitted` — cross-namespace reference without ReferenceGrant
  - `InvalidCertificateRef` — TLS Secret not found or wrong type
  - `BackendNotFound` — Service referenced in backendRefs doesn't exist
  - `PortNotFound` — port in backendRefs doesn't match any Service port

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: `kubectl get gateway` shows Gateway but `Programmed: False` → no controller installed, or gatewayClassName is wrong
  - Scenario 2: HTTPRoute created but traffic not flowing → check `kubectl describe httproute` — `Accepted: False` because `allowedRoutes` doesn't permit the HTTPRoute's namespace
  - Scenario 3: HTTPS not working on Gateway → listener `ResolvedRefs: False` — TLS Secret name typo or Secret doesn't exist
  - Scenario 4: Path `/api` works but `/api/v1` doesn't → `path.type: Exact` instead of `PathPrefix`
  - Scenario 5: HTTPRoute with cross-namespace backendRef returns 500 → missing ReferenceGrant in target namespace
  - Scenario 6: Two HTTPRoutes with conflicting rules → traffic goes to the one with more specific match (longest path prefix wins)
  - Scenario 7: Canary split not working → weight fields missing or both set to 0

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch (Service → Pods)?**
  - Service gets 0 Endpoints/EndpointSlices
  - Gateway controller sees no upstreams for that backend
  - HTTPRoute status may still show `ResolvedRefs: True` (Service exists, controller can't see endpoint count)
  - Result: **503 Service Unavailable** — traffic routed to empty backend
  - The issue is NOT in Gateway/HTTPRoute — it's downstream at Service level

- **What Happens If Ports Mismatch?**
  - `backendRefs[].port` doesn't match Service `.spec.ports[].port`
  - HTTPRoute condition: `ResolvedRefs: False` with reason `PortNotFound` (controller-dependent)
  - Or traffic routes to wrong port → connection refused from Pod
  - Result: **503 or connection reset**

- **What Happens If Gateway Controller Image Is Wrong?**
  - Controller Pod enters `ImagePullBackOff` or `CrashLoopBackOff`
  - ALL Gateways and HTTPRoutes managed by that controller stop working
  - Existing connections drop, Gateway status stops updating
  - No new route changes are processed

- **What Happens If GatewayClass Is Wrong / Missing?**
  - `gatewayClassName` on Gateway references non-existent GatewayClass
  - Gateway is stored in etcd but never processed
  - Gateway status: `Accepted: False` (or no status at all)
  - All attached HTTPRoutes are effectively orphaned

- **What Happens If CRDs Are Not Installed?**
  - `kubectl apply -f gateway.yaml` → `error: the server doesn't have a resource type "gateways"`
  - Can't create Gateway, HTTPRoute, GatewayClass, or ReferenceGrant resources
  - This is the **most fundamental failure** — must install CRDs first

- **What Happens If ReferenceGrant Is Missing?**
  - HTTPRoute in namespace-a references Service in namespace-b
  - Without ReferenceGrant in namespace-b, the cross-namespace reference is denied
  - HTTPRoute condition: `ResolvedRefs: False`, reason: `RefNotPermitted`
  - Traffic to that backendRef drops, other backendRefs in the same rule may still work

- **What Happens If Dependencies Are Missing (Service/Pods)?**
  - Missing Service → `ResolvedRefs: False`, reason: `BackendNotFound`
  - Service exists, no Pods → 503 from controller (no upstream endpoints)
  - Pods exist but not Ready → 503 (endpoints not populated)

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Verify Gateway API CRDs are installed
  kubectl get crd | grep gateway.networking.k8s.io

  # 2. Check GatewayClass exists and has a valid controller
  kubectl get gatewayclass

  # 3. Check Gateway exists and status
  kubectl get gateway -A
  kubectl describe gateway <name> -n <namespace>

  # 4. Check HTTPRoute exists and status
  kubectl get httproute -A
  kubectl describe httproute <name> -n <namespace>

  # 5. Verify Gateway controller is running
  kubectl get pods -A | grep -i gateway    # (controller namespace varies)

  # 6. Check backend Service exists
  kubectl get svc <backend-svc> -n <namespace>

  # 7. Check Service has Endpoints
  kubectl get endpoints <backend-svc> -n <namespace>

  # 8. Check backend Pods are running and Ready
  kubectl get pods -l <selector-labels> -n <namespace>

  # 9. Check controller logs
  kubectl logs -n <controller-ns> <controller-pod> --tail=50

  # 10. Check ReferenceGrant (if cross-namespace)
  kubectl get referencegrant -A

  # 11. Test from inside the cluster
  kubectl run tmp --image=busybox --rm -it -- wget -qO- http://<svc-name>:<port>
  ```

- **Order of Debugging Steps**
  1. Are Gateway API CRDs installed? → `kubectl get crd | grep gateway`
  2. Is a Gateway controller running? → `kubectl get pods -A | grep gateway`
  3. Does GatewayClass exist? → `kubectl get gatewayclass`
  4. Does Gateway exist and is it `Programmed: True`? → `kubectl get gateway`
  5. Does HTTPRoute exist and is it `Accepted: True`? → `kubectl get httproute`
  6. Does the backend Service exist in the correct namespace?
  7. Does the Service have Endpoints?
  8. Are backend Pods Running and Ready?
  9. If cross-namespace: does ReferenceGrant exist? → `kubectl get referencegrant -n <target-ns>`
  10. Check controller logs for specific errors

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get gateway` — `Programmed`? `Accepted`? → If False, it's a Gateway/controller issue
  - `kubectl describe httproute` → check `Status.Parents[].Conditions` — `Accepted: False`? Read the reason
  - `kubectl get endpoints <backend>` → empty? → label/selector mismatch on Service
  - `kubectl logs <controller-pod> --tail=20` → 502/503? → backend issue

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix wrong gatewayClassName on Gateway
  kubectl edit gateway <name> -n <namespace>
  # → Correct spec.gatewayClassName

  # Fix wrong parentRefs on HTTPRoute
  kubectl edit httproute <name> -n <namespace>
  # → Correct parentRefs[].name, namespace, sectionName

  # Fix wrong backend Service name or port
  kubectl edit httproute <name> -n <namespace>
  # → Correct rules[].backendRefs[].name and port

  # Fix allowedRoutes on Gateway listener
  kubectl edit gateway <name> -n <namespace>
  # → Change allowedRoutes.namespaces.from to "All" or "Selector"

  # Install missing Gateway API CRDs
  kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/standard-install.yaml

  # Create missing TLS Secret
  kubectl create secret tls app-tls --cert=tls.crt --key=tls.key -n <namespace>

  # Fix or create ReferenceGrant for cross-namespace routing
  kubectl apply -f - <<EOF
  apiVersion: gateway.networking.k8s.io/v1beta1
  kind: ReferenceGrant
  metadata:
    name: allow-from-other-ns
    namespace: target-ns
  spec:
    from:
      - group: gateway.networking.k8s.io
        kind: HTTPRoute
        namespace: source-ns
    to:
      - group: ""
        kind: Service
  EOF

  # Fix path type
  kubectl edit httproute <name>
  # → Change path.type from Exact to PathPrefix (or vice versa)

  # Fix Service selector (if no Endpoints)
  kubectl edit svc <backend-svc>
  # → Correct selector labels to match Pod labels
  ```

- **Validate Fix**
  ```bash
  # Check Gateway is now Programmed
  kubectl get gateway <name> -n <namespace>
  # → PROGRAMMED=True

  # Check HTTPRoute is Accepted
  kubectl describe httproute <name> -n <namespace>
  # → Status.Parents[].Conditions: Accepted=True, ResolvedRefs=True

  # Check Endpoints populated
  kubectl get endpoints <backend-svc>

  # Test from inside cluster
  kubectl run tmp --image=busybox --rm -it -- wget -qO- http://<svc-name>:<port>

  # Test via Gateway
  curl -H "Host: app.example.com" http://<gateway-ip>/
  ```

- **Confirm Recovery**
  - Gateway shows `Programmed: True` and `Accepted: True` ✅
  - All listeners show `ResolvedRefs: True` ✅
  - HTTPRoute shows `Accepted: True` for all parentRefs ✅
  - `curl` via Gateway returns expected HTTP response ✅
  - Controller logs show no upstream errors ✅
  - Endpoints are populated for backend Services ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

```yaml
# 0. Install Gateway API CRDs (if not installed)
# kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/standard-install.yaml

---
# 1. Deploy backend application
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# 2. Create backend Service
apiVersion: v1
kind: Service
metadata:
  name: web-svc
  namespace: default
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
---
# 3. Create GatewayClass (if controller provides one, skip this)
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: example-gc
spec:
  controllerName: example.com/gateway-controller
---
# 4. Create Gateway
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: example-gateway
  namespace: default
spec:
  gatewayClassName: example-gc
  listeners:
    - name: http
      protocol: HTTP
      port: 80
      allowedRoutes:
        namespaces:
          from: All
---
# 5. Create HTTPRoute
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: web-route
  namespace: default
spec:
  parentRefs:
    - name: example-gateway
      namespace: default
  hostnames:
    - "web.example.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: web-svc
          port: 80
```

- **Validation Steps**
  ```bash
  kubectl apply -f gateway-example.yaml

  # Verify GatewayClass
  kubectl get gatewayclass

  # Verify Gateway — should show Programmed=True
  kubectl get gateway example-gateway
  kubectl describe gateway example-gateway

  # Verify HTTPRoute — should show Accepted=True
  kubectl get httproute web-route
  kubectl describe httproute web-route

  # Verify backend Endpoints
  kubectl get endpoints web-svc
  # → Should show 2 Pod IPs

  # Test routing
  curl -H "Host: web.example.com" http://<gateway-ip>/
  # → Should return NGINX default page
  ```

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Wrong gatewayClassName | Change `example-gc` → `wrong-gc` | Gateway `Accepted: False` |
| Wrong parentRef | Change HTTPRoute parentRef name → `nonexistent-gateway` | HTTPRoute `Accepted: False` |
| Wrong sectionName | Add `sectionName: wrong-listener` to parentRef | HTTPRoute not attached to any listener |
| Wrong backend Service | Change `web-svc` → `web-svc-wrong` | HTTPRoute `ResolvedRefs: False` |
| Wrong backend port | Change `port: 80` → `port: 9999` | 503 errors, possible `ResolvedRefs: False` |
| Hostname mismatch | Gateway listener hostname `*.example.com`, HTTPRoute hostname `other.dev` | HTTPRoute rejected, `NoMatchingListenerHostname` |
| Namespace restriction | Set `allowedRoutes.namespaces.from: Same`, move HTTPRoute to other ns | HTTPRoute `Accepted: False`, `NotAllowedByListeners` |
| Missing TLS Secret | Add HTTPS listener referencing non-existent secret | Listener `ResolvedRefs: False` |
| Break Service selector | Edit Service selector to mismatch Pods | 0 Endpoints → 503 |
| Delete CRDs | `kubectl delete -f standard-install.yaml` | ALL Gateway resources vanish |
| Cross-ns without ReferenceGrant | HTTPRoute in ns-a, backendRef to Service in ns-b, no ReferenceGrant | `RefNotPermitted` |
| Weight = 0 on all backends | Set all backendRef weights to 0 | No traffic routed, 500 errors |
| Remove parentRefs | Delete parentRefs from HTTPRoute | Route orphaned, no traffic |
| Port conflict | Two listeners with same port and conflicting hostnames | Listener `Conflicted: True` |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get gateway -A
  kubectl get httproute -A
  ```

- **How to Narrow Issue**
  1. Gateway `Programmed: True`? → If no: GatewayClass / controller issue
  2. HTTPRoute `Accepted: True`? → If no: read the condition reason
  3. Describe HTTPRoute → check each parent condition and `ResolvedRefs`
  4. Backend Service exists? → `kubectl get svc`
  5. Endpoints populated? → `kubectl get endpoints <svc>`
  6. Controller logs? → `kubectl logs <controller-pod> --tail=30`

- **How to Confirm Root Cause**
  - Trace: `GatewayClass → Gateway → HTTPRoute → Service → Endpoints → Pods`
  - Read `kubectl describe` status conditions — the **reason** field tells you exactly what's wrong
  - Gateway API has much better status reporting than Ingress — trust the status conditions
  - Compare YAML field values against actual resource names/namespaces

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - Gateway API resource model: GatewayClass → Gateway → HTTPRoute
  - Role-oriented design: platform admin (Gateway) vs developer (HTTPRoute)
  - Difference from Ingress — why Gateway API exists
  - YAML structure for GatewayClass, Gateway, and HTTPRoute
  - Listener configuration (port, protocol, hostname, TLS, allowedRoutes)
  - HTTPRoute matching (path, headers, method)
  - Backend references (backendRefs, ports, weights)
  - Cross-namespace routing with ReferenceGrant
  - Status conditions and how to read them (Accepted, Programmed, ResolvedRefs)
  - Installing Gateway API CRDs in a cluster

- **What to Focus On**
  - Writing Gateway + HTTPRoute YAML under time pressure
  - Understanding parentRefs and how routes attach to Gateways
  - Debugging using status conditions (much richer than Ingress status)
  - Setting up `allowedRoutes` correctly for multi-namespace scenarios
  - Traffic splitting with weights in backendRefs
  - Creating ReferenceGrant for cross-namespace routing
  - TLS termination on Gateway listeners

- **What to Practice Repeatedly**
  - Create full Gateway + HTTPRoute → verify → break → fix cycle
  - Write HTTPRoute YAML from memory using `kubectl explain`
  - Debug "HTTPRoute not Accepted" scenarios
  - Debug cross-namespace routing failures (ReferenceGrant)
  - Switch between PathPrefix and Exact matching
  - Set up weighted traffic splitting (canary pattern)
  - Multi-listener Gateway with HTTP + HTTPS

- **What to Ignore (Deep Internals)**
  - Gateway controller source code or architecture
  - Advanced filter types (RequestMirror, ExtensionRef)
  - Controller-specific parametersRef configurations
  - TCPRoute, UDPRoute, TLSRoute (less likely on CKA)
  - GRPCRoute specifics (unless explicitly asked)
  - Policy attachment (BackendTLSPolicy, etc.)
  - Custom Gateway controller development

- **5 Quiz Questions**
  1. What are the three core resources in the Gateway API and what does each do?
     → *GatewayClass (defines controller implementation), Gateway (defines listeners — ports/protocols/TLS), HTTPRoute (defines routing rules — paths/backends/weights)*
  2. An HTTPRoute shows `Accepted: False` with reason `NotAllowedByListeners`. What's the cause?
     → *The Gateway listener's `allowedRoutes` doesn't permit routes from the HTTPRoute's namespace or kind. Fix: change `allowedRoutes.namespaces.from` to `All` or add a namespace selector*
  3. How does cross-namespace routing work in Gateway API?
     → *An HTTPRoute in namespace-a can reference a Service in namespace-b ONLY if a ReferenceGrant in namespace-b explicitly allows it. Without ReferenceGrant, the reference is denied with `RefNotPermitted`*
  4. How do you implement canary traffic splitting with Gateway API?
     → *Use multiple `backendRefs` in an HTTPRoute rule with `weight` fields, e.g., stable weight: 90, canary weight: 10*
  5. What is the key difference between Gateway API and Ingress?
     → *Gateway API separates infrastructure (Gateway — managed by platform admin) from routing (HTTPRoute — managed by developer). It's more expressive (header matching, weighted routing, filters built into the API spec) and supports multi-namespace routing with explicit delegation via allowedRoutes and ReferenceGrant*

## 1️⃣3️⃣ Component Interaction Map

- **Gateway API ↔ Pods**
  - Gateway API does NOT directly interact with Pods
  - Traffic path: Gateway Controller → HTTPRoute rules → Service → Endpoints → Pods
  - If Pods not Ready → removed from Endpoints → Gateway returns 503
  - **When things go wrong:** Pod crashes → endpoint removed → traffic to that backend fails; if all Pods down → 503 for the route

- **Gateway API ↔ Services**
  - HTTPRoute references Services in `backendRefs[].name`
  - Service must be in the same namespace as HTTPRoute (or cross-ns with ReferenceGrant)
  - `backendRefs[].port` must match `.spec.ports[].port` of the Service
  - **When things go wrong:** wrong Service name → `ResolvedRefs: False`; wrong port → 503; Service deleted → route becomes unhealthy

- **Gateway API ↔ Nodes**
  - Gateway controller typically runs as a Deployment on Nodes
  - Controller Pods may be exposed via NodePort or LoadBalancer Service
  - **When things go wrong:** controller Pod unschedulable → Gateway stops processing; all routes go stale

- **Gateway API ↔ Scheduler**
  - Scheduler places Gateway controller Pods on Nodes
  - Scheduler does NOT interact with Gateway/HTTPRoute resources directly
  - **When things go wrong:** controller Pod Pending → insufficient resources or taints blocking scheduling → all Gateways down

- **Gateway API ↔ Controller Manager**
  - Gateway API controllers are **separate** from kube-controller-manager
  - They run as their own Deployments (e.g., envoy-gateway, nginx-gateway-fabric)
  - They watch Gateway, HTTPRoute, Service resources via API Server
  - **When things go wrong:** controller crashed → no reconciliation → stale routing; new routes ignored; status conditions stop updating

- **Gateway API ↔ etcd**
  - All Gateway API resources (GatewayClass, Gateway, HTTPRoute, ReferenceGrant) are stored in etcd
  - Controllers read them via API Server (never directly from etcd)
  - **When things go wrong:** etcd unavailable → can't create/update Gateway resources; existing running configuration may continue working until controller restarts

- **Gateway API ↔ Network Layer**
  - Gateway operates at **Layer 7** (HTTP/HTTPS) for HTTPRoute, Layer 4 for TCPRoute
  - Relies on kube-proxy and CNI for underlying network connectivity
  - Gateway controller Pods must reach backend Pod IPs via the CNI network
  - **When things go wrong:** CNI broken → controller can't reach Pods → 502/504; NetworkPolicy blocking controller → traffic dropped silently

- **Gateway API ↔ Storage Layer**
  - No direct interaction with storage
  - Gateway controllers may use ConfigMaps for configuration
  - TLS Secrets are referenced for certificate data (not PV/PVC)

## Gateway API vs Ingress — Quick Comparison (CKA Reference)

| Aspect | Ingress | Gateway API |
|---|---|---|
| API Group | `networking.k8s.io/v1` | `gateway.networking.k8s.io/v1` |
| Resources | 1 (Ingress) | 3+ (GatewayClass, Gateway, HTTPRoute, ReferenceGrant) |
| Role separation | None — single resource | Platform admin (Gateway) vs Developer (HTTPRoute) |
| Routing features | Path + Host only | Path, Host, Headers, Method, Query params, Weights |
| Annotations | Required for advanced features | Built-in via filters and matches |
| Cross-namespace | Not supported | First-class with ReferenceGrant |
| Traffic splitting | Requires controller-specific annotations | Built-in with backendRefs weights |
| Status reporting | Minimal (ADDRESS only) | Rich conditions (Accepted, Programmed, ResolvedRefs) |
| TLS config | On the Ingress resource | On the Gateway listener |
| CRD installation | Built into Kubernetes | Separate CRD install required |

> **CKA Exam Tip:** Gateway API is newer to CKA. The most likely exam patterns are: (1) Create a Gateway + HTTPRoute to route traffic to backend Services, (2) Fix broken Gateway API configuration (wrong parentRefs, missing ReferenceGrant, wrong backendRefs), (3) Configure TLS on a Gateway listener. Practice reading the status conditions — they tell you exactly what's wrong, unlike Ingress. Use `kubectl explain gateway.spec.listeners` and `kubectl explain httproute.spec.rules` to quickly check field names during the exam.
