Let's assume the starting Corefile is:

```txt
.:53 {
    errors
    health
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure
       fallthrough in-addr.arpa ip6.arpa
       ttl 30
    }
    prometheus :9153
    forward . /etc/resolv.conf
    cache 30
    loop
    reload
    loadbalance
}
```

# The "Everything Together" Lab

This is a realistic exam/practice configuration combining multiple changes:

```txt
corp.local:53 {
    forward . 10.20.30.40
}

.:53 {
    errors
    log
    health
    ready

    rewrite name old.company.com new.company.com

    hosts {
        10.10.10.10 db.internal
        fallthrough
    }

    template IN A facebook.com {
        answer "{{ .Name }} 60 IN A 127.0.0.1"
    }

    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure
       fallthrough in-addr.arpa ip6.arpa
       ttl 300
    }

    prometheus :9153

    forward . 8.8.8.8 1.1.1.1

    cache 300

    loop
    reload
    loadbalance
}
```

Memorize the processing order:

```txt
log
↓
rewrite
↓
hosts
↓
template
↓
kubernetes
↓
forward
↓
cache
↓
loadbalance
```

If you understand what each stage does and why it appears in that order, you'll be comfortable with almost any CoreDNS configuration task likely to appear in CKA/CKS labs or day-to-day cluster operations.
