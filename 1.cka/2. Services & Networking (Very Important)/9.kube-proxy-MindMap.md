# ⚡ Kubernetes kube-proxy — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Implements the **Service abstraction** — makes ClusterIP, NodePort, and LoadBalancer Services actually work
  - Runs on **every Node** as a DaemonSet (or static Pod in some setups)
  - Watches the API Server for Service and Endpoint changes, then programs **network rules** on the Node
  - Routes traffic destined for a Service ClusterIP to the actual backend Pod IPs
  - Performs Layer 4 (TCP/UDP) load balancing across backend Pods

- **Real-World Use Case**
  - Pod calls `http://my-svc:80` → DNS resolves to ClusterIP (e.g., `10.96.45.12`) → kube-proxy's rules on the Node intercept traffic to `10.96.45.12` → forward to a backend Pod IP (e.g., `10.244.1.5:8080`)
  - NodePort Service: external traffic to `NodeIP:30080` → kube-proxy forwards to backend Pod
  - Load balancing: distributes connections across multiple Pod replicas
  - Session affinity: sticky sessions when `sessionAffinity: ClientIP` is configured

- **Where It Fits in Cluster Architecture**
  - Runs on **every Node** (control plane and worker Nodes)
  - Sits between **Services** and **Pods** at the network level
  - Does NOT proxy traffic itself (despite the name) — it programs rules that the kernel enforces
  - Three modes: **iptables** (default), **IPVS**, and **userspace** (legacy)
  - Flow: `Pod → ClusterIP → iptables/IPVS rules (set by kube-proxy) → backend Pod IP`
  - Configuration stored in ConfigMap `kube-proxy` in `kube-system`

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **Every Service** — ClusterIP, NodePort, LoadBalancer, and ExternalName depend on kube-proxy for traffic routing
  - **Every Pod** accessing other Pods via Service names/IPs
  - **CoreDNS** — its `kube-dns` Service routing depends on kube-proxy rules
  - **Ingress Controllers** — route through Services, which depend on kube-proxy
  - **Health checks** via Service — liveness/readiness probes through Service IPs

- **What Resources It Interacts With**
  - **Services** — watches all Service objects for ClusterIP, ports, type changes
  - **Endpoints / EndpointSlices** — watches to know which Pod IPs back each Service
  - **ConfigMap `kube-proxy`** (`kube-system`) — runtime configuration (mode, CIDR, etc.)
  - **Nodes** — programs iptables/IPVS rules on each Node's kernel
  - **API Server** — watches for real-time Service/Endpoint updates

- **Upstream Dependencies**
  - API Server must be reachable (kube-proxy watches it continuously)
  - CNI must provide Pod networking (kube-proxy routes to Pod IPs — those IPs must be routable)
  - iptables or IPVS kernel modules must be available on Nodes
  - ConfigMap `kube-proxy` must exist with valid configuration

- **Downstream Dependencies**
  - ALL Services depend on kube-proxy
  - If kube-proxy is down on a Node → Services are unreachable FROM that Node
  - CoreDNS Service depends on kube-proxy → DNS breaks if kube-proxy fails
  - Ingress, monitoring, logging — anything using Services

## 3️⃣ Recognize Structure (YAML Skeleton)

### kube-proxy DaemonSet (Reference — rarely modified directly)

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: kube-proxy
  namespace: kube-system
  labels:
    k8s-app: kube-proxy
spec:
  selector:
    matchLabels:
      k8s-app: kube-proxy
  template:
    metadata:
      labels:
        k8s-app: kube-proxy
    spec:
      hostNetwork: true                   # Uses Node's network namespace
      serviceAccountName: kube-proxy
      tolerations:
        - operator: Exists                # Runs on ALL Nodes including control plane
      containers:
        - name: kube-proxy
          image: registry.k8s.io/kube-proxy:v1.30.0
          command:
            - /usr/local/bin/kube-proxy
            - --config=/var/lib/kube-proxy/config.conf
          volumeMounts:
            - name: kube-proxy
              mountPath: /var/lib/kube-proxy
            - name: xtables-lock
              mountPath: /run/xtables.lock
            - name: lib-modules
              mountPath: /lib/modules
              readOnly: true
      volumes:
        - name: kube-proxy
          configMap:
            name: kube-proxy              # ConfigMap with config
        - name: xtables-lock
          hostPath:
            path: /run/xtables.lock
            type: FileOrCreate
        - name: lib-modules
          hostPath:
            path: /lib/modules
```

### kube-proxy ConfigMap (THE KEY CONFIG — What You'll Actually Edit)

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: kube-proxy
  namespace: kube-system
data:
  config.conf: |
    apiVersion: kubeproxy.config.k8s.io/v1alpha1
    kind: KubeProxyConfiguration
    mode: "iptables"                      # iptables (default) | ipvs | userspace
    clusterCIDR: "10.244.0.0/16"          # Pod network CIDR
    metricsBindAddress: "0.0.0.0:10249"   # Metrics endpoint
    iptables:
      syncPeriod: "30s"                   # How often iptables rules are refreshed
      minSyncPeriod: "1s"
    ipvs:
      syncPeriod: "30s"
      scheduler: "rr"                    # rr (round-robin), lc (least-connection), etc.
    conntrack:
      maxPerCore: 32768
      min: 131072
```

- **Key Sections Explained**
  - `mode` — the proxy mode: `iptables` (default, most common), `ipvs` (higher performance), `userspace` (legacy)
  - `clusterCIDR` — Pod network CIDR — must match actual CNI configuration
  - `iptables.syncPeriod` — how frequently kube-proxy refreshes iptables rules
  - `hostNetwork: true` — DaemonSet uses Node's network (required to manage Node-level rules)
  - Tolerations: `operator: Exists` — ensures kube-proxy runs on ALL Nodes

- **Common Indentation Traps**
  - ConfigMap contains a **nested YAML** (config.conf) inside the data field — double indentation
  - `mode:` must be a string — `"iptables"` not `iptables` (though both often work)
  - `clusterCIDR` must match the CNI's Pod CIDR exactly
  - DaemonSet `hostNetwork: true` is critical — removing it breaks kube-proxy networking

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `mode` (iptables/ipvs) | Determines how rules are programmed | Wrong mode → rules not created → Services unreachable |
| `clusterCIDR` | Pod network range | Wrong CIDR → kube-proxy can't identify Pod traffic → routing fails |
| `hostNetwork: true` | DaemonSet networking mode | Without it → kube-proxy can't manage Node iptables |
| `image` version | Must match cluster version | Version mismatch → compatibility issues → CrashLoopBackOff |
| `config.conf` (ConfigMap) | All runtime config | Missing/corrupt → kube-proxy can't start |
| DaemonSet tolerations | Run on all Nodes | Missing → kube-proxy skips tainted Nodes → Services broken on those Nodes |
| `xtables-lock` volume | iptables lock file | Missing → iptables contention errors |
| `lib-modules` volume | Kernel modules for IPVS/iptables | Missing → can't load required kernel modules |

- **Field Relationships**
  - `mode: iptables` → programs iptables NAT rules on each Node
  - `mode: ipvs` → programs IPVS virtual servers on each Node (needs `ip_vs` kernel module)
  - `clusterCIDR` ↔ CNI Pod CIDR (must match — e.g., Calico `192.168.0.0/16` or Flannel `10.244.0.0/16`)
  - ConfigMap `kube-proxy` ↔ volume mount at `/var/lib/kube-proxy/config.conf`
  - DaemonSet selector `k8s-app: kube-proxy` ↔ Pod labels
  - Service `spec.ports[].port` → kube-proxy rules → Endpoint Pod IPs + targetPort

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - Is kube-proxy DaemonSet running on ALL Nodes? (`DESIRED` = `READY`)
  - What mode is configured? (iptables or IPVS)
  - Is the ConfigMap `kube-proxy` present?
  - Does the image version match the cluster version?
  - Is `hostNetwork: true` set?

- **Usually Irrelevant for CKA**
  - `conntrack` tuning parameters
  - `metricsBindAddress` details
  - `syncPeriod` fine-tuning
  - `minSyncPeriod` values
  - nodePortAddresses configuration
  - UDP timeout settings

- **Red Flags to Detect in 10 Seconds**
  - ❌ kube-proxy Pods not running on all Nodes (`DESIRED ≠ READY`)
  - ❌ kube-proxy Pod in `CrashLoopBackOff` → config error or missing modules
  - ❌ ConfigMap `kube-proxy` missing → Pods can't start
  - ❌ Wrong `clusterCIDR` in config → Pod traffic routing broken
  - ❌ Image version doesn't match cluster version → potential incompatibility
  - ❌ Missing tolerations → kube-proxy not scheduled on tainted Nodes

## 6️⃣ What to Notice In

### kubectl describe daemonset kube-proxy -n kube-system

- **What Signals Failure**
  - `Desired Number of Nodes Scheduled: 3` but `Number Ready: 2` → one Node missing kube-proxy
  - Events: `FailedCreate`, `FailedScheduling` → can't place Pod on a Node
  - Wrong image or missing ConfigMap in Pod template
  - Missing tolerations → won't schedule on tainted Nodes

- **Where to Look First**
  - `Desired` vs `Current` vs `Ready` numbers
  - Pod template: image, volumes, tolerations
  - Events section

- **Fields That Matter Most**
  - Node count match (Desired = Ready = Available)
  - Image version
  - Volume mounts (ConfigMap, xtables-lock, lib-modules)

### kubectl logs -n kube-system \<kube-proxy-pod\>

- **What Signals Failure**
  - `failed to load config` → ConfigMap missing or corrupt
  - `can't open iptables` → xtables-lock permission issue
  - `IPVS: Can't initialize ipvs` → IPVS kernel modules not loaded
  - `Failed to retrieve node IP` → Node networking issue
  - `connection refused` to API Server → kube-proxy can't watch Services
  - `couldn't get current iptables rules` → iptables binary missing or permission issue

- **Where to Look First**
  - First few lines — initialization errors
  - FATAL or ERROR level messages
  - Connection errors to API Server

- **Fields That Matter Most**
  - Error type (config, iptables, IPVS, API Server connectivity)
  - Mode that kube-proxy is using (confirms iptables vs IPVS)
  - "Using X proxier" message confirms which mode is active

### kubectl exec (Verifying kube-proxy's effect)

- **Key Commands**
  ```bash
  # Check iptables rules on a Node (from kube-proxy Pod or Node SSH)
  # Since kube-proxy uses hostNetwork, exec into kube-proxy Pod:
  kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep <service-name>
  kubectl exec -n kube-system <kube-proxy-pod> -- iptables -t nat -L KUBE-SERVICES -n

  # Check IPVS rules (if IPVS mode)
  kubectl exec -n kube-system <kube-proxy-pod> -- ipvsadm -Ln

  # Test Service connectivity FROM a regular Pod
  kubectl exec <app-pod> -- wget -qO- --timeout=3 http://<svc-clusterip>:<port>
  kubectl exec <app-pod> -- wget -qO- --timeout=3 http://<svc-name>:<port>

  # Check kube-proxy config
  kubectl exec -n kube-system <kube-proxy-pod> -- cat /var/lib/kube-proxy/config.conf
  ```

- **What Signals Failure**
  - No iptables rules for a Service → kube-proxy not syncing that Service
  - `iptables-save` shows stale rules (old Pod IPs) → kube-proxy not refreshing
  - Service ClusterIP times out from a Pod → kube-proxy rules missing on that Node
  - IPVS mode but `ipvsadm` shows no virtual servers → IPVS not initialized

### kubectl explain

- **Key Usage**
  ```bash
  kubectl explain service.spec.type
  kubectl explain service.spec.sessionAffinity
  kubectl explain service.spec.externalTrafficPolicy
  ```
- Helps understand Service fields that kube-proxy implements
- Not a separate `kubectl explain` for kube-proxy itself (it's a system component, not an API resource)

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ kube-proxy Pod not running on a Node → Services unreachable FROM that Node
  - ❌ ConfigMap `kube-proxy` deleted or corrupt → kube-proxy crashes on restart
  - ❌ Wrong `mode` (e.g., `ipvs` without kernel modules) → kube-proxy CrashLoopBackOff
  - ❌ `clusterCIDR` mismatch with CNI → incorrect source NAT behavior
  - ❌ Missing tolerations → kube-proxy doesn't run on tainted Nodes (including control plane)
  - ❌ Image version mismatch → API incompatibility → errors syncing rules
  - ❌ iptables binary missing or broken on Node → can't program rules
  - ❌ kube-proxy can't reach API Server → stale rules, new Services not routed

- **Status Values and Their Meaning**
  - `Running` on every Node → healthy ✅
  - `CrashLoopBackOff` → config error, missing modules, or API Server unreachable
  - `Pending` → can't schedule (taint/toleration issue, resource limits)
  - `ImagePullBackOff` → wrong image or registry unreachable
  - DaemonSet shows `DESIRED ≠ READY` → some Nodes missing kube-proxy

- **Event Messages to Recognize**
  - `FailedToCreateRoute` → routing issue
  - `MissingClusterCIDR` → clusterCIDR not configured
  - `FailedToSetUpSystemLogging` → minor, usually non-fatal
  - `BackOff` → check kube-proxy logs for real cause

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: Services unreachable from one specific Node → kube-proxy Pod not running on that Node → check DaemonSet, tolerations
  - Scenario 2: ALL Services broken cluster-wide → kube-proxy DaemonSet deleted or ConfigMap corrupted
  - Scenario 3: ClusterIP works, but NodePort doesn't → kube-proxy mode issue or firewall blocking NodePort range
  - Scenario 4: New Services don't work but old ones do → kube-proxy can't reach API Server → stale rules
  - Scenario 5: kube-proxy CrashLoopBackOff → ConfigMap has invalid YAML or IPVS modules not loaded

## 8️⃣ Misconfiguration Behavior

- **What Happens If kube-proxy Is Not Running on a Node?**
  - Services are unreachable FROM Pods on that Node
  - DNS may also break (kube-dns Service routing depends on kube-proxy)
  - Pods on that Node can still reach other Pods by direct Pod IP
  - Traffic TO Pods on that Node (from other Nodes with working kube-proxy) still works
  - **Key insight:** kube-proxy failure affects OUTBOUND Service access from that Node

- **What Happens If Mode Is Wrong?**
  - `mode: ipvs` without `ip_vs` kernel module → kube-proxy fails to start → CrashLoopBackOff
  - `mode: userspace` → works but very slow, high CPU (legacy mode)
  - Switching mode requires restarting kube-proxy on all Nodes
  - Old rules from previous mode may linger → manual cleanup needed

- **What Happens If `clusterCIDR` Is Wrong?**
  - Source NAT behavior may be incorrect
  - `externalTrafficPolicy: Local` may not work properly
  - NodePort traffic may show wrong source IP
  - Pod-to-Service traffic generally still works (iptables rules still match ClusterIP)

- **What Happens If Image Is Wrong?**
  - Missing image → `ImagePullBackOff` → no kube-proxy → Service routing breaks on that Node
  - Wrong version → potential API incompatibility → may crash or sync failures
  - Very old image → may not support newer Service features (e.g., EndpointSlices)

- **What Happens If ConfigMap Is Missing?**
  - kube-proxy Pods can't mount config → fail to start → CrashLoopBackOff
  - Error: `failed to load config from /var/lib/kube-proxy/config.conf`
  - ALL kube-proxy Pods affected → cluster-wide Service routing failure

- **What Happens If API Server Is Unreachable?**
  - kube-proxy can't watch Service/Endpoint changes
  - Existing iptables/IPVS rules remain (stale but functional for existing Services)
  - New Services won't get rules → unreachable
  - Deleted Services still have rules → stale routes (may route to dead Pods)

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Check kube-proxy DaemonSet — all Nodes covered?
  kubectl get daemonset kube-proxy -n kube-system

  # 2. Check all kube-proxy Pods
  kubectl get pods -n kube-system -l k8s-app=kube-proxy -o wide

  # 3. Check kube-proxy logs on the problematic Node
  kubectl logs -n kube-system <kube-proxy-pod-on-affected-node> --tail=30

  # 4. Verify ConfigMap exists
  kubectl get configmap kube-proxy -n kube-system

  # 5. Check what mode kube-proxy is running
  kubectl get configmap kube-proxy -n kube-system -o yaml | grep mode

  # 6. Test Service connectivity from affected Node
  kubectl exec <pod-on-affected-node> -- wget -qO- --timeout=3 http://<svc-clusterip>:<port>

  # 7. Check iptables rules on the Node
  kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep KUBE-SVC | head -20

  # 8. Verify Service and Endpoints exist
  kubectl get svc <service-name>
  kubectl get endpoints <service-name>

  # 9. Check if issue is Node-specific or cluster-wide
  # Test from Pods on DIFFERENT Nodes
  kubectl exec <pod-on-node1> -- wget -qO- --timeout=3 http://<svc>:<port>
  kubectl exec <pod-on-node2> -- wget -qO- --timeout=3 http://<svc>:<port>
  ```

- **Order of Debugging Steps**
  1. Is kube-proxy running on all Nodes? → `kubectl get ds kube-proxy -n kube-system`
  2. Is the issue Node-specific or cluster-wide? → test from Pods on different Nodes
  3. Node-specific → check kube-proxy Pod on that Node → logs, status
  4. Cluster-wide → check ConfigMap, DaemonSet definition, image
  5. kube-proxy running but Services broken → check iptables rules on the Node
  6. Rules exist but stale → kube-proxy can't reach API Server

- **How to Detect Issue Under 60 Seconds**
  ```bash
  # 15s: Check DaemonSet
  kubectl get ds kube-proxy -n kube-system
  # → DESIRED ≠ READY? → Node missing kube-proxy

  # 15s: Check Pod status
  kubectl get pods -n kube-system -l k8s-app=kube-proxy
  # → CrashLoopBackOff? → check logs

  # 15s: Check logs
  kubectl logs -n kube-system <kube-proxy-pod> --tail=10
  # → Config error? Module missing? API unreachable?

  # 15s: Test Service
  kubectl exec <any-pod> -- wget -qO- --timeout=3 http://kubernetes.default.svc:443
  # → Timeout? → kube-proxy rules missing
  ```

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix ConfigMap (mode, clusterCIDR, etc.)
  kubectl edit configmap kube-proxy -n kube-system
  # → Fix mode, clusterCIDR, or other config values
  # → Restart kube-proxy to pick up changes:
  kubectl rollout restart daemonset kube-proxy -n kube-system

  # Fix DaemonSet (image, tolerations, volumes)
  kubectl edit daemonset kube-proxy -n kube-system
  # → Fix image version, add tolerations, fix volume mounts

  # Fix missing tolerations (kube-proxy not on control plane)
  kubectl patch daemonset kube-proxy -n kube-system --type=merge -p '{
    "spec": {"template": {"spec": {"tolerations": [
      {"operator": "Exists"}
    ]}}}
  }'

  # Fix wrong image version
  kubectl set image daemonset/kube-proxy -n kube-system \
    kube-proxy=registry.k8s.io/kube-proxy:v1.30.0

  # Recreate missing ConfigMap (emergency minimal config)
  kubectl create configmap kube-proxy -n kube-system \
    --from-literal=config.conf='
  apiVersion: kubeproxy.config.k8s.io/v1alpha1
  kind: KubeProxyConfiguration
  mode: "iptables"
  clusterCIDR: "10.244.0.0/16"
  '
  kubectl rollout restart daemonset kube-proxy -n kube-system

  # Force restart all kube-proxy Pods
  kubectl rollout restart daemonset kube-proxy -n kube-system

  # If IPVS mode fails, switch to iptables
  kubectl edit configmap kube-proxy -n kube-system
  # → Change mode: "ipvs" to mode: "iptables"
  kubectl rollout restart daemonset kube-proxy -n kube-system
  ```

- **Validate Fix**
  ```bash
  # All kube-proxy Pods running
  kubectl get ds kube-proxy -n kube-system
  # → DESIRED = CURRENT = READY = AVAILABLE

  kubectl get pods -n kube-system -l k8s-app=kube-proxy
  # → All Running, 0 restarts

  # Test Service connectivity
  kubectl run test --image=busybox:1.36 --rm -it --restart=Never \
    -- wget -qO- --timeout=3 http://kubernetes.default.svc:443
  # → Should get a response (even if HTTPS error — connection established = routing works)

  # Check iptables rules populated
  kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep KUBE-SVC | wc -l
  # → Should return a number > 0
  ```

- **Confirm Recovery**
  - DaemonSet: DESIRED = READY on all Nodes ✅
  - No kube-proxy Pods in CrashLoopBackOff ✅
  - `iptables-save | grep KUBE-SVC` returns rules ✅
  - Service ClusterIP accessible from Pods ✅
  - NodePort accessible externally (if applicable) ✅
  - CoreDNS (kube-dns Service) working → DNS resolution succeeds ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

kube-proxy is pre-installed in kubeadm clusters. Validate the existing setup:

```bash
# Verify kube-proxy components
kubectl get daemonset kube-proxy -n kube-system
kubectl get pods -n kube-system -l k8s-app=kube-proxy -o wide
kubectl get configmap kube-proxy -n kube-system

# Check current mode
kubectl get configmap kube-proxy -n kube-system -o yaml | grep mode

# Verify iptables rules exist for an existing Service
kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep kube-dns

# Create a test Service and verify routing
kubectl create deployment web --image=nginx --replicas=2
kubectl expose deployment web --port=80
kubectl get svc web
kubectl exec <any-pod> -- wget -qO- --timeout=3 http://web.default.svc:80
# → Should return nginx HTML

# Verify iptables rules for the new Service
kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep web
```

- **Validation Steps**
  - DaemonSet shows DESIRED = READY = number of Nodes
  - All kube-proxy Pods are Running
  - ConfigMap exists with valid config
  - iptables rules exist for Services
  - Service ClusterIP is reachable from Pods

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Delete ConfigMap | `kubectl delete cm kube-proxy -n kube-system` | Existing Pods keep running (config cached); new Pods after restart can't start |
| Restart after ConfigMap delete | `kubectl rollout restart ds kube-proxy -n kube-system` | All kube-proxy Pods CrashLoopBackOff → Service routing stops on affected Nodes |
| Scale to 0 (sort of) | Delete DaemonSet: `kubectl delete ds kube-proxy -n kube-system` | All kube-proxy Pods removed → existing iptables rules linger but go stale |
| Wrong mode | Edit ConfigMap → `mode: "ipvs"` (if IPVS modules not loaded) | kube-proxy crashes → CrashLoopBackOff |
| Wrong image | `kubectl set image ds/kube-proxy -n kube-system kube-proxy=nginx:latest` | kube-proxy Pods run but can't function (wrong binary) → Service routing breaks |
| Remove tolerations | Edit DaemonSet → remove all tolerations | kube-proxy evicted from tainted Nodes → Services broken on those Nodes |
| Wrong clusterCIDR | Edit ConfigMap → change clusterCIDR to `192.168.0.0/16` (if using Flannel 10.244.x) | Source NAT issues; `externalTrafficPolicy: Local` may break |
| Corrupt config.conf | Edit ConfigMap → add invalid YAML in config.conf | kube-proxy can't parse config → CrashLoopBackOff on restart |
| Delete kube-proxy Pods on one Node | `kubectl delete pod <kube-proxy-pod-on-node>` | Pod auto-recreated by DaemonSet (self-healing test — should recover) |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get daemonset kube-proxy -n kube-system
  kubectl get pods -n kube-system -l k8s-app=kube-proxy
  ```

- **How to Narrow Issue**
  1. DaemonSet DESIRED ≠ READY? → Missing Pods → check tolerations, node count
  2. Pods CrashLoopBackOff? → check logs → config error? module missing?
  3. Pods Running but Services broken? → check iptables rules on the Node
  4. One Node affected? → check kube-proxy Pod on that specific Node
  5. All Nodes affected? → ConfigMap or DaemonSet issue

- **How to Confirm Root Cause**
  - `kubectl logs -n kube-system <kube-proxy-pod> --tail=20` → error message tells you exactly what's wrong
  - `kubectl get cm kube-proxy -n kube-system -o yaml` → verify mode, clusterCIDR
  - `kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep KUBE-SVC` → rules exist?

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - kube-proxy architecture: DaemonSet + ConfigMap + iptables/IPVS rules
  - Three proxy modes: iptables (default), IPVS, userspace (legacy)
  - How kube-proxy translates Services → iptables NAT rules
  - Relationship between kube-proxy and Services/Endpoints
  - `externalTrafficPolicy` impact (Local vs Cluster) — kube-proxy enforces this
  - `sessionAffinity` — kube-proxy implements sticky sessions
  - How NodePort works at the iptables level

- **What to Focus On**
  - Verifying kube-proxy is running on all Nodes
  - Diagnosing "Service unreachable" by checking kube-proxy health
  - Editing kube-proxy ConfigMap to change mode or fix config
  - Restarting kube-proxy DaemonSet after config changes
  - Checking iptables rules to confirm Service routing exists
  - Understanding that kube-proxy failure affects the NODE it runs on

- **What to Practice Repeatedly**
  - Check kube-proxy DaemonSet health: `kubectl get ds kube-proxy -n kube-system`
  - Read kube-proxy logs for error messages
  - Edit ConfigMap, restart DaemonSet, verify
  - Test Service connectivity before and after kube-proxy changes
  - Inspect iptables rules for a specific Service

- **What to Ignore (Deep Internals)**
  - Detailed iptables chain logic (KUBE-SVC, KUBE-SEP, KUBE-MARK-MASQ chains)
  - IPVS scheduling algorithms internals
  - Connection tracking table details
  - kube-proxy source code or Go implementation
  - nftables backend (experimental in newer versions)
  - Kernel module compilation

- **5 Quiz Questions**
  1. What happens if kube-proxy is not running on Node-2 but is running on Node-1 and Node-3?
     → *Services are unreachable FROM Pods on Node-2. Pods on Node-2 can't use ClusterIP or NodePort. Node-1 and Node-3 are unaffected.*
  2. What is the default kube-proxy mode and how does it work?
     → *iptables mode. kube-proxy programs iptables NAT rules on each Node that DNAT Service ClusterIPs to backend Pod IPs. Kernel handles actual packet forwarding — kube-proxy does NOT sit in the data path.*
  3. You changed the kube-proxy ConfigMap but Services still use old config. Why?
     → *kube-proxy doesn't auto-reload ConfigMap changes. Must restart: `kubectl rollout restart daemonset kube-proxy -n kube-system`*
  4. kube-proxy is in CrashLoopBackOff with error "can't initialize IPVS." What's the fix?
     → *IPVS kernel modules (`ip_vs`, `ip_vs_rr`, etc.) are not loaded on the Node. Either load them or switch mode to `iptables` in the ConfigMap.*
  5. A Pod can reach another Pod by IP but NOT via its Service ClusterIP. What component is likely broken?
     → *kube-proxy — it's responsible for programming the rules that translate ClusterIP to Pod IPs.*

## 1️⃣3️⃣ Component Interaction Map

- **kube-proxy ↔ Pods**
  - kube-proxy does NOT interact with Pods directly
  - It programs Node-level rules that redirect Pod traffic destined for ClusterIPs
  - kube-proxy itself runs as a Pod (DaemonSet) with `hostNetwork: true`
  - **When things go wrong:** kube-proxy Pod crashes → Node-level rules not updated → Pods on that Node can't reach Services by ClusterIP

- **kube-proxy ↔ Services**
  - kube-proxy watches ALL Service objects via API Server
  - For each Service, creates iptables/IPVS rules mapping ClusterIP:port → Pod IPs:targetPort
  - Implements `sessionAffinity`, `externalTrafficPolicy`, NodePort allocation
  - **When things go wrong:** kube-proxy down → new Services get no rules → unreachable; existing rules go stale → route to dead Pods

- **kube-proxy ↔ Nodes**
  - Runs on EVERY Node (DaemonSet with universal toleration)
  - Uses `hostNetwork: true` to access Node's network namespace
  - Programs iptables/IPVS rules in the Node's kernel
  - **When things go wrong:** kube-proxy missing on a Node → that Node has no Service routing rules → Pods on that Node can't reach ANY Service

- **kube-proxy ↔ Scheduler**
  - Scheduler places kube-proxy Pods on Nodes (DaemonSet behavior)
  - kube-proxy itself doesn't interact with scheduling
  - **When things go wrong:** Scheduler can't place kube-proxy (taint/resource issue) → Node has no kube-proxy → Service routing fails on that Node

- **kube-proxy ↔ Controller Manager**
  - DaemonSet controller (in controller manager) ensures kube-proxy Pod on every Node
  - Endpoint controller creates Endpoint/EndpointSlice objects that kube-proxy watches
  - **When things go wrong:** Controller manager down → DaemonSet can't recreate crashed kube-proxy Pods; Endpoints not updated → stale routing

- **kube-proxy ↔ etcd**
  - No direct interaction — kube-proxy reads via API Server
  - Service and Endpoint records stored in etcd
  - **When things go wrong:** etcd down → API Server can't serve → kube-proxy can't sync → stale rules

- **kube-proxy ↔ Network Layer (CNI)**
  - kube-proxy provides Service-level routing (ClusterIP → Pod IP)
  - CNI provides Pod-level networking (Pod IP assignment, Pod-to-Pod connectivity)
  - They are COMPLEMENTARY — kube-proxy needs CNI-assigned Pod IPs to be routable
  - **When things go wrong:** CNI broken → Pod IPs unreachable → even if kube-proxy rules exist, traffic can't reach Pods; kube-proxy broken → Pod IPs reachable but ClusterIPs are not

- **kube-proxy ↔ Storage Layer**
  - No interaction — kube-proxy is stateless
  - All state derived from API Server watchers
  - Configuration from ConfigMap (not PV/PVC)

## Quick Reference Cheat Sheet

```bash
# =============================================
# KUBE-PROXY QUICK REFERENCE
# =============================================

# Check DaemonSet health
kubectl get daemonset kube-proxy -n kube-system

# List all kube-proxy Pods (with Node placement)
kubectl get pods -n kube-system -l k8s-app=kube-proxy -o wide

# View kube-proxy logs
kubectl logs -n kube-system <kube-proxy-pod> --tail=30

# View kube-proxy config
kubectl get configmap kube-proxy -n kube-system -o yaml

# Check proxy mode
kubectl get cm kube-proxy -n kube-system -o yaml | grep mode

# Edit kube-proxy config
kubectl edit configmap kube-proxy -n kube-system

# Restart kube-proxy (after config change)
kubectl rollout restart daemonset kube-proxy -n kube-system

# Check iptables rules for a Service
kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep <service-name>

# Count total Service rules
kubectl exec -n kube-system <kube-proxy-pod> -- iptables-save | grep KUBE-SVC | wc -l

# Check IPVS rules (if IPVS mode)
kubectl exec -n kube-system <kube-proxy-pod> -- ipvsadm -Ln

# Test Service connectivity
kubectl run test --image=busybox:1.36 --rm -it --restart=Never \
  -- wget -qO- --timeout=3 http://<service-name>:<port>

# =============================================
# PROXY MODES
# =============================================
# iptables (default):  Programs iptables NAT rules. Kernel handles forwarding.
# ipvs:                Programs IPVS virtual servers. Higher performance. Needs ip_vs module.
# userspace (legacy):  kube-proxy proxies in userspace. Slow. Don't use.

# =============================================
# KEY RELATIONSHIP
# =============================================
# Pod calls ClusterIP → iptables DNAT → backend Pod IP
# kube-proxy watches: API Server → Services + Endpoints → programs rules
# kube-proxy failure on Node X → Services broken FROM Node X
```

> **CKA Exam Tip:** kube-proxy questions typically appear as "Services are not reachable from a node" troubleshooting scenarios. The fastest path: (1) `kubectl get ds kube-proxy -n kube-system` — check DESIRED vs READY, (2) check logs on the failing Node's kube-proxy Pod, (3) verify ConfigMap exists. Remember: kube-proxy failure is **Node-local** — if one Node's kube-proxy is down, only Pods ON that Node lose Service routing. Direct Pod-to-Pod by IP still works because that's CNI's job, not kube-proxy's.
