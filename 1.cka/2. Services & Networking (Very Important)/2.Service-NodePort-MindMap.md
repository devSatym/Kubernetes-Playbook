# 🧠 Kubernetes Mind Map — Service (NodePort)

---

## 1️⃣ What It Does

- **Core Purpose**
  - Exposes a Service on a **static port on every Node's IP** in the cluster
  - Makes the application reachable from **outside the cluster** via `<NodeIP>:<NodePort>`
  - Kubernetes automatically creates a **ClusterIP** underneath — NodePort builds on top of ClusterIP
  - kube-proxy programs iptables/IPVS rules on every Node to forward `<NodeIP>:<NodePort>` → `ClusterIP:port` → `PodIP:targetPort`

- **Real-World Use Case**
  - Quick external access during **development or testing** without a cloud load balancer
  - On-premise clusters where a LoadBalancer provider is not available
  - Exposing monitoring dashboards (Grafana, Prometheus) for external access
  - Simple demo environments where you want to hit `http://<NodeIP>:30080`
  - **Not recommended for production** — prefer LoadBalancer or Ingress

- **Where It Fits in Cluster Architecture**
  - Lives in the **Services & Networking** domain of CKA
  - Hierarchy: **NodePort → ClusterIP → Endpoints → Pods**
  - kube-proxy on every Node opens the `nodePort` and forwards traffic
  - CoreDNS still provides internal DNS resolution (same as ClusterIP)
  - Traffic flow: External client → `<AnyNodeIP>:<NodePort>` → kube-proxy → Pod(s)
  - Even if a Node doesn't run the target Pod, it still accepts traffic on the NodePort and routes it to the correct Node

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **External clients** — users/tools connecting from outside the cluster via `<NodeIP>:<NodePort>`
  - **LoadBalancer Services** — build on top of NodePort (NodePort is auto-created)
  - **Pods** — backend Pods receive the traffic routed through the NodePort
  - **Deployments / ReplicaSets** — expose their Pods externally through NodePort
  - **Ingress controllers** — some basic setups use NodePort to expose the Ingress controller itself

- **What Resources It Interacts With**
  - **Endpoints / EndpointSlices** — auto-created; list of Pod IPs matching the selector
  - **Pods** — selector ↔ labels matching (identical to ClusterIP)
  - **ClusterIP** — auto-created underneath the NodePort Service
  - **kube-proxy** — programs iptables/IPVS rules AND opens the nodePort on every Node
  - **CoreDNS** — registers `<svc>.<ns>.svc.cluster.local` A record (internal access)
  - **Node firewall / Security Groups** — must allow traffic on the nodePort range

- **Upstream and Downstream Dependencies**
  - Upstream: Pods must exist with matching labels → Endpoints get populated
  - Upstream: kube-proxy must be running on all Nodes → nodePort is opened and traffic is forwarded
  - Upstream: **Node firewall** must allow inbound traffic on port range `30000–32767`
  - Upstream: CoreDNS must be healthy → internal DNS discovery works
  - Downstream: External clients, LoadBalancer Services, monitoring tools consuming this NodePort
  - Downstream: If NodePort is broken → external access is lost; internal ClusterIP access may still work

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: v1                  # Always v1 for Service
kind: Service                   # Resource type
metadata:
  name: my-nodeport-svc         # Service name (used in DNS internally)
  namespace: default            # Namespace scoping
  labels:                       # Labels ON the Service itself (for organization)
    app: my-app
spec:
  type: NodePort                # ⚠️ MUST specify — not the default
  selector:                     # ⚠️ CRITICAL — matches Pod labels
    app: my-app
  ports:
    - name: http                # Port name (required if multiple ports)
      protocol: TCP             # TCP (default) | UDP | SCTP
      port: 80                  # ⚠️ ClusterIP port (internal consumer access)
      targetPort: 8080          # ⚠️ Pod port (where traffic is forwarded)
      nodePort: 30080           # ⚠️ External port on every Node (30000–32767)
```

- **Key Sections Explained**
  - `apiVersion: v1` — Service is a core API resource, always `v1`
  - `kind: Service` — identifies the resource type
  - `metadata.name` — becomes the DNS hostname inside the cluster
  - `spec.type: NodePort` — **must be explicitly set** (default is ClusterIP)
  - `spec.selector` — **the most critical field** — selects Pods by label match
  - `spec.ports[].port` — the ClusterIP port (internal consumers connect to this)
  - `spec.ports[].targetPort` — the port on the Pod where traffic lands
  - `spec.ports[].nodePort` — the external port opened on every Node; **optional** (auto-assigned from 30000–32767 if omitted)

- **Common Indentation Traps**
  - `selector` is under `spec`, NOT under `metadata`
  - `ports` is a **list** (starts with `-`), not a map
  - `nodePort` is inside each port entry, at the same level as `port` and `targetPort`
  - `type: NodePort` — the **N** and **P** are capitalized (case-sensitive!)
  - Forgetting `name` field when specifying **multiple ports** → validation error
  - `nodePort` must be in range `30000–32767` — out-of-range → API rejected

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `spec.type` | Must be `NodePort` to enable external access | Default is ClusterIP → **no external exposure** |
| `spec.selector` | Selects which Pods receive traffic | No Endpoints → Service has **no backends** → connection refused |
| `spec.ports[].port` | ClusterIP port for internal consumers | Wrong port → internal consumers can't connect |
| `spec.ports[].targetPort` | Port on the Pod receiving traffic | Traffic sent to wrong Pod port → connection refused |
| `spec.ports[].nodePort` | External port on every Node (30000–32767) | Out of range → API error; conflict → creation fails |
| `metadata.name` | DNS name for internal access | Wrong name → DNS lookup fails for internal consumers |
| `metadata.namespace` | Scopes the Service | Cross-namespace calls need `<svc>.<namespace>.svc.cluster.local` |
| `spec.ports[].protocol` | TCP/UDP/SCTP | Mismatch with app protocol → silent failures |

- **Field Relationships**
  - `spec.selector` ↔ `pod.metadata.labels` — **must match exactly**
  - `spec.ports[].port` = ClusterIP port; `spec.ports[].targetPort` = Pod port; `spec.ports[].nodePort` = Node port
  - Three-way port mapping: External → `nodePort` → `port` (ClusterIP) → `targetPort` (Pod)
  - If `targetPort` is omitted → defaults to same value as `port`
  - If `nodePort` is omitted → auto-assigned from 30000–32767 range
  - `nodePort` must be unique across all NodePort Services in the entire cluster
  - One Service can expose **multiple ports** (each needs a unique `name` and can have its own `nodePort`)

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (< 5 seconds)**
  - `spec.type: NodePort` — is it actually set? (omitting = ClusterIP!)
  - `spec.selector` — Are labels present? Do they match Pod labels?
  - `spec.ports` — Is `port`, `targetPort`, and `nodePort` correct?
  - `nodePort` range — is it within `30000–32767`?
  - `metadata.name` and `metadata.namespace` — correct?

- **Usually Irrelevant for CKA**
  - `spec.sessionAffinity` — rarely tested
  - `spec.clusterIP` — auto-assigned, don't set manually
  - `spec.externalTrafficPolicy` — advanced topic, rarely in CKA
  - `spec.ipFamilyPolicy` — dual-stack details not CKA priority
  - `status` section — auto-managed

- **Red Flags to Detect in 10 Seconds**
  - ❌ `type` is missing or set to `ClusterIP` when NodePort was intended → **no external access**
  - ❌ `selector` is empty or missing → **no Endpoints**
  - ❌ `selector` labels don't match any existing Pod labels → **0 Endpoints**
  - ❌ `targetPort` doesn't match any `containerPort` on the Pod → **traffic black hole**
  - ❌ `nodePort` outside `30000–32767` → **API validation error**
  - ❌ `nodePort` conflicts with another Service → **creation fails**
  - ❌ Typo in `type` (e.g., `Nodeport` instead of `NodePort`) → API error or defaults to ClusterIP

## 6️⃣ What to Notice In

### `kubectl describe service <svc-name>`
- **Where to Look First**
  - `Type` — confirm it says `NodePort` (not ClusterIP)
  - `NodePort` line — shows the assigned nodePort number
  - `Endpoints` line — should list Pod IPs; `<none>` = **broken selector**
  - `Selector` — verify it matches Pod labels
  - `Port` / `TargetPort` — confirm mapping is correct
- **Failure Signals**
  - `Endpoints: <none>` → no Pods match the selector
  - `Type: ClusterIP` → type was not set to NodePort
  - Missing `NodePort` line → Service is not NodePort type
  - Port mismatch → traffic routing will fail silently

### `kubectl get endpoints <svc-name>`
- **Where to Look First**
  - IP addresses listed — if empty, selector mismatch or no ready Pods
  - Port number — must match Pod `containerPort`
- **Failure Signals**
  - Empty Endpoints → no matching **Ready** Pods (check labels + readiness probes)
  - Fewer Endpoints than expected → some Pods not Ready

### `kubectl logs <pod>`
- **Where to Look First**
  - Application startup logs — is the app actually listening on the `targetPort`?
  - Errors like "address already in use" or "bind failed"
- **Failure Signals**
  - App not listening → `targetPort` is correct but app misconfigured
  - Connection refused in logs from consumers → Service not routing properly

### `kubectl exec -it <pod> -- curl <svc-name>:<port>`
- **Where to Look First**
  - Response from the target application — proves internal connectivity
  - Also test `curl <NodeIP>:<NodePort>` from a Pod to verify external path
  - DNS resolution — `nslookup <svc-name>` to test CoreDNS
- **Failure Signals**
  - `curl: (7) Failed to connect` → Service/Endpoints broken or targetPort wrong
  - `Could not resolve host` → CoreDNS issue or wrong Service name
  - Timeout → kube-proxy issue or NetworkPolicy blocking

### `kubectl explain service.spec`
- **Where to Look First**
  - Quickly check field names, types, and whether they're required
  - `kubectl explain service.spec.ports` — shows `nodePort`, `port`, `targetPort` fields
  - `kubectl explain service.spec.type` — confirms valid type values
  - Useful during CKA when unsure of exact field name or nodePort range

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - `type` not set to `NodePort` → only ClusterIP created, no external access
  - Selector labels don't match Pod labels → `Endpoints: <none>`
  - `targetPort` doesn't match Pod `containerPort` → connection refused
  - `nodePort` already in use by another Service → creation fails with `port is already allocated`
  - `nodePort` out of range → API rejection: `provided port is not in the valid range`
  - Node firewall/Security Group blocks the nodePort → external access timeout
  - Service in wrong namespace → internal DNS lookup fails

- **Status Values to Recognize**
  - Service itself doesn't have Pod-like statuses (no CrashLoopBackOff, Pending, etc.)
  - Instead, check:
    - `Endpoints: <none>` → **the main red flag** (same as ClusterIP)
    - Pod statuses: CrashLoopBackOff, Pending → Pods aren't Ready → removed from Endpoints
    - `curl <NodeIP>:<NodePort>` → connection refused → targetPort mismatch or no Endpoints
    - `curl <NodeIP>:<NodePort>` → timeout → firewall blocking or kube-proxy down
    - NodePort accessible on one Node but not another → kube-proxy issue on that specific Node

- **Event Messages to Recognize**
  - `Error: provided port is not in the valid range. The range of valid ports is 30000-32767` → nodePort out of range
  - `Error: Service "x" is invalid: spec.ports[0].nodePort: Invalid value: ... provided port is already allocated` → port conflict
  - Check **Pod events** for scheduling/readiness issues: `kubectl describe pod <pod>` → Events section
  - Check **Endpoints events**: `kubectl get events --field-selector involvedObject.kind=Endpoints`

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: "Service created but not reachable from outside" → type is ClusterIP instead of NodePort
  - Scenario 2: "NodePort Service shows `Endpoints: <none>`" → selector mismatch or Pods not Ready
  - Scenario 3: "`curl <NodeIP>:<NodePort>` returns connection refused" → targetPort doesn't match containerPort
  - Scenario 4: "`curl <NodeIP>:<NodePort>` times out" → firewall blocking or kube-proxy down on that Node
  - Scenario 5: "Can reach Service internally via ClusterIP but not via NodePort" → kube-proxy issue or firewall
  - Scenario 6: "Service creation fails" → nodePort conflict or out of range
  - Scenario 7: "Intermittent failures from external" → some Pods unhealthy, removed from Endpoints

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch?**
  - Service selector doesn't match any Pod labels
  - Endpoints object is created but has **zero addresses**
  - `kubectl describe svc` shows `Endpoints: <none>`
  - Both `<NodeIP>:<NodePort>` AND `<ClusterIP>:<port>` → **connection refused**
  - The Service itself looks healthy — `kubectl get svc` shows ClusterIP and NodePort assigned
  - **Silent failure** — no error/event on the Service itself

- **What Happens If Ports Mismatch?**
  - `targetPort` doesn't match Pod's `containerPort`
  - Endpoints exist (IPs are populated — label matching is fine)
  - Traffic forwarded to Pod on wrong port → **connection refused**
  - `nodePort` wrong → external clients connect to wrong port → timeout or "connection refused"
  - `port` wrong → internal consumers connect to wrong ClusterIP port → timeout
  - If `targetPort` is a **named port** and the name doesn't exist on the Pod → traffic fails

- **What Happens If Image Is Wrong?**
  - Pod goes to `ImagePullBackOff` or `ErrImagePull`
  - Pod is **not Ready** → removed from Endpoints
  - Service has zero Endpoints → connection refused on both NodePort and ClusterIP
  - Fix the Deployment image → Pod becomes Ready → auto-added to Endpoints

- **What Happens If Resource Limits Are Wrong?**
  - Pod may be `OOMKilled` → CrashLoopBackOff → not Ready → removed from Endpoints
  - CPU throttling → app slow to respond → potential timeouts through NodePort
  - If all Pods crash → Service has zero Endpoints → both NodePort and ClusterIP return connection refused

- **What Happens If Dependencies Are Missing?**
  - If Pods depend on a ConfigMap/Secret that doesn't exist → Pod stuck in `ContainerCreating`
  - Pod not Ready → not in Endpoints → Service has no backends
  - If Node firewall blocks the nodePort range → external access times out, but internal ClusterIP access works fine
  - If kube-proxy is down on a specific Node → NodePort on that Node doesn't work, other Nodes may still work

## 9️⃣ Fast Detection Strategy

- **Exact kubectl Commands to Run (in order)**
  ```bash
  # Step 1: Check Service exists, type is NodePort, and nodePort is assigned
  kubectl get svc <svc-name> -n <namespace>

  # Step 2: Check Endpoints — THE MOST IMPORTANT CHECK
  kubectl get endpoints <svc-name> -n <namespace>

  # Step 3: Describe Service — look at Type, NodePort, Selector, and Endpoints
  kubectl describe svc <svc-name> -n <namespace>

  # Step 4: Check Pod labels match the selector
  kubectl get pods -n <namespace> --show-labels
  kubectl get pods -n <namespace> -l <selector-key>=<selector-value>

  # Step 5: Check Pod readiness
  kubectl get pods -n <namespace> -o wide

  # Step 6: Test NodePort externally (from a Node or from inside a Pod)
  curl -v <NodeIP>:<NodePort>
  # Or from inside a Pod:
  kubectl exec -it <pod> -- curl <NodeIP>:<NodePort>

  # Step 7: Test internal ClusterIP access (isolates NodePort-specific issues)
  kubectl exec -it <pod> -- curl <svc-name>:<port>

  # Step 8: Check kube-proxy and CoreDNS if all above looks correct
  kubectl get pods -n kube-system -l k8s-app=kube-proxy
  kubectl get pods -n kube-system -l k8s-app=kube-dns

  # Step 9: Check if the nodePort is actually listening on the Node
  # (from the Node itself)
  ss -tlnp | grep <nodePort>
  ```

- **Order of Debugging Steps**
  1. Service exists and type is NodePort? → `kubectl get svc`
  2. Endpoints populated? → `kubectl get endpoints` ← **80% of issues found here**
  3. Selector matches? → compare `describe svc` selector vs `get pods --show-labels`
  4. Pods Ready? → `kubectl get pods`
  5. Internal access works? → `curl <svc-name>:<port>` from inside a Pod
  6. External access works? → `curl <NodeIP>:<NodePort>`
  7. If internal works but external doesn't → firewall / kube-proxy issue
  8. If neither works → Endpoints / selector / targetPort issue

- **How to Detect Issue Under 60 Seconds**
  - Run `kubectl describe svc <name>` → check `Type` (must be NodePort) and `Endpoints` line
  - If `Endpoints: <none>` → selector mismatch or Pods not Ready → compare labels
  - If Endpoints exist → `curl <NodeIP>:<NodePort>` → if refused → targetPort wrong
  - If timeout → firewall blocking nodePort on the Node
  - If internal ClusterIP works but NodePort doesn't → kube-proxy or firewall issue

## 🔟 Fast Fix Strategy

- **How to Fix Using kubectl**
  ```bash
  # Fix type — change ClusterIP to NodePort
  kubectl edit svc <svc-name>
  # Change spec.type: ClusterIP → spec.type: NodePort

  # Fix selector mismatch — edit Service selector
  kubectl edit svc <svc-name> -n <namespace>
  # Change spec.selector to match Pod labels

  # Or fix Pod labels to match Service selector
  kubectl label pod <pod-name> app=my-app --overwrite

  # Fix port mismatch
  kubectl edit svc <svc-name>
  # Update targetPort to match Pod containerPort

  # Fix nodePort conflict — change to a different port
  kubectl edit svc <svc-name>
  # Change nodePort to an unoccupied port in 30000–32767

  # Fix via patch (faster for CKA)
  kubectl patch svc <svc-name> -p '{"spec":{"type":"NodePort"}}'
  kubectl patch svc <svc-name> -p '{"spec":{"selector":{"app":"correct-label"}}}'
  kubectl patch svc <svc-name> -p '{"spec":{"ports":[{"port":80,"targetPort":8080,"nodePort":30080}]}}'

  # Fix by re-applying corrected YAML
  kubectl apply -f corrected-service.yaml

  # If Pod image is wrong → fix Deployment
  kubectl set image deployment/<deploy-name> <container>=<correct-image>
  ```

- **How to Validate Fix**
  ```bash
  # Check Endpoints are now populated
  kubectl get endpoints <svc-name>

  # Verify type is NodePort and nodePort is assigned
  kubectl get svc <svc-name>

  # Verify Pods are Ready
  kubectl get pods -l <selector>

  # Test internal access
  kubectl exec -it <pod> -- curl <svc-name>:<port>

  # Test external access via NodePort
  curl <NodeIP>:<NodePort>
  ```

- **How to Confirm Recovery**
  - `kubectl get endpoints <svc-name>` → shows Pod IPs
  - `kubectl describe svc <svc-name>` → Endpoints populated, Type is NodePort, NodePort assigned
  - `curl <NodeIP>:<NodePort>` returns expected response
  - `curl <svc-name>:<port>` from inside a Pod returns expected response
  - Consumer application logs show successful connections

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 – Create Working Example

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# nodeport-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 80
      nodePort: 30080
```

- **Validation Steps**
  ```bash
  kubectl apply -f deployment.yaml
  kubectl apply -f nodeport-service.yaml
  kubectl get svc web-nodeport                     # Should show TYPE=NodePort, PORT 80:30080
  kubectl get endpoints web-nodeport               # Should show 3 Pod IPs
  kubectl describe svc web-nodeport                # Endpoints populated, NodePort assigned

  # Test internal access
  kubectl run test --rm -it --image=busybox --restart=Never -- wget -qO- web-nodeport:80

  # Test external access (from a Node or machine with access to Node IP)
  curl <NodeIP>:30080

  # Verify on multiple Nodes (traffic works on ANY Node IP)
  curl <Node1-IP>:30080
  curl <Node2-IP>:30080
  ```

### Step 2 – Break It Intentionally

| # | What to Break | How | Expected Symptom |
|---|---|---|---|
| 1 | **Remove type: NodePort** | `kubectl edit svc web-nodeport` → change `type: ClusterIP` | NodePort disappears, external access lost, internal still works |
| 2 | **Mismatch selector** | Change selector to `app: wrong` | Endpoints: `<none>`, both internal and external fail |
| 3 | **Wrong targetPort** | Change `targetPort: 9999` | Endpoints exist but `curl` → connection refused (both paths) |
| 4 | **Wrong nodePort (out of range)** | Change `nodePort: 80` | API error: port is not in valid range 30000–32767 |
| 5 | **Conflicting nodePort** | Create 2 Services with same `nodePort: 30080` | Second Service creation fails: port is already allocated |
| 6 | **Wrong Pod image** | `kubectl set image deploy/web-app nginx=nginx:nonexistent` | Pods → ImagePullBackOff → Endpoints empty |
| 7 | **Delete all Pods** | `kubectl scale deploy/web-app --replicas=0` | Endpoints: `<none>`, both access paths fail |
| 8 | **Add failing readiness probe** | Add probe for wrong path → Pods not Ready | Endpoints empty, both access paths fail |
| 9 | **Block nodePort in firewall** | `iptables -I INPUT -p tcp --dport 30080 -j DROP` (on Node) | External access times out, internal ClusterIP still works |
| 10 | **Remove selector entirely** | `kubectl edit svc` → delete `selector` block | Endpoints become empty, no auto-population |

### Step 3 – Debug Without Looking at Solution

- **What command to run first**
  - `kubectl get svc web-nodeport` — verify Type is NodePort and nodePort is assigned
  - `kubectl get endpoints web-nodeport` — this reveals 80% of issues instantly
- **How to narrow the issue**
  - Endpoints empty? → `kubectl describe svc` → check Selector → compare with `kubectl get pods --show-labels`
  - Endpoints exist? → `kubectl exec -- curl web-nodeport:80` → works? → internal is fine
  - Internal works but external doesn't? → `curl <NodeIP>:30080` → timeout? → firewall/kube-proxy issue
  - Internal also fails? → `targetPort` mismatch → `kubectl describe svc` vs `kubectl describe pod`
  - No NodePort in `kubectl get svc`? → type is not NodePort → `kubectl describe svc` → check Type
- **How to confirm root cause**
  - Selector mismatch: `kubectl get pods -l app=web` returns 0 results but Pods exist with other labels
  - Port mismatch: `kubectl describe svc` shows wrong targetPort vs `kubectl describe pod` containerPort
  - Type wrong: `kubectl get svc` shows `ClusterIP` in TYPE column instead of `NodePort`
  - Pod not Ready: `kubectl get pods` shows Ready 0/1
  - Firewall: `curl <NodeIP>:30080` times out but `kubectl exec -- curl <svc>:80` works

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - NodePort Service creation (imperative and declarative)
  - Three-way port mapping: `nodePort` ↔ `port` ↔ `targetPort`
  - NodePort range: `30000–32767` (configurable via `--service-node-port-range` on apiserver)
  - Selector ↔ label matching mechanism (same as ClusterIP)
  - Endpoints auto-population behavior
  - DNS resolution: `<svc>.<ns>.svc.cluster.local` (internal only; external uses NodeIP)
  - Relationship: NodePort ⊃ ClusterIP ⊃ Endpoints

- **What to Focus On**
  - Creating NodePort Services quickly: `kubectl expose deployment <name> --port=80 --target-port=8080 --type=NodePort`
  - Setting specific nodePort: `--node-port=30080` (imperative) or in YAML
  - Debugging empty Endpoints (identical to ClusterIP debugging)
  - Verifying external access: `curl <NodeIP>:<NodePort>`
  - Verifying internal access: `curl <svc-name>:<port>` from inside a Pod
  - Understanding `kubectl describe svc` output — Type, NodePort, Endpoints
  - Differentiating between internal (ClusterIP) vs external (NodePort) access paths

- **What to Practice Repeatedly**
  - Create a Deployment + NodePort Service from scratch in under 2 minutes
  - Debug a broken NodePort Service (selector mismatch, type wrong, port mismatch) in under 60 seconds
  - Use `kubectl expose --type=NodePort` as a shortcut
  - Verify traffic flow: external client → NodePort → Pod
  - Change Service type from ClusterIP to NodePort and back

- **What to Ignore (Deep Internals Not Required)**
  - iptables rules / IPVS configuration details
  - `externalTrafficPolicy` (Local vs Cluster) — advanced, rarely in CKA
  - `healthCheckNodePort` — auto-created when externalTrafficPolicy is Local
  - Source NAT behavior details
  - Custom `--service-node-port-range` configuration (just know it's configurable)

- **5 Quiz Questions**
  1. **Q:** What is the default nodePort range in Kubernetes?
     - **A:** `30000–32767`. It can be changed via `--service-node-port-range` on the API server.
  2. **Q:** You create a NodePort Service but forget to specify `nodePort`. What happens?
     - **A:** Kubernetes auto-assigns a free port from the `30000–32767` range. The Service is created successfully.
  3. **Q:** A NodePort Service has `Endpoints: <none>`. External access returns connection refused. What is the most likely cause?
     - **A:** No Pods have matching labels, or Pods exist but are not in Ready state.
  4. **Q:** You can `curl <svc-name>:80` from inside a Pod, but `curl <NodeIP>:30080` times out. What should you check?
     - **A:** Node firewall/Security Groups blocking port 30080, or kube-proxy not running on that Node.
  5. **Q:** Two NodePort Services try to use the same `nodePort: 30080`. What happens?
     - **A:** The second Service creation fails with error: `provided port is already allocated`.

## 1️⃣3️⃣ Component Interaction Map

```
                                     ┌─────────────┐
                                     │   CoreDNS   │
                                     │ (DNS A rec) │
                                     └──────┬──────┘
                                            │ resolves svc name → ClusterIP
                                            │ (internal DNS only)
                                            ▼
┌──────────────┐   <NodeIP>:<NodePort>   ┌───────────────────┐   iptables/IPVS   ┌──────────┐
│   External   │ ──────────────────────► │     Service        │ ────────────────► │  Pod(s)  │
│   Client     │                         │   (NodePort)       │   (kube-proxy)    │ :targetP │
└──────────────┘                         └─────────┬─────────┘                    └──────────┘
                                                   │
┌──────────────┐   <ClusterIP>:<port>              │
│   Internal   │ ──────────────────────────────────┘
│   Consumer   │        (also works)     ┌──────────────────┐
│    Pod       │                         │    Endpoints /    │
└──────────────┘                         │  EndpointSlices   │
                                         │  (auto-managed)   │
                                         └──────────────────┘
```

- **Pods**
  - Service selects Pods via labels
  - Only **Ready** Pods appear in Endpoints
  - Pod deletion → auto-removed from Endpoints
  - Pod IP change (restart) → Endpoints updated automatically
  - Pod can be on any Node; NodePort routes traffic across Nodes

- **Services ↔ Other Services**
  - NodePort Service automatically creates a ClusterIP underneath
  - LoadBalancer Service automatically creates a NodePort underneath
  - Hierarchy: **LoadBalancer → NodePort → ClusterIP → Endpoints → Pods**

- **Nodes**
  - kube-proxy on **every Node** opens the `nodePort` and programs routing rules
  - External traffic can arrive on **any Node**, even one not running the target Pod
  - kube-proxy forwards to the correct Pod regardless of which Node receives the traffic
  - If a Node is drained/unschedulable → it still handles NodePort traffic (kube-proxy still runs)
  - Node firewall must allow inbound traffic on the nodePort

- **Scheduler**
  - No direct interaction — Scheduler places Pods; Service discovers them via labels
  - If Scheduler can't place a Pod → Pod stays Pending → not in Endpoints
  - NodePort works regardless of where Pods are scheduled

- **Controller Manager**
  - **Endpoints Controller** watches Pod labels and readiness → populates Endpoints
  - **EndpointSlice Controller** (modern) → manages EndpointSlice objects
  - If Controller Manager is down → Endpoints are **not updated** (stale)

- **etcd**
  - Stores Service, Endpoints, and EndpointSlice objects
  - Stores the nodePort allocation registry (conflict detection)
  - If etcd is down → no new Services can be created; existing ones continue to work (cached kube-proxy rules)

- **Network Layer**
  - kube-proxy → opens nodePort on all Nodes AND programs iptables/IPVS rules
  - CNI plugin → provides Pod-to-Pod networking (required for cross-Node forwarding)
  - NetworkPolicies → can block traffic to/from Pods behind a Service (both internal and NodePort paths)
  - External traffic hits Node's physical/virtual NIC → kube-proxy forwards

- **Storage Layer**
  - No direct interaction — Services are purely networking constructs
  - However, if a Pod behind a Service needs a PVC and it's not bound → Pod stays Pending → not in Endpoints

- **Interaction Behavior When Things Go Wrong**
  - kube-proxy down on one Node → NodePort on that Node fails; other Nodes still work
  - kube-proxy down on all Nodes → all NodePort access fails; ClusterIP may also fail
  - CoreDNS down → internal DNS resolution fails; NodePort (via IP) still works
  - Endpoints Controller down → Endpoints become stale; new Pods won't be added/removed
  - CNI plugin issues → Pod-to-Pod networking fails → even with correct Endpoints, cross-Node forwarding breaks
  - Node firewall blocks nodePort → timeout from external; internal ClusterIP access unaffected

## ⚡ Quick Reference Cheat Sheet

```bash
# Create NodePort Service imperatively
kubectl expose deployment <name> --name=<svc-name> --port=80 --target-port=8080 --type=NodePort

# Create with specific nodePort (must edit YAML — imperative doesn't support --node-port directly)
# Use: kubectl expose, then kubectl edit svc <name> to set nodePort

# Get Service (check TYPE column and PORT column shows nodePort)
kubectl get svc <name> -o wide

# Check Endpoints (MOST IMPORTANT)
kubectl get endpoints <name>

# Describe Service (check Type, NodePort, Endpoints)
kubectl describe svc <name>

# Test internal DNS
kubectl exec -it <pod> -- nslookup <svc-name>

# Test internal connectivity (ClusterIP path)
kubectl exec -it <pod> -- curl -s <svc-name>:<port>

# Test external connectivity (NodePort path)
curl <NodeIP>:<NodePort>

# Get Node IPs (to test NodePort)
kubectl get nodes -o wide

# Quick selector check
kubectl get pods -l <key>=<value>

# Patch type to NodePort
kubectl patch svc <name> -p '{"spec":{"type":"NodePort"}}'

# Patch selector
kubectl patch svc <name> -p '{"spec":{"selector":{"app":"correct"}}}'

# Patch ports including nodePort
kubectl patch svc <name> --type='json' -p='[{"op":"replace","path":"/spec/ports/0/targetPort","value":8080}]'

# Quick create: Deployment + NodePort Service in one shot
kubectl create deployment web --image=nginx --port=80
kubectl expose deployment web --type=NodePort --port=80 --target-port=80
```
