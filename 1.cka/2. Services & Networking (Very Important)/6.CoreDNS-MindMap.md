# 🌐 Kubernetes CoreDNS — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Provides **cluster-internal DNS resolution** for all Kubernetes Services and Pods
  - Translates Service names (e.g., `my-svc.default.svc.cluster.local`) to ClusterIP addresses
  - Enables service discovery — Pods find other Services by name, not by IP
  - Replaces the older kube-dns (CoreDNS is default since Kubernetes 1.13)
  - Runs as a Deployment with 2 replicas in the `kube-system` namespace

- **Real-World Use Case**
  - Pod A calls `http://backend-svc:8080` → CoreDNS resolves `backend-svc` to a ClusterIP → traffic reaches backend Pods
  - Cross-namespace resolution: `http://api-svc.production.svc.cluster.local`
  - Headless Service resolution: returns individual Pod IPs instead of a single ClusterIP
  - External DNS forwarding: Pods resolving `google.com` → CoreDNS forwards to upstream DNS (e.g., `/etc/resolv.conf` on the Node)

- **Where It Fits in Cluster Architecture**
  - Runs as `coredns` Deployment in `kube-system` namespace
  - Exposed via `kube-dns` Service (ClusterIP, typically `10.96.0.10`)
  - Every Pod's `/etc/resolv.conf` is auto-configured to point `nameserver` to the `kube-dns` Service IP
  - Sits between **all** Pod-to-Service communication
  - Flow: `Pod DNS query → kube-dns Service (10.96.0.10) → CoreDNS Pods → respond with ClusterIP or Pod IPs`
  - Configuration lives in a **ConfigMap** named `coredns` in `kube-system`

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **Every Pod** in the cluster that uses DNS-based service discovery
  - Services accessed by name (virtually all microservice communication)
  - Ingress Controllers resolving backend Service names
  - Operators, controllers, and add-ons that resolve internal Service names
  - Pods resolving external domains (e.g., pulling images, calling external APIs)

- **What Resources It Interacts With**
  - **ConfigMap `coredns`** (`kube-system`) — Corefile configuration
  - **Service `kube-dns`** (`kube-system`) — ClusterIP fronting CoreDNS Pods
  - **Deployment `coredns`** (`kube-system`) — manages CoreDNS Pod replicas
  - **Endpoints `kube-dns`** — links the kube-dns Service to CoreDNS Pod IPs
  - **Kubernetes API Server** — CoreDNS watches Services/Endpoints/Pods to build DNS records
  - **Node's `/etc/resolv.conf`** — upstream DNS for forwarding external queries

- **Upstream Dependencies**
  - API Server must be reachable (CoreDNS watches it for Service/Endpoint changes)
  - CNI (network plugin) must be functional — CoreDNS Pods need network connectivity
  - Node DNS (`/etc/resolv.conf`) — for forwarding external queries
  - `kube-dns` Service must exist with correct ClusterIP

- **Downstream Dependencies**
  - ALL Pods depend on CoreDNS for name resolution
  - If CoreDNS is down → DNS resolution fails cluster-wide → cascading failures
  - Ingress Controllers, monitoring, logging — anything using Service names internally

## 3️⃣ Recognize Structure (YAML Skeleton)

### CoreDNS Deployment (for reference — rarely created from scratch)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: coredns
  namespace: kube-system
  labels:
    k8s-app: kube-dns                  # Label used by kube-dns Service selector
spec:
  replicas: 2                          # HA — always 2 replicas
  selector:
    matchLabels:
      k8s-app: kube-dns
  template:
    metadata:
      labels:
        k8s-app: kube-dns
    spec:
      serviceAccountName: coredns
      tolerations:
        - key: CriticalAddonsOnly
          operator: Exists
        - key: node-role.kubernetes.io/control-plane
          effect: NoSchedule
      containers:
        - name: coredns
          image: registry.k8s.io/coredns/coredns:v1.11.1
          ports:
            - containerPort: 53
              name: dns
              protocol: UDP
            - containerPort: 53
              name: dns-tcp
              protocol: TCP
          volumeMounts:
            - name: config-volume
              mountPath: /etc/coredns
              readOnly: true
          args:
            - -conf
            - /etc/coredns/Corefile
      volumes:
        - name: config-volume
          configMap:
            name: coredns              # References the coredns ConfigMap
            items:
              - key: Corefile
                path: Corefile
```

### CoreDNS ConfigMap (Corefile) — **THE ONE YOU'LL EDIT IN CKA**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors                         # Log errors
        health {                       # Health check endpoint on :8080
            lameduck 5s
        }
        ready                          # Readiness endpoint on :8181
        kubernetes cluster.local in-addr.arpa ip6.arpa {   # Kubernetes plugin
            pods insecure               # Pod DNS records
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153               # Metrics endpoint
        forward . /etc/resolv.conf {   # Forward external queries upstream
            max_concurrent 1000
        }
        cache 30                       # Cache DNS responses for 30s
        loop                           # Detect and halt forwarding loops
        reload                         # Auto-reload Corefile on ConfigMap change
        loadbalance                    # Round-robin DNS responses
    }
```

- **Key Sections Explained**
  - `.:53` — listen on all interfaces, port 53
  - `kubernetes cluster.local` — the plugin that answers queries for cluster Services/Pods
  - `forward . /etc/resolv.conf` — forward non-cluster queries to Node's upstream DNS
  - `cache 30` — cache results for 30 seconds
  - `loop` — detects DNS forwarding loops and stops CoreDNS (prevents infinite loops)
  - `reload` — auto-reloads config when ConfigMap changes (no Pod restart needed)
  - `health` / `ready` — liveness and readiness probe endpoints

- **Common Indentation Traps**
  - Corefile uses **brace-based syntax**, NOT YAML indentation — spaces within `{ }` blocks
  - Each plugin must be on its own line inside the `.:53 { }` block
  - `forward . /etc/resolv.conf` — the `.` means "all unmatched queries" — missing dot = broken forwarding
  - `kubernetes cluster.local` — wrong domain (`cluster.local`) = Services not resolvable

## 4️⃣ Critical Fields Only (CKA Focus)

| Field / Config | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `kubernetes cluster.local` | Resolves cluster Services | Wrong domain → ALL Service DNS fails |
| `forward . /etc/resolv.conf` | External DNS resolution | External domains unresolvable from Pods |
| `cache 30` | Reduces DNS latency | Removed → higher API Server load, slower resolution |
| `loop` | Prevents forwarding loops | Removed → potential infinite DNS loops crash CoreDNS |
| `ConfigMap name: coredns` | Config source for CoreDNS | Renamed/deleted → CoreDNS can't start or uses stale config |
| `replicas: 2` | High availability | Set to 0 → no DNS in cluster |
| `kube-dns Service ClusterIP` | DNS nameserver for all Pods | Wrong IP → no Pod can resolve any name |
| Label `k8s-app: kube-dns` | Connects Deployment ↔ Service | Mismatch → `kube-dns` Service has 0 Endpoints → DNS fails |
| `containerPort: 53 (UDP & TCP)` | DNS protocol ports | Wrong port → DNS queries never reach CoreDNS |
| Pod `/etc/resolv.conf` | Points Pods to CoreDNS | Wrong nameserver → Pod can't resolve anything |

- **Field Relationships**
  - `kube-dns` Service selector (`k8s-app: kube-dns`) ↔ CoreDNS Pod labels (`k8s-app: kube-dns`)
  - `kube-dns` Service port `53` ↔ CoreDNS container port `53` (both UDP and TCP)
  - CoreDNS ConfigMap `coredns` ↔ Volume mount in Deployment at `/etc/coredns/Corefile`
  - `kubelet --cluster-dns` flag ↔ `kube-dns` Service ClusterIP (how Pods get their nameserver)
  - `forward . /etc/resolv.conf` ↔ Node's `/etc/resolv.conf` (upstream DNS servers)

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - Is the `coredns` Deployment running with 2/2 replicas?
  - Does the `kube-dns` Service exist with the expected ClusterIP?
  - Is the `coredns` ConfigMap present in `kube-system`?
  - Does the Corefile contain `kubernetes cluster.local`?
  - Is `forward . /etc/resolv.conf` present for external DNS?

- **Usually Irrelevant for CKA**
  - Prometheus metrics configuration
  - Detailed cache tuning parameters
  - `lameduck` timing in health checks
  - `loadbalance` plugin details
  - CoreDNS image version specifics

- **Red Flags to Detect in 10 Seconds**
  - ❌ `coredns` Pods in `CrashLoopBackOff` → Corefile syntax error or DNS loop
  - ❌ `kube-dns` Service missing → ALL DNS resolution broken
  - ❌ ConfigMap `coredns` missing or empty → CoreDNS can't load config
  - ❌ `kubernetes` plugin missing from Corefile → cluster Services not resolvable
  - ❌ `forward` plugin pointing to wrong upstream → external DNS broken
  - ❌ Labels mismatch between Deployment and `kube-dns` Service → 0 Endpoints
  - ❌ `loop` plugin detects a loop → CoreDNS crashes intentionally (by design)

## 6️⃣ What to Notice In

### kubectl describe deployment coredns -n kube-system

- **What Signals Failure**
  - `Replicas: 2 desired | 0 available` → Pods not scheduling or crashing
  - Events: `FailedCreate`, `BackOff` → underlying Pod issues
  - `Image:` field shows wrong or unreachable image
  - `Volumes:` not referencing the `coredns` ConfigMap

- **Where to Look First**
  - Ready replicas count
  - Pod template → image, volume mounts, tolerations
  - Events section at the bottom

- **Fields That Matter Most**
  - `Replicas` — both desired and available
  - `Pod Template → Containers → Image`
  - `Pod Template → Volumes → ConfigMap name`

### kubectl logs -n kube-system -l k8s-app=kube-dns

- **What Signals Failure**
  - `plugin/loop: Loop ... detected` → DNS forwarding loop → CoreDNS exits
  - `plugin/kubernetes: ... connection refused` → can't reach API Server
  - `Failed to list *v1.Service` → RBAC issue, can't watch Services
  - `[FATAL] plugin/loop: ... Seen "HINFO" ...` → loop detection triggered
  - `Error: ... open /etc/coredns/Corefile: no such file or directory` → ConfigMap not mounted

- **Where to Look First**
  - FATAL or ERROR level messages
  - Plugin-specific errors (loop, kubernetes, forward)
  - Connection refused / timeout messages

- **Fields That Matter Most**
  - Plugin name in error (tells you which feature broke)
  - Error description (connection, config, loop)

### kubectl exec (Testing DNS from a Pod)

- **Useful Commands**
  ```bash
  # Test DNS resolution from any Pod
  kubectl exec -it <pod> -- nslookup kubernetes.default
  kubectl exec -it <pod> -- nslookup <service-name>.<namespace>.svc.cluster.local

  # Test with specific DNS server
  kubectl exec -it <pod> -- nslookup kubernetes.default 10.96.0.10

  # Check Pod's resolv.conf
  kubectl exec -it <pod> -- cat /etc/resolv.conf

  # Test external DNS resolution
  kubectl exec -it <pod> -- nslookup google.com

  # Use a debug Pod if none available
  kubectl run dns-test --image=busybox:1.36 --rm -it -- nslookup kubernetes.default
  ```

- **What Signals Failure**
  - `** server can't find ...: NXDOMAIN` → Service doesn't exist or wrong name
  - `** server can't find ...: SERVFAIL` → CoreDNS is broken or unreachable
  - `;; connection timed out; no servers could be reached` → kube-dns Service down, CoreDNS Pods down, or network issue
  - `nameserver` in `/etc/resolv.conf` doesn't match `kube-dns` ClusterIP → kubelet misconfiguration

### kubectl explain

- **Key Usage**
  ```bash
  kubectl explain service.spec.clusterIP
  kubectl explain pod.spec.dnsPolicy
  kubectl explain pod.spec.dnsConfig
  ```
- Quickly verify `dnsPolicy` options: `Default`, `ClusterFirst`, `ClusterFirstWithHostNet`, `None`
- Check `dnsConfig` fields for custom DNS settings

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ Corefile syntax error → CoreDNS Pods crash (`CrashLoopBackOff`)
  - ❌ `loop` detected → CoreDNS exits intentionally (Node's `/etc/resolv.conf` points back to CoreDNS)
  - ❌ ConfigMap `coredns` deleted → Pods can't mount Corefile → crash
  - ❌ `kube-dns` Service deleted → nameserver IP unreachable → all DNS fails
  - ❌ Label mismatch on CoreDNS Pods → `kube-dns` Service has 0 Endpoints
  - ❌ Wrong `kubernetes` plugin domain → internal Services not resolvable
  - ❌ `forward` pointing to non-existent upstream → external resolution fails
  - ❌ Network Policy blocking DNS traffic on port 53 → silent DNS failures
  - ❌ CoreDNS Pods not scheduled (taint issues) → no DNS

- **Status Values and Their Meaning**
  - `Running` (2/2) → healthy ✅
  - `CrashLoopBackOff` → Corefile error, DNS loop, or missing ConfigMap
  - `Pending` → can't schedule (resource constraints, taints, no nodes)
  - `ContainerCreating` → waiting for image pull or volume mount

- **Event Messages to Recognize**
  - `plugin/loop: Loop ... detected for zone "."` → forwarding loop → CoreDNS crashes
  - `Failed to list *v1.Service: Unauthorized` → ServiceAccount RBAC broken
  - `MountVolume.SetUp failed: configmap "coredns" not found` → ConfigMap missing
  - `Back-off restarting failed container` → check logs for root cause

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: Pods can't resolve Service names → CoreDNS Pods in CrashLoopBackOff → check logs → Corefile syntax error
  - Scenario 2: Internal DNS works, external DNS fails → `forward` plugin misconfigured or removed
  - Scenario 3: DNS works for some Pods, fails for others → check Pod's `dnsPolicy` (set to `Default` instead of `ClusterFirst`)
  - Scenario 4: CoreDNS keeps crashing → `loop` plugin detects loop → Node resolv.conf pointing to cluster DNS
  - Scenario 5: New Service not resolving → CoreDNS can't reach API Server → check connectivity

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch (CoreDNS Pods ↔ kube-dns Service)?**
  - `kube-dns` Service gets 0 Endpoints
  - Pods send DNS queries to kube-dns ClusterIP → no backend → connection timeout
  - Result: **ALL DNS resolution fails cluster-wide**
  - Error in Pods: `;; connection timed out; no servers could be reached`

- **What Happens If Ports Mismatch?**
  - CoreDNS listens on port 53 (UDP+TCP) — if changed in Corefile but not in Service/Deployment
  - `kube-dns` Service forwards to wrong port → DNS queries unanswered
  - Result: **DNS timeout for all Pods**

- **What Happens If Image Is Wrong?**
  - CoreDNS Pods enter `ImagePullBackOff` or `ErrImagePull`
  - Existing DNS cache expires → DNS resolution fails across the cluster
  - If image is wrong version with breaking config changes → `CrashLoopBackOff`

- **What Happens If Resource Limits Are Too Low?**
  - CoreDNS Pods get `OOMKilled` under load
  - DNS resolution intermittently fails — works sometimes, times out other times
  - Pod restarts frequently → brief DNS outages during restarts

- **What Happens If ConfigMap Is Missing/Corrupt?**
  - Missing ConfigMap → volume mount fails → Pod can't start → `CrashLoopBackOff`
  - Corrupt Corefile syntax → CoreDNS parses and fails → `CrashLoopBackOff`
  - Missing `kubernetes` plugin → internal Services resolve as NXDOMAIN
  - Missing `forward` plugin → external domains (google.com) resolve as NXDOMAIN

- **What Happens If `dnsPolicy` Is Wrong on a Pod?**
  - `dnsPolicy: Default` → Pod uses Node's DNS, NOT CoreDNS → can't resolve cluster Service names
  - `dnsPolicy: None` without `dnsConfig` → Pod has no DNS at all
  - `dnsPolicy: ClusterFirst` (default) → correct behavior, uses CoreDNS

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Check if CoreDNS Pods are running
  kubectl get pods -n kube-system -l k8s-app=kube-dns

  # 2. Check kube-dns Service exists and has Endpoints
  kubectl get svc kube-dns -n kube-system
  kubectl get endpoints kube-dns -n kube-system

  # 3. Quick DNS test from a temporary Pod
  kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup kubernetes.default

  # 4. Check CoreDNS logs for errors
  kubectl logs -n kube-system -l k8s-app=kube-dns --tail=30

  # 5. Check the ConfigMap (Corefile)
  kubectl get configmap coredns -n kube-system -o yaml

  # 6. Check a failing Pod's resolv.conf
  kubectl exec <pod> -- cat /etc/resolv.conf

  # 7. Check Pod's dnsPolicy
  kubectl get pod <pod> -o jsonpath='{.spec.dnsPolicy}'

  # 8. Describe CoreDNS Deployment
  kubectl describe deployment coredns -n kube-system

  # 9. Check for NetworkPolicies blocking DNS
  kubectl get networkpolicies -A
  ```

- **Order of Debugging Steps**
  1. Are CoreDNS Pods running? → `kubectl get pods -n kube-system -l k8s-app=kube-dns`
  2. Quick DNS test → `kubectl run dns-test --image=busybox:1.36 --rm -it -- nslookup kubernetes.default`
  3. CoreDNS Pods crashing? → check logs for `loop`, config errors
  4. kube-dns Service and Endpoints healthy? → `kubectl get svc,endpoints kube-dns -n kube-system`
  5. ConfigMap correct? → `kubectl get cm coredns -n kube-system -o yaml`
  6. Pod-specific? → check `dnsPolicy`, `/etc/resolv.conf` in the failing Pod

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get pods -n kube-system -l k8s-app=kube-dns` → Not Running? **Root cause is CoreDNS Pods**
  - `kubectl run dns-test --image=busybox:1.36 --rm -it -- nslookup kubernetes.default` → Times out? **DNS is broken**
  - `kubectl logs -n kube-system -l k8s-app=kube-dns --tail=10` → `loop detected`? **Forwarding loop**
  - `kubectl get endpoints kube-dns -n kube-system` → Empty? **Label mismatch**

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix Corefile syntax / config errors
  kubectl edit configmap coredns -n kube-system
  # → Fix the Corefile content, save → CoreDNS auto-reloads (reload plugin)

  # Force restart CoreDNS after config fix
  kubectl rollout restart deployment coredns -n kube-system

  # Fix loop detection (Node resolv.conf pointing to cluster DNS)
  # Option 1: Fix Node's /etc/resolv.conf to point to real upstream DNS
  # Option 2: Change forward target in Corefile to a specific upstream IP
  kubectl edit configmap coredns -n kube-system
  # → Change: forward . /etc/resolv.conf
  # → To:    forward . 8.8.8.8 8.8.4.4

  # Restore deleted kube-dns Service (emergency)
  kubectl expose deployment coredns -n kube-system \
    --name=kube-dns --port=53 --target-port=53 --protocol=UDP --type=ClusterIP
  # (Note: You ALSO need TCP port 53 — this is a quick emergency fix)

  # Fix missing ConfigMap — recreate minimal Corefile
  kubectl create configmap coredns -n kube-system --from-literal=Corefile='
  .:53 {
      errors
      health
      ready
      kubernetes cluster.local in-addr.arpa ip6.arpa {
          pods insecure
          fallthrough in-addr.arpa ip6.arpa
          ttl 30
      }
      forward . 8.8.8.8
      cache 30
      loop
      reload
      loadbalance
  }'

  # Fix label mismatch
  kubectl label pods -n kube-system -l <wrong-label> k8s-app=kube-dns --overwrite

  # Fix Pod dnsPolicy
  kubectl edit pod <pod-name>
  # → Change dnsPolicy from Default to ClusterFirst (requires Pod recreation)
  ```

- **Validate Fix**
  ```bash
  # Verify CoreDNS Pods are running
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  # → 2/2 Running

  # Verify DNS resolution works
  kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
    -- nslookup kubernetes.default
  # → Should return the API Server ClusterIP

  # Verify external DNS works
  kubectl run dns-test2 --image=busybox:1.36 --rm -it --restart=Never \
    -- nslookup google.com
  # → Should return IPs

  # Verify kube-dns Endpoints exist
  kubectl get endpoints kube-dns -n kube-system
  # → Should show CoreDNS Pod IPs
  ```

- **Confirm Recovery**
  - CoreDNS Pods: 2/2 Running, no restarts ✅
  - `nslookup kubernetes.default` succeeds from test Pod ✅
  - `nslookup <any-service>.<namespace>.svc.cluster.local` succeeds ✅
  - External DNS resolution works (google.com) ✅
  - No error/fatal messages in CoreDNS logs ✅
  - `kube-dns` Endpoints populated with Pod IPs ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

CoreDNS is pre-installed in any kubeadm cluster. Instead, validate the existing setup:

```bash
# Verify CoreDNS components exist
kubectl get deployment coredns -n kube-system
kubectl get svc kube-dns -n kube-system
kubectl get configmap coredns -n kube-system
kubectl get endpoints kube-dns -n kube-system

# Test DNS — create a test Service and resolve it
kubectl create deployment web --image=nginx
kubectl expose deployment web --port=80
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup web.default.svc.cluster.local
# → Should resolve to the ClusterIP of the web Service

# Test external DNS
kubectl run dns-test2 --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup google.com
# → Should resolve

# View current Corefile
kubectl get configmap coredns -n kube-system -o yaml
```

- **Validation Steps**
  - Deployment shows 2/2 READY
  - `kube-dns` Service has ClusterIP (usually 10.96.0.10)
  - Endpoints show 2 Pod IPs
  - `nslookup` from test Pod succeeds for both internal and external names

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Delete ConfigMap | `kubectl delete cm coredns -n kube-system` | New CoreDNS Pods can't start; existing ones keep running until restart |
| Scale to 0 | `kubectl scale deploy coredns -n kube-system --replicas=0` | ALL DNS resolution fails cluster-wide |
| Break Corefile syntax | Edit ConfigMap → add random text in Corefile | CoreDNS crashes → `CrashLoopBackOff` (after reload) |
| Remove `kubernetes` plugin | Edit ConfigMap → delete `kubernetes cluster.local` block | Internal Services return NXDOMAIN; external still works |
| Remove `forward` plugin | Edit ConfigMap → delete `forward . /etc/resolv.conf` line | Internal DNS works; external (google.com) returns NXDOMAIN |
| Change domain | Edit → `kubernetes wrong.domain` instead of `cluster.local` | All `svc.cluster.local` queries return NXDOMAIN |
| Create DNS loop | Edit → `forward . <kube-dns-ClusterIP>` (point to itself) | `loop` plugin detects → CoreDNS crashes |
| Break labels | `kubectl label pod -n kube-system <coredns-pod> k8s-app-` (remove label) | kube-dns Service loses that Endpoint → partial DNS failure |
| Wrong dnsPolicy on Pod | Create Pod with `dnsPolicy: Default` | That Pod can't resolve cluster Service names |
| Add restrictive NetworkPolicy | Create NetworkPolicy in kube-system blocking port 53 | DNS queries blocked → timeout |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup kubernetes.default
  ```

- **How to Narrow Issue**
  1. CoreDNS Pods running? → No → check events, logs, ConfigMap
  2. Pods running but DNS fails? → check kube-dns Service & Endpoints
  3. Internal fails, external works? → Corefile `kubernetes` plugin issue
  4. External fails, internal works? → Corefile `forward` plugin issue
  5. Only specific Pod fails? → check that Pod's `dnsPolicy` and `/etc/resolv.conf`

- **How to Confirm Root Cause**
  - Check CoreDNS logs: `kubectl logs -n kube-system -l k8s-app=kube-dns --tail=30`
  - Check ConfigMap: `kubectl get cm coredns -n kube-system -o yaml`
  - Check Endpoints: `kubectl get endpoints kube-dns -n kube-system`
  - Match the error to the break: loop → loop message; NXDOMAIN → missing plugin; timeout → Pods down

  - Trace: Pod resolv.conf → kube-dns Service → CoreDNS Pods → API Server
  - Create custom DNS entries using ConfigMap `hosts` plugin

- **What to Ignore (Deep Internals)**
  - CoreDNS plugin development or Go source code
  - Advanced plugin chains (etcd backend, route53, etc.)
  - CoreDNS performance tuning at scale
  - DNSSEC configuration
  - CoreDNS compilation or custom builds

- **5 Quiz Questions**
  1. A Pod can reach a Service by IP but NOT by name. What's the most likely issue?
     → *CoreDNS is down or Pod's dnsPolicy is set to `Default` instead of `ClusterFirst`*
  2. CoreDNS Pods are in CrashLoopBackOff. Logs show "Loop detected." What's the cause and fix?
     → *Node's `/etc/resolv.conf` points to the kube-dns ClusterIP, creating a forwarding loop. Fix: change `forward . /etc/resolv.conf` to `forward . 8.8.8.8` in the coredns ConfigMap*
  3. What is the DNS name format for a Service called `db` in namespace `production`?
     → *`db.production.svc.cluster.local`*
  4. Where is the CoreDNS configuration stored and how do you edit it?
     → *ConfigMap `coredns` in `kube-system` namespace. Edit with `kubectl edit cm coredns -n kube-system`*
  5. A Headless Service resolves differently from a ClusterIP Service. How?
     → *Headless Service (clusterIP: None) returns individual Pod IPs as A records, not a single ClusterIP*

## 1️⃣3️⃣ Component Interaction Map

- **CoreDNS ↔ Pods**
  - Every Pod's `/etc/resolv.conf` points `nameserver` to the `kube-dns` Service ClusterIP
  - Pods send all DNS queries to CoreDNS (when `dnsPolicy: ClusterFirst`)
  - **When things go wrong:** CoreDNS down → Pod DNS queries timeout → app connection failures; wrong `dnsPolicy` → Pod bypasses CoreDNS entirely

- **CoreDNS ↔ Services**
  - CoreDNS creates DNS A records for every Service: `<svc>.<ns>.svc.cluster.local → ClusterIP`
  - CoreDNS creates SRV records for named ports
  - Headless Services → CoreDNS returns Pod IPs directly
  - CoreDNS itself is accessed via the `kube-dns` Service
  - **When things go wrong:** CoreDNS can't watch API Server → DNS records become stale; new Services don't get DNS entries

- **CoreDNS ↔ Nodes**
  - CoreDNS Pods run on Nodes (scheduled by scheduler)
  - CoreDNS uses Node's `/etc/resolv.conf` for upstream DNS forwarding
  - kubelet on each Node configures Pod DNS via `--cluster-dns` flag
  - **When things go wrong:** Node `/etc/resolv.conf` points to kube-dns IP → forwarding loop; kubelet wrong `--cluster-dns` → Pods get wrong nameserver

- **CoreDNS ↔ Scheduler**
  - Scheduler places CoreDNS Pods on available Nodes
  - CoreDNS has tolerations for control-plane nodes
  - **When things go wrong:** No schedulable Node → CoreDNS Pods Pending → no DNS in cluster

- **CoreDNS ↔ Controller Manager**
  - Deployment controller (in kube-controller-manager) manages CoreDNS replicas
  - Ensures 2 replicas are always running
  - **When things go wrong:** Controller manager down → can't scale or recreate CoreDNS Pods if they fail

- **CoreDNS ↔ etcd**
  - CoreDNS does NOT read from etcd directly
  - It watches Service/Endpoint resources via API Server (which reads from etcd)
  - **When things go wrong:** etcd down → API Server down → CoreDNS can't sync new Service records (but existing cached records still work briefly)

- **CoreDNS ↔ Network Layer**
  - CoreDNS needs CNI to be functional for network connectivity
  - DNS queries use UDP/TCP port 53
  - kube-proxy manages the kube-dns Service routing (ClusterIP → Pod IPs)
  - **When things go wrong:** CNI broken → CoreDNS Pods can't communicate → DNS fails; NetworkPolicy blocking port 53 → DNS queries silently dropped; kube-proxy broken → kube-dns Service unreachable

- **CoreDNS ↔ Storage Layer**
  - No persistent storage required
  - Configuration stored in ConfigMap (not PV/PVC)
  - CoreDNS is stateless — all data fetched from API Server at runtime
  - **When things go wrong:** ConfigMap deleted → CoreDNS can't load config on restart

## Quick Reference Cheat Sheet

```bash
# Check CoreDNS health
kubectl get pods -n kube-system -l k8s-app=kube-dns
kubectl get svc kube-dns -n kube-system
kubectl get endpoints kube-dns -n kube-system

# View Corefile
kubectl get cm coredns -n kube-system -o yaml

# Edit Corefile
kubectl edit cm coredns -n kube-system

# Restart CoreDNS (after config change, if reload plugin doesn't pick it up)
kubectl rollout restart deployment coredns -n kube-system

# Check CoreDNS logs
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=30

# Test internal DNS
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup kubernetes.default

# Test cross-namespace DNS
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup <svc>.<namespace>.svc.cluster.local

# Test external DNS
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup google.com

# Check Pod's DNS configuration
kubectl exec <pod> -- cat /etc/resolv.conf

# Check Pod's dnsPolicy
kubectl get pod <pod> -o jsonpath='{.spec.dnsPolicy}'

# DNS record format
# Service:  <svc-name>.<namespace>.svc.cluster.local
# Pod:      <pod-ip-dashed>.<namespace>.pod.cluster.local
# SRV:      _<port-name>._<protocol>.<svc>.<ns>.svc.cluster.local
```

If you see "loop detected," change `forward . /etc/resolv.conf` to `forward . 8.8.8.8` in the `coredns` ConfigMap. Know the Corefile structure — you WILL need to edit it.
