# 🧠 Kubernetes Mind Map — Service (ClusterIP)

---

## 1️⃣ What It Does

- **Core Purpose**
  - Provides a **stable virtual IP** (ClusterIP) to reach a set of Pods inside the cluster
  - Acts as an **internal load balancer** — distributes traffic across matching Pods
  - Decouples consumers from individual Pod IPs (Pods are ephemeral; Service IP is permanent)
  - Creates a **DNS entry** (`<svc-name>.<namespace>.svc.cluster.local`) for service discovery

- **Real-World Use Case**
  - Frontend Pods → ClusterIP Service → Backend Pods (e.g., a web app calling a REST API)
  - Application Pods → ClusterIP Service → Database Pods (e.g., connecting to MySQL/PostgreSQL)
  - Microservice-to-microservice communication within the cluster
  - Any internal traffic that should **never be exposed** outside the cluster

- **Where It Fits in Cluster Architecture**
  - Lives in the **Services & Networking** domain of CKA
  - `kube-proxy` on every Node programs **iptables / IPVS rules** to route ClusterIP traffic to Pod endpoints
  - CoreDNS resolves the Service name to the ClusterIP
  - Endpoints Controller watches Pod labels and populates the **Endpoints** object automatically
  - Sits between consumer Pods and provider Pods — the "glue" of internal networking

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **Pods** — consumers use the Service name/IP to reach backend Pods
  - **Deployments / ReplicaSets** — expose their Pods through a Service
  - **Ingress** — routes external traffic to a ClusterIP Service (backend)
  - **NetworkPolicies** — may reference Service selectors indirectly via Pod labels
  - **HorizontalPodAutoscaler** — scaling Pods behind a Service

- **What Resources It Interacts With**
  - **Endpoints / EndpointSlices** — auto-created; list of Pod IPs matching the selector
  - **Pods** — selector ↔ labels matching is the core mechanism
  - **CoreDNS** — registers `<svc>.<ns>.svc.cluster.local` A record
  - **kube-proxy** — programs iptables/IPVS rules on every Node for the ClusterIP

- **Upstream and Downstream Dependencies**
  - Upstream: Pods must exist with matching labels → Endpoints get populated
  - Upstream: kube-proxy must be running on Nodes → traffic routing works
  - Upstream: CoreDNS must be healthy → DNS-based discovery works
  - Downstream: Ingress, other Services, or Pods consuming this Service
  - Downstream: If Service is broken → all consumers see connection refused / timeouts

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: v1                  # Always v1 for Service
kind: Service                   # Resource type
metadata:
  name: my-service              # Service name (used in DNS)
  namespace: default            # Namespace scoping
  labels:                       # Labels ON the Service itself (for organization)
    app: my-app
spec:
  type: ClusterIP               # Default type — can be omitted
  selector:                     # ⚠️ CRITICAL — matches Pod labels
    app: my-app
  ports:
    - name: http                # Port name (required if multiple ports)
      protocol: TCP             # TCP (default) | UDP | SCTP
      port: 80                  # ⚠️ Service port (what consumers connect to)
      targetPort: 8080          # ⚠️ Pod port (where traffic is forwarded)
```

- **Key Sections Explained**
  - `apiVersion: v1` — Service is a core API resource, always `v1`
  - `kind: Service` — identifies the resource type
  - `metadata.name` — becomes the DNS hostname inside the cluster
  - `spec.type: ClusterIP` — default type; can be omitted and it still creates a ClusterIP
  - `spec.selector` — **the most critical field** — selects Pods by label match
  - `spec.ports` — defines the port mapping from Service → Pod

- **Common Indentation Traps**
  - `selector` is under `spec`, NOT under `metadata`
  - `ports` is a **list** (starts with `-`), not a map
  - `targetPort` is inside each port entry, at the same level as `port`
  - Forgetting `name` field when specifying **multiple ports** → validation error
  - `selector` labels must be **flat key-value pairs** (no nesting)

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `spec.selector` | Selects which Pods receive traffic | No Endpoints → Service has **no backends** → connection refused |
| `spec.ports[].port` | The port consumers connect to | Consumers connecting to wrong port → connection refused |
| `spec.ports[].targetPort` | The port on the Pod receiving traffic | Traffic sent to wrong Pod port → connection refused or wrong app |
| `spec.type` | Determines exposure scope | Wrong type → unexpected external exposure or no access |
| `metadata.name` | DNS name for the Service | Wrong name → DNS lookup fails for consumers |
| `metadata.namespace` | Scopes the Service | Cross-namespace calls need `<svc>.<namespace>.svc.cluster.local` |
| `spec.ports[].protocol` | TCP/UDP/SCTP | Mismatch with app protocol → silent failures |
| `spec.ports[].name` | Required for multi-port Services | Missing → API validation error with multiple ports |

- **Field Relationships**
  - `spec.selector` ↔ `pod.metadata.labels` — **must match exactly**
  - `spec.ports[].port` = what consumers use; `spec.ports[].targetPort` = what Pod listens on
  - `targetPort` can be a **number** or a **named port** (referencing `containerPort` name)
  - If `targetPort` is omitted → defaults to same value as `port`
  - One Service can expose **multiple ports** (each needs a unique `name`)

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (< 5 seconds)**
  - `spec.selector` — Are labels present? Do they match Pod labels?
  - `spec.ports` — Is `port` and `targetPort` correct?
  - `spec.type` — Is it ClusterIP (or omitted, which defaults to ClusterIP)?
  - `metadata.name` and `metadata.namespace` — correct?

- **Usually Irrelevant for CKA**
  - `spec.sessionAffinity` — rarely tested
  - `spec.clusterIP` — auto-assigned, don't set manually unless required
  - `spec.ipFamilyPolicy` — dual-stack details not CKA priority
  - `status` section — auto-managed

- **Red Flags to Detect in 10 Seconds**
  - ❌ `selector` is empty or missing → **no Endpoints**
  - ❌ `selector` labels don't match any existing Pod labels → **0 Endpoints**
  - ❌ `targetPort` doesn't match any `containerPort` on the Pod → **traffic black hole**
  - ❌ `port` is wrong → consumers connect to wrong port → timeout
  - ❌ `type: NodePort` or `LoadBalancer` when ClusterIP was intended → **unintended external exposure**

## 6️⃣ What to Notice In

### `kubectl describe service <svc-name>`
- **Where to Look First**
  - `Endpoints` line — should list Pod IPs; `<none>` = **broken selector**
  - `Selector` — verify it matches Pod labels
  - `Port` / `TargetPort` — confirm mapping is correct
  - `Type` — confirm it's ClusterIP
- **Failure Signals**
  - `Endpoints: <none>` → no Pods match the selector
  - Missing or wrong `Selector` → wrong Pods or no Pods selected
  - Port mismatch → traffic routing will fail silently

### `kubectl get endpoints <svc-name>`
- **Where to Look First**
  - IP addresses listed — if empty, selector mismatch or no ready Pods
  - Port number — must match Pod `containerPort`
- **Failure Signals**
  - Empty Endpoints → no matching **Ready** Pods (check labels + readiness probes)
  - Fewer Endpoints than expected → some Pods not Ready

### `kubectl logs <pod>`
- **Where to Look First**
  - Application startup logs — is the app actually listening on the `targetPort`?
  - Errors like "address already in use" or "bind failed"
- **Failure Signals**
  - App not listening → `targetPort` is correct but app misconfigured
  - Connection refused in logs from consumers → Service not routing properly

### `kubectl exec -it <pod> -- curl <svc-name>:<port>`
- **Where to Look First**
  - Response from the target application — proves end-to-end connectivity
  - DNS resolution — `nslookup <svc-name>` to test CoreDNS
- **Failure Signals**
  - `curl: (7) Failed to connect` → Service/Endpoints broken or targetPort wrong
  - `Could not resolve host` → CoreDNS issue or wrong Service name
  - Timeout → kube-proxy issue or NetworkPolicy blocking

### `kubectl explain service.spec`
- **Where to Look First**
  - Quickly check field names, types, and whether they're required
  - Useful during CKA when unsure of exact field name
  - `kubectl explain service.spec.ports` — shows port fields
  - `kubectl explain service.spec.selector` — confirms it's a map

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - Selector labels don't match Pod labels → `Endpoints: <none>`
  - `targetPort` doesn't match Pod `containerPort` → connection refused
  - `port` wrong → consumers connecting to wrong port
  - Service in wrong namespace → DNS lookup fails
  - Pod not Ready (failing readiness probe) → removed from Endpoints

- **Status Values to Recognize**
  - Service itself doesn't have status like Pods (no CrashLoopBackOff, Pending, etc.)
  - Instead, check:
    - `Endpoints: <none>` → **the main red flag**
    - Pod statuses: CrashLoopBackOff, Pending → Pods aren't Ready → removed from Endpoints
    - `curl` returns connection refused → targetPort mismatch
    - `curl` returns timeout → kube-proxy / NetworkPolicy issue

- **Event Messages to Recognize**
  - Services rarely generate events themselves
  - Check **Pod events** instead: `kubectl describe pod <pod>` → Events section
  - Check **Endpoints events**: `kubectl get events --field-selector involvedObject.kind=Endpoints`

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: "Service created but application unreachable" → selector mismatch
  - Scenario 2: "Service Endpoints show `<none>`" → no Pods with matching labels or Pods not Ready
  - Scenario 3: "Can reach Service by IP but not by name" → CoreDNS issue
  - Scenario 4: "curl returns connection refused" → targetPort doesn't match containerPort
  - Scenario 5: "Intermittent failures" → some Pods unhealthy, removed from Endpoints
  - Scenario 6: "Cross-namespace Service unreachable" → using short name instead of FQDN

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch?**
  - Service selector doesn't match any Pod labels
  - Endpoints object is created but has **zero addresses**
  - `kubectl describe svc` shows `Endpoints: <none>`
  - All traffic to the Service → **connection refused**
  - The Service itself looks healthy — `kubectl get svc` shows a ClusterIP
  - **Silent failure** — no error/event on the Service

- **What Happens If Ports Mismatch?**
  - `targetPort` doesn't match Pod's `containerPort`
  - Endpoints exist (IPs are populated — label matching is fine)
  - Traffic is forwarded to Pod on wrong port → **connection refused**
  - `port` wrong → consumers connect to wrong Service port → timeout or refused
  - If `targetPort` is a **named port** and the name doesn't exist on the Pod → traffic fails

- **What Happens If Image Is Wrong?**
  - Pod goes to `ImagePullBackOff` or `ErrImagePull`
  - Pod is **not Ready** → removed from Endpoints
  - Service has zero Endpoints → connection refused
  - Fix the Deployment image → Pod becomes Ready → auto-added to Endpoints

- **What Happens If Resource Limits Are Wrong?**
  - Pod may be `OOMKilled` → CrashLoopBackOff → not Ready → removed from Endpoints
  - CPU throttling → app slow to respond → potential timeouts through the Service
  - If all Pods crash → Service has zero Endpoints

- **What Happens If Dependencies Are Missing?**
  - If Pods depend on a ConfigMap/Secret that doesn't exist → Pod stuck in `ContainerCreating`
  - Pod not Ready → not in Endpoints → Service has no backends
  - If consumer depends on a Service that doesn't exist → DNS lookup fails → `Could not resolve host`

## 9️⃣ Fast Detection Strategy

- **Exact kubectl Commands to Run (in order)**
  ```bash
  # Step 1: Check Service exists and has a ClusterIP
  kubectl get svc <svc-name> -n <namespace>

  # Step 2: Check Endpoints — THE MOST IMPORTANT CHECK
  kubectl get endpoints <svc-name> -n <namespace>

  # Step 3: Describe Service — look at Selector and Endpoints
  kubectl describe svc <svc-name> -n <namespace>

  # Step 4: Check Pod labels match the selector
  kubectl get pods -n <namespace> --show-labels
  kubectl get pods -n <namespace> -l <selector-key>=<selector-value>

  # Step 5: Check Pod readiness
  kubectl get pods -n <namespace> -o wide

  # Step 6: Test DNS resolution
  kubectl exec -it <any-pod> -- nslookup <svc-name>

  # Step 7: Test connectivity end-to-end
  kubectl exec -it <any-pod> -- curl -v <svc-name>:<port>

  # Step 8: Check kube-proxy and CoreDNS if all above looks correct
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  kubectl get pods -n kube-system -l k8s-app=kube-proxy
  ```

- **Order of Debugging Steps**
  1. Service exists? → `kubectl get svc`
  2. Endpoints populated? → `kubectl get endpoints` ← **80% of issues found here**
  3. Selector matches? → compare `describe svc` selector vs `get pods --show-labels`
  4. Pods Ready? → `kubectl get pods`
  5. DNS works? → `nslookup` from inside a Pod
  6. Port correct? → `curl` from inside a Pod
  7. Network layer? → check kube-proxy, CoreDNS, NetworkPolicies

- **How to Detect Issue Under 60 Seconds**
  - Run `kubectl describe svc <name>` → look at `Endpoints` line
  - If `<none>` → selector mismatch or Pods not Ready → compare labels
  - If Endpoints exist → `kubectl exec -- curl <svc>:<port>` → if refused → targetPort wrong
  - If timeout → NetworkPolicy or kube-proxy issue

## 🔟 Fast Fix Strategy

- **How to Fix Using kubectl**
  ```bash
  # Fix selector mismatch — edit Service selector
  kubectl edit svc <svc-name> -n <namespace>
  # Change spec.selector to match Pod labels

  # Or fix Pod labels to match Service selector
  kubectl label pod <pod-name> app=my-app --overwrite

  # Fix port mismatch
  kubectl edit svc <svc-name>
  # Update targetPort to match Pod containerPort

  # Fix via patch (faster for CKA)
  kubectl patch svc <svc-name> -p '{"spec":{"selector":{"app":"correct-label"}}}'
  kubectl patch svc <svc-name> -p '{"spec":{"ports":[{"port":80,"targetPort":8080}]}}'

  # Fix by re-applying corrected YAML
  kubectl apply -f corrected-service.yaml

  # If Pod image is wrong → fix Deployment
  kubectl set image deployment/<deploy-name> <container>=<correct-image>
  ```

- **How to Validate Fix**
  ```bash
  # Check Endpoints are now populated
  kubectl get endpoints <svc-name>

  # Verify Pods are Ready
  kubectl get pods -l <selector>

  # Test end-to-end
  kubectl exec -it <pod> -- curl <svc-name>:<port>
  ```

- **How to Confirm Recovery**
  - `kubectl get endpoints <svc-name>` → shows Pod IPs
  - `kubectl describe svc <svc-name>` → Endpoints line is populated
  - `curl` from inside a Pod returns expected response
  - Consumer application logs show successful connections

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 – Create Working Example

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 80
```

- **Validation Steps**
  ```bash
  kubectl apply -f deployment.yaml
  kubectl apply -f service.yaml
  kubectl get svc web-service
  kubectl get endpoints web-service          # Should show 3 Pod IPs
  kubectl describe svc web-service           # Endpoints populated
  kubectl run test --rm -it --image=busybox --restart=Never -- wget -qO- web-service:80
  ```

### Step 2 – Break It Intentionally

| # | What to Break | How | Expected Symptom |
|---|---|---|---|
| 1 | **Remove selector labels** | `kubectl edit svc web-service` → delete `selector` | Endpoints: `<none>`, connection refused |
| 2 | **Mismatch selector** | Change selector to `app: wrong` | Endpoints: `<none>` |
| 3 | **Wrong targetPort** | Change `targetPort: 9999` | Endpoints exist but `curl` → connection refused |
| 4 | **Wrong port** | Change `port: 9999` | Must curl on 9999; old port stops working |
| 5 | **Wrong Pod image** | `kubectl set image deploy/web-app nginx=nginx:nonexistent` | Pods → ImagePullBackOff → Endpoints empty |
| 6 | **Delete Pods (all)** | `kubectl scale deploy/web-app --replicas=0` | Endpoints: `<none>` |
| 7 | **Wrong namespace** | Create Service in `different-ns` | DNS lookup fails from default namespace |
| 8 | **Add failing readiness probe** | Add probe for wrong path → Pods not Ready | Endpoints empty |

### Step 3 – Debug Without Looking at Solution

- **What command to run first**
  - `kubectl get endpoints web-service` — this reveals 80% of issues instantly
- **How to narrow the issue**
  - Endpoints empty? → `kubectl describe svc` → check Selector → compare with `kubectl get pods --show-labels`
  - Endpoints exist? → `kubectl exec -- curl web-service:80` → connection refused? → check targetPort
  - DNS fails? → `kubectl exec -- nslookup web-service` → check CoreDNS
- **How to confirm root cause**
  - Selector mismatch: `kubectl get pods -l app=web` returns 0 results but Pods exist with other labels
  - Port mismatch: `kubectl describe svc` shows wrong targetPort vs `kubectl describe pod` containerPort
  - Pod not Ready: `kubectl get pods` shows Ready 0/1

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - ClusterIP Service creation (imperative and declarative)
  - Selector ↔ label matching mechanism
  - Port mapping: `port` vs `targetPort` vs `containerPort`
  - Endpoints auto-population behavior
  - DNS resolution: `<svc>.<ns>.svc.cluster.local`
  - Cross-namespace Service access
  - Headless Services (`clusterIP: None`) — returns Pod IPs directly

- **What to Focus On**
  - Creating Services quickly: `kubectl expose deployment <name> --port=80 --target-port=8080`
  - Debugging empty Endpoints
  - Verifying connectivity with `curl` / `wget` from inside Pods
  - Understanding `kubectl describe svc` output
  - Multi-port Service definitions

- **What to Practice Repeatedly**
  - Create a Deployment + ClusterIP Service from scratch in under 2 minutes
  - Debug a broken Service (selector mismatch) in under 60 seconds
  - Use `kubectl expose` as a shortcut
  - Cross-namespace DNS resolution

- **What to Ignore (Deep Internals Not Required)**
  - iptables rules / IPVS configuration details
  - kube-proxy source code or modes (iptables vs IPVS vs userspace)
  - Internal ClusterIP allocation CIDR management
  - `sessionAffinity` deep configuration
  - IPv4/IPv6 dual-stack details

- **5 Quiz Questions**
  1. **Q:** A Service has `selector: app: web` but `kubectl get endpoints` shows `<none>`. What is the most likely cause?
     - **A:** No Pods have the label `app: web`, or Pods exist but are not in Ready state.
  2. **Q:** What is the difference between `port` and `targetPort` in a Service spec?
     - **A:** `port` is what consumers connect to on the Service; `targetPort` is the port on the Pod where traffic is forwarded.
  3. **Q:** How do you access a Service named `backend` in namespace `prod` from a Pod in namespace `dev`?
     - **A:** `curl backend.prod.svc.cluster.local:<port>` or `curl backend.prod:<port>`
  4. **Q:** You create a Service but omit `spec.type`. What type is it?
     - **A:** `ClusterIP` — it is the default Service type.
  5. **Q:** What happens if you set `targetPort: 9090` but the Pod container listens on port `8080`?
     - **A:** Traffic is forwarded to port 9090 on the Pod, which has nothing listening → connection refused.

## 1️⃣3️⃣ Component Interaction Map

```
                  ┌─────────────┐
                  │   CoreDNS   │
                  │ (DNS A rec) │
                  └──────┬──────┘
                         │ resolves svc name → ClusterIP
                         ▼
┌──────────┐   ClusterIP:Port   ┌─────────────────┐    iptables/IPVS    ┌──────────┐
│ Consumer │ ────────────────► │    Service        │ ──────────────────► │  Pod(s)  │
│   Pod    │                   │  (ClusterIP)      │    (kube-proxy)     │ :targetP │
└──────────┘                   └────────┬──────────┘                     └──────────┘
                                        │
                                        ▼
                               ┌──────────────────┐
                               │    Endpoints /    │
                               │  EndpointSlices   │
                               │  (auto-managed)   │
                               └──────────────────┘
```

- **Pods**
  - Service selects Pods via labels
  - Only **Ready** Pods appear in Endpoints
  - Pod deletion → auto-removed from Endpoints
  - Pod IP change (restart) → Endpoints updated automatically

- **Services ↔ Other Services**
  - Services don't directly interact with each other
  - Chained via Pods: Service A → Pod A → Service B → Pod B

- **Nodes**
  - kube-proxy on every Node programs routing rules for ClusterIP
  - ClusterIP is a **virtual IP** — not bound to any specific Node
  - Accessible from any Node or Pod in the cluster

- **Scheduler**
  - No direct interaction — Scheduler places Pods; Service discovers them via labels
  - If Scheduler can't place a Pod → Pod stays Pending → not in Endpoints

- **Controller Manager**
  - **Endpoints Controller** watches Pod labels and readiness → populates Endpoints
  - **EndpointSlice Controller** (modern) → manages EndpointSlice objects
  - If Controller Manager is down → Endpoints are **not updated** (stale)

- **etcd**
  - Stores Service, Endpoints, and EndpointSlice objects
  - If etcd is down → no new Services can be created; existing ones continue to work (kube-proxy cached rules)

- **Network Layer**
  - kube-proxy → programs iptables/IPVS rules
  - CNI plugin → provides Pod-to-Pod networking (required for traffic to actually reach Pods)
  - NetworkPolicies → can block traffic to/from Pods behind a Service

- **Storage Layer**
  - No direct interaction — Services are purely networking constructs
  - However, if a Pod behind a Service needs a PVC and it's not bound → Pod stays Pending → not in Endpoints

- **Interaction Behavior When Things Go Wrong**
  - kube-proxy down → existing rules may work; new Services won't be routed
  - CoreDNS down → DNS resolution fails; direct ClusterIP access still works
  - Endpoints Controller down → Endpoints become stale; new Pods won't be added
  - CNI plugin issues → Pod-to-Pod networking fails → Service traffic fails even with correct Endpoints

```bash
kubectl exec -it <pod> -- curl -s <svc-name>:<port>

# Quick selector check
kubectl get pods -l <key>=<value>

# Patch selector
kubectl patch svc <name> -p '{"spec":{"selector":{"app":"correct"}}}'

# Patch port
kubectl patch svc <name> --type='json' -p='[{"op":"replace","path":"/spec/ports/0/targetPort","value":8080}]'
```
