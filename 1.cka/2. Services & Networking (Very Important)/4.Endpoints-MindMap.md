# рҹ§  Kubernetes Mind Map вҖ” Endpoints

---

## 1пёҸвғЈ What It Does

- **Core Purpose**
  - An Endpoints object is a **list of IP:port pairs** that a Service routes traffic to
  - Automatically created and managed by the **Endpoints Controller** when a Service has a `selector`
  - Acts as the **bridge between a Service and its backing Pods** вҖ” translates selector-based matching into actual routable addresses
  - kube-proxy reads Endpoints to program **iptables/IPVS rules** on every Node
  - Without Endpoints, a Service is just a DNS name with nothing behind it вҶ’ connection refused

- **Real-World Use Case**
  - **Auto-managed (most common):** Service with `selector` вҶ’ Endpoints auto-populated with matching Ready Pod IPs
  - **Manually managed:** Service **without** `selector` вҶ’ you create Endpoints yourself to point to **external resources** (e.g., an external database, legacy server, or a service outside the cluster)
  - Debugging: checking Endpoints is the **#1 diagnostic step** for broken Services
  - CKA exam: questions about "why is my Service not working" almost always trace back to empty Endpoints

- **Where It Fits in Cluster Architecture**
  - Lives in the **Services & Networking** domain of CKA
  - Created in the **same namespace** and with the **same name** as the parent Service
  - Traffic flow: Client вҶ’ Service (DNS вҶ’ ClusterIP) вҶ’ kube-proxy reads Endpoints вҶ’ forwards to Pod IP:Port
  - **Endpoints Controller** (in Controller Manager) watches Pods and Services вҶ’ keeps Endpoints in sync
  - Modern replacement: **EndpointSlices** (more scalable), but Endpoints are still used and CKA-relevant

---

## 2пёҸвғЈ What Depends On It

- **Which Components Rely On It**
  - **kube-proxy** вҖ” reads Endpoints to program iptables/IPVS forwarding rules on every Node
  - **CoreDNS** вҖ” for Headless Services, reads Endpoints to return individual Pod A records
  - **Services** вҖ” every Service depends on Endpoints to have backends to route to
  - **Ingress controllers** вҖ” read Endpoints (or EndpointSlices) to configure backend routing
  - **NetworkPolicies** вҖ” indirectly; traffic allowed to Pods is the same set as in Endpoints

- **What Resources It Interacts With**
  - **Service** вҖ” Endpoints must have the **same name and namespace** as the Service
  - **Pods** вҖ” Pod IPs populate the Endpoints `addresses` list; only **Ready** Pods are included
  - **Endpoints Controller** вҖ” automatically creates/updates/deletes Endpoints when Pods or Services change
  - **kube-proxy** вҖ” consumes Endpoints to build forwarding rules
  - **EndpointSlices** вҖ” modern counterpart; Endpoints and EndpointSlices coexist

- **Upstream and Downstream Dependencies**
  - Upstream: **Service** must exist вҶ’ Endpoints object created with the same name
  - Upstream: **Pods** must exist, be Ready, and match the Service selector вҶ’ IPs added to Endpoints
  - Upstream: **Endpoints Controller** must be running (part of Controller Manager) вҶ’ Endpoints stay in sync
  - Downstream: **kube-proxy** вҶ’ if Endpoints are empty, no iptables rules вҶ’ connection refused
  - Downstream: **CoreDNS** вҶ’ for Headless Services, DNS depends directly on Endpoints
  - Downstream: **Ingress** вҶ’ backends are resolved via Endpoints

## 3пёҸвғЈ Recognize Structure (YAML Skeleton)

### Auto-managed Endpoints (created automatically вҖ” you rarely write this YAML)

```yaml
apiVersion: v1                    # Always v1 for Endpoints
kind: Endpoints                   # Resource type (plural!)
metadata:
  name: my-service                # вҡ пёҸ MUST match the Service name
  namespace: default              # вҡ пёҸ MUST match the Service namespace
subsets:                          # List of endpoint subsets
  - addresses:                    # вҡ пёҸ Ready Pod IPs
      - ip: 10.244.1.5
        targetRef:
          kind: Pod
          name: my-pod-abc
          namespace: default
      - ip: 10.244.2.8
        targetRef:
          kind: Pod
          name: my-pod-def
          namespace: default
    notReadyAddresses: []         # Pods that exist but are NOT Ready
    ports:
      - name: http                # Must match Service port name
        port: 8080                # вҡ пёҸ The actual port on the Pod
        protocol: TCP
```

### Manually created Endpoints (for external services вҖ” CKA-relevant)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db              # Service without selector
spec:
  ports:
    - port: 3306
      targetPort: 3306
  # вҡ пёҸ NO selector вҖ” Endpoints must be created manually
---
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db              # вҡ пёҸ MUST match Service name exactly
  namespace: default
subsets:
  - addresses:
      - ip: 192.168.1.100        # External IP address
    ports:
      - port: 3306
        protocol: TCP
```

- **Key Sections Explained**
  - `apiVersion: v1` вҖ” Endpoints is a core API resource, always `v1`
  - `kind: Endpoints` вҖ” **plural** (not "Endpoint") вҖ” common gotcha!
  - `metadata.name` вҖ” **must exactly match** the parent Service name
  - `metadata.namespace` вҖ” **must exactly match** the parent Service namespace
  - `subsets[].addresses` вҖ” list of IPs of **Ready** Pods (or manually specified IPs)
  - `subsets[].notReadyAddresses` вҖ” list of IPs of Pods that are **not Ready** (auto-managed)
  - `subsets[].ports` вҖ” port numbers and protocols matching the Service ports
  - `targetRef` вҖ” reference to the Pod (auto-populated; not needed for manual Endpoints)

- **Common Indentation Traps**
  - `subsets` is a **list** (starts with `-`), and each subset has `addresses` and `ports`
  - `addresses` is a **list** of objects, each with an `ip` field
  - `ports` inside `subsets` is the **Pod port** (not the Service port)
  - For manual Endpoints: name must match Service name **exactly** (case-sensitive)
  - `kind: Endpoints` is **plural** вҖ” writing `kind: Endpoint` вҶ’ API error

## 4пёҸвғЈ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `metadata.name` | Must match the Service name exactly | Endpoints won't associate with the Service вҶ’ Service has no backends |
| `metadata.namespace` | Must match the Service namespace | Same as above вҖ” namespace mismatch = no association |
| `subsets[].addresses[].ip` | The Pod (or external) IP that receives traffic | Wrong IP вҶ’ traffic sent to wrong destination |
| `subsets[].ports[].port` | The port on the Pod/target | Mismatch вҶ’ connection refused at the target |
| `subsets[].ports[].protocol` | TCP/UDP/SCTP | Mismatch вҶ’ silent traffic failure |
| `subsets[].ports[].name` | Must match Service port name | Mismatch вҶ’ kube-proxy can't map Service port to Endpoint port |
| `subsets[].notReadyAddresses` | Pods that exist but are not Ready | Non-empty = Pods exist but failing readiness probes |

- **Field Relationships**
  - `endpoints.metadata.name` вҶ” `service.metadata.name` вҖ” **must be identical**
  - `endpoints.subsets[].ports[].name` вҶ” `service.spec.ports[].name` вҖ” **must match** for multi-port Services
  - `endpoints.subsets[].ports[].port` = the actual Pod `containerPort` (= Service's `targetPort`)
  - `endpoints.subsets[].addresses[].ip` = Pod IP (from `kubectl get pods -o wide`)
  - Service with selector вҶ’ Endpoints **auto-managed** (don't edit manually вҖ” changes will be overwritten)
  - Service without selector вҶ’ Endpoints **must be manually created and maintained**

## 5пёҸвғЈ How Much Should You Notice in YAML?

- **Scan Immediately (< 5 seconds)**
  - `metadata.name` вҖ” does it match the Service name?
  - `subsets[].addresses` вҖ” are there any IPs listed? Empty = no backends
  - `subsets[].ports[].port` вҖ” does it match the Pod's `containerPort`?
  - Is the Endpoints object even present? (`kubectl get endpoints <svc-name>`)

- **Usually Irrelevant for CKA**
  - `targetRef` fields (auto-populated, informational)
  - `notReadyAddresses` details (just know it exists вҖ” indicates unhealthy Pods)
  - `resourceVersion`, `uid`, `creationTimestamp` (metadata noise)
  - EndpointSlice details (know it's the modern replacement, but Endpoints is still tested)

- **Red Flags to Detect in 10 Seconds**
  - вқҢ `subsets` is empty or missing вҶ’ **no backends** вҶ’ Service returns connection refused
  - вқҢ `addresses` is empty but `notReadyAddresses` has entries вҶ’ Pods exist but not Ready
  - вқҢ `metadata.name` doesn't match Service name вҶ’ orphaned Endpoints, Service has no backends
  - вқҢ Port in Endpoints doesn't match Pod `containerPort` вҶ’ traffic to wrong port
  - вқҢ Endpoints object doesn't exist at all вҶ’ Service has no selector, no manual Endpoints created

## 6пёҸвғЈ What to Notice In

### `kubectl describe endpoints <name>`
- **Where to Look First**
  - `Addresses` вҖ” list of Pod IPs (Ready Pods)
  - `NotReadyAddresses` вҖ” Pods that exist but are NOT Ready
  - `Ports` вҖ” port numbers and names
- **Failure Signals**
  - `Addresses: <none>` + `NotReadyAddresses` has IPs вҶ’ Pods exist but failing readiness probes
  - `Addresses: <none>` + `NotReadyAddresses: <none>` вҶ’ no Pods match the selector at all
  - Port mismatch with what you expect вҶ’ wrong `targetPort` on the Service

### `kubectl get endpoints <name>`
- **Where to Look First**
  - The `ENDPOINTS` column вҖ” should show comma-separated `IP:Port` pairs
  - `<none>` вҶ’ **broken** вҖ” no backends
  - Count of IPs вҖ” should match number of Ready Pods
- **Failure Signals**
  - `<none>` вҶ’ selector mismatch, no Pods, or Pods not Ready
  - Fewer IPs than expected вҶ’ some Pods not Ready
  - Wrong port number вҶ’ `targetPort` mismatch

### `kubectl logs <pod>`
- **Where to Look First**
  - Not directly related to Endpoints, but relevant for debugging **why** a Pod isn't Ready
  - Check application startup вҖ” is it listening on the expected port?
  - Check readiness probe logs вҖ” is the probe path/port correct?
- **Failure Signals**
  - App crash/not starting вҶ’ Pod not Ready вҶ’ removed from Endpoints
  - "bind failed" / "address in use" вҶ’ app can't listen on expected port

### `kubectl exec -it <pod> -- curl <svc-name>:<port>`
- **Where to Look First**
  - Does the Service respond? If yes вҶ’ Endpoints are working
  - If connection refused вҶ’ Endpoints empty or `targetPort` wrong
- **Failure Signals**
  - Connection refused вҶ’ check `kubectl get endpoints` first
  - Timeout вҶ’ kube-proxy issue or NetworkPolicy

### `kubectl explain endpoints`
- **Where to Look First**
  - `kubectl explain endpoints.subsets` вҖ” structure of addresses and ports
  - `kubectl explain endpoints.subsets.addresses` вҖ” shows `ip`, `targetRef`, etc.
  - Useful during CKA when writing manual Endpoints YAML

## 7пёҸвғЈ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - Service selector doesn't match Pod labels вҶ’ Endpoints auto-populated with **zero** addresses
  - Manual Endpoints name doesn't match Service name вҶ’ Service has no backends
  - Manual Endpoints port doesn't match Pod's actual listening port вҶ’ connection refused
  - Pod failing readiness probe вҶ’ moved to `notReadyAddresses` вҶ’ removed from active `addresses`
  - All Pods in CrashLoopBackOff вҶ’ Endpoints completely empty
  - Editing auto-managed Endpoints manually вҶ’ **overwritten** by Endpoints Controller within seconds

- **Status Values to Recognize**
  - Endpoints don't have their own "status" like Pods вҖ” they're either populated or empty
  - Key indicator: `kubectl get endpoints <name>` вҶ’ `ENDPOINTS` column
    - `<none>` вҶ’ **empty** вҖ” the most critical red flag
    - `10.244.1.5:8080,10.244.2.8:8080` вҶ’ healthy, two backends
  - Related Pod statuses that cause empty Endpoints:
    - `Pending` вҶ’ Pod not scheduled вҶ’ not in Endpoints
    - `CrashLoopBackOff` вҶ’ Pod not Ready вҶ’ not in Endpoints
    - `ImagePullBackOff` вҶ’ Pod not Ready вҶ’ not in Endpoints
    - `ContainerCreating` вҶ’ Pod not Ready вҶ’ not in Endpoints

- **Event Messages to Recognize**
  - Endpoints themselves rarely generate events
  - Check **Service** events: `kubectl describe svc <name>` вҶ’ Events
  - Check **Pod** events: `kubectl describe pod <name>` вҶ’ Events (for readiness probe failures)
  - `kubectl get events --field-selector involvedObject.kind=Endpoints` вҶ’ rare but possible

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: "Service created but `Endpoints: <none>`" вҶ’ selector mismatch or no Ready Pods
  - Scenario 2: "Endpoints show fewer IPs than expected" вҶ’ some Pods failing readiness probes
  - Scenario 3: "Service for external database not working" вҶ’ manual Endpoints name mismatch or wrong IP
  - Scenario 4: "I edited Endpoints but changes keep reverting" вҶ’ Service has a selector вҶ’ auto-managed вҶ’ don't edit manually
  - Scenario 5: "Service worked yesterday, Endpoints are empty today" вҶ’ Pods were deleted/scaled to 0 or image broken
  - Scenario 6: "Service returns wrong data" вҶ’ Endpoints point to wrong Pods (selector too broad)

## 8пёҸвғЈ Misconfiguration Behavior

- **What Happens If Labels Mismatch?**
  - Service selector doesn't match any Pod labels
  - Endpoints Controller creates Endpoints object but `subsets` has **zero addresses**
  - `kubectl get endpoints <svc>` shows `<none>`
  - All traffic to the Service вҶ’ **connection refused**
  - Fix: either change Service selector or add correct labels to Pods

- **What Happens If Ports Mismatch?**
  - Service `targetPort` doesn't match Pod `containerPort`
  - Endpoints still get populated (IPs are correct вҖ” label match works)
  - But the port in Endpoints reflects `targetPort` вҶ’ kube-proxy forwards to wrong port on Pod
  - Result: **connection refused** at the Pod level
  - Fix: align `targetPort` in Service with `containerPort` in Pod spec

- **What Happens If Image Is Wrong?**
  - Pod goes to `ImagePullBackOff` / `ErrImagePull` вҶ’ not Ready
  - Pod IP moved from `addresses` to `notReadyAddresses` (or never added)
  - Endpoints become empty if all Pods are affected
  - Service returns connection refused
  - Fix: correct the image вҶ’ Pod becomes Ready вҶ’ auto-added back to Endpoints

- **What Happens If Resource Limits Are Wrong?**
  - Pod `OOMKilled` вҶ’ restart вҶ’ CrashLoopBackOff вҶ’ not Ready вҶ’ removed from Endpoints
  - Endpoints may fluctuate: Pod starts вҶ’ briefly Ready вҶ’ OOMKilled вҶ’ removed
  - Intermittent connection failures from the Service
  - Fix: increase memory limits вҶ’ Pod stabilizes вҶ’ stays in Endpoints

- **What Happens If Dependencies Are Missing?**
  - Pod needs ConfigMap/Secret that doesn't exist вҶ’ stuck in `ContainerCreating` вҶ’ not Ready вҶ’ not in Endpoints
  - Pod needs PVC that can't bind вҶ’ stuck in `Pending` вҶ’ not in Endpoints
  - Manual Endpoints pointing to an IP that's unreachable вҶ’ Endpoints populated but traffic fails
  - Service without selector + no manual Endpoints created вҶ’ Endpoints object doesn't exist at all

## 9пёҸвғЈ Fast Detection Strategy

- **Exact kubectl Commands to Run (in order)**
  ```bash
  # Step 1: Check Endpoints exist and have addresses вҖ” THE MOST IMPORTANT CHECK
  kubectl get endpoints <svc-name> -n <namespace>

  # Step 2: Describe Endpoints вҖ” see addresses vs notReadyAddresses
  kubectl describe endpoints <svc-name> -n <namespace>

  # Step 3: Check Service selector
  kubectl describe svc <svc-name> -n <namespace>
  # Look at Selector and Endpoints lines

  # Step 4: Check Pod labels match the selector
  kubectl get pods -n <namespace> --show-labels
  kubectl get pods -n <namespace> -l <selector-key>=<selector-value>

  # Step 5: Check Pod readiness
  kubectl get pods -n <namespace>
  # Look at READY column (e.g., 1/1 vs 0/1)

  # Step 6: For manual Endpoints вҖ” verify name matches Service
  kubectl get svc <name> -n <namespace>
  kubectl get endpoints <name> -n <namespace>
  # Names must be identical

  # Step 7: Verify port in Endpoints matches Pod containerPort
  kubectl get endpoints <name> -o yaml
  kubectl describe pod <pod> | grep -i "port"

  # Step 8: Test connectivity end-to-end
  kubectl exec -it <pod> -- curl <svc-name>:<port>
  ```

- **Order of Debugging Steps**
  1. Endpoints exist? вҶ’ `kubectl get endpoints <svc-name>`
  2. Endpoints populated or `<none>`? вҶҗ **80% of issues detected here**
  3. If `<none>` вҶ’ check Service selector: `kubectl describe svc`
  4. Selector matches Pods? вҶ’ `kubectl get pods -l <selector>`
  5. Pods Ready? вҶ’ `kubectl get pods` вҶ’ READY column
  6. If Pods not Ready вҶ’ `kubectl describe pod` вҶ’ Events + readiness probe
  7. Port correct? вҶ’ compare Endpoints port vs Pod containerPort
  8. For manual Endpoints вҶ’ verify name matches Service name exactly

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get endpoints <svc-name>` вҶ’ `<none>`? вҶ’ **problem found**
  - If `<none>` вҶ’ `kubectl describe svc <svc-name>` вҶ’ look at `Selector`
  - `kubectl get pods -l <selector>` вҶ’ 0 results? вҶ’ selector mismatch
  - If Pods found but `READY 0/1` вҶ’ readiness probe failing вҶ’ `kubectl describe pod`
  - If Endpoints have IPs but Service doesn't work вҶ’ port mismatch

## рҹ”ҹ Fast Fix Strategy

- **How to Fix Using kubectl**
  ```bash
  # Fix 1: Selector mismatch вҖ” change Service selector to match Pod labels
  kubectl edit svc <svc-name>
  # Update spec.selector to match Pod labels

  # Fix 2: Or fix Pod labels to match Service selector
  kubectl label pod <pod-name> app=my-app --overwrite

  # Fix 3: Port mismatch вҖ” fix Service targetPort
  kubectl edit svc <svc-name>
  # Change targetPort to match Pod containerPort

  # Fix 4: Pods not Ready вҖ” fix the root cause
  kubectl describe pod <pod> # Find why it's not Ready
  # Common fixes:
  kubectl set image deployment/<name> <container>=<correct-image>  # Fix image
  kubectl edit deployment/<name>  # Fix readiness probe path/port

  # Fix 5: Manual Endpoints name mismatch
  kubectl delete endpoints <wrong-name>
  # Recreate with correct name matching the Service
  kubectl apply -f corrected-endpoints.yaml

  # Fix 6: Manual Endpoints wrong IP
  kubectl edit endpoints <name>
  # Or patch:
  kubectl patch endpoints <name> --type='json' \
    -p='[{"op":"replace","path":"/subsets/0/addresses/0/ip","value":"192.168.1.200"}]'

  # Fix 7: Create missing manual Endpoints
  cat <<EOF | kubectl apply -f -
  apiVersion: v1
  kind: Endpoints
  metadata:
    name: <service-name>
  subsets:
    - addresses:
        - ip: <target-ip>
      ports:
        - port: <target-port>
  EOF
  ```

- **How to Validate Fix**
  ```bash
  # Check Endpoints are now populated
  kubectl get endpoints <svc-name>

  # Verify Pod IPs are in the Endpoints
  kubectl get pods -o wide  # Get Pod IPs
  kubectl get endpoints <svc-name>  # Should list same IPs

  # Test connectivity
  kubectl exec -it <pod> -- curl <svc-name>:<port>
  ```

- **How to Confirm Recovery**
  - `kubectl get endpoints <svc-name>` вҶ’ shows IP:port pairs
  - `kubectl describe svc <svc-name>` вҶ’ `Endpoints` line is populated
  - `kubectl exec -- curl <svc-name>:<port>` вҶ’ returns expected response
  - Number of Endpoint IPs matches number of Ready Pods

## 1пёҸвғЈ1пёҸвғЈ Hands-On Learning Cycle

### Step 1 вҖ“ Create Working Example

#### Example A: Auto-managed Endpoints (Service with selector)

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
```

#### Example B: Manual Endpoints (Service without selector)

```yaml
# external-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: external-api
spec:
  ports:
    - port: 443
      targetPort: 443
---
# external-endpoints.yaml
apiVersion: v1
kind: Endpoints
metadata:
  name: external-api            # вҡ пёҸ Must match Service name
subsets:
  - addresses:
      - ip: 203.0.113.50        # External IP
    ports:
      - port: 443
```

- **Validation Steps**
  ```bash
  # Auto-managed
  kubectl apply -f deployment.yaml -f service.yaml
  kubectl get endpoints web-service           # Should show 3 Pod IPs
  kubectl describe endpoints web-service      # addresses list has 3 entries

  # Manual
  kubectl apply -f external-service.yaml -f external-endpoints.yaml
  kubectl get endpoints external-api          # Should show 203.0.113.50:443
  kubectl describe svc external-api           # Endpoints line shows the IP

  # Test
  kubectl run test --rm -it --image=busybox --restart=Never -- wget -qO- web-service:80
  ```

### Step 2 вҖ“ Break It Intentionally

| # | What to Break | How | Expected Symptom |
|---|---|---|---|
| 1 | **Mismatch Service selector** | `kubectl edit svc web-service` вҶ’ change selector to `app: wrong` | Endpoints become `<none>` |
| 2 | **Remove Pod labels** | `kubectl label pod <pod> app-` (remove label) | That Pod disappears from Endpoints |
| 3 | **Wrong targetPort** | `kubectl edit svc` вҶ’ change `targetPort: 9999` | Endpoints port changes to 9999; traffic вҶ’ connection refused |
| 4 | **Wrong image** | `kubectl set image deploy/web-app nginx=nginx:nonexistent` | Pods вҶ’ ImagePullBackOff вҶ’ Endpoints deplete |
| 5 | **Scale to 0** | `kubectl scale deploy/web-app --replicas=0` | Endpoints become `<none>` |
| 6 | **Manual Endpoints name mismatch** | Create Endpoints with name `external-api-wrong` | Service `external-api` has no Endpoints вҶ’ connection refused |
| 7 | **Manual Endpoints wrong IP** | Edit Endpoints вҶ’ set IP to `1.2.3.4` (unreachable) | Endpoints populated but traffic times out |
| 8 | **Delete Endpoints for auto-managed Service** | `kubectl delete endpoints web-service` | Endpoints Controller **recreates** them within seconds! |
| 9 | **Add failing readiness probe** | Edit Deployment вҶ’ wrong readiness probe path | Pods not Ready вҶ’ moved to `notReadyAddresses` вҶ’ Endpoints empty |
| 10 | **Edit auto-managed Endpoints manually** | `kubectl edit endpoints web-service` вҶ’ change an IP | Changes **overwritten** by Endpoints Controller instantly |

### Step 3 вҖ“ Debug Without Looking at Solution

- **What command to run first**
  - `kubectl get endpoints <svc-name>` вҖ” this reveals 80% of issues instantly
- **How to narrow the issue**
  - Endpoints `<none>`? вҶ’ is it auto-managed (Service has selector) or manual?
  - Auto-managed + empty вҶ’ `kubectl get pods -l <selector>` вҶ’ do Pods match?
  - Auto-managed + Pods exist but not in Endpoints вҶ’ `kubectl get pods` вҶ’ READY column вҶ’ readiness probe issue
  - Manual + empty вҶ’ does Endpoints name match Service name?
  - Endpoints populated but Service doesn't work вҶ’ check port numbers
- **How to confirm root cause**
  - Selector mismatch: `kubectl describe svc` selector vs `kubectl get pods --show-labels`
  - Readiness issue: `kubectl describe pod` вҶ’ Events вҶ’ readiness probe failure
  - Name mismatch: `kubectl get svc` name vs `kubectl get endpoints` name
  - Port mismatch: `kubectl get endpoints -o yaml` port vs `kubectl describe pod` containerPort

  - Topology-aware Endpoints routing
  - `publishNotReadyAddresses` deep behavior

- **5 Quiz Questions**
  1. **Q:** A Service named `backend` is created with a selector. What must the auto-created Endpoints object be named?
     - **A:** `backend` вҖ” Endpoints must have the **exact same name** as the Service.
  2. **Q:** `kubectl get endpoints my-svc` shows `<none>`. What are the top 3 causes?
     - **A:** (1) Service selector doesn't match any Pod labels, (2) Pods exist but are not Ready, (3) No Pods exist at all.
  3. **Q:** You manually edit the Endpoints of a Service that has a selector. What happens?
     - **A:** Your changes are **overwritten within seconds** by the Endpoints Controller. Auto-managed Endpoints cannot be manually edited.
  4. **Q:** How do you create Endpoints for an external database at `10.0.0.50:5432`?
     - **A:** Create a Service **without selector** and a matching Endpoints object with the same name, specifying `ip: 10.0.0.50` and `port: 5432` in `subsets`.
  5. **Q:** Endpoints show 2 IPs but the Deployment has 3 replicas. What's the most likely cause?
     - **A:** One Pod is not passing its readiness probe вҖ” it exists but is not Ready, so it's in `notReadyAddresses` instead of `addresses`.

- **Pods**
  - Pod IPs populate the `addresses` list in Endpoints
  - Only **Ready** Pods are in `addresses`; not-Ready Pods are in `notReadyAddresses`
  - Pod deletion вҶ’ IP removed from Endpoints
  - Pod restart (new IP) вҶ’ Endpoints updated automatically
  - Pod label change вҶ’ may be added/removed from Endpoints based on selector match

- **Services**
  - Every Service with a selector has an auto-managed Endpoints object with the same name
  - Service without a selector вҶ’ no auto-managed Endpoints вҶ’ must create manually
  - Service reads Endpoints via name matching (same name, same namespace)
  - `kubectl describe svc` shows the `Endpoints` line from this Endpoints object

- **Nodes**
  - kube-proxy on every Node reads Endpoints вҶ’ programs iptables/IPVS rules
  - If Endpoints change вҶ’ kube-proxy updates rules (eventual consistency, usually < 1 second)
  - Endpoints are cluster-wide вҖ” every Node's kube-proxy sees the same Endpoints

- **Scheduler**
  - No direct interaction with Endpoints
  - Indirectly: if a Pod can't be scheduled вҶ’ Pod stays Pending вҶ’ not Ready вҶ’ not in Endpoints

- **Controller Manager**
  - **Endpoints Controller** is the key component вҖ” runs inside Controller Manager
  - Watches: all Pods and all Services
  - When a Pod becomes Ready / not Ready / deleted вҶ’ updates Endpoints
  - When a Service selector changes вҶ’ recalculates Endpoints
  - If Controller Manager is down вҶ’ **Endpoints become stale** (not updated)

- **etcd**
  - Stores Endpoints objects
  - If etcd is down вҶ’ Endpoints can't be updated; existing kube-proxy rules still work (cached)

- **Network Layer**
  - kube-proxy reads Endpoints to know where to forward traffic
  - If Endpoints are empty вҶ’ kube-proxy has no backends вҶ’ connection refused
  - If Endpoints are stale вҶ’ traffic may be sent to deleted/moved Pods вҶ’ errors
  - CNI plugin вҶ’ provides the Pod-to-Pod connectivity that Endpoints rely on

- **Storage Layer**
  - No direct interaction
  - Indirect: Pod needing a PVC that can't bind вҶ’ Pod Pending вҶ’ not in Endpoints

- **Interaction Behavior When Things Go Wrong**
  - Endpoints Controller down вҶ’ Endpoints become stale; Pods added/removed but Endpoints don't reflect it
  - kube-proxy down вҶ’ can't program forwarding rules; even with correct Endpoints, traffic can't be routed
  - CoreDNS down вҶ’ DNS fails, but if client uses ClusterIP directly, kube-proxy rules (based on Endpoints) still work
  - etcd down вҶ’ Endpoints can't be updated; existing rules continue working
  - Pod flapping (Ready вҶ’ not Ready вҶ’ Ready) вҶ’ Endpoints constantly updated вҶ’ kube-proxy constantly reprogramming вҶ’ possible brief connection failures

## вҡЎ Quick Reference Cheat Sheet

```bash
# Check Endpoints (THE SINGLE MOST IMPORTANT COMMAND for Service debugging)
kubectl get endpoints <svc-name>

# Describe Endpoints (see addresses vs notReadyAddresses)
kubectl describe endpoints <svc-name>

# View Endpoints YAML
kubectl get endpoints <svc-name> -o yaml

# Check Service вҶ’ Endpoints relationship
kubectl describe svc <svc-name>  # Look at "Endpoints" line

# Check Pod labels match Service selector
kubectl get pods --show-labels
kubectl get pods -l <key>=<value>

# Check Pod readiness
kubectl get pods  # READY column

# Test connectivity through Service
kubectl exec -it <pod> -- curl <svc-name>:<port>

# Create manual Endpoints for external service
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Endpoints
metadata:
  name: <must-match-service-name>
subsets:
  - addresses:
      - ip: <external-ip>
    ports:
      - port: <port>
EOF

# Patch manual Endpoints
kubectl patch endpoints <name> --type='json' \
  -p='[{"op":"replace","path":"/subsets/0/addresses/0/ip","value":"<new-ip>"}]'

# Delete and recreate manual Endpoints
kubectl delete endpoints <name>
kubectl apply -f corrected-endpoints.yaml

# Verify Endpoints match Pod IPs
kubectl get pods -o wide          # Pod IPs
kubectl get endpoints <svc-name>  # Should list same IPs

# Quick: does Service have backends?
kubectl get endpoints <svc-name> | grep -v '<none>'
```
