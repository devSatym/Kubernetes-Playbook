# 🧠 Kubernetes Mind Map — Endpoints

---

## 1️⃣ What It Does

- **Core Purpose**
  - An Endpoints object is a **list of IP:port pairs** that a Service routes traffic to
  - Automatically created and managed by the **Endpoints Controller** when a Service has a `selector`
  - Acts as the **bridge between a Service and its backing Pods** — translates selector-based matching into actual routable addresses
  - kube-proxy reads Endpoints to program **iptables/IPVS rules** on every Node
  - Without Endpoints, a Service is just a DNS name with nothing behind it → connection refused

- **Real-World Use Case**
  - **Auto-managed (most common):** Service with `selector` → Endpoints auto-populated with matching Ready Pod IPs
  - **Manually managed:** Service **without** `selector` → you create Endpoints yourself to point to **external resources** (e.g., an external database, legacy server, or a service outside the cluster)
  - Debugging: checking Endpoints is the **#1 diagnostic step** for broken Services
  - CKA exam: questions about "why is my Service not working" almost always trace back to empty Endpoints

- **Where It Fits in Cluster Architecture**
  - Lives in the **Services & Networking** domain of CKA
  - Created in the **same namespace** and with the **same name** as the parent Service
  - Traffic flow: Client → Service (DNS → ClusterIP) → kube-proxy reads Endpoints → forwards to Pod IP:Port
  - **Endpoints Controller** (in Controller Manager) watches Pods and Services → keeps Endpoints in sync
  - Modern replacement: **EndpointSlices** (more scalable), but Endpoints are still used and CKA-relevant

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **kube-proxy** — reads Endpoints to program iptables/IPVS forwarding rules on every Node
  - **CoreDNS** — for Headless Services, reads Endpoints to return individual Pod A records
  - **Services** — every Service depends on Endpoints to have backends to route to
  - **Ingress controllers** — read Endpoints (or EndpointSlices) to configure backend routing
  - **NetworkPolicies** — indirectly; traffic allowed to Pods is the same set as in Endpoints

- **What Resources It Interacts With**
  - **Service** — Endpoints must have the **same name and namespace** as the Service
  - **Pods** — Pod IPs populate the Endpoints `addresses` list; only **Ready** Pods are included
  - **Endpoints Controller** — automatically creates/updates/deletes Endpoints when Pods or Services change
  - **kube-proxy** — consumes Endpoints to build forwarding rules
  - **EndpointSlices** — modern counterpart; Endpoints and EndpointSlices coexist

- **Upstream and Downstream Dependencies**
  - Upstream: **Service** must exist → Endpoints object created with the same name
  - Upstream: **Pods** must exist, be Ready, and match the Service selector → IPs added to Endpoints
  - Upstream: **Endpoints Controller** must be running (part of Controller Manager) → Endpoints stay in sync
  - Downstream: **kube-proxy** → if Endpoints are empty, no iptables rules → connection refused
  - Downstream: **CoreDNS** → for Headless Services, DNS depends directly on Endpoints
  - Downstream: **Ingress** → backends are resolved via Endpoints

## 3️⃣ Recognize Structure (YAML Skeleton)

### Auto-managed Endpoints (created automatically — you rarely write this YAML)

```yaml
apiVersion: v1                    # Always v1 for Endpoints
kind: Endpoints                   # Resource type (plural!)
metadata:
  name: my-service                # ⚠️ MUST match the Service name
  namespace: default              # ⚠️ MUST match the Service namespace
subsets:                          # List of endpoint subsets
  - addresses:                    # ⚠️ Ready Pod IPs
      - ip: 10.244.1.5
        targetRef:
          kind: Pod
          name: my-pod-abc
          namespace: default
      - ip: 10.244.2.8
        targetRef:
          kind: Pod
          name: my-pod-def
          namespace: default
    notReadyAddresses: []         # Pods that exist but are NOT Ready
    ports:
      - name: http                # Must match Service port name
        port: 8080                # ⚠️ The actual port on the Pod
        protocol: TCP
```

### Manually created Endpoints (for external services — CKA-relevant)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db              # Service without selector
spec:
  ports:
    - port: 3306
      targetPort: 3306
  # ⚠️ NO selector — Endpoints must be created manually
---
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db              # ⚠️ MUST match Service name exactly
  namespace: default
subsets:
  - addresses:
      - ip: 192.168.1.100        # External IP address
    ports:
      - port: 3306
        protocol: TCP
```

- **Key Sections Explained**
  - `apiVersion: v1` — Endpoints is a core API resource, always `v1`
  - `kind: Endpoints` — **plural** (not "Endpoint") — common gotcha!
  - `metadata.name` — **must exactly match** the parent Service name
  - `metadata.namespace` — **must exactly match** the parent Service namespace
  - `subsets[].addresses` — list of IPs of **Ready** Pods (or manually specified IPs)
  - `subsets[].notReadyAddresses` — list of IPs of Pods that are **not Ready** (auto-managed)
  - `subsets[].ports` — port numbers and protocols matching the Service ports
  - `targetRef` — reference to the Pod (auto-populated; not needed for manual Endpoints)

- **Common Indentation Traps**
  - `subsets` is a **list** (starts with `-`), and each subset has `addresses` and `ports`
  - `addresses` is a **list** of objects, each with an `ip` field
  - `ports` inside `subsets` is the **Pod port** (not the Service port)
  - For manual Endpoints: name must match Service name **exactly** (case-sensitive)
  - `kind: Endpoints` is **plural** — writing `kind: Endpoint` → API error

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `metadata.name` | Must match the Service name exactly | Endpoints won't associate with the Service → Service has no backends |
| `metadata.namespace` | Must match the Service namespace | Same as above — namespace mismatch = no association |
| `subsets[].addresses[].ip` | The Pod (or external) IP that receives traffic | Wrong IP → traffic sent to wrong destination |
| `subsets[].ports[].port` | The port on the Pod/target | Mismatch → connection refused at the target |
| `subsets[].ports[].protocol` | TCP/UDP/SCTP | Mismatch → silent traffic failure |
| `subsets[].ports[].name` | Must match Service port name | Mismatch → kube-proxy can't map Service port to Endpoint port |
| `subsets[].notReadyAddresses` | Pods that exist but are not Ready | Non-empty = Pods exist but failing readiness probes |

- **Field Relationships**
  - `endpoints.metadata.name` ↔ `service.metadata.name` — **must be identical**
  - `endpoints.subsets[].ports[].name` ↔ `service.spec.ports[].name` — **must match** for multi-port Services
  - `endpoints.subsets[].ports[].port` = the actual Pod `containerPort` (= Service's `targetPort`)
  - `endpoints.subsets[].addresses[].ip` = Pod IP (from `kubectl get pods -o wide`)
  - Service with selector → Endpoints **auto-managed** (don't edit manually — changes will be overwritten)
  - Service without selector → Endpoints **must be manually created and maintained**

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (< 5 seconds)**
  - `metadata.name` — does it match the Service name?
  - `subsets[].addresses` — are there any IPs listed? Empty = no backends
  - `subsets[].ports[].port` — does it match the Pod's `containerPort`?
  - Is the Endpoints object even present? (`kubectl get endpoints <svc-name>`)

- **Usually Irrelevant for CKA**
  - `targetRef` fields (auto-populated, informational)
  - `notReadyAddresses` details (just know it exists — indicates unhealthy Pods)
  - `resourceVersion`, `uid`, `creationTimestamp` (metadata noise)
  - EndpointSlice details (know it's the modern replacement, but Endpoints is still tested)

- **Red Flags to Detect in 10 Seconds**
  - ❌ `subsets` is empty or missing → **no backends** → Service returns connection refused
  - ❌ `addresses` is empty but `notReadyAddresses` has entries → Pods exist but not Ready
  - ❌ `metadata.name` doesn't match Service name → orphaned Endpoints, Service has no backends
  - ❌ Port in Endpoints doesn't match Pod `containerPort` → traffic to wrong port
  - ❌ Endpoints object doesn't exist at all → Service has no selector, no manual Endpoints created

## 6️⃣ What to Notice In

### `kubectl describe endpoints <name>`
- **Where to Look First**
  - `Addresses` — list of Pod IPs (Ready Pods)
  - `NotReadyAddresses` — Pods that exist but are NOT Ready
  - `Ports` — port numbers and names
- **Failure Signals**
  - `Addresses: <none>` + `NotReadyAddresses` has IPs → Pods exist but failing readiness probes
  - `Addresses: <none>` + `NotReadyAddresses: <none>` → no Pods match the selector at all
  - Port mismatch with what you expect → wrong `targetPort` on the Service

### `kubectl get endpoints <name>`
- **Where to Look First**
  - The `ENDPOINTS` column — should show comma-separated `IP:Port` pairs
  - `<none>` → **broken** — no backends
  - Count of IPs — should match number of Ready Pods
- **Failure Signals**
  - `<none>` → selector mismatch, no Pods, or Pods not Ready
  - Fewer IPs than expected → some Pods not Ready
  - Wrong port number → `targetPort` mismatch

### `kubectl logs <pod>`
- **Where to Look First**
  - Not directly related to Endpoints, but relevant for debugging **why** a Pod isn't Ready
  - Check application startup — is it listening on the expected port?
  - Check readiness probe logs — is the probe path/port correct?
- **Failure Signals**
  - App crash/not starting → Pod not Ready → removed from Endpoints
  - "bind failed" / "address in use" → app can't listen on expected port

### `kubectl exec -it <pod> -- curl <svc-name>:<port>`
- **Where to Look First**
  - Does the Service respond? If yes → Endpoints are working
  - If connection refused → Endpoints empty or `targetPort` wrong
- **Failure Signals**
  - Connection refused → check `kubectl get endpoints` first
  - Timeout → kube-proxy issue or NetworkPolicy

### `kubectl explain endpoints`
- **Where to Look First**
  - `kubectl explain endpoints.subsets` — structure of addresses and ports
  - `kubectl explain endpoints.subsets.addresses` — shows `ip`, `targetRef`, etc.
  - Useful during CKA when writing manual Endpoints YAML

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - Service selector doesn't match Pod labels → Endpoints auto-populated with **zero** addresses
  - Manual Endpoints name doesn't match Service name → Service has no backends
  - Manual Endpoints port doesn't match Pod's actual listening port → connection refused
  - Pod failing readiness probe → moved to `notReadyAddresses` → removed from active `addresses`
  - All Pods in CrashLoopBackOff → Endpoints completely empty
  - Editing auto-managed Endpoints manually → **overwritten** by Endpoints Controller within seconds

- **Status Values to Recognize**
  - Endpoints don't have their own "status" like Pods — they're either populated or empty
  - Key indicator: `kubectl get endpoints <name>` → `ENDPOINTS` column
    - `<none>` → **empty** — the most critical red flag
    - `10.244.1.5:8080,10.244.2.8:8080` → healthy, two backends
  - Related Pod statuses that cause empty Endpoints:
    - `Pending` → Pod not scheduled → not in Endpoints
    - `CrashLoopBackOff` → Pod not Ready → not in Endpoints
    - `ImagePullBackOff` → Pod not Ready → not in Endpoints
    - `ContainerCreating` → Pod not Ready → not in Endpoints

- **Event Messages to Recognize**
  - Endpoints themselves rarely generate events
  - Check **Service** events: `kubectl describe svc <name>` → Events
  - Check **Pod** events: `kubectl describe pod <name>` → Events (for readiness probe failures)
  - `kubectl get events --field-selector involvedObject.kind=Endpoints` → rare but possible

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: "Service created but `Endpoints: <none>`" → selector mismatch or no Ready Pods
  - Scenario 2: "Endpoints show fewer IPs than expected" → some Pods failing readiness probes
  - Scenario 3: "Service for external database not working" → manual Endpoints name mismatch or wrong IP
  - Scenario 4: "I edited Endpoints but changes keep reverting" → Service has a selector → auto-managed → don't edit manually
  - Scenario 5: "Service worked yesterday, Endpoints are empty today" → Pods were deleted/scaled to 0 or image broken
  - Scenario 6: "Service returns wrong data" → Endpoints point to wrong Pods (selector too broad)

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch?**
  - Service selector doesn't match any Pod labels
  - Endpoints Controller creates Endpoints object but `subsets` has **zero addresses**
  - `kubectl get endpoints <svc>` shows `<none>`
  - All traffic to the Service → **connection refused**
  - Fix: either change Service selector or add correct labels to Pods

- **What Happens If Ports Mismatch?**
  - Service `targetPort` doesn't match Pod `containerPort`
  - Endpoints still get populated (IPs are correct — label match works)
  - But the port in Endpoints reflects `targetPort` → kube-proxy forwards to wrong port on Pod
  - Result: **connection refused** at the Pod level
  - Fix: align `targetPort` in Service with `containerPort` in Pod spec

- **What Happens If Image Is Wrong?**
  - Pod goes to `ImagePullBackOff` / `ErrImagePull` → not Ready
  - Pod IP moved from `addresses` to `notReadyAddresses` (or never added)
  - Endpoints become empty if all Pods are affected
  - Service returns connection refused
  - Fix: correct the image → Pod becomes Ready → auto-added back to Endpoints

- **What Happens If Resource Limits Are Wrong?**
  - Pod `OOMKilled` → restart → CrashLoopBackOff → not Ready → removed from Endpoints
  - Endpoints may fluctuate: Pod starts → briefly Ready → OOMKilled → removed
  - Intermittent connection failures from the Service
  - Fix: increase memory limits → Pod stabilizes → stays in Endpoints

- **What Happens If Dependencies Are Missing?**
  - Pod needs ConfigMap/Secret that doesn't exist → stuck in `ContainerCreating` → not Ready → not in Endpoints
  - Pod needs PVC that can't bind → stuck in `Pending` → not in Endpoints
  - Manual Endpoints pointing to an IP that's unreachable → Endpoints populated but traffic fails
  - Service without selector + no manual Endpoints created → Endpoints object doesn't exist at all

## 9️⃣ Fast Detection Strategy

- **Exact kubectl Commands to Run (in order)**
  ```bash
  # Step 1: Check Endpoints exist and have addresses — THE MOST IMPORTANT CHECK
  kubectl get endpoints <svc-name> -n <namespace>

  # Step 2: Describe Endpoints — see addresses vs notReadyAddresses
  kubectl describe endpoints <svc-name> -n <namespace>

  # Step 3: Check Service selector
  kubectl describe svc <svc-name> -n <namespace>
  # Look at Selector and Endpoints lines

  # Step 4: Check Pod labels match the selector
  kubectl get pods -n <namespace> --show-labels
  kubectl get pods -n <namespace> -l <selector-key>=<selector-value>

  # Step 5: Check Pod readiness
  kubectl get pods -n <namespace>
  # Look at READY column (e.g., 1/1 vs 0/1)

  # Step 6: For manual Endpoints — verify name matches Service
  kubectl get svc <name> -n <namespace>
  kubectl get endpoints <name> -n <namespace>
  # Names must be identical

  # Step 7: Verify port in Endpoints matches Pod containerPort
  kubectl get endpoints <name> -o yaml
  kubectl describe pod <pod> | grep -i "port"

  # Step 8: Test connectivity end-to-end
  kubectl exec -it <pod> -- curl <svc-name>:<port>
  ```

- **Order of Debugging Steps**
  1. Endpoints exist? → `kubectl get endpoints <svc-name>`
  2. Endpoints populated or `<none>`? ← **80% of issues detected here**
  3. If `<none>` → check Service selector: `kubectl describe svc`
  4. Selector matches Pods? → `kubectl get pods -l <selector>`
  5. Pods Ready? → `kubectl get pods` → READY column
  6. If Pods not Ready → `kubectl describe pod` → Events + readiness probe
  7. Port correct? → compare Endpoints port vs Pod containerPort
  8. For manual Endpoints → verify name matches Service name exactly

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get endpoints <svc-name>` → `<none>`? → **problem found**
  - If `<none>` → `kubectl describe svc <svc-name>` → look at `Selector`
  - `kubectl get pods -l <selector>` → 0 results? → selector mismatch
  - If Pods found but `READY 0/1` → readiness probe failing → `kubectl describe pod`
  - If Endpoints have IPs but Service doesn't work → port mismatch

## 🔟 Fast Fix Strategy

- **How to Fix Using kubectl**
  ```bash
  # Fix 1: Selector mismatch — change Service selector to match Pod labels
  kubectl edit svc <svc-name>
  # Update spec.selector to match Pod labels

  # Fix 2: Or fix Pod labels to match Service selector
  kubectl label pod <pod-name> app=my-app --overwrite

  # Fix 3: Port mismatch — fix Service targetPort
  kubectl edit svc <svc-name>
  # Change targetPort to match Pod containerPort

  # Fix 4: Pods not Ready — fix the root cause
  kubectl describe pod <pod> # Find why it's not Ready
  # Common fixes:
  kubectl set image deployment/<name> <container>=<correct-image>  # Fix image
  kubectl edit deployment/<name>  # Fix readiness probe path/port

  # Fix 5: Manual Endpoints name mismatch
  kubectl delete endpoints <wrong-name>
  # Recreate with correct name matching the Service
  kubectl apply -f corrected-endpoints.yaml

  # Fix 6: Manual Endpoints wrong IP
  kubectl edit endpoints <name>
  # Or patch:
  kubectl patch endpoints <name> --type='json' \
    -p='[{"op":"replace","path":"/subsets/0/addresses/0/ip","value":"192.168.1.200"}]'

  # Fix 7: Create missing manual Endpoints
  cat <<EOF | kubectl apply -f -
  apiVersion: v1
  kind: Endpoints
  metadata:
    name: <service-name>
  subsets:
    - addresses:
        - ip: <target-ip>
      ports:
        - port: <target-port>
  EOF
  ```

- **How to Validate Fix**
  ```bash
  # Check Endpoints are now populated
  kubectl get endpoints <svc-name>

  # Verify Pod IPs are in the Endpoints
  kubectl get pods -o wide  # Get Pod IPs
  kubectl get endpoints <svc-name>  # Should list same IPs

  # Test connectivity
  kubectl exec -it <pod> -- curl <svc-name>:<port>
  ```

- **How to Confirm Recovery**
  - `kubectl get endpoints <svc-name>` → shows IP:port pairs
  - `kubectl describe svc <svc-name>` → `Endpoints` line is populated
  - `kubectl exec -- curl <svc-name>:<port>` → returns expected response
  - Number of Endpoint IPs matches number of Ready Pods

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 – Create Working Example

#### Example A: Auto-managed Endpoints (Service with selector)

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
```

#### Example B: Manual Endpoints (Service without selector)

```yaml
# external-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: external-api
spec:
  ports:
    - port: 443
      targetPort: 443
---
# external-endpoints.yaml
apiVersion: v1
kind: Endpoints
metadata:
  name: external-api            # ⚠️ Must match Service name
subsets:
  - addresses:
      - ip: 203.0.113.50        # External IP
    ports:
      - port: 443
```

- **Validation Steps**
  ```bash
  # Auto-managed
  kubectl apply -f deployment.yaml -f service.yaml
  kubectl get endpoints web-service           # Should show 3 Pod IPs
  kubectl describe endpoints web-service      # addresses list has 3 entries

  # Manual
  kubectl apply -f external-service.yaml -f external-endpoints.yaml
  kubectl get endpoints external-api          # Should show 203.0.113.50:443
  kubectl describe svc external-api           # Endpoints line shows the IP

  # Test
  kubectl run test --rm -it --image=busybox --restart=Never -- wget -qO- web-service:80
  ```

### Step 2 – Break It Intentionally

| # | What to Break | How | Expected Symptom |
|---|---|---|---|
| 1 | **Mismatch Service selector** | `kubectl edit svc web-service` → change selector to `app: wrong` | Endpoints become `<none>` |
| 2 | **Remove Pod labels** | `kubectl label pod <pod> app-` (remove label) | That Pod disappears from Endpoints |
| 3 | **Wrong targetPort** | `kubectl edit svc` → change `targetPort: 9999` | Endpoints port changes to 9999; traffic → connection refused |
| 4 | **Wrong image** | `kubectl set image deploy/web-app nginx=nginx:nonexistent` | Pods → ImagePullBackOff → Endpoints deplete |
| 5 | **Scale to 0** | `kubectl scale deploy/web-app --replicas=0` | Endpoints become `<none>` |
| 6 | **Manual Endpoints name mismatch** | Create Endpoints with name `external-api-wrong` | Service `external-api` has no Endpoints → connection refused |
| 7 | **Manual Endpoints wrong IP** | Edit Endpoints → set IP to `1.2.3.4` (unreachable) | Endpoints populated but traffic times out |
| 8 | **Delete Endpoints for auto-managed Service** | `kubectl delete endpoints web-service` | Endpoints Controller **recreates** them within seconds! |
| 9 | **Add failing readiness probe** | Edit Deployment → wrong readiness probe path | Pods not Ready → moved to `notReadyAddresses` → Endpoints empty |
| 10 | **Edit auto-managed Endpoints manually** | `kubectl edit endpoints web-service` → change an IP | Changes **overwritten** by Endpoints Controller instantly |

### Step 3 – Debug Without Looking at Solution

- **What command to run first**
  - `kubectl get endpoints <svc-name>` — this reveals 80% of issues instantly
- **How to narrow the issue**
  - Endpoints `<none>`? → is it auto-managed (Service has selector) or manual?
  - Auto-managed + empty → `kubectl get pods -l <selector>` → do Pods match?
  - Auto-managed + Pods exist but not in Endpoints → `kubectl get pods` → READY column → readiness probe issue
  - Manual + empty → does Endpoints name match Service name?
  - Endpoints populated but Service doesn't work → check port numbers
- **How to confirm root cause**
  - Selector mismatch: `kubectl describe svc` selector vs `kubectl get pods --show-labels`
  - Readiness issue: `kubectl describe pod` → Events → readiness probe failure
  - Name mismatch: `kubectl get svc` name vs `kubectl get endpoints` name
  - Port mismatch: `kubectl get endpoints -o yaml` port vs `kubectl describe pod` containerPort

  - Understanding why Endpoints are empty: selector mismatch, Pods not Ready, no Pods
  - Writing manual Endpoints YAML (name must match Service)
  - Recognizing that auto-managed Endpoints can't be manually edited (overwritten instantly)
  - Port mapping: Endpoints port = Pod `containerPort` = Service `targetPort`

- **What to Practice Repeatedly**
  - Check Endpoints for a Service in under 5 seconds
  - Create a Service + manual Endpoints for an external resource
  - Debug empty Endpoints: follow selector → labels → Pod readiness chain
  - Intentionally break and fix selector-Endpoints relationship

- **What to Ignore (Deep Internals Not Required)**
  - EndpointSlice internal structure and controller mechanics
  - Endpoints Controller source code / reconciliation loop details
  - `targetRef` fields and their usage
  - Topology-aware Endpoints routing
  - `publishNotReadyAddresses` deep behavior

- **5 Quiz Questions**
  1. **Q:** A Service named `backend` is created with a selector. What must the auto-created Endpoints object be named?
     - **A:** `backend` — Endpoints must have the **exact same name** as the Service.
  2. **Q:** `kubectl get endpoints my-svc` shows `<none>`. What are the top 3 causes?
     - **A:** (1) Service selector doesn't match any Pod labels, (2) Pods exist but are not Ready, (3) No Pods exist at all.
  3. **Q:** You manually edit the Endpoints of a Service that has a selector. What happens?
     - **A:** Your changes are **overwritten within seconds** by the Endpoints Controller. Auto-managed Endpoints cannot be manually edited.
  4. **Q:** How do you create Endpoints for an external database at `10.0.0.50:5432`?
     - **A:** Create a Service **without selector** and a matching Endpoints object with the same name, specifying `ip: 10.0.0.50` and `port: 5432` in `subsets`.
  5. **Q:** Endpoints show 2 IPs but the Deployment has 3 replicas. What's the most likely cause?
     - **A:** One Pod is not passing its readiness probe — it exists but is not Ready, so it's in `notReadyAddresses` instead of `addresses`.

## 1️⃣3️⃣ Component Interaction Map

```
                                   ┌──────────────────────┐
                                   │   Endpoints Controller│
                                   │   (Controller Manager)│
                                   └───────────┬──────────┘
                                               │ watches Pods + Services
                                               │ populates/updates Endpoints
                                               ▼
┌──────────┐   DNS/ClusterIP   ┌──────────┐   reads    ┌──────────────┐         ┌──────────┐
│ Consumer │ ────────────────► │ Service  │ ──────────► │  Endpoints   │ ◄─────► │  Pod(s)  │
│   Pod    │                   │          │             │  (IP:Port    │  IPs    │ :contPort│
└──────────┘                   └──────────┘             │   pairs)     │         └──────────┘
                                                        └──────┬───────┘
                                                               │
                                                        ┌──────▼───────┐
                                                        │  kube-proxy  │
                                                        │  (iptables/  │
                                                        │   IPVS)      │
                                                        └──────────────┘
```

- **Pods**
  - Pod IPs populate the `addresses` list in Endpoints
  - Only **Ready** Pods are in `addresses`; not-Ready Pods are in `notReadyAddresses`
  - Pod deletion → IP removed from Endpoints
  - Pod restart (new IP) → Endpoints updated automatically
  - Pod label change → may be added/removed from Endpoints based on selector match

- **Services**
  - Every Service with a selector has an auto-managed Endpoints object with the same name
  - Service without a selector → no auto-managed Endpoints → must create manually
  - Service reads Endpoints via name matching (same name, same namespace)
  - `kubectl describe svc` shows the `Endpoints` line from this Endpoints object

- **Nodes**
  - kube-proxy on every Node reads Endpoints → programs iptables/IPVS rules
  - If Endpoints change → kube-proxy updates rules (eventual consistency, usually < 1 second)
  - Endpoints are cluster-wide — every Node's kube-proxy sees the same Endpoints

- **Scheduler**
  - No direct interaction with Endpoints
  - Indirectly: if a Pod can't be scheduled → Pod stays Pending → not Ready → not in Endpoints

- **Controller Manager**
  - **Endpoints Controller** is the key component — runs inside Controller Manager
  - Watches: all Pods and all Services
  - When a Pod becomes Ready / not Ready / deleted → updates Endpoints
  - When a Service selector changes → recalculates Endpoints
  - If Controller Manager is down → **Endpoints become stale** (not updated)

- **etcd**
  - Stores Endpoints objects
  - If etcd is down → Endpoints can't be updated; existing kube-proxy rules still work (cached)

- **Network Layer**
  - kube-proxy reads Endpoints to know where to forward traffic
  - If Endpoints are empty → kube-proxy has no backends → connection refused
  - If Endpoints are stale → traffic may be sent to deleted/moved Pods → errors
  - CNI plugin → provides the Pod-to-Pod connectivity that Endpoints rely on

- **Storage Layer**
  - No direct interaction
  - Indirect: Pod needing a PVC that can't bind → Pod Pending → not in Endpoints

- **Interaction Behavior When Things Go Wrong**
  - Endpoints Controller down → Endpoints become stale; Pods added/removed but Endpoints don't reflect it
  - kube-proxy down → can't program forwarding rules; even with correct Endpoints, traffic can't be routed
  - CoreDNS down → DNS fails, but if client uses ClusterIP directly, kube-proxy rules (based on Endpoints) still work
  - etcd down → Endpoints can't be updated; existing rules continue working
  - Pod flapping (Ready → not Ready → Ready) → Endpoints constantly updated → kube-proxy constantly reprogramming → possible brief connection failures

## ⚡ Quick Reference Cheat Sheet

```bash
# Check Endpoints (THE SINGLE MOST IMPORTANT COMMAND for Service debugging)
kubectl get endpoints <svc-name>

# Describe Endpoints (see addresses vs notReadyAddresses)
kubectl describe endpoints <svc-name>

# View Endpoints YAML
kubectl get endpoints <svc-name> -o yaml

# Check Service → Endpoints relationship
kubectl describe svc <svc-name>  # Look at "Endpoints" line

# Check Pod labels match Service selector
kubectl get pods --show-labels
kubectl get pods -l <key>=<value>

# Check Pod readiness
kubectl get pods  # READY column

# Test connectivity through Service
kubectl exec -it <pod> -- curl <svc-name>:<port>

# Create manual Endpoints for external service
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Endpoints
metadata:
  name: <must-match-service-name>
subsets:
  - addresses:
      - ip: <external-ip>
    ports:
      - port: <port>
EOF

# Patch manual Endpoints
kubectl patch endpoints <name> --type='json' \
  -p='[{"op":"replace","path":"/subsets/0/addresses/0/ip","value":"<new-ip>"}]'

# Delete and recreate manual Endpoints
kubectl delete endpoints <name>
kubectl apply -f corrected-endpoints.yaml

# Verify Endpoints match Pod IPs
kubectl get pods -o wide          # Pod IPs
kubectl get endpoints <svc-name>  # Should list same IPs

# Quick: does Service have backends?
kubectl get endpoints <svc-name> | grep -v '<none>'
```
