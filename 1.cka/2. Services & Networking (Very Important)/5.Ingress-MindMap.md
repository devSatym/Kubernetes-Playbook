# 🌐 Kubernetes Ingress — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Manages external HTTP/HTTPS access to Services within a cluster
  - Provides URL-based routing, host-based routing, TLS termination, and load balancing at Layer 7
  - Single entry point for multiple backend Services — replaces the need for multiple NodePort/LoadBalancer Services
  - Acts as a "smart router" — maps incoming HTTP requests to backend Services based on rules

- **Real-World Use Case**
  - Route `app.example.com` → frontend Service, `api.example.com` → backend Service
  - Path-based routing: `/api` → api-svc, `/web` → web-svc under single domain
  - TLS termination — HTTPS at the Ingress, plain HTTP to Pods
  - Canary deployments — route a percentage of traffic to new version
  - Centralized access control and rate limiting

- **Where It Fits in Cluster Architecture**
  - Lives at the **edge** of the cluster — between external clients and internal Services
  - Ingress **resource** (YAML object) defines rules
  - Ingress **Controller** (Pod running NGINX/Traefik/HAProxy) enforces rules
  - Flow: `Client → Ingress Controller Pod → ClusterIP Service → Pods`
  - Requires a working Ingress Controller — the Ingress resource alone does NOTHING

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - External users/clients accessing HTTP/HTTPS applications
  - Frontend applications needing domain-based routing
  - TLS-secured applications depending on Ingress for cert termination
  - Monitoring dashboards exposed externally

- **What Resources It Interacts With**
  - **Services (ClusterIP)** — backends that Ingress routes traffic to
  - **Pods** — ultimate traffic destination behind Services
  - **Secrets (TLS)** — stores TLS certificates referenced by Ingress
  - **IngressClass** — selects which Ingress Controller handles this Ingress
  - **Ingress Controller** — NGINX, Traefik, etc. — the actual enforcement engine
  - **ConfigMaps** — Ingress Controller configuration (e.g., nginx-configuration)

- **Upstream Dependencies**
  - Ingress Controller **must be deployed** (e.g., ingress-nginx)
  - IngressClass resource must exist (Kubernetes 1.18+)
  - Backend Services must exist and have healthy endpoints
  - TLS Secrets must exist before Ingress references them
  - DNS must resolve hostnames to Ingress Controller's external IP

- **Downstream Dependencies**
  - ClusterIP Services (backends)
  - Pods serving actual application traffic
  - Endpoints connecting Services to Pods

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: networking.k8s.io/v1       # MUST be networking.k8s.io/v1 (NOT extensions/v1beta1)
kind: Ingress
metadata:
  name: my-ingress                     # Name of the Ingress resource
  namespace: default                   # Namespace — must match backend Services
  annotations:                         # Controller-specific behavior
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx              # Which Ingress Controller handles this (required in 1.18+)
  tls:                                 # TLS termination config
    - hosts:
        - app.example.com
      secretName: app-tls-secret       # TLS Secret (type kubernetes.io/tls)
  rules:                               # Routing rules
    - host: app.example.com            # Host-based routing (optional)
      http:
        paths:
          - path: /                    # URL path to match
            pathType: Prefix           # Prefix | Exact | ImplementationSpecific
            backend:
              service:
                name: my-service       # Backend Service name
                port:
                  number: 80           # Service port (NOT Pod/container port)
  defaultBackend:                      # Catch-all backend (optional)
    service:
      name: fallback-svc
      port:
        number: 80
```

- **Key Sections Explained**
  - `apiVersion: networking.k8s.io/v1` — the only valid API for 1.22+ (extensions/v1beta1 removed)
  - `ingressClassName` — replaces the old `kubernetes.io/ingress.class` annotation
  - `rules` — array of host + path combinations, each pointing to a backend Service
  - `tls` — references a Secret of type `kubernetes.io/tls` containing `tls.crt` and `tls.key`
  - `defaultBackend` — handles requests that match no rule
  - `pathType` — **REQUIRED** field (Prefix, Exact, ImplementationSpecific)

- **Common Indentation Traps**
  - `paths` is nested under `http` → under `rules[].host`
  - `backend.service.name` and `backend.service.port.number` — deep nesting, easy to misplace
  - `tls` is at the same level as `rules` (under `spec`), NOT nested inside `rules`
  - `secretName` under `tls` must match an existing Secret — typo = silent failure
  - `pathType` is REQUIRED — omitting it causes validation error

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `ingressClassName` | Selects the controller | Ingress is ignored — no controller picks it up |
| `rules[].host` | Host-based routing | Requests to that hostname won't match |
| `rules[].http.paths[].path` | Path matching | Requests to that path won't route |
| `rules[].http.paths[].pathType` | Required — Prefix/Exact | API validation error if missing |
| `backend.service.name` | Target Service | 404/503 errors — traffic has no destination |
| `backend.service.port.number` | Service port | 503 — wrong port on the Service |
| `tls[].secretName` | TLS certificate | HTTPS fails — falls back to default fake cert |
| `tls[].hosts` | Hostnames for TLS | Cert mismatch warnings in browser |
| `defaultBackend` | Catch-all | Unmatched requests get 404 from controller |
| `annotations` | Controller-specific behavior | Rewrite, rate-limit, etc. won't work |

- **Field Relationships**
  - `backend.service.name` ↔ must match an existing Service name in the same namespace
  - `backend.service.port.number` ↔ must match the Service's `.spec.ports[].port` (NOT targetPort)
  - `tls[].secretName` ↔ must reference a Secret of type `kubernetes.io/tls`
  - `tls[].hosts` ↔ should match `rules[].host` for proper TLS routing
  - `ingressClassName` ↔ must match an existing IngressClass resource

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - `apiVersion` — must be `networking.k8s.io/v1`
  - `ingressClassName` — is it set? Does it match your controller?
  - `rules` — are hosts and paths present?
  - `backend.service.name` — does the Service exist?
  - `backend.service.port.number` — correct port?
  - `pathType` — is it present? (Prefix vs Exact matters)

- **Usually Irrelevant for CKA**
  - Detailed annotation tuning (rate limits, custom headers)
  - Advanced canary/weight annotations
  - Session affinity via annotations
  - Custom error pages configuration

- **Red Flags to Detect in 10 Seconds**
  - ❌ Missing `ingressClassName` or using deprecated annotation
  - ❌ `pathType` missing — won't pass validation
  - ❌ `backend.service.port.number` pointing to containerPort instead of Service port
  - ❌ `secretName` referencing a non-existent Secret
  - ❌ Namespace mismatch — Ingress and Service must be in the SAME namespace
  - ❌ No Ingress Controller deployed in the cluster
  - ❌ Old apiVersion `extensions/v1beta1` on 1.22+ cluster

## 6️⃣ What to Notice In

### kubectl describe ingress \<name\>

- **What Signals Failure**
  - `Default backend: default-http-backend:80 (<error: endpoints "default-http-backend" not found>)`
  - Rules showing `<none>` for backend — misconfigured Service reference
  - Events: `"No service found"`, `"Error obtaining endpoints"`
  - Empty `Address` field — controller hasn't assigned an IP yet
  - `TLS: <secret-name> terminates <host>` — verify Secret exists

- **Where to Look First**
  - `Rules` table — check each host/path → backend mapping
  - `Address` — external IP/hostname assigned by controller
  - `Events` section — controller reconciliation errors

- **Fields That Matter Most**
  - `Rules` — correct Service name and port
  - `Address` — should show the Ingress Controller's external IP
  - `TLS` — confirms cert is being used
  - `Events` — any warnings about missing backends or Secrets

### kubectl logs \<ingress-controller-pod\>

- **What Signals Failure**
  - `upstream connect error or disconnect/reset before headers`
  - `no endpoints found for service`
  - `SSL certificate error` or `fake certificate`
  - `connection refused` — backend Service/Pod not reachable
  - `404 Not Found` in access logs — path doesn't match any rule

- **Where to Look First**
  - Error-level logs for upstream connection failures
  - Access logs showing 502/503/504 status codes
  - Configuration reload errors after Ingress changes

- **Fields That Matter Most**
  - HTTP status codes (502 = bad backend, 503 = no endpoints, 504 = timeout)
  - Upstream server address
  - Error descriptions

### kubectl exec (Testing from inside the cluster)

- **Useful Commands**
  ```bash
  # Test backend Service directly
  kubectl exec -it test-pod -- curl http://my-service:80

  # Test Ingress Controller directly
  kubectl exec -it test-pod -- curl -H "Host: app.example.com" http://<ingress-controller-svc-ip>

  # Verify DNS resolution
  kubectl exec -it test-pod -- nslookup my-service
  ```

- **What Signals Failure**
  - `Connection refused` — Service exists but no healthy Pods
  - `Could not resolve host` — Service doesn't exist or DNS issue
  - `Empty reply from server` — Pod is running but app not responding
  - HTTP 502/503 — Service exists, but Pods not ready

### kubectl explain ingress

- **Key Usage**
  ```bash
  kubectl explain ingress.spec.rules
  kubectl explain ingress.spec.rules.http.paths.backend
  kubectl explain ingress.spec.tls
  kubectl explain ingress.spec.ingressClassName
  ```
- During CKA — quickly verify field names and nesting
- Confirm `pathType` is required
- Check backend service reference structure

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ No Ingress Controller deployed → Ingress resource ignored entirely
  - ❌ Wrong `ingressClassName` → controller doesn't pick it up
  - ❌ Service name typo → 503 Service Temporarily Unavailable
  - ❌ Wrong Service port → 503 (port exists on Service, not container)
  - ❌ Missing `pathType` → API validation rejection
  - ❌ TLS Secret missing → HTTPS uses controller's fake/default certificate
  - ❌ Namespace mismatch (Ingress in ns-a, Service in ns-b) → 503
  - ❌ Service has no Endpoints (label mismatch) → 503
  - ❌ Path not matching (Exact vs Prefix confusion) → 404

- **Status / Error Indicators**
  - Ingress itself doesn't have "status" like Pods — check the **Ingress Controller Pod**
  - `ADDRESS` field empty → controller hasn't processed the Ingress
  - Events: `"Rejected"`, `"error obtaining endpoints"`, `"no upstream host"`
  - Controller Pod logs: `502 Bad Gateway`, `503 Service Unavailable`

- **Event Messages to Recognize**
  - `Successfully synced` — Ingress processed correctly ✅
  - `Error obtaining Endpoints for Service` — Service has no Pods
  - `Rejected: no <IngressClass> found` — missing IngressClass
  - `Secret "xxx" not found` — TLS misconfiguration

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: Ingress created but not working → no Ingress Controller deployed
  - Scenario 2: Ingress returns 503 → backend Service has no Endpoints (labels mismatch)
  - Scenario 3: HTTPS showing invalid cert → TLS Secret name typo
  - Scenario 4: `/api` path works but `/api/v1` doesn't → `pathType: Exact` instead of `Prefix`
  - Scenario 5: One host works, another doesn't → missing rule for second host

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch (Service → Pods)?**
  - Service gets 0 Endpoints
  - Ingress Controller sees "no upstream servers" for that backend
  - Result: **503 Service Temporarily Unavailable**
  - Ingress itself looks healthy — problem is downstream

- **What Happens If Ports Mismatch?**
  - `backend.service.port.number` doesn't match Service `.spec.ports[].port`
  - Result: **503 error** — Ingress Controller can't connect to non-existent Service port
  - If Service port exists but `targetPort` wrong → connection refused from Pod

- **What Happens If Image Is Wrong (Ingress Controller)?**
  - Ingress Controller Pod enters `ImagePullBackOff` or `CrashLoopBackOff`
  - ALL Ingress rules stop working — no controller to enforce them
  - Existing connections may drop

- **What Happens If IngressClass / Controller Is Missing?**
  - Ingress resource is created but never processed
  - `ADDRESS` remains empty
  - No events from any controller
  - Traffic never reaches backend Services

- **What Happens If TLS Secret Is Missing?**
  - HTTPS requests get the controller's **default fake certificate**
  - Browser shows "Your connection is not private" / certificate warning
  - HTTP routing may still work — only TLS is broken

- **What Happens If Dependencies Are Missing (Service/Pods)?**
  - Missing Service → 503, controller logs "service not found"
  - Service exists, no Pods → 503, "no endpoints"
  - Pods exist but not Ready → 503, endpoints not populated

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Check Ingress resource exists and has ADDRESS
  kubectl get ingress -A

  # 2. Describe the Ingress — check rules, backends, events
  kubectl describe ingress <name> -n <namespace>

  # 3. Verify Ingress Controller is running
  kubectl get pods -n ingress-nginx    # (or wherever controller lives)

  # 4. Verify IngressClass exists
  kubectl get ingressclass

  # 5. Check backend Service exists
  kubectl get svc <backend-svc> -n <namespace>

  # 6. Check Service has Endpoints
  kubectl get endpoints <backend-svc> -n <namespace>

  # 7. Check backend Pods are running and Ready
  kubectl get pods -l <selector-labels> -n <namespace>

  # 8. Check Ingress Controller logs for errors
  kubectl logs -n ingress-nginx <controller-pod> --tail=50

  # 9. Test from inside the cluster
  kubectl run test --image=busybox --rm -it -- wget -qO- http://<service-name>
  ```

- **Order of Debugging Steps**
  1. Does the Ingress exist? → `kubectl get ingress`
  2. Does it have an ADDRESS? → If no, controller issue
  3. Does the Ingress Controller exist and is it Running? → `kubectl get pods -n ingress-nginx`
  4. Is the IngressClass correct? → `kubectl get ingressclass`
  5. Does the backend Service exist in the same namespace?
  6. Does the Service have Endpoints?
  7. Are the Pods behind the Service Running and Ready?
  8. Check Ingress Controller logs for specific errors

- **How to Detect Issue Under 60 Seconds**
  - `kubectl get ingress` — no ADDRESS? → controller not picking it up
  - `kubectl describe ingress <name>` — read Events and Rules
  - `kubectl get endpoints <backend-svc>` — empty? → label/selector mismatch
  - `kubectl logs <controller-pod> -n ingress-nginx --tail=20` — 502/503? → backend issue

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix wrong Service name or port
  kubectl edit ingress <name> -n <namespace>
  # → Correct backend.service.name and backend.service.port.number

  # Fix missing IngressClass
  kubectl patch ingress <name> -n <namespace> \
    --type=merge -p '{"spec":{"ingressClassName":"nginx"}}'

  # Fix path type
  kubectl edit ingress <name>
  # → Set pathType: Prefix (or Exact)

  # Fix TLS Secret reference
  kubectl edit ingress <name>
  # → Correct secretName under tls section

  # Create missing TLS Secret
  kubectl create secret tls app-tls-secret \
    --cert=tls.crt --key=tls.key -n <namespace>

  # Fix Service selector (if Service has no Endpoints)
  kubectl edit svc <backend-svc>
  # → Correct selector labels to match Pod labels
  ```

- **Validate Fix**
  ```bash
  # Check Ingress has ADDRESS now
  kubectl get ingress <name>

  # Describe — events should show "Successfully synced"
  kubectl describe ingress <name>

  # Test externally
  curl -H "Host: app.example.com" http://<ingress-ip>/

  # Test from inside cluster
  kubectl run test --image=busybox --rm -it -- wget -qO- http://<svc-name>
  ```

- **Confirm Recovery**
  - Ingress has an ADDRESS assigned ✅
  - `describe ingress` shows correct rules with valid backends ✅
  - `curl` returns expected HTTP response ✅
  - Controller logs show no upstream errors ✅
  - Endpoints are populated for backend Service ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

```yaml
# 1. Deploy backend application
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
---
# 2. Create backend Service
apiVersion: v1
kind: Service
metadata:
  name: web-svc
  namespace: default
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
---
# 3. Create Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-ingress
  namespace: default
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - host: web.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-svc
                port:
                  number: 80
```

- **Validation Steps**
  ```bash
  kubectl apply -f ingress-example.yaml
  kubectl get ingress web-ingress
  # → Should show ADDRESS after a few seconds
  kubectl describe ingress web-ingress
  # → Rules should list web-svc:80
  kubectl get endpoints web-svc
  # → Should show 2 Pod IPs
  curl -H "Host: web.example.com" http://<ingress-controller-ip>/
  # → Should return NGINX default page
  ```

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Wrong Service name | Change `web-svc` → `web-svc-wrong` | 503, controller logs: "service not found" |
| Wrong Service port | Change `port.number: 80` → `8080` | 503, port mismatch |
| Remove pathType | Delete `pathType: Prefix` | API validation error on apply |
| Wrong ingressClassName | Change `nginx` → `traefik` | Ingress ignored, no ADDRESS |
| Remove host | Delete `host: web.example.com` | Matches ALL hosts (catch-all) — may work differently than expected |
| Wrong path | Change `/` → `/api` with `pathType: Exact` | Only `/api` matches, `/` returns 404 |
| Missing TLS Secret | Add `tls` block referencing non-existent secret | HTTPS uses fake cert, browser warns |
| Break Service selector | Edit Service selector labels to mismatch Pods | 0 Endpoints → 503 |
| Delete Ingress Controller | `kubectl delete deploy -n ingress-nginx ingress-nginx-controller` | ALL Ingress rules stop working, all traffic fails |
| Wrong namespace | Move Ingress to different namespace than Service | 503, backend not found |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get ingress -A
  kubectl describe ingress <name>
  ```

- **How to Narrow Issue**
  1. ADDRESS present? → Yes = controller sees it; No = ingressClassName / controller issue
  2. Describe events show errors? → Read the error message
  3. Backend Service exists? → `kubectl get svc`
  4. Endpoints populated? → `kubectl get endpoints <svc>`
  5. Controller logs? → `kubectl logs <controller-pod> --tail=30`

- **How to Confirm Root Cause**
  - Compare Ingress YAML → Service name/port → Service selector → Pod labels
  - Trace the full chain: Ingress → Service → Endpoints → Pods
  - If any link is broken, that's the root cause

## 1️⃣3️⃣ Component Interaction Map

- **Ingress ↔ Pods**
  - Ingress does NOT directly interact with Pods
  - Traffic path: Ingress Controller → Service (ClusterIP) → Endpoints → Pods
  - If Pods are not Ready → they are removed from Endpoints → Ingress gets 503

- **Ingress ↔ Services**
  - Ingress references Services by name in `backend.service.name`
  - Service must be in the **same namespace** as the Ingress
  - Service port in Ingress must match `.spec.ports[].port` of the Service
  - **When things go wrong:** wrong Service name → 503; wrong port → 503

- **Ingress ↔ Nodes**
  - Ingress Controller typically runs as a DaemonSet or Deployment on Nodes
  - NodePort or LoadBalancer Service exposes the Ingress Controller externally
  - **When things go wrong:** Controller Pod unschedulable → no Node available → all Ingress rules fail

- **Ingress ↔ Scheduler**
  - Scheduler places Ingress Controller Pods on Nodes
  - Scheduler does NOT interact with Ingress resources directly
  - **When things go wrong:** Controller Pod Pending → no matching Node (taints, resources)

- **Ingress ↔ Controller Manager**
  - Ingress Controller (e.g., nginx-ingress-controller) watches Ingress resources via API Server
  - It is NOT the kube-controller-manager — it's a separate controller
  - **When things go wrong:** Ingress Controller crashed → no reconciliation → stale routing rules

- **Ingress ↔ etcd**
  - Ingress resources are stored in etcd like all Kubernetes objects
  - Ingress Controller reads them via API Server (never directly from etcd)
  - **When things go wrong:** etcd unavailable → can't create/update Ingress resources

- **Ingress ↔ Network Layer**
  - Ingress operates at **Layer 7** (HTTP/HTTPS) — not Layer 4
  - Relies on kube-proxy and CNI for underlying network connectivity
  - Ingress Controller Pod must be able to reach backend Pod IPs
  - **When things go wrong:** CNI broken → Ingress Controller can't reach Pods → 502/504; Network Policy blocking Ingress Controller → traffic dropped silently

- **Ingress ↔ Storage Layer**
  - No direct interaction with storage
  - Ingress Controller **may** use ConfigMaps for configuration (not PVs)

kubectl create ingress my-ing --rule="host/path=svc:port" --class=nginx --dry-run=client -o yaml
