# 🧠 Kubernetes Mind Map — Service (Headless)

---

## 1️⃣ What It Does

- **Core Purpose**
  - A Service with `clusterIP: None` — Kubernetes does **not** allocate a virtual IP
  - DNS query for the Service name returns **individual Pod IPs** (A records) instead of a single ClusterIP
  - Allows clients to discover and connect to **specific Pods directly** — no load-balancing proxy layer
  - Essential for **stateful workloads** where each Pod has a unique identity (e.g., database replicas)
  - When used with a StatefulSet, each Pod gets a **stable DNS name**: `<pod-name>.<svc-name>.<ns>.svc.cluster.local`

- **Real-World Use Case**
  - **StatefulSets** — databases (MySQL, PostgreSQL, MongoDB), message brokers (Kafka, RabbitMQ), distributed systems (Zookeeper, etcd)
  - Clients needing to connect to a **specific replica** (e.g., always talk to the primary DB, not just any replica)
  - Service discovery where the application itself does its own load balancing / leader election
  - Kafka consumers needing to connect to **specific broker Pods** by DNS name
  - StatefulSet Pods discovering peers for **cluster formation** (e.g., etcd peer discovery)

- **Where It Fits in Cluster Architecture**
  - Lives in the **Services & Networking** domain of CKA
  - No kube-proxy rules → no iptables/IPVS for this Service (no virtual IP to route)
  - CoreDNS returns **Pod A records** directly when queried
  - Endpoints Controller still populates Endpoints (Pod IPs) — the same as regular Services
  - Paired with **StatefulSet** for stable network identity: `<pod-name>.<headless-svc>.<ns>.svc.cluster.local`
  - Without StatefulSet: returns all matching Pod IPs (round-robin DNS), but no per-Pod stable names

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **StatefulSets** — `spec.serviceName` must point to a Headless Service for stable Pod DNS
  - **Pods** — consumers query the Headless Service DNS to get individual Pod IPs
  - **Applications with custom load balancing** — app-level routing, leader election, sharding
  - **Distributed databases** — peer discovery (each node finds other nodes via DNS)
  - **Operators** — custom controllers often rely on Headless Services for member discovery

- **What Resources It Interacts With**
  - **Endpoints / EndpointSlices** — auto-created; list of Pod IPs matching the selector (same as regular Service)
  - **Pods** — selector ↔ labels matching populates Endpoints; each Pod gets a DNS A record
  - **CoreDNS** — returns multiple A records (one per Pod) instead of a single ClusterIP
  - **StatefulSet** — references the Headless Service via `spec.serviceName`
  - **PersistentVolumeClaims** — often paired with StatefulSet + Headless Service for stateful workloads

- **Upstream and Downstream Dependencies**
  - Upstream: Pods must exist with matching labels → Endpoints get populated → DNS returns Pod IPs
  - Upstream: CoreDNS must be healthy → DNS resolution returns Pod A records
  - Upstream: StatefulSet must reference correct `serviceName` → stable per-Pod DNS works
  - Downstream: Applications consuming the Headless Service DNS for peer discovery / direct Pod access
  - Downstream: If Headless Service is broken → StatefulSet Pods lose stable DNS names → peer discovery fails

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: v1                  # Always v1 for Service
kind: Service                   # Resource type
metadata:
  name: my-headless-svc         # Service name (used in DNS)
  namespace: default            # Namespace scoping
  labels:                       # Labels ON the Service itself
    app: my-app
spec:
  clusterIP: None               # ⚠️ THE KEY FIELD — makes it headless
  selector:                     # ⚠️ CRITICAL — matches Pod labels
    app: my-app
  ports:
    - name: http                # Port name (required if multiple ports)
      protocol: TCP             # TCP (default) | UDP | SCTP
      port: 80                  # ⚠️ Service port (used in DNS SRV records)
      targetPort: 8080          # ⚠️ Pod port (where traffic is forwarded)
```

- **Key Sections Explained**
  - `apiVersion: v1` — Service is a core API resource, always `v1`
  - `kind: Service` — identifies the resource type
  - `metadata.name` — becomes the DNS name; for StatefulSets, also part of Pod DNS: `<pod>.<svc-name>.<ns>.svc.cluster.local`
  - `spec.clusterIP: None` — **the defining field** — tells Kubernetes not to assign a virtual IP
  - `spec.selector` — selects Pods whose IPs will be returned by DNS
  - `spec.ports` — still define ports for DNS SRV records and Endpoint population; traffic goes directly to Pod IPs
  - `spec.type` — **omit this** or set to `ClusterIP`; headless only works with ClusterIP type

- **Common Indentation Traps**
  - `clusterIP: None` is under `spec`, at the same level as `selector` and `ports`
  - `None` must be capitalized — `none` or `"None"` (quoted) also works, but `NONE` does not
  - `selector` is under `spec`, NOT under `metadata`
  - `ports` is a **list** (starts with `-`), not a map
  - Forgetting `clusterIP: None` → creates a regular ClusterIP Service (common mistake!)
  - If `clusterIP` is set to any IP (e.g., `clusterIP: 10.96.0.1`) → it's NOT headless

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Wrong |
|---|---|---|
| `spec.clusterIP` | Must be `None` to make it headless | Any other value → regular ClusterIP Service, not headless |
| `spec.selector` | Selects which Pods' IPs are returned by DNS | No selector → Endpoints not auto-managed; DNS returns nothing |
| `spec.ports[].port` | Defines port for DNS SRV records | Missing → SRV records incomplete; Pod discovery may fail |
| `spec.ports[].targetPort` | Port on the Pod receiving traffic | Mismatch → clients connect to wrong Pod port |
| `metadata.name` | DNS name AND part of StatefulSet Pod DNS | Must match StatefulSet's `spec.serviceName` exactly |
| `metadata.namespace` | Scopes the Service | Must be same namespace as the StatefulSet/Pods |
| `spec.type` | Must be `ClusterIP` (or omitted) | Setting `NodePort` or `LoadBalancer` → not headless, gets a ClusterIP assigned |

- **Field Relationships**
  - `spec.clusterIP: None` + `spec.selector` = Headless Service with auto-managed Endpoints
  - `spec.clusterIP: None` + no `spec.selector` = Headless Service WITHOUT auto-managed Endpoints (manual Endpoints)
  - `metadata.name` ↔ `statefulset.spec.serviceName` — **must match exactly** for stable Pod DNS
  - `spec.selector` ↔ `pod.metadata.labels` — **must match exactly** for Endpoints population
  - `spec.ports[].port` = used for DNS SRV records; `spec.ports[].targetPort` = actual Pod port
  - With StatefulSet: each Pod gets DNS `<pod-name>.<svc-name>.<ns>.svc.cluster.local`
  - Without StatefulSet: DNS returns all matching Pod IPs as A records (no per-Pod names)

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (< 5 seconds)**
  - `spec.clusterIP: None` — is it actually set? This is the **only** thing that makes it headless
  - `spec.selector` — present and matching Pod labels?
  - `metadata.name` — does it match the StatefulSet's `spec.serviceName`?
  - `spec.type` — must be `ClusterIP` or omitted (NOT `NodePort` or `LoadBalancer`)

- **Usually Irrelevant for CKA**
  - `spec.sessionAffinity` — not applicable (no proxy layer)
  - `spec.publishNotReadyAddresses` — advanced; default `false` is fine for most cases
  - `spec.ipFamilyPolicy` — dual-stack details not CKA priority
  - `status` section — auto-managed, empty for headless

- **Red Flags to Detect in 10 Seconds**
  - ❌ `clusterIP` is missing or set to an IP → **not headless** — creates a regular ClusterIP Service
  - ❌ `selector` is empty or missing → **no auto-managed Endpoints** → DNS returns nothing
  - ❌ `selector` labels don't match any Pod labels → Endpoints empty → DNS returns nothing
  - ❌ `metadata.name` doesn't match StatefulSet's `spec.serviceName` → **Pods have no stable DNS**
  - ❌ `type: NodePort` or `type: LoadBalancer` → Kubernetes assigns a ClusterIP → **not headless**
  - ❌ `ports` missing entirely → DNS SRV records incomplete

## 6️⃣ What to Notice In

### `kubectl describe service <svc-name>`
- **Where to Look First**
  - `Type` — should be `ClusterIP`
  - `IP` line — should show `None` (confirms headless)
  - `Endpoints` line — should list Pod IPs; `<none>` = **broken selector**
  - `Selector` — verify it matches Pod labels
  - `Port` / `TargetPort` — confirm mapping is correct
- **Failure Signals**
  - `IP: 10.x.x.x` (any IP) → NOT headless — `clusterIP: None` was not set
  - `Endpoints: <none>` → no Pods match the selector or Pods are not Ready
  - Missing `Selector` → Endpoints not auto-managed

### `kubectl get endpoints <svc-name>`
- **Where to Look First**
  - IP addresses listed — if empty, selector mismatch or no ready Pods
  - Number of IPs — should match number of Ready Pods
  - Port number — must match Pod `containerPort`
- **Failure Signals**
  - Empty Endpoints → no matching **Ready** Pods
  - Fewer IPs than expected Pods → some Pods not Ready
  - With StatefulSet: Endpoints should list all StatefulSet Pods

### `kubectl logs <pod>`
- **Where to Look First**
  - Application startup logs — is the app listening on the `targetPort`?
  - Peer discovery logs — is the app finding other Pods via DNS?
  - Cluster formation logs — do distributed systems (etcd, Kafka) discover peers?
- **Failure Signals**
  - "Cannot resolve peer" / "DNS lookup failed" → Headless Service not working
  - App not listening on expected port → `targetPort` mismatch
  - Timeout connecting to peer → selector mismatch or Pod not Ready

### `kubectl exec -it <pod> -- nslookup <svc-name>`
- **Where to Look First**
  - DNS response should return **multiple A records** (one per Pod) — NOT a single ClusterIP
  - Each A record is a Pod IP
  - For StatefulSet: `nslookup <pod-name>.<svc-name>` should resolve to that specific Pod IP
- **Failure Signals**
  - Returns a **single IP** (ClusterIP) → Service is NOT headless (`clusterIP: None` missing)
  - Returns `** server can't find` → Service doesn't exist or DNS issue
  - Returns zero addresses → Endpoints empty (selector mismatch / Pods not Ready)

### `kubectl explain service.spec.clusterIP`
- **Where to Look First**
  - Confirms `clusterIP` accepts `None` as a value
  - Useful during CKA if unsure of exact syntax
  - `kubectl explain service.spec` — overview of all spec fields

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - `clusterIP: None` not set → creates a regular ClusterIP Service instead of headless
  - Selector labels don't match Pod labels → `Endpoints: <none>` → DNS returns nothing
  - `metadata.name` doesn't match StatefulSet's `spec.serviceName` → Pods have no stable DNS
  - Service type set to `NodePort` or `LoadBalancer` → gets a ClusterIP assigned → not headless
  - Headless Service in wrong namespace → StatefulSet can't find it
  - Missing `ports` section → DNS SRV records incomplete

- **Status Values to Recognize**
  - Headless Services don't have Pod-like statuses
  - Check:
    - `kubectl describe svc` → `IP: None` (confirms headless) vs an actual IP (not headless)
    - `Endpoints: <none>` → **the main red flag**
    - Pod statuses: CrashLoopBackOff, Pending → Pods not Ready → removed from Endpoints
    - `nslookup` returns single IP → not headless
    - `nslookup` returns no addresses → Endpoints empty

- **Event Messages to Recognize**
  - Headless Services rarely generate events themselves
  - Check **Pod events**: `kubectl describe pod <pod>` → Events section
  - StatefulSet events: `kubectl describe statefulset <name>` → check for Service-related errors
  - If StatefulSet's `serviceName` doesn't match → no explicit error, just **no stable DNS**

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: "StatefulSet Pods can't discover peers" → Headless Service name doesn't match `spec.serviceName`
  - Scenario 2: "`nslookup <svc-name>` returns a single IP" → `clusterIP: None` not set → regular Service
  - Scenario 3: "`nslookup <svc-name>` returns no addresses" → selector mismatch or Pods not Ready
  - Scenario 4: "DNS for `<pod-name>.<svc-name>` doesn't resolve" → Service name doesn't match StatefulSet's `serviceName`
  - Scenario 5: "Database cluster can't form — peers not found" → Headless Service selector wrong → Endpoints empty
  - Scenario 6: "Headless Service created but `kubectl describe` shows a ClusterIP" → type was set to NodePort/LoadBalancer

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch?**
  - Service selector doesn't match any Pod labels
  - Endpoints object is created but has **zero addresses**
  - DNS query for the Service name returns **no A records**
  - `nslookup <svc-name>` → "Name or service not known" or empty answer
  - StatefulSet Pod DNS (`<pod>.<svc>`) also **doesn't resolve**
  - **Silent failure** — no error/event on the Service itself
  - Distributed systems fail cluster formation — "cannot find peers"

- **What Happens If Ports Mismatch?**
  - `targetPort` doesn't match Pod's `containerPort`
  - Endpoints exist (IPs populated — label matching is fine)
  - Clients get correct Pod IPs from DNS but connect to wrong port → **connection refused**
  - DNS SRV records show wrong port → clients using SRV records connect to wrong port
  - Harder to detect than with ClusterIP — must compare ports manually

- **What Happens If Image Is Wrong?**
  - Pod goes to `ImagePullBackOff` or `ErrImagePull`
  - Pod is **not Ready** → removed from Endpoints
  - DNS returns fewer (or zero) Pod IPs
  - StatefulSet blocks rolling update — higher-ordinal Pods not created until lower ones are Ready
  - Peer discovery degrades — some peers unreachable

- **What Happens If Resource Limits Are Wrong?**
  - Pod may be `OOMKilled` → CrashLoopBackOff → not Ready → removed from Endpoints
  - DNS returns fewer Pod IPs → peer discovery returns incomplete list
  - Distributed systems may lose quorum if enough Pods are removed

- **What Happens If Dependencies Are Missing?**
  - If StatefulSet references a Headless Service that doesn't exist → **StatefulSet still creates Pods** but Pods have **no stable DNS**
  - If Pods depend on a ConfigMap/Secret that doesn't exist → Pod stuck in `ContainerCreating` → not Ready → not in Endpoints
  - If PVC can't bind (wrong storageClass) → Pod stuck in `Pending` → not Ready → not in Endpoints → DNS returns nothing
  - Missing `clusterIP: None` → regular Service → DNS returns single IP instead of Pod IPs → **most common dependency error**

## 9️⃣ Fast Detection Strategy

- **Exact kubectl Commands to Run (in order)**
  ```bash
  # Step 1: Check Service exists and clusterIP is None
  kubectl get svc <svc-name> -n <namespace>
  # CLUSTER-IP column should show "None"

  # Step 2: Describe Service — confirm IP: None and check Endpoints
  kubectl describe svc <svc-name> -n <namespace>

  # Step 3: Check Endpoints — verify Pod IPs are populated
  kubectl get endpoints <svc-name> -n <namespace>

  # Step 4: Check Pod labels match the selector
  kubectl get pods -n <namespace> --show-labels
  kubectl get pods -n <namespace> -l <selector-key>=<selector-value>

  # Step 5: Test DNS — should return MULTIPLE A records (Pod IPs)
  kubectl exec -it <any-pod> -- nslookup <svc-name>
  # Or for more detail:
  kubectl exec -it <any-pod> -- nslookup <svc-name>.<namespace>.svc.cluster.local

  # Step 6: Test per-Pod DNS (if StatefulSet)
  kubectl exec -it <any-pod> -- nslookup <pod-name>.<svc-name>.<namespace>.svc.cluster.local

  # Step 7: Check Pod readiness
  kubectl get pods -n <namespace> -o wide

  # Step 8: Check StatefulSet references correct serviceName
  kubectl get statefulset <name> -o jsonpath='{.spec.serviceName}'

  # Step 9: Check CoreDNS is healthy
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  ```

- **Order of Debugging Steps**
  1. Service exists and clusterIP is `None`? → `kubectl get svc` (CLUSTER-IP column)
  2. If CLUSTER-IP shows an IP → **not headless** → check `clusterIP: None` in YAML
  3. Endpoints populated? → `kubectl get endpoints` ← **80% of issues found here**
  4. Selector matches? → compare `describe svc` selector vs `get pods --show-labels`
  5. Pods Ready? → `kubectl get pods`
  6. DNS returns Pod IPs? → `nslookup <svc-name>` from inside a Pod
  7. Per-Pod DNS resolves? → `nslookup <pod-name>.<svc-name>` (StatefulSet only)
  8. StatefulSet `serviceName` matches? → `kubectl get sts -o yaml | grep serviceName`

- **How to Detect Issue Under 60 Seconds**
  - Run `kubectl get svc <name>` → check CLUSTER-IP column → must be `None`
  - If it shows an IP → `clusterIP: None` is missing → fix immediately
  - If `None` → `kubectl get endpoints <name>` → are Pod IPs present?
  - If Endpoints empty → selector mismatch or Pods not Ready
  - If Endpoints populated → `kubectl exec -- nslookup <svc-name>` → confirm DNS returns Pod IPs

## 🔟 Fast Fix Strategy

- **How to Fix Using kubectl**
  ```bash
  # ⚠️ IMPORTANT: clusterIP is IMMUTABLE — you cannot change it after creation
  # If clusterIP is wrong, you must DELETE and RECREATE the Service

  # Fix: Service is not headless (clusterIP was assigned)
  kubectl delete svc <svc-name>
  # Then recreate with clusterIP: None
  kubectl apply -f corrected-headless-service.yaml

  # Fix selector mismatch
  kubectl edit svc <svc-name> -n <namespace>
  # Change spec.selector to match Pod labels

  # Or fix Pod labels to match Service selector
  kubectl label pod <pod-name> app=my-app --overwrite

  # Fix port mismatch
  kubectl edit svc <svc-name>
  # Update targetPort to match Pod containerPort

  # Fix via patch (faster for CKA)
  kubectl patch svc <svc-name> -p '{"spec":{"selector":{"app":"correct-label"}}}'

  # Fix StatefulSet serviceName mismatch — must delete and recreate StatefulSet
  # serviceName is immutable
  kubectl delete sts <name> --cascade=orphan  # Keep Pods running
  # Edit YAML: fix spec.serviceName
  kubectl apply -f corrected-statefulset.yaml

  # If Pod image is wrong → fix StatefulSet
  kubectl set image statefulset/<sts-name> <container>=<correct-image>
  ```

- **How to Validate Fix**
  ```bash
  # Confirm headless (CLUSTER-IP = None)
  kubectl get svc <svc-name>

  # Check Endpoints are populated
  kubectl get endpoints <svc-name>

  # Verify DNS returns multiple Pod IPs (not a single ClusterIP)
  kubectl exec -it <pod> -- nslookup <svc-name>

  # Verify per-Pod DNS (StatefulSet)
  kubectl exec -it <pod> -- nslookup <pod-name>.<svc-name>

  # Verify Pods are Ready
  kubectl get pods -l <selector>
  ```

- **How to Confirm Recovery**
  - `kubectl get svc <name>` → CLUSTER-IP column shows `None`
  - `kubectl get endpoints <name>` → shows all Pod IPs
  - `nslookup <svc-name>` → returns multiple A records (Pod IPs)
  - `nslookup <pod-name>.<svc-name>` → resolves to specific Pod IP (StatefulSet)
  - Application logs show successful peer discovery / cluster formation

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 – Create Working Example

```yaml
# headless-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-headless
  labels:
    app: web
spec:
  clusterIP: None
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 80
---
# statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: web-headless       # ⚠️ Must match Headless Service name
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
```

- **Validation Steps**
  ```bash
  kubectl apply -f headless-service.yaml
  kubectl apply -f statefulset.yaml

  # Verify Service is headless
  kubectl get svc web-headless                     # CLUSTER-IP should show "None"

  # Verify Endpoints are populated
  kubectl get endpoints web-headless               # Should show 3 Pod IPs

  # Verify DNS returns multiple Pod IPs
  kubectl run dns-test --rm -it --image=busybox --restart=Never -- nslookup web-headless
  # Should return 3 A records

  # Verify per-Pod DNS (stable DNS for StatefulSet Pods)
  kubectl run dns-test --rm -it --image=busybox --restart=Never -- nslookup web-0.web-headless
  kubectl run dns-test --rm -it --image=busybox --restart=Never -- nslookup web-1.web-headless
  kubectl run dns-test --rm -it --image=busybox --restart=Never -- nslookup web-2.web-headless
  # Each should resolve to its respective Pod IP

  # Verify connectivity
  kubectl run test --rm -it --image=busybox --restart=Never -- wget -qO- web-0.web-headless:80
  ```

### Step 2 – Break It Intentionally

| # | What to Break | How | Expected Symptom |
|---|---|---|---|
| 1 | **Remove `clusterIP: None`** | Delete and recreate Service without `clusterIP: None` | `kubectl get svc` shows a ClusterIP → DNS returns single IP, not Pod IPs |
| 2 | **Mismatch selector** | `kubectl edit svc web-headless` → change selector to `app: wrong` | Endpoints: `<none>`, DNS returns nothing |
| 3 | **Wrong targetPort** | Change `targetPort: 9999` | DNS returns Pod IPs, but connecting to them → connection refused |
| 4 | **Mismatch serviceName** | Delete StatefulSet (--cascade=orphan), recreate with `serviceName: wrong-name` | Per-Pod DNS (`web-0.web-headless`) stops resolving |
| 5 | **Wrong Pod image** | `kubectl set image sts/web nginx=nginx:nonexistent` | Pods → ImagePullBackOff → Endpoints deplete one by one |
| 6 | **Delete Headless Service** | `kubectl delete svc web-headless` | All DNS resolution fails, per-Pod DNS gone |
| 7 | **Scale to 0** | `kubectl scale sts web --replicas=0` | Endpoints empty, DNS returns zero A records |
| 8 | **Add failing readiness probe** | Edit StatefulSet → add readiness probe for wrong path | Pods not Ready → removed from Endpoints → DNS returns nothing |
| 9 | **Wrong namespace** | Create Headless Service in `different-ns` | StatefulSet Pods in default ns have no DNS |
| 10 | **Set type to NodePort** | `kubectl edit svc` → set `type: NodePort` | ClusterIP gets assigned → no longer headless |

### Step 3 – Debug Without Looking at Solution

- **What command to run first**
  - `kubectl get svc web-headless` — check CLUSTER-IP column shows `None`
  - `kubectl get endpoints web-headless` — check Pod IPs are present
- **How to narrow the issue**
  - CLUSTER-IP shows an IP? → Service is NOT headless → delete and recreate with `clusterIP: None`
  - CLUSTER-IP is `None` but Endpoints empty? → `kubectl describe svc` → check Selector → compare with `kubectl get pods --show-labels`
  - Endpoints populated but per-Pod DNS fails? → check `kubectl get sts -o yaml | grep serviceName` → must match Service name
  - Endpoints populated and DNS works but connection fails? → check `targetPort` vs `containerPort`
- **How to confirm root cause**
  - Not headless: `kubectl get svc` CLUSTER-IP column has an IP address
  - Selector mismatch: `kubectl get pods -l app=web` returns 0 results but Pods exist with other labels
  - serviceName mismatch: `kubectl get sts web -o jsonpath='{.spec.serviceName}'` doesn't match Service name
  - Port mismatch: `kubectl describe svc` targetPort vs `kubectl describe pod` containerPort differ

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - Headless Service creation (`clusterIP: None`)
  - Difference: regular Service DNS (single A record) vs Headless DNS (multiple A records)
  - StatefulSet + Headless Service pairing for stable Pod DNS
  - Per-Pod DNS format: `<pod-name>.<svc-name>.<ns>.svc.cluster.local`
  - Endpoints auto-population (same mechanism as ClusterIP)
  - Headless without selector (manual Endpoints — rare but testable)

- **What to Focus On**
  - Writing correct Headless Service YAML (don't forget `clusterIP: None`)
  - Matching `metadata.name` with StatefulSet's `spec.serviceName`
  - Verifying DNS behavior: `nslookup` from inside a Pod → expect multiple A records
  - Verifying per-Pod DNS for StatefulSets
  - Understanding that `clusterIP` is **immutable** — must delete/recreate to fix
  - Debugging Endpoints (same skills as ClusterIP)

- **What to Practice Repeatedly**
  - Create a StatefulSet + Headless Service from scratch in under 3 minutes
  - Verify headless behavior using `nslookup` (multiple A records vs single)
  - Debug per-Pod DNS issues: check `serviceName` match
  - Fix a non-headless Service by deleting and recreating with `clusterIP: None`
  - Compare `kubectl get svc` output: regular vs headless (CLUSTER-IP column)

- **What to Ignore (Deep Internals Not Required)**
  - CoreDNS internal config for headless Services
  - EndpointSlice internal mechanics
  - `publishNotReadyAddresses` deep behavior (know it exists, not the details)
  - DNS TTL values for Pod A records
  - Custom DNS policies / dnsConfig

- **5 Quiz Questions**
  1. **Q:** What makes a Service "headless"?
     - **A:** Setting `spec.clusterIP: None`. Kubernetes doesn't allocate a virtual IP; DNS returns individual Pod IPs instead.
  2. **Q:** A StatefulSet's `spec.serviceName` is set to `db-svc` but the Headless Service is named `database`. What happens?
     - **A:** StatefulSet Pods are created but don't get stable per-Pod DNS names. `nslookup <pod-name>.db-svc` doesn't resolve. Fix: rename the Service to `db-svc` or update StatefulSet's `serviceName`.
  3. **Q:** `kubectl get svc my-svc` shows `CLUSTER-IP = 10.96.0.15`. Is this headless?
     - **A:** No. A headless Service shows `None` in the CLUSTER-IP column. This is a regular ClusterIP Service.
  4. **Q:** You realize a Service should be headless but it was created with a ClusterIP. Can you `kubectl edit` to add `clusterIP: None`?
     - **A:** No. `clusterIP` is **immutable** after creation. You must delete the Service and recreate it with `clusterIP: None`.
  5. **Q:** `nslookup my-headless-svc` from inside a Pod returns no addresses. Endpoints show `<none>`. What's wrong?
     - **A:** The Service selector doesn't match any Pod labels, or no Pods are in Ready state. Check: `kubectl describe svc` → Selector → compare with `kubectl get pods --show-labels`.

## 1️⃣3️⃣ Component Interaction Map

```
                              ┌─────────────┐
                              │   CoreDNS   │
                              │ (A records) │
                              └──────┬──────┘
                                     │ returns Pod IPs directly
                                     │ (NOT a single ClusterIP)
                                     ▼
┌──────────────┐   nslookup / DNS   ┌───────────────────┐
│   Consumer   │ ──────────────────►│  Headless Service  │
│   Pod        │                    │  (clusterIP: None) │
└──────┬───────┘                    └─────────┬─────────┘
       │                                      │
       │   direct Pod IP connection           ▼
       │                           ┌──────────────────┐
       └──────────────────────────►│    Pod(s)        │
           (no proxy layer)        │  :targetPort     │
                                   └──────────────────┘
                                            ▲
                                   ┌──────────────────┐
                                   │   Endpoints /     │
                                   │  EndpointSlices   │
                                   │  (auto-managed)   │
                                   └──────────────────┘
```

- **Pods**
  - Service selects Pods via labels → Pod IPs populate Endpoints
  - Only **Ready** Pods appear in Endpoints → appear in DNS
  - Each Pod IP becomes a separate DNS A record
  - With StatefulSet: each Pod gets a stable DNS name `<pod-name>.<svc-name>.<ns>.svc.cluster.local`
  - Pod deletion → auto-removed from Endpoints → removed from DNS
  - Pod recreation (StatefulSet) → same name, new IP → DNS updated automatically

- **Services ↔ Other Services**
  - Headless Services do NOT route through kube-proxy — no iptables/IPVS rules
  - A regular ClusterIP/NodePort Service can coexist alongside a Headless Service for the same Pods
  - Common pattern: Headless Service for peer discovery + ClusterIP Service for client access

- **Nodes**
  - kube-proxy does **NOT** program any rules for Headless Services (no virtual IP to route)
  - Traffic goes directly from consumer Pod to target Pod via CNI networking
  - Node-to-Node connectivity (CNI) is critical — traffic is direct, not proxied

- **Scheduler**
  - No direct interaction — Scheduler places Pods; Service discovers them via labels
  - If Scheduler can't place a Pod → Pod stays Pending → not in Endpoints → missing from DNS
  - For StatefulSet: Pods are created in order; if Pod-0 can't be scheduled, Pod-1 is never created

- **Controller Manager**
  - **Endpoints Controller** watches Pod labels and readiness → populates Endpoints
  - **StatefulSet Controller** manages Pod identity and ordering
  - If Controller Manager is down → Endpoints become stale; new Pods won't appear in DNS

- **etcd**
  - Stores Service, Endpoints, and EndpointSlice objects
  - Stores StatefulSet state (ordinal assignments, Pod identities)
  - If etcd is down → no new Services or Pods can be created

- **Network Layer**
  - **No kube-proxy involvement** — traffic goes directly to Pod IPs
  - CNI plugin → provides Pod-to-Pod networking (critical since traffic is direct)
  - NetworkPolicies → can block traffic to/from Pods behind a Headless Service
  - DNS resolution via CoreDNS is the **only** discovery mechanism (no virtual IP)

- **Storage Layer**
  - StatefulSet + Headless Service often paired with `volumeClaimTemplates` → each Pod gets its own PVC
  - If PVC can't bind → Pod stays Pending → not in Endpoints → missing from DNS
  - PV/PVC issues cascade: no storage → no Pod → no Endpoint → no DNS record

- **Interaction Behavior When Things Go Wrong**
  - CoreDNS down → **total failure** — headless depends entirely on DNS (no virtual IP fallback)
  - Endpoints Controller down → Endpoints become stale; new/deleted Pods not reflected in DNS
  - CNI plugin issues → Pod-to-Pod networking fails → clients have correct Pod IPs but can't reach them
  - StatefulSet Controller down → Pods not managed; existing DNS entries may become stale
  - Regular ClusterIP Service for same Pods → still works (kube-proxy path) even if Headless is broken

## ⚡ Quick Reference Cheat Sheet

```bash
# Create Headless Service (no imperative shortcut — must use YAML or set clusterIP)
# Imperative workaround:
kubectl create service clusterip my-headless --clusterip="None" --tcp=80:80

# Verify headless (CLUSTER-IP column must show "None")
kubectl get svc <name>

# Check Endpoints
kubectl get endpoints <name>

# Describe Service (confirm IP: None)
kubectl describe svc <name>

# Test DNS — should return MULTIPLE A records (Pod IPs)
kubectl exec -it <pod> -- nslookup <svc-name>

# Test per-Pod DNS (StatefulSet)
kubectl exec -it <pod> -- nslookup <pod-name>.<svc-name>

# Check StatefulSet serviceName
kubectl get sts <name> -o jsonpath='{.spec.serviceName}'

# Quick selector check
kubectl get pods -l <key>=<value>

# ⚠️ clusterIP is IMMUTABLE — to fix, delete and recreate
kubectl delete svc <name>
kubectl apply -f corrected-headless.yaml

# Patch selector (works)
kubectl patch svc <name> -p '{"spec":{"selector":{"app":"correct"}}}'

# Patch targetPort (works)
kubectl patch svc <name> --type='json' -p='[{"op":"replace","path":"/spec/ports/0/targetPort","value":8080}]'

# Quick DNS comparison: headless vs regular
kubectl exec -it <pod> -- nslookup <headless-svc>     # Multiple A records
kubectl exec -it <pod> -- nslookup <regular-svc>      # Single A record (ClusterIP)

# Create StatefulSet + Headless Service pair
# Remember: sts.spec.serviceName MUST match headless Service metadata.name
```
