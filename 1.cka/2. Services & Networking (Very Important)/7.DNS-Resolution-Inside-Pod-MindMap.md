# 🔍 DNS Resolution Inside Pod — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Enables Pods to resolve Service names, Pod names, and external hostnames to IP addresses
  - Every Pod gets a `/etc/resolv.conf` file injected by kubelet at creation time
  - This file tells the Pod WHERE to send DNS queries (nameserver) and HOW to search (search domains)
  - Without DNS resolution, Pods must use raw IP addresses — no service discovery

- **Real-World Use Case**
  - Pod calls `http://backend-svc:8080` → `/etc/resolv.conf` search domains expand it → `backend-svc.default.svc.cluster.local` → resolved by CoreDNS → returns ClusterIP
  - Cross-namespace: `http://db-svc.production:5432` → resolves via search domain → `db-svc.production.svc.cluster.local`
  - External: `https://api.github.com` → CoreDNS forwards to upstream DNS → returns public IP
  - Headless Service: `nslookup my-headless-svc` → returns individual Pod IPs (not a ClusterIP)

- **Where It Fits in Cluster Architecture**
  - Pod-level concern — each Pod has its OWN DNS configuration
  - Kubelet injects DNS settings based on Pod's `dnsPolicy` and optional `dnsConfig`
  - DNS query path: `Pod → /etc/resolv.conf → nameserver (kube-dns ClusterIP) → CoreDNS Pods → response`
  - Controlled by three things: **dnsPolicy**, **dnsConfig**, and **CoreDNS configuration**
  - Part of the "Services & Networking" CKA domain — appears in troubleshooting scenarios frequently

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - **Every application Pod** using Service names to communicate
  - Init containers resolving dependencies before main container starts
  - Sidecar containers (e.g., Envoy, Fluentd) resolving Service endpoints
  - Health checks using HTTP probes with hostnames (rare, but possible)
  - Any Pod code doing DNS-based service discovery

- **What Resources It Interacts With**
  - **CoreDNS Deployment** — the actual DNS server answering queries
  - **kube-dns Service** — ClusterIP that Pods send DNS queries to (usually `10.96.0.10`)
  - **kubelet** — injects `/etc/resolv.conf` into each Pod based on `dnsPolicy`
  - **Pod spec (`dnsPolicy`, `dnsConfig`)** — controls DNS behavior per-Pod
  - **Services** — the objects being resolved (ClusterIP, Headless, ExternalName)
  - **Endpoints / EndpointSlices** — CoreDNS reads these to build DNS records

- **Upstream Dependencies**
  - kubelet must be running and configured with `--cluster-dns` and `--cluster-domain`
  - CoreDNS Pods must be running and healthy
  - `kube-dns` Service must exist with correct ClusterIP
  - CNI must be functional (Pod network connectivity to kube-dns Service)
  - kube-proxy must route traffic to kube-dns ClusterIP correctly

- **Downstream Dependencies**
  - All inter-Pod communication via Service names
  - External API calls from Pods
  - Any hostname-based configuration inside containers

## 3️⃣ Recognize Structure (YAML Skeleton)

### Pod with DNS Configuration

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: dns-example
  namespace: default
spec:
  dnsPolicy: ClusterFirst              # Controls which DNS server the Pod uses
  dnsConfig:                           # Optional — fine-tune DNS settings
    nameservers:
      - 8.8.8.8                        # Additional nameserver (max 3 total)
    searches:
      - my-custom.svc.cluster.local    # Additional search domain
      - my-custom.local
    options:
      - name: ndots                    # Number of dots before absolute lookup
        value: "5"
      - name: timeout
        value: "3"
      - name: attempts
        value: "2"
  containers:
    - name: app
      image: busybox:1.36
      command: ["sleep", "3600"]
```

### Resulting /etc/resolv.conf Inside the Pod

```
nameserver 10.96.0.10                  # kube-dns Service ClusterIP
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

- **Key Sections Explained**
  - `dnsPolicy` — the MOST important field — determines DNS behavior entirely
    - `ClusterFirst` (default) — use CoreDNS for everything, forward external queries
    - `Default` — use Node's DNS settings (NOT CoreDNS) — breaks cluster Service resolution
    - `ClusterFirstWithHostNet` — like `ClusterFirst` but for Pods using `hostNetwork: true`
    - `None` — no auto-configuration; YOU must provide `dnsConfig` manually
  - `dnsConfig` — optional overrides ADDED to the auto-generated resolv.conf
    - `nameservers` — extra DNS servers (appended, max 3 total including auto-injected)
    - `searches` — extra search domains (appended, max 6 total, 256 chars)
    - `options` — DNS resolver options like `ndots`, `timeout`, `attempts`
  - `ndots:5` — if a name has fewer than 5 dots, the search domains are tried FIRST before treating it as absolute
    - This is why `nslookup my-svc` → tries `my-svc.default.svc.cluster.local` first ✅

- **Common Indentation Traps**
  - `dnsPolicy` is at `spec` level (same level as `containers`) — NOT inside containers
  - `dnsConfig` is also at `spec` level — NOT inside containers
  - `dnsConfig.nameservers` is an array — each entry prefixed with `-`
  - `dnsConfig.options` items are objects with `name` and optional `value` (value is a string, not int)
  - `dnsPolicy: None` WITHOUT `dnsConfig` → Pod has empty resolv.conf → NO DNS at all

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `dnsPolicy` | Controls entire DNS behavior of the Pod | `Default` → can't resolve cluster Services; `None` without config → no DNS |
| `dnsConfig.nameservers` | Adds custom DNS servers | Wrong IP → DNS queries to that server timeout |
| `dnsConfig.searches` | Custom search domains | Too many → exceeds 6-domain limit → error; wrong domain → NXDOMAIN |
| `dnsConfig.options[ndots]` | Controls when search domains are used | Too low → short names treated as absolute → NXDOMAIN; too high → unnecessary lookups |
| `hostNetwork: true` | Pod uses Node's network namespace | Without `ClusterFirstWithHostNet`, Pod uses Node DNS, not CoreDNS |
| `/etc/resolv.conf` (resultant) | Actual DNS config used at runtime | Wrong nameserver IP → all DNS fails |

- **Field Relationships**
  - `dnsPolicy: ClusterFirst` + kubelet `--cluster-dns=10.96.0.10` → Pod gets `nameserver 10.96.0.10`
  - `dnsPolicy: ClusterFirst` + Pod in namespace `prod` → search domains include `prod.svc.cluster.local`
  - `dnsPolicy: None` requires `dnsConfig` → otherwise Pod has NO DNS configuration at all
  - `dnsPolicy: Default` → uses Node's `/etc/resolv.conf` → bypasses CoreDNS
  - `hostNetwork: true` + `dnsPolicy: ClusterFirst` → actually behaves like `Default` (uses Node DNS!) — need `ClusterFirstWithHostNet`
  - `ndots:5` + name like `my-svc` (0 dots, < 5) → search domains appended first before absolute lookup

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - `dnsPolicy` — is it set? What value? (`ClusterFirst` is correct for most cases)
  - `hostNetwork: true` present? → then `dnsPolicy` must be `ClusterFirstWithHostNet`
  - `dnsConfig` present? → if `dnsPolicy: None`, is `dnsConfig` actually provided?
  - Any custom `nameservers` or `searches` that override cluster defaults?

- **Usually Irrelevant for CKA**
  - Detailed `ndots` tuning for performance optimization
  - Custom `timeout` and `attempts` values
  - `single-request` or `single-request-reopen` options
  - EDNS0 options

- **Red Flags to Detect in 10 Seconds**
  - ❌ `dnsPolicy: Default` on a Pod that needs to resolve cluster Services
  - ❌ `dnsPolicy: None` without any `dnsConfig` specified
  - ❌ `hostNetwork: true` with `dnsPolicy: ClusterFirst` (should be `ClusterFirstWithHostNet`)
  - ❌ `dnsConfig.nameservers` pointing to a non-existent DNS server
  - ❌ `ndots: "0"` → short Service names treated as absolute → NXDOMAIN for `my-svc`
  - ❌ Missing `dnsPolicy` entirely on a Pod that can't resolve DNS (defaults to `ClusterFirst`, but worth verifying)

## 6️⃣ What to Notice In

### kubectl describe pod \<name\>

- **What Signals Failure**
  - `DNS Policy: Default` on a Pod that should use cluster DNS
  - Events showing container failures related to hostname resolution
  - Init container stuck → possibly waiting for a Service name to resolve
  - Container `CrashLoopBackOff` with exit code related to connection/resolution failure

- **Where to Look First**
  - `DNS Policy:` line — confirms which DNS mode is active
  - `DNS Config:` section — shows any custom DNS overrides
  - Events — any scheduling or network-related issues

- **Fields That Matter Most**
  - `DNS Policy`
  - `DNS Config` (nameservers, searches, options)
  - `Host Network` — true/false
  - Container status and restart count

### kubectl logs \<pod\>

- **What Signals Failure**
  - `dial tcp: lookup <service-name>: no such host` → DNS resolution failing
  - `could not resolve host` → DNS not working for that name
  - `getaddrinfo ENOTFOUND` (Node.js) → DNS failure
  - `Name or service not known` → resolv.conf misconfigured
  - Connection timeouts to Service names → DNS working but Service unreachable (different issue)

- **Where to Look First**
  - Error messages containing hostnames or "DNS" or "resolve"
  - Connection errors to other Services by name
  - Application startup failures mentioning database or API hosts

- **Fields That Matter Most**
  - The hostname that failed to resolve
  - Whether it's a cluster-internal name or external name
  - Timing — fails immediately (NXDOMAIN) vs timeout (nameserver unreachable)

### kubectl exec (The Primary DNS Debug Tool)

- **Key Commands**
  ```bash
  # Check Pod's DNS configuration
  kubectl exec <pod> -- cat /etc/resolv.conf

  # Test internal DNS
  kubectl exec <pod> -- nslookup kubernetes.default
  kubectl exec <pod> -- nslookup <service>.<namespace>.svc.cluster.local

  # Test short name resolution (uses search domains)
  kubectl exec <pod> -- nslookup <service-name>

  # Test external DNS
  kubectl exec <pod> -- nslookup google.com

  # Test with specific DNS server
  kubectl exec <pod> -- nslookup kubernetes.default 10.96.0.10

  # Detailed DNS query (if dig available)
  kubectl exec <pod> -- dig <service>.default.svc.cluster.local

  # Check DNS response time
  kubectl exec <pod> -- time nslookup kubernetes.default
  ```

- **What Signals Failure**
  - `** server can't find ...: NXDOMAIN` → name doesn't exist or search domains wrong
  - `** server can't find ...: SERVFAIL` → CoreDNS itself is broken
  - `;; connection timed out; no servers could be reached` → nameserver unreachable (CoreDNS down or wrong IP in resolv.conf)
  - `nameserver` in resolv.conf is NOT the kube-dns ClusterIP → wrong `dnsPolicy`
  - Missing search domains → short names won't resolve

### kubectl explain pod.spec.dnsPolicy / pod.spec.dnsConfig

- **Key Usage**
  ```bash
  kubectl explain pod.spec.dnsPolicy
  kubectl explain pod.spec.dnsConfig
  kubectl explain pod.spec.dnsConfig.nameservers
  kubectl explain pod.spec.dnsConfig.searches
  kubectl explain pod.spec.dnsConfig.options
  ```
- Quickly check allowed `dnsPolicy` values during exam
- Verify `dnsConfig` structure and field names
- Confirm that `dnsConfig.options[].value` is string type

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ `dnsPolicy: Default` → Pod uses Node DNS → can't resolve `my-svc.default.svc.cluster.local`
  - ❌ `dnsPolicy: None` without `dnsConfig` → empty resolv.conf → no DNS at all
  - ❌ `hostNetwork: true` + `dnsPolicy: ClusterFirst` → silently uses Node DNS → cluster names fail
  - ❌ Custom `nameservers` in `dnsConfig` overriding CoreDNS → internal names unresolvable
  - ❌ `ndots: "0"` → `nslookup my-svc` treated as absolute FQDN → NXDOMAIN (search domains skipped)
  - ❌ Too many search domains (>6) → Pod creation fails
  - ❌ CoreDNS down → all Pods lose DNS resolution
  - ❌ kube-dns Service missing → nameserver IP unreachable → DNS timeout

- **Status Values / Behaviors**
  - Pod itself is `Running` — DNS misconfiguration does NOT crash the Pod
  - Application inside the Pod crashes → `CrashLoopBackOff` due to unresolvable dependencies
  - Init container hangs → waiting for DNS to resolve a Service name
  - Pod `Running` but app returns errors → "no such host" in logs

- **Event Messages to Recognize**
  - DNS issues rarely show in Pod Events — they manifest as **application errors in logs**
  - Container restart events due to failed health checks (if probes use Service names)
  - Init container timeout events (stuck resolving names)

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: Pod can't connect to a Service by name, but CAN connect by IP → `dnsPolicy: Default` or CoreDNS is down
  - Scenario 2: Pod A resolves Services, Pod B doesn't → check Pod B's `dnsPolicy` (likely `Default` or `None`)
  - Scenario 3: Pod with `hostNetwork: true` can't resolve cluster Services → needs `ClusterFirstWithHostNet`
  - Scenario 4: All Pods in cluster fail DNS → CoreDNS Pods are down or `kube-dns` Service missing
  - Scenario 5: Pod resolves `my-svc.default.svc.cluster.local` but NOT `my-svc` → `ndots` set too low or search domains missing

## 8️⃣ Misconfiguration Behavior

- **What Happens If `dnsPolicy` Is Wrong?**
  - `Default` instead of `ClusterFirst`:
    - Pod's `/etc/resolv.conf` = Node's `/etc/resolv.conf`
    - `nameserver` = Node's upstream DNS (e.g., `8.8.8.8` or ISP DNS)
    - External names resolve ✅ — `google.com` works
    - Cluster names fail ❌ — `my-svc.default.svc.cluster.local` = NXDOMAIN
    - **This is the SNEAKIEST failure** — Pod is Running, external works, only internal fails
  - `None` without `dnsConfig`:
    - Pod's `/etc/resolv.conf` is empty
    - ALL DNS resolution fails — both internal and external
    - App crashes immediately on any name lookup

- **What Happens If `hostNetwork: true` With Wrong Policy?**
  - `hostNetwork: true` + `dnsPolicy: ClusterFirst` → Kubernetes silently treats it as `Default`
  - Pod gets Node's resolv.conf → can't resolve cluster Services
  - Fix: change to `dnsPolicy: ClusterFirstWithHostNet`
  - This is a common CKA trap — the Pod spec looks correct but DNS fails

- **What Happens If `ndots` Is Misconfigured?**
  - `ndots: 5` (default) → name `my-svc` has 0 dots (< 5) → search domains tried FIRST
    - Resolves as `my-svc.default.svc.cluster.local` ✅
  - `ndots: 0` → name `my-svc` has 0 dots (≥ 0) → treated as absolute FQDN first
    - Tries `my-svc.` as-is → NXDOMAIN → then falls back to search domains (slower, may fail)
  - `ndots: 1` → name `my-svc` has 0 dots (< 1) → search domains tried first ✅
    - But `my-svc.production` (1 dot, ≥ 1) → treated as absolute → may fail before search

- **What Happens If Nameserver Is Wrong in `dnsConfig`?**
  - Custom `nameservers: ["1.2.3.4"]` where 1.2.3.4 doesn't exist
  - DNS queries to that server timeout (5-30 seconds delay)
  - If it replaces CoreDNS nameserver → all cluster DNS fails
  - If it's additional → queries may fall through to correct server (slower)

- **What Happens If CoreDNS Dependencies Are Missing?**
  - CoreDNS Pods down → nameserver (10.96.0.10) unreachable → DNS timeout in ALL Pods
  - `kube-dns` Service deleted → nameserver IP has no backend → same timeout
  - CoreDNS ConfigMap syntax error → CoreDNS CrashLoop → same timeout
  - CNI broken → CoreDNS Pods exist but unreachable → same timeout

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. Check the failing Pod's DNS configuration
  kubectl exec <pod> -- cat /etc/resolv.conf

  # 2. Test internal DNS from the Pod
  kubectl exec <pod> -- nslookup kubernetes.default

  # 3. Test external DNS from the Pod
  kubectl exec <pod> -- nslookup google.com

  # 4. Check Pod's dnsPolicy
  kubectl get pod <pod> -o jsonpath='{.spec.dnsPolicy}{"\n"}'

  # 5. Check if hostNetwork is true
  kubectl get pod <pod> -o jsonpath='{.spec.hostNetwork}{"\n"}'

  # 6. Test DNS from a KNOWN-GOOD Pod (isolate Pod-specific vs cluster-wide)
  kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never \
    -- nslookup kubernetes.default

  # 7. Check CoreDNS health (if cluster-wide issue)
  kubectl get pods -n kube-system -l k8s-app=kube-dns

  # 8. Check kube-dns Service
  kubectl get svc kube-dns -n kube-system

  # 9. Check CoreDNS logs
  kubectl logs -n kube-system -l k8s-app=kube-dns --tail=20
  ```

- **Order of Debugging Steps**
  1. Is it Pod-specific or cluster-wide? → test from failing Pod AND a new debug Pod
  2. Pod-specific → check `dnsPolicy`, `hostNetwork`, `dnsConfig` on that Pod
  3. Cluster-wide → check CoreDNS Pods, kube-dns Service, CoreDNS logs
  4. Internal-only failure → `dnsPolicy: Default` or CoreDNS `kubernetes` plugin issue
  5. External-only failure → CoreDNS `forward` plugin issue
  6. Both fail → CoreDNS down OR nameserver unreachable

- **How to Detect Issue Under 60 Seconds**
  ```bash
  # 30-second path:
  kubectl exec <failing-pod> -- cat /etc/resolv.conf
  # → nameserver NOT 10.96.0.10? → dnsPolicy issue → FIX THE POD

  kubectl exec <failing-pod> -- nslookup kubernetes.default
  # → NXDOMAIN? → search domains wrong or dnsPolicy wrong
  # → Timeout? → CoreDNS unreachable → check CoreDNS Pods

  kubectl get pods -n kube-system -l k8s-app=kube-dns
  # → Not Running? → CoreDNS is the problem
  ```

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix wrong dnsPolicy (requires Pod recreation)
  # Option 1: If managed by Deployment
  kubectl edit deployment <deploy-name>
  # → Change dnsPolicy to ClusterFirst (or ClusterFirstWithHostNet if hostNetwork: true)

  # Option 2: Standalone Pod — delete and recreate with correct spec
  kubectl get pod <pod> -o yaml > /tmp/pod-fix.yaml
  # Edit /tmp/pod-fix.yaml → change dnsPolicy
  kubectl delete pod <pod>
  kubectl apply -f /tmp/pod-fix.yaml

  # Fix hostNetwork Pod DNS
  kubectl edit deployment <deploy>
  # → Add: dnsPolicy: ClusterFirstWithHostNet

  # Add custom DNS config
  kubectl patch deployment <deploy> --type=merge -p '{
    "spec": {"template": {"spec": {
      "dnsConfig": {
        "nameservers": ["10.96.0.10"],
        "searches": ["default.svc.cluster.local", "svc.cluster.local", "cluster.local"]
      }
    }}}
  }'

  # Fix CoreDNS if that's the root cause
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  kubectl rollout restart deployment coredns -n kube-system

  # Fix CoreDNS ConfigMap
  kubectl edit cm coredns -n kube-system
  ```

- **Validate Fix**
  ```bash
  # Confirm Pod's resolv.conf is correct now
  kubectl exec <pod> -- cat /etc/resolv.conf
  # → Should show: nameserver 10.96.0.10
  # → Should show: search <ns>.svc.cluster.local svc.cluster.local cluster.local

  # Test internal DNS
  kubectl exec <pod> -- nslookup kubernetes.default
  # → Should return API Server ClusterIP

  # Test a specific Service
  kubectl exec <pod> -- nslookup <service-name>
  # → Should return the Service ClusterIP

  # Test external DNS
  kubectl exec <pod> -- nslookup google.com
  # → Should return IPs

  # Verify dnsPolicy
  kubectl get pod <pod> -o jsonpath='{.spec.dnsPolicy}'
  # → Should show ClusterFirst (or ClusterFirstWithHostNet)
  ```

- **Confirm Recovery**
  - `/etc/resolv.conf` shows correct nameserver (kube-dns ClusterIP) ✅
  - Search domains include `<namespace>.svc.cluster.local` ✅
  - `nslookup kubernetes.default` succeeds ✅
  - Application logs no longer show "no such host" or resolution errors ✅
  - Pod is Running without restarts ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

```yaml
# Pod with default DNS (ClusterFirst)
apiVersion: v1
kind: Pod
metadata:
  name: dns-default
  namespace: default
spec:
  containers:
    - name: test
      image: busybox:1.36
      command: ["sleep", "3600"]
---
# Pod with explicit dnsConfig
apiVersion: v1
kind: Pod
metadata:
  name: dns-custom
  namespace: default
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
      - 10.96.0.10
    searches:
      - default.svc.cluster.local
      - svc.cluster.local
      - cluster.local
    options:
      - name: ndots
        value: "5"
  containers:
    - name: test
      image: busybox:1.36
      command: ["sleep", "3600"]
---
# Pod with hostNetwork
apiVersion: v1
kind: Pod
metadata:
  name: dns-hostnet
  namespace: default
spec:
  hostNetwork: true
  dnsPolicy: ClusterFirstWithHostNet
  containers:
    - name: test
      image: busybox:1.36
      command: ["sleep", "3600"]
```

- **Validation Steps**
  ```bash
  kubectl apply -f dns-pods.yaml

  # Check default Pod
  kubectl exec dns-default -- cat /etc/resolv.conf
  kubectl exec dns-default -- nslookup kubernetes.default
  # → nameserver should be kube-dns ClusterIP, nslookup should succeed

  # Check custom Pod
  kubectl exec dns-custom -- cat /etc/resolv.conf
  kubectl exec dns-custom -- nslookup kubernetes.default
  # → resolv.conf should match dnsConfig, nslookup should succeed

  # Check hostNetwork Pod
  kubectl exec dns-hostnet -- cat /etc/resolv.conf
  kubectl exec dns-hostnet -- nslookup kubernetes.default
  # → Should use CoreDNS (because ClusterFirstWithHostNet), nslookup should succeed
  ```

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Set `dnsPolicy: Default` | Edit Pod spec | External DNS works, cluster Service names fail (NXDOMAIN) |
| Set `dnsPolicy: None` without `dnsConfig` | Edit Pod spec | ALL DNS fails — empty resolv.conf |
| `hostNetwork: true` + `ClusterFirst` | Edit Pod spec | Silently uses Node DNS → cluster names fail |
| Set `ndots: "0"` | Add to `dnsConfig.options` | Short names like `my-svc` may fail (depends on resolver behavior) |
| Wrong nameserver in `dnsConfig` | `nameservers: ["1.2.3.4"]` | DNS queries timeout (5-30s delay) |
| Remove search domains | `dnsPolicy: None` + only `nameservers` | Short names fail; only FQDNs work (`my-svc.default.svc.cluster.local`) |
| Kill CoreDNS | `kubectl scale deploy coredns -n kube-system --replicas=0` | ALL Pods lose DNS resolution — cluster-wide outage |
| Delete kube-dns Service | `kubectl delete svc kube-dns -n kube-system` | Nameserver IP unreachable → DNS timeout everywhere |
| Add too many search domains | `dnsPolicy: None` + 8 search domains | Pod creation may fail (exceeds limit) |
| Wrong search domain | Replace `cluster.local` with `cluster.wrong` | Service names → NXDOMAIN |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl exec <pod> -- cat /etc/resolv.conf
  kubectl exec <pod> -- nslookup kubernetes.default
  ```

- **How to Narrow Issue**
  1. Is resolv.conf correct? → nameserver = kube-dns IP? search domains present?
  2. Yes → test DNS: `nslookup kubernetes.default`
  3. NXDOMAIN → wrong search domains or CoreDNS misconfigured
  4. Timeout → nameserver unreachable → CoreDNS down or kube-dns Service missing
  5. Only this Pod fails? → `dnsPolicy` issue on this Pod
  6. ALL Pods fail? → CoreDNS / kube-dns issue

- **How to Confirm Root Cause**
  - Compare `/etc/resolv.conf` of failing Pod vs working Pod
  - Check `dnsPolicy` of failing Pod: `kubectl get pod <pod> -o yaml | grep dnsPolicy`
  - If resolv.conf is wrong → fix `dnsPolicy` or `dnsConfig`
  - If resolv.conf is correct but DNS fails → CoreDNS is the problem

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - All four `dnsPolicy` values and their exact behavior
  - How kubelet populates `/etc/resolv.conf` using `--cluster-dns` and `--cluster-domain`
  - DNS search domain expansion: `my-svc` → `my-svc.default.svc.cluster.local`
  - `ndots` behavior and why it defaults to 5
  - `dnsConfig` fields: `nameservers`, `searches`, `options`

- **What to Focus On**
  - Diagnosing Pod-specific DNS failures vs cluster-wide DNS failures
  - Fixing `dnsPolicy` to restore DNS for a specific Pod
  - Using `kubectl exec` + `nslookup` + `cat /etc/resolv.conf` as primary debug tools
  - `hostNetwork: true` → `ClusterFirstWithHostNet` requirement
  - `dnsPolicy: None` + `dnsConfig` for full manual control scenarios

- **What to Practice Repeatedly**
  - Create Pods with each `dnsPolicy` → verify `/etc/resolv.conf` → test DNS
  - Break DNS on a single Pod → detect → fix → validate
  - Differentiate "Pod DNS issue" vs "CoreDNS issue" in under 30 seconds
  - Write `dnsConfig` from memory using `kubectl explain`
  - Debug `hostNetwork` Pods with DNS failures

- **What to Ignore (Deep Internals)**
  - DNS caching strategies and TTL optimization
  - glibc/musl resolver implementation differences
  - `single-request` / `single-request-reopen` options
  - EDNS and DNS packet size tuning
  - Application-level DNS caching (JVM, etc.)

- **5 Quiz Questions**
  1. A Pod with `dnsPolicy: Default` cannot resolve `my-svc`. External DNS works fine. Why?
     → *`dnsPolicy: Default` uses the Node's DNS, not CoreDNS. Cluster Service names are only resolved by CoreDNS. Fix by changing to `ClusterFirst`.*
  2. What does the `ndots:5` option in `/etc/resolv.conf` mean?
     → *If a DNS query has fewer than 5 dots, the search domains from `/etc/resolv.conf` are appended FIRST before trying the name as an absolute FQDN.*
  3. A Pod with `hostNetwork: true` and `dnsPolicy: ClusterFirst` can't resolve cluster Service names. Why?
     → *When `hostNetwork: true`, `ClusterFirst` behaves like `Default`. Must use `ClusterFirstWithHostNet` to get CoreDNS.*
  4. A Pod has `dnsPolicy: None` and no `dnsConfig`. What happens?
     → *The Pod gets an empty `/etc/resolv.conf`. ALL DNS resolution fails — both internal and external.*
  5. What are the search domains in a default Pod's `/etc/resolv.conf` in namespace `prod`?
     → *`prod.svc.cluster.local svc.cluster.local cluster.local`*

## 1️⃣3️⃣ Component Interaction Map

- **DNS Resolution ↔ Pods**
  - Every Pod has `/etc/resolv.conf` injected by kubelet
  - DNS behavior controlled by Pod's `dnsPolicy` and `dnsConfig`
  - **When things go wrong:** Wrong `dnsPolicy` → wrong resolv.conf → DNS fails for that specific Pod only; other Pods unaffected

- **DNS Resolution ↔ Services**
  - Services get DNS A records in CoreDNS: `<name>.<ns>.svc.cluster.local`
  - ClusterIP Services → DNS returns the ClusterIP
  - Headless Services → DNS returns Pod IPs
  - ExternalName Services → DNS returns a CNAME record
  - **When things go wrong:** Service doesn't exist → NXDOMAIN; Service exists but no Endpoints → DNS resolves but connection fails

- **DNS Resolution ↔ Nodes**
  - kubelet on each Node configures Pod DNS via `--cluster-dns` and `--cluster-domain`
  - Node's `/etc/resolv.conf` used when `dnsPolicy: Default`
  - Node's `/etc/resolv.conf` used by CoreDNS for upstream forwarding
  - **When things go wrong:** kubelet misconfigured `--cluster-dns` → ALL Pods on that Node get wrong nameserver; Node's resolv.conf points to cluster DNS → loop

- **DNS Resolution ↔ Scheduler**
  - Scheduler places CoreDNS Pods on Nodes
  - No direct interaction with Pod DNS configuration
  - **When things go wrong:** CoreDNS Pods can't be scheduled → no DNS cluster-wide

- **DNS Resolution ↔ Controller Manager**
  - ReplicaSet controller ensures CoreDNS replicas are maintained
  - No direct interaction with Pod DNS settings
  - **When things go wrong:** Controller manager down → CoreDNS Pods not recreated if they fail

- **DNS Resolution ↔ etcd**
  - No direct interaction
  - Service and Endpoint records stored in etcd → watched by CoreDNS via API Server
  - **When things go wrong:** etcd down → no new Services discoverable → stale DNS records

- **DNS Resolution ↔ Network Layer**
  - DNS queries are UDP/TCP packets on port 53
  - Require functional CNI for Pod-to-Pod communication
  - kube-proxy routes kube-dns ClusterIP traffic to CoreDNS Pod IPs
  - **When things go wrong:** CNI broken → Pod can't reach CoreDNS → DNS timeout; kube-proxy broken → kube-dns ClusterIP unreachable; NetworkPolicy blocking port 53 → DNS silently fails

- **DNS Resolution ↔ Storage Layer**
  - No interaction with storage
  - DNS configuration is ephemeral — injected at Pod creation, not persisted

## Quick Reference Cheat Sheet

```bash
# =============================================
# DNS POLICY QUICK REFERENCE
# =============================================
# ClusterFirst (default)  → CoreDNS → cluster + external DNS ✅
# Default                 → Node DNS → external only, NO cluster DNS ❌
# ClusterFirstWithHostNet → CoreDNS for hostNetwork Pods ✅
# None                    → Manual only via dnsConfig

# =============================================
# DNS DEBUGGING COMMANDS
# =============================================

# Check Pod's DNS config
kubectl exec <pod> -- cat /etc/resolv.conf

# Check Pod's dnsPolicy
kubectl get pod <pod> -o jsonpath='{.spec.dnsPolicy}{"\n"}'

# Check hostNetwork
kubectl get pod <pod> -o jsonpath='{.spec.hostNetwork}{"\n"}'

# Test internal DNS
kubectl exec <pod> -- nslookup kubernetes.default
kubectl exec <pod> -- nslookup <svc>.<ns>.svc.cluster.local

# Test external DNS
kubectl exec <pod> -- nslookup google.com

# Quick debug Pod
kubectl run dns-debug --image=busybox:1.36 --rm -it --restart=Never \
  -- nslookup kubernetes.default

# Check CoreDNS health
kubectl get pods -n kube-system -l k8s-app=kube-dns

# =============================================
# DNS RECORD FORMATS
# =============================================
# Service:      <svc>.<ns>.svc.cluster.local
# Pod:          <pod-ip-dashed>.<ns>.pod.cluster.local
# SRV:          _<port>._<proto>.<svc>.<ns>.svc.cluster.local
# Headless Pod: <pod-name>.<svc>.<ns>.svc.cluster.local (if subdomain set)

# =============================================
# RESOLV.CONF (EXPECTED FOR ClusterFirst IN "default" NAMESPACE)
# =============================================
# nameserver 10.96.0.10
# search default.svc.cluster.local svc.cluster.local cluster.local
# options ndots:5
```

Nine times out of ten, it's a `dnsPolicy: Default` or a `hostNetwork: true` without `ClusterFirstWithHostNet`. This check takes 10 seconds and solves most DNS questions on the exam.
