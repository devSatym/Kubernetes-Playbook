This is exactly how you should learn CoreDNS for CKA/CKS and real-world troubleshooting:

**For every change, know:**

1. **Where** to add it
2. **Before what plugin**
3. **After what plugin**
4. **Why placement matters**
5. **How to verify**

---

# Rule #1: Plugin Execution Order Matters

CoreDNS executes plugins **top → bottom**.

```txt
.:53 {
    plugin1
    plugin2
    plugin3
}
```

Request flow:

```txt
DNS Query
    ↓
plugin1
    ↓
plugin2
    ↓
plugin3
```

A plugin can:

* Answer the query
* Modify the query
* Pass it to next plugin

Therefore placement matters.

# Q1: Enable DNS Query Logging

### Task

Log all DNS requests.

### Change

Before:

```txt
.:53 {
    errors
    health
    kubernetes cluster.local
}
```

After:

```txt
.:53 {
    errors
    log
    health
    kubernetes cluster.local
}
```

### Why Here?

Put near top.

You want logging to see queries before other plugins process them.

### Verify

```bash
kubectl logs -n kube-system deploy/coredns
```

# Q2: Cache DNS Responses for 5 Minutes

### Before

```txt
cache 30
```

### After

```txt
cache 300
```

### Where?

Near bottom.

Typical:

```txt
forward . /etc/resolv.conf
cache 300
loop
reload
```

### Why?

Cache should happen after lookup result is obtained.

# Q3: Add Static Entry

Resolve:

```txt
db.internal
```

to:

```txt
10.10.10.10
```

### Before

```txt
kubernetes cluster.local
forward . /etc/resolv.conf
```

### After

```txt
hosts {
    10.10.10.10 db.internal
    fallthrough
}

kubernetes cluster.local
forward . /etc/resolv.conf
```

### Why Here?

Hosts should check first.

Flow:

```txt
Query
 ↓
hosts
 ↓
kubernetes
 ↓
forward
```

# Q4: Forward Corp Domain to Corporate DNS

### Task

```txt
*.corp.local
```

should go to:

```txt
10.20.30.40
```

### Add Before Main Server Block

```txt
corp.local {
    forward . 10.20.30.40
}
```

Then:

```txt
.:53 {
...
}
```

### Why?

Separate zone.

CoreDNS matches:

```txt
app.corp.local
```

before matching:

```txt
.
```

# Q7: Block Domain

### Task

Block:

```txt
facebook.com
```

### Add Near Top

```txt
template IN A facebook.com {
    answer "{{ .Name }} 60 IN A 127.0.0.1"
}
```

### Why?

Need to answer before forwarding.

Flow:

```txt
Query
 ↓
template
 ↓
answer
```

Request stops.

# Q14: Troubleshoot Pod DNS

### Question

A pod cannot resolve:

```txt
google.com
```

### Checks

```bash
kubectl get pods -n kube-system
```

```bash
kubectl logs -n kube-system deploy/coredns
```

```bash
kubectl get svc -n kube-system kube-dns
```

```bash
kubectl exec pod -- nslookup google.com
```

```bash
kubectl exec pod -- cat /etc/resolv.conf
```

# Most Important Placement Rules

```txt
rewrite
hosts
template
```

⬆ Top

Reason:
They modify or answer queries early.

---

```txt
kubernetes
```

Middle

Reason:
Resolves cluster services.

---

```txt
forward
```

After kubernetes

Reason:
Unknown queries go outside cluster.

---

```txt
cache
loadbalance
```

Bottom

Reason:
Work on completed answers.

---

A good mental model:

```txt
Request
 ↓
log
 ↓
rewrite
 ↓
hosts
 ↓
template
 ↓
kubernetes
 ↓
forward
 ↓
cache
 ↓
loadbalance
 ↓
Response
```

If you can explain **why each plugin sits where it sits in that flow**, you've reached the level needed for most CKA/CKS CoreDNS questions and many real production DNS troubleshooting scenarios.
