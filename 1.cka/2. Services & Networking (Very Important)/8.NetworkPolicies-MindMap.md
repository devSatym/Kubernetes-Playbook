# 🛡️ Kubernetes Network Policies — CKA Mind Map

---

## 1️⃣ What It Does

- **Core Purpose**
  - Controls **ingress (incoming)** and **egress (outgoing)** network traffic to/from Pods
  - Acts as a Pod-level firewall — defines which Pods can talk to which Pods, namespaces, or IP ranges
  - By default, all Pods can communicate freely — NetworkPolicy restricts this (whitelist model)
  - Enforced by the **CNI plugin** (Calico, Cilium, Weave) — NOT by Kubernetes itself
  - Once ANY NetworkPolicy selects a Pod, that Pod becomes **isolated** — only explicitly allowed traffic passes

- **Real-World Use Case**
  - Isolate `production` namespace from `development` — prevent cross-env communication
  - Database Pods only accept traffic from backend Pods — block frontend direct access
  - Restrict Pods from accessing external internet (egress control)
  - Allow only specific namespaces to access shared services (e.g., monitoring)
  - Compliance: PCI-DSS, HIPAA — enforce network segmentation between workloads

- **Where It Fits in Cluster Architecture**
  - Lives at the **network layer** between Pods — below Services, above CNI
  - NetworkPolicy resource is a Kubernetes API object stored in etcd
  - Enforcement is done by **CNI plugin** — the NetworkPolicy resource is useless without a compatible CNI
  - Works at **Layer 3/4** (IP + port) — NOT Layer 7 (HTTP path/headers)
  - Applies per-namespace — NetworkPolicy in namespace A cannot control Pods in namespace B

---

## 2️⃣ What Depends On It

- **Which Components Rely On It**
  - Security teams enforcing network segmentation
  - Compliance requirements (isolation between environments)
  - Multi-tenant clusters — tenant isolation
  - Any Pod that needs network access control

- **What Resources It Interacts With**
  - **Pods** — selected by `spec.podSelector` (target Pods)
  - **Pods in rules** — `from/to` blocks with `podSelector` (allowed sources/destinations)
  - **Namespaces** — `namespaceSelector` in rules (allow/deny cross-namespace traffic)
  - **IP blocks** — `ipBlock` (CIDR ranges for external traffic)
  - **CNI Plugin** — Calico, Cilium, Weave Net — the enforcement engine
  - **Labels** — both Pod labels and namespace labels are heavily used

- **Upstream Dependencies**
  - **CNI plugin that supports NetworkPolicy** — MUST be deployed (Flannel alone does NOT support NetworkPolicy)
  - Labels on Pods and Namespaces — the policy uses selectors
  - Namespace must exist for the policy to be applied in

- **Downstream Dependencies**
  - All Pods selected by the policy become isolated
  - Services routing to those Pods are indirectly affected (traffic blocked = Service appears broken)
  - DNS resolution can break if egress to CoreDNS is not explicitly allowed

## 3️⃣ Recognize Structure (YAML Skeleton)

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: my-network-policy
  namespace: default                    # Policy applies to THIS namespace only
spec:
  podSelector:                          # WHICH Pods this policy targets
    matchLabels:
      app: backend                      # Pods with label app=backend
  policyTypes:                          # MUST specify — Ingress, Egress, or both
    - Ingress
    - Egress
  ingress:                              # Rules for INCOMING traffic
    - from:                             # Sources allowed
        - podSelector:                  # Allow from Pods with specific labels
            matchLabels:
              app: frontend
        - namespaceSelector:            # Allow from Pods in specific namespaces
            matchLabels:
              env: production
        - ipBlock:                      # Allow from specific IP ranges
            cidr: 10.0.0.0/8
            except:
              - 10.0.1.0/24
      ports:                            # On which ports (optional)
        - protocol: TCP
          port: 8080
  egress:                               # Rules for OUTGOING traffic
    - to:                               # Destinations allowed
        - podSelector:
            matchLabels:
              app: database
      ports:
        - protocol: TCP
          port: 5432
    - to:                               # Allow DNS resolution (CRITICAL!)
        - namespaceSelector: {}         # Any namespace
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

- **Key Sections Explained**
  - `podSelector` — selects target Pods in the SAME namespace; empty `{}` = ALL Pods in namespace
  - `policyTypes` — MUST list `Ingress`, `Egress`, or both — determines which direction is restricted
  - `ingress[].from` — array of sources; items are OR'd (any match = allow)
  - `egress[].to` — array of destinations; items are OR'd
  - `ports` — optional; if omitted, all ports allowed for that rule
  - `from`/`to` items: `podSelector`, `namespaceSelector`, `ipBlock` — can be combined

- **⚠️ CRITICAL: AND vs OR Logic**
  - **Separate list items under `from`/`to` are OR'd** (any ONE match allows traffic):
    ```yaml
    from:
      - podSelector: {matchLabels: {app: a}}    # OR
      - namespaceSelector: {matchLabels: {env: prod}}  # OR
    ```
    This means: allow from Pods with `app=a` in ANY namespace **OR** from ANY Pod in `env=prod` namespaces
  - **Combined in SAME item = AND'd** (BOTH must match):
    ```yaml
    from:
      - podSelector: {matchLabels: {app: a}}
        namespaceSelector: {matchLabels: {env: prod}}
    ```
    This means: allow from Pods with `app=a` that are ALSO in `env=prod` namespaces

- **Common Indentation Traps**
  - ❌ AND vs OR confusion — **single `-` with both selectors = AND; separate `-` = OR** (this is the #1 CKA trap)
  - ❌ `podSelector: {}` under `spec` = selects ALL Pods (deny-all policy if no rules)
  - ❌ Empty `ingress: []` = no ingress allowed; but `ingress:` with items = allow those items
  - ❌ `ports` must be under the same `ingress`/`egress` rule item, not at spec level
  - ❌ Missing `policyTypes` when only `egress` is defined → policy may not apply egress correctly

## 4️⃣ Critical Fields Only (CKA Focus)

| Field | Why It Matters | What Breaks If Misconfigured |
|---|---|---|
| `podSelector` | Selects which Pods are affected | Wrong labels → wrong Pods isolated OR no Pods matched |
| `policyTypes` | Declares if Ingress/Egress/both are controlled | Missing `Egress` → egress not restricted even if `egress` rules exist |
| `ingress[].from[].podSelector` | Source Pod filter | Wrong labels → legitimate traffic blocked |
| `ingress[].from[].namespaceSelector` | Source namespace filter | Wrong labels → cross-namespace traffic blocked or over-allowed |
| `egress[].to` | Destination filter | Missing DNS egress → **ALL DNS breaks** for selected Pods |
| `ports` | Port filter | Wrong port → traffic blocked; missing ports → all ports allowed for that rule |
| `ipBlock.cidr` | External IP ranges | Wrong CIDR → external traffic blocked or over-allowed |
| `ipBlock.except` | Exclusion from CIDR | Missing exclusions → unintended IPs allowed |

- **Field Relationships**
  - `spec.podSelector` labels ↔ actual Pod labels in the SAME namespace
  - `from[].podSelector` ↔ source Pod labels (in same namespace by default)
  - `from[].namespaceSelector` ↔ namespace labels (`kubectl get ns --show-labels`)
  - When BOTH `podSelector` AND `namespaceSelector` in same item → AND logic
  - When separate items → OR logic
  - `policyTypes: [Ingress]` without `ingress` rules → **deny ALL ingress** to selected Pods
  - `policyTypes: [Egress]` without `egress` rules → **deny ALL egress** from selected Pods

## 5️⃣ How Much Should You Notice in YAML?

- **Scan Immediately (5 seconds)**
  - `podSelector` — which Pods are targeted?
  - `policyTypes` — Ingress? Egress? Both?
  - `from/to` rules — are sources/destinations correct?
  - DNS egress allowed? — if `Egress` is in `policyTypes`, DNS (port 53) MUST be allowed
  - AND vs OR — are selectors in same item (AND) or separate items (OR)?

- **Usually Irrelevant for CKA**
  - Detailed CIDR subnetting calculations
  - IPv6 rules
  - Named ports (rarely tested)
  - `endPort` (port ranges) — newer feature, rarely in CKA
  - Specific CNI plugin configuration

- **Red Flags to Detect in 10 Seconds**
  - ❌ `policyTypes: [Egress]` without DNS egress rule → Pods can't resolve names
  - ❌ `podSelector: {}` with `policyTypes: [Ingress, Egress]` and no rules → total isolation (deny-all)
  - ❌ `from` with combined `podSelector` + `namespaceSelector` when OR was intended → too restrictive
  - ❌ Separate `podSelector` + `namespaceSelector` when AND was intended → too permissive
  - ❌ Missing `policyTypes` when egress rules are defined → egress may not be enforced
  - ❌ Policy in wrong namespace → targets no Pods

## 6️⃣ What to Notice In

### kubectl describe networkpolicy \<name\>

- **What Signals Failure**
  - `PodSelector:` shows unexpected or empty selector
  - `Allowing ingress traffic:` / `Allowing egress traffic:` sections show wrong rules
  - `Not affecting Ingress/Egress` when you expected enforcement
  - `<none>` under allowing traffic → all traffic of that type is blocked

- **Where to Look First**
  - `PodSelector:` — are the right Pods targeted?
  - `Allowing ingress/egress traffic:` — are the right sources/destinations listed?
  - `Affecting Ingress/Egress` — is the direction you want actually enforced?

- **Fields That Matter Most**
  - `PodSelector` — verify it matches intended Pods
  - Policy type enforcement — Ingress, Egress, or both
  - Listed rules — each allowed source/destination with ports

### kubectl logs (Not directly useful for NetworkPolicy)

- **Key Insight**
  - NetworkPolicy itself produces NO logs — it's enforcement by CNI
  - Look at **application logs** for connection failures:
    - `Connection refused` → target Pod reachable but port blocked or app not listening
    - `Connection timed out` → traffic blocked by NetworkPolicy (packet silently dropped)
    - `No route to host` → network-level issue, possibly NetworkPolicy
  - Some CNIs (Calico, Cilium) can be configured to log denied traffic — but this is CNI-specific, not CKA-relevant

- **Key Difference**
  - `Connection timed out` = likely NetworkPolicy blocking (packets dropped silently)
  - `Connection refused` = traffic reaches Pod but port is closed (NetworkPolicy is NOT blocking)

### kubectl exec (Primary testing tool for NetworkPolicy)

- **Key Commands**
  ```bash
  # Test connectivity FROM a Pod TO another Pod/Service
  kubectl exec <source-pod> -- wget -qO- --timeout=3 http://<dest-svc>:<port>
  kubectl exec <source-pod> -- nc -zv <dest-pod-ip> <port> -w 3
  kubectl exec <source-pod> -- curl --connect-timeout 3 http://<dest-svc>:<port>

  # Test DNS resolution (critical — egress must allow port 53)
  kubectl exec <source-pod> -- nslookup kubernetes.default

  # Test external connectivity
  kubectl exec <source-pod> -- wget -qO- --timeout=3 http://google.com
  ```

- **What Signals Failure**
  - `wget: download timed out` → traffic blocked (NetworkPolicy or network issue)
  - `nslookup` times out → egress to DNS (port 53) is blocked
  - Connectivity works from Pod A but not Pod B → NetworkPolicy selectively blocking

### kubectl explain networkpolicy

- **Key Usage**
  ```bash
  kubectl explain networkpolicy.spec.podSelector
  kubectl explain networkpolicy.spec.ingress
  kubectl explain networkpolicy.spec.ingress.from
  kubectl explain networkpolicy.spec.egress.to
  kubectl explain networkpolicy.spec.policyTypes
  ```
- Quickly verify `from`/`to` selector structure
- Confirm `ipBlock` fields (`cidr`, `except`)
- Check `ports` field structure

## 7️⃣ Failure Patterns (Very Important)

- **Common Misconfigurations**
  - ❌ Egress policy without DNS port 53 rule → Pods can't resolve any names → app "can't connect" to Services by name
  - ❌ AND vs OR confusion → policy too restrictive (AND) or too permissive (OR)
  - ❌ `podSelector: {}` (all Pods) with deny-all intention but accidentally blocking critical traffic
  - ❌ Policy targets wrong namespace (created in wrong namespace)
  - ❌ `namespaceSelector` matching by namespace name instead of label → namespace labels not set
  - ❌ CNI doesn't support NetworkPolicy (e.g., Flannel without Calico) → policy created but NEVER enforced
  - ❌ Missing `policyTypes: [Egress]` but `egress` rules present → egress not restricted
  - ❌ Allow rule ports not matching actual application ports → traffic blocked

- **Status Values**
  - NetworkPolicy has **NO status field** — it's either created or not
  - Effects are observed through connectivity testing, NOT through `kubectl get`
  - Pods targeted by a NetworkPolicy show NO special status or condition

- **Event Messages to Recognize**
  - NetworkPolicy itself generates NO events
  - Diagnosis is purely by **testing connectivity** and checking policy rules
  - Application Pods may show events related to health check failures (if probes are blocked by NetworkPolicy)

- **Real Exam-Style Failure Scenarios**
  - Scenario 1: Pod A can't reach Pod B after applying a NetworkPolicy → check if Pod B's ingress rule allows Pod A's labels/namespace
  - Scenario 2: All Pods in a namespace suddenly can't resolve DNS → an egress deny-all policy was applied without allowing port 53
  - Scenario 3: NetworkPolicy exists but has no effect → CNI doesn't support NetworkPolicy (Flannel without Calico overlay)
  - Scenario 4: Want to allow only `frontend` Pods to reach `backend` Pods, but all Pods in the namespace can reach → AND vs OR mistake in `from` rules
  - Scenario 5: Cross-namespace traffic blocked → `namespaceSelector` uses wrong label or namespace label not set

## 8️⃣ Misconfiguration Behavior

- **What Happens If Labels Mismatch (podSelector)?**
  - `spec.podSelector` doesn't match any Pod → policy exists but affects NOTHING
  - `from[].podSelector` wrong labels → legitimate source Pods blocked → connection timeout
  - Pods not matched by ANY NetworkPolicy → remain fully open (default allow-all)
  - **Behavior:** no error, no warning — policy silently has no effect or wrong effect

- **What Happens If Ports Mismatch?**
  - Policy allows port 8080 but app listens on 80 → traffic on port 80 blocked
  - Policy allows TCP but app uses UDP → traffic blocked (protocol matters)
  - Missing `ports` entirely in a rule → ALL ports allowed for that rule (may be too permissive)
  - **Behavior:** connection timeout on wrong port; works on allowed port

- **What Happens If CNI Doesn't Support NetworkPolicy?**
  - NetworkPolicy resource created successfully (no API error)
  - Policy stored in etcd — `kubectl get networkpolicy` shows it
  - But **ZERO enforcement** — all traffic flows freely
  - This is the most dangerous misconfiguration — false sense of security
  - Common with vanilla Flannel (must add Calico for enforcement)

- **What Happens If Egress DNS Is Not Allowed?**
  - Pods can't resolve ANY hostname (internal or external)
  - `nslookup kubernetes.default` → times out
  - Application logs: "no such host", "could not resolve"
  - Pods can still communicate by raw IP — only name resolution breaks
  - **Fix:** add egress rule allowing UDP+TCP port 53 to CoreDNS

- **What Happens If Namespace Labels Are Missing?**
  - `namespaceSelector: {matchLabels: {env: prod}}` but namespace has no `env: prod` label
  - Rule matches ZERO namespaces → cross-namespace traffic blocked
  - `kubectl get ns --show-labels` → verify labels
  - **Fix:** `kubectl label namespace <ns> env=prod`

## 9️⃣ Fast Detection Strategy

- **Exact Commands to Run (in order)**
  ```bash
  # 1. List all NetworkPolicies in the namespace
  kubectl get networkpolicy -n <namespace>

  # 2. Describe the NetworkPolicy — check rules
  kubectl describe networkpolicy <name> -n <namespace>

  # 3. Check which Pods are selected by the policy
  kubectl get pods -n <namespace> -l <selector-labels>

  # 4. Test connectivity FROM source Pod TO destination
  kubectl exec <source-pod> -n <ns> -- wget -qO- --timeout=3 http://<dest-svc>:<port>

  # 5. Test DNS from the affected Pod
  kubectl exec <pod> -n <ns> -- nslookup kubernetes.default

  # 6. Test connectivity by IP (bypass DNS)
  kubectl exec <source-pod> -- wget -qO- --timeout=3 http://<dest-pod-ip>:<port>

  # 7. Check Pod labels
  kubectl get pods -n <namespace> --show-labels

  # 8. Check namespace labels
  kubectl get ns --show-labels

  # 9. Verify CNI supports NetworkPolicy
  kubectl get pods -n kube-system | grep -E 'calico|cilium|weave'

  # 10. Temporarily delete policy to confirm it's the cause
  kubectl delete networkpolicy <name> -n <namespace>
  # → If connectivity restores → policy was the issue → re-apply with fix
  ```

- **Order of Debugging Steps**
  1. Is a NetworkPolicy selecting the Pod? → `kubectl get networkpolicy -n <ns>`
  2. Does connectivity work WITHOUT the policy? → delete policy temporarily
  3. What does the policy allow? → `kubectl describe networkpolicy <name>`
  4. Do Pod labels match `podSelector`? → `kubectl get pods --show-labels`
  5. Do source/dest labels match `from`/`to` rules? → verify label values
  6. Do namespace labels match `namespaceSelector`? → `kubectl get ns --show-labels`
  7. Is DNS allowed? → `kubectl exec <pod> -- nslookup kubernetes.default`
  8. Are correct ports specified? → compare policy ports vs actual app port

- **How to Detect Issue Under 60 Seconds**
  ```bash
  # Step 1 (10s): Check if NetworkPolicy exists
  kubectl get networkpolicy -n <namespace>

  # Step 2 (10s): Test connectivity
  kubectl exec <source> -- wget -qO- --timeout=3 http://<dest-svc>:<port>
  # → Timeout = blocked

  # Step 3 (20s): Check what the policy allows
  kubectl describe networkpolicy <name> -n <namespace>
  # → Read "Allowing ingress/egress traffic" → compare with intended traffic

  # Step 4 (20s): Verify labels match
  kubectl get pods --show-labels -n <namespace>
  kubectl get ns --show-labels
  ```

## 🔟 Fast Fix Strategy

- **Fix Using kubectl edit / patch / apply**
  ```bash
  # Fix podSelector to target correct Pods
  kubectl edit networkpolicy <name> -n <namespace>
  # → Correct matchLabels under podSelector

  # Add missing DNS egress rule
  kubectl edit networkpolicy <name> -n <namespace>
  # → Add under egress:
  #   - to:
  #       - namespaceSelector: {}
  #         podSelector:
  #           matchLabels:
  #             k8s-app: kube-dns
  #     ports:
  #       - protocol: UDP
  #         port: 53
  #       - protocol: TCP
  #         port: 53

  # Fix AND vs OR (separate items for OR)
  kubectl edit networkpolicy <name> -n <namespace>
  # → Move podSelector and namespaceSelector to separate list items if OR intended

  # Add missing namespace label
  kubectl label namespace <ns> env=prod

  # Add missing Pod label
  kubectl label pod <pod> app=frontend

  # Delete overly restrictive policy
  kubectl delete networkpolicy <name> -n <namespace>

  # Apply corrected policy from file
  kubectl apply -f corrected-netpol.yaml
  ```

- **Validate Fix**
  ```bash
  # Verify policy looks correct
  kubectl describe networkpolicy <name> -n <namespace>

  # Test connectivity again
  kubectl exec <source-pod> -- wget -qO- --timeout=3 http://<dest-svc>:<port>
  # → Should succeed now

  # Test DNS
  kubectl exec <pod> -- nslookup kubernetes.default
  # → Should resolve

  # Test from a Pod that should be BLOCKED (verify policy still enforces)
  kubectl exec <unauthorized-pod> -- wget -qO- --timeout=3 http://<dest-svc>:<port>
  # → Should timeout (still blocked) ✅
  ```

- **Confirm Recovery**
  - Allowed traffic flows successfully ✅
  - Blocked traffic is still blocked ✅
  - DNS resolution works for affected Pods ✅
  - Application logs show no connection timeouts ✅
  - `kubectl describe networkpolicy` shows expected rules ✅

## 1️⃣1️⃣ Hands-On Learning Cycle

### Step 1 — Create Working Example

```yaml
# 1. Create namespaces with labels
apiVersion: v1
kind: Namespace
metadata:
  name: netpol-test
  labels:
    env: test
---
# 2. Backend Pod (the one we'll protect)
apiVersion: v1
kind: Pod
metadata:
  name: backend
  namespace: netpol-test
  labels:
    app: backend
    tier: api
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80
---
# 3. Frontend Pod (allowed to access backend)
apiVersion: v1
kind: Pod
metadata:
  name: frontend
  namespace: netpol-test
  labels:
    app: frontend
    tier: web
spec:
  containers:
    - name: busybox
      image: busybox:1.36
      command: ["sleep", "3600"]
---
# 4. Rogue Pod (should be blocked)
apiVersion: v1
kind: Pod
metadata:
  name: rogue
  namespace: netpol-test
  labels:
    app: rogue
spec:
  containers:
    - name: busybox
      image: busybox:1.36
      command: ["sleep", "3600"]
---
# 5. Backend Service
apiVersion: v1
kind: Service
metadata:
  name: backend-svc
  namespace: netpol-test
spec:
  selector:
    app: backend
  ports:
    - port: 80
      targetPort: 80
---
# 6. NetworkPolicy — only frontend can reach backend on port 80
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-policy
  namespace: netpol-test
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 80
```

- **Validation Steps**
  ```bash
  kubectl apply -f netpol-example.yaml

  # Frontend CAN reach backend (allowed) ✅
  kubectl exec -n netpol-test frontend -- wget -qO- --timeout=3 http://backend-svc
  # → Returns nginx HTML

  # Rogue CANNOT reach backend (blocked) ❌
  kubectl exec -n netpol-test rogue -- wget -qO- --timeout=3 http://backend-svc
  # → Times out (blocked by NetworkPolicy)

  # Verify policy
  kubectl describe networkpolicy backend-policy -n netpol-test
  ```

### Step 2 — Break It Intentionally

| Break | How | Expected Behavior |
|---|---|---|
| Wrong `podSelector` | Change `app: backend` → `app: wrong` | Policy targets no Pods → backend is unprotected → rogue CAN reach it |
| Wrong `from` selector | Change `app: frontend` → `app: wrong` in ingress | Frontend blocked → times out; rogue still blocked |
| Remove `ports` | Delete `ports` section from ingress | All ports allowed from frontend (more permissive than intended) |
| Add Egress deny-all | Add `policyTypes: [Ingress, Egress]`, no egress rules | Backend can't make ANY outbound connections, including DNS |
| Egress without DNS | Add egress rules for app traffic but forget port 53 | Backend can't resolve names → apps fail on name lookups |
| AND vs OR mistake | Combine `podSelector` + `namespaceSelector` in same item | Too restrictive — requires BOTH to match |
| Wrong namespace | Create policy in `default` instead of `netpol-test` | Policy exists but targets nothing in `netpol-test` |
| Deny-all policy | `podSelector: {}`, `policyTypes: [Ingress]`, no `ingress` rules | ALL Pods in namespace isolated — no inbound traffic at all |
| Wrong port | Change `port: 80` → `port: 8080` | Frontend reaches backend but port 80 blocked → timeout |
| Missing namespace label | Use `namespaceSelector: {matchLabels: {env: test}}` but remove `env: test` label from namespace | Cross-namespace rule matches nothing → traffic blocked |

### Step 3 — Debug Without Looking at Solution

- **What Command to Run First**
  ```bash
  kubectl get networkpolicy -n <namespace>
  kubectl exec <source> -- wget -qO- --timeout=3 http://<dest-svc>:<port>
  ```

- **How to Narrow Issue**
  1. Does a NetworkPolicy exist? → `kubectl get networkpolicy -n <ns>`
  2. Delete the policy temporarily → does connectivity restore? → policy is the cause
  3. What does the policy target? → `kubectl describe networkpolicy <name>`
  4. Do Pod labels match the policy's `podSelector`? → `kubectl get pods --show-labels`
  5. Do source labels match `from` rules? → compare labels
  6. Are correct ports listed? → compare with actual app port
  7. Is DNS allowed (if egress policy)? → `kubectl exec <pod> -- nslookup kubernetes.default`

- **How to Confirm Root Cause**
  - Delete policy → connectivity works → re-examine the policy rules
  - Read `kubectl describe networkpolicy` output carefully
  - Check AND vs OR logic by examining indentation of `from`/`to` items
  - Verify namespace labels with `kubectl get ns --show-labels`

## 1️⃣2️⃣ Topic-wise Breakdown (CKA Domain Alignment)

- **What to Learn**
  - NetworkPolicy YAML structure: `podSelector`, `policyTypes`, `ingress`, `egress`
  - AND vs OR logic in `from`/`to` selectors — **#1 CKA trap**
  - Default behavior: no policy = all traffic allowed; any policy selecting a Pod = Pod is isolated
  - How to write deny-all, allow-specific, and egress policies
  - DNS egress rule — **always include port 53** when restricting egress
  - `namespaceSelector` and namespace labels
  - `ipBlock` for CIDR-based rules

- **What to Focus On**
  - Writing NetworkPolicy from scratch under time pressure
  - AND vs OR distinction — practice both forms side-by-side
  - Always test with TWO Pods — one that should be allowed, one that should be blocked
  - Egress + DNS pattern — memorize the DNS egress rule
  - Deny-all-ingress, deny-all-egress, deny-all-both templates

- **What to Practice Repeatedly**
  - Create allow-ingress-from-specific-Pods policy → test → verify
  - Create egress policy with DNS → test external and internal connectivity
  - Break AND/OR → observe behavior change → fix
  - Cross-namespace policies with `namespaceSelector` + namespace labels
  - Deny-all → add exceptions → validate each opening

- **What to Ignore (Deep Internals)**
  - CNI plugin configuration details (Calico Felix, Cilium eBPF)
  - iptables rules generated by NetworkPolicy
  - NetworkPolicy performance tuning
  - Logging and auditing of denied traffic (CNI-specific)
  - Layer 7 policies (Cilium CiliumNetworkPolicy — beyond standard API)

- **5 Quiz Questions**
  1. You apply a NetworkPolicy with `podSelector: {matchLabels: {app: db}}` and `policyTypes: [Ingress]` but no `ingress` rules. What happens to Pods with label `app: db`?
     → *ALL ingress traffic to those Pods is denied. `policyTypes: [Ingress]` with no rules = deny-all-ingress for selected Pods.*
  2. What is the difference between these two `from` blocks?
     ```yaml
     # Block A (separate items)
     from:
       - podSelector: {matchLabels: {app: web}}
       - namespaceSelector: {matchLabels: {env: prod}}
     # Block B (same item)
     from:
       - podSelector: {matchLabels: {app: web}}
         namespaceSelector: {matchLabels: {env: prod}}
     ```
     → *Block A (OR): allows from any Pod with `app=web` in ANY namespace, OR from any Pod in `env=prod` namespaces. Block B (AND): allows ONLY from Pods with `app=web` that are ALSO in `env=prod` namespaces.*
  3. You create an egress NetworkPolicy but now Pods can't resolve DNS. Why?
     → *Egress policy restricts ALL outgoing traffic. DNS queries (UDP/TCP port 53) to CoreDNS are also blocked. Add an egress rule allowing port 53.*
  4. A NetworkPolicy exists but isn't blocking anything. What's the most likely cause?
     → *The CNI plugin doesn't support NetworkPolicy (e.g., vanilla Flannel). The policy is stored but never enforced.*
  5. How do you create a deny-all-ingress-and-egress policy for all Pods in a namespace?
     → *`podSelector: {}`, `policyTypes: [Ingress, Egress]`, no `ingress` or `egress` rules.*

## 1️⃣3️⃣ Component Interaction Map

- **NetworkPolicy ↔ Pods**
  - `spec.podSelector` targets Pods by labels in the same namespace
  - Selected Pods become **isolated** — only explicitly allowed traffic passes
  - Unselected Pods remain fully open (no implicit deny-all)
  - **When things go wrong:** wrong labels → wrong Pods isolated; app breaks for unintended Pods; unmatched Pods remain unprotected

- **NetworkPolicy ↔ Services**
  - NetworkPolicy does NOT directly reference Services — it works at Pod level
  - But blocking Pod-to-Pod traffic effectively "breaks" Services
  - Traffic to a Service's ClusterIP is evaluated against backend Pod's NetworkPolicy
  - **When things go wrong:** policy blocks traffic to Service's backend Pods → Service appears broken (timeout) even though Service and Endpoints exist

- **NetworkPolicy ↔ Nodes**
  - NetworkPolicy does NOT affect Node-to-Pod communication by default
  - kubelet health checks (liveness/readiness probes) are typically NOT blocked
  - Node-level network policies are CNI-specific (not standard NetworkPolicy)
  - **When things go wrong:** some CNIs may block kubelet probes → Pods marked NotReady → Services remove Pod from Endpoints

- **NetworkPolicy ↔ Scheduler**
  - No interaction — scheduler places Pods regardless of NetworkPolicy
  - NetworkPolicy is enforced post-scheduling by CNI
  - **When things go wrong:** N/A — scheduling is independent

- **NetworkPolicy ↔ Controller Manager**
  - No direct interaction
  - Controller manager doesn't enforce NetworkPolicy
  - **When things go wrong:** N/A

- **NetworkPolicy ↔ etcd**
  - NetworkPolicy resource stored in etcd
  - CNI watches NetworkPolicy changes via API Server
  - **When things go wrong:** etcd down → can't create/update policies; existing policies remain enforced by CNI

- **NetworkPolicy ↔ Network Layer (CNI)**
  - **CNI implements the enforcement** — this is the MOST critical interaction
  - Calico → uses iptables/eBPF rules
  - Cilium → uses eBPF programs
  - Flannel → does NOT support NetworkPolicy (need Calico or Canal alongside)
  - **When things go wrong:** CNI doesn't support it → policy exists but has zero effect (DANGEROUS); CNI crashes → enforcement may stop; CNI misconfigured → partial enforcement

- **NetworkPolicy ↔ Storage Layer**
  - No interaction — NetworkPolicy is purely network-level

## Quick Reference Cheat Sheet

```bash
# =============================================
# NETWORK POLICY QUICK REFERENCE
# =============================================

# List all NetworkPolicies
kubectl get networkpolicy -A
kubectl get netpol -A                  # shortname

# Describe policy
kubectl describe networkpolicy <name> -n <ns>

# Test connectivity (use --timeout to avoid waiting forever)
kubectl exec <pod> -- wget -qO- --timeout=3 http://<svc>:<port>
kubectl exec <pod> -- nc -zv <ip> <port> -w 3

# Check Pod labels
kubectl get pods -n <ns> --show-labels

# Check namespace labels
kubectl get ns --show-labels

# Label a namespace (for namespaceSelector)
kubectl label namespace <ns> env=prod

# =============================================
# DENY-ALL TEMPLATES
# =============================================

# Deny ALL ingress to all Pods in namespace:
# podSelector: {}
# policyTypes: [Ingress]
# (no ingress rules)

# Deny ALL egress from all Pods in namespace:
# podSelector: {}
# policyTypes: [Egress]
# (no egress rules)

# Deny ALL traffic (both directions):
# podSelector: {}
# policyTypes: [Ingress, Egress]
# (no ingress or egress rules)

# =============================================
# DNS EGRESS RULE (MEMORIZE THIS!)
# =============================================
# egress:
#   - to:
#       - namespaceSelector: {}
#         podSelector:
#           matchLabels:
#             k8s-app: kube-dns
#     ports:
#       - protocol: UDP
#         port: 53
#       - protocol: TCP
#         port: 53

# =============================================
# AND vs OR CHEAT SHEET
# =============================================
# OR (separate items with -):
# from:
#   - podSelector: {matchLabels: {app: a}}      ← Match A
#   - namespaceSelector: {matchLabels: {env: p}} ← OR Match B
#
# AND (same item, no second -):
# from:
#   - podSelector: {matchLabels: {app: a}}       ← Match A
#     namespaceSelector: {matchLabels: {env: p}}  ← AND Match B
```

---

> **CKA Exam Tip:** NetworkPolicy questions appear in almost every CKA exam. Memorize three things: (1) the deny-all template, (2) the DNS egress rule, and (3) AND vs OR logic. The AND vs OR trap alone can cost you the question. When writing policies under time pressure, always test with TWO Pods — one that should pass and one that should be blocked. If you restrict egress, **ALWAYS include port 53 for DNS** — forgetting this is the #1 mistake.
